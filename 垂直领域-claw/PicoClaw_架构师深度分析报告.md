## 一、执行摘要

### 1.1 架构定位

**PicoClaw** 是由 Sipeed 发起的开源项目，定位为"超轻量级个人 AI 助手"，是 NanoBot 的 Go 语言重构版本。项目通过 **AI 自举（AI-Bootstrapped）** 方式开发，95% 的核心代码由 AI Agent 生成，人类在循环中进行优化。

### 1.2 核心架构特征

| 维度 | 架构特征 | 成熟度 |
|------|----------|:------:|
| **内存占用** | <10MB（核心功能） | ⭐⭐⭐⭐⭐ |
| **启动速度** | <1秒（0.6GHz 单核） | ⭐⭐⭐⭐⭐ |
| **部署形态** | 单二进制文件 | ⭐⭐⭐⭐⭐ |
| **硬件支持** | RISC-V、ARM、MIPS、x86、LoongArch | ⭐⭐⭐⭐⭐ |
| **AI 自举** | 95% 代码由 AI 生成 | ⭐⭐⭐⭐ |

### 1.3 关键架构指标

```
代码规模:
├── Go 代码: ~60K 行 (442 个文件)
├── Web UI: React 前端
└── 支持平台: 15+ 渠道，8+ 架构

性能指标:
├── 内存占用: <10MB
├── 启动时间: <1s
├── 二进制大小: ~8MB
└── 硬件成本: $10 级设备
```

## 三、技术选型深度分析

### 3.1 Go 语言选型理由

| 理由 | 说明 |
|------|------|
| **静态二进制** | 单文件部署，无依赖，适合嵌入式设备 |
| **内存效率** | 比 Python/Node.js 内存占用少 90%+ |
| **并发模型** | Goroutine + Channel 天然适合消息处理 |
| **交叉编译** | 原生支持多架构 (ARM/MIPS/RISC-V) |
| **启动速度** | 无解释器启动开销，毫秒级启动 |

### 3.2 核心依赖库

```go
// go.mod 关键依赖
require (
    github.com/spf13/cobra v1.10.2          // CLI 框架
    github.com/modelcontextprotocol/go-sdk v1.4.1  // MCP 协议
    github.com/openai/openai-go/v3 v3.22.0  // OpenAI API
    github.com/anthropics/anthropic-sdk-go v1.26.0 // Anthropic API
    github.com/mymmrac/telego v1.7.0        // Telegram Bot
    github.com/bwmarrin/discordgo v0.29.0   // Discord Bot
    go.mau.fi/whatsmeow v0.0.0-...          // WhatsApp (可选)
    modernc.org/sqlite v1.46.1              // SQLite (CGO-free)
    github.com/rs/zerolog v1.34.0           // 结构化日志
)
```

### 3.3 数据存储方案

| 存储类型 | 实现 | 用途 |
|----------|------|------|
| **会话历史** | JSONL (append-only) | 消息历史，支持截断但不删除 |
| **技能配置** | SQLite | 技能元数据和状态 |
| **状态管理** | JSON 文件 (原子写入) | 通道状态、设备状态 |
| **媒体文件** | 文件系统 + 清理器 | 临时媒体文件管理 |
| **配置** | JSON + 环境变量 | 应用配置 |

## 五、数据流与状态管理

### 5.1 请求处理流程

```
用户消息 → Channel → ChannelManager Queue → Worker Pool → 
┌─────────────────────────────────────────────────────────────┐
│                        Agent Loop                           │
│  1. 路由解析 → 选择 Agent → 构建上下文                       │
│  2. 加载历史 → 调用 LLM → 执行工具                          │
│  3. 保存回复 → 发送响应 → 可选摘要                          │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
                       Event Stream (SSE)
                              │
                              ▼
                      Channel Response
```

### 5.2 状态管理

**JSONL 存储设计**

```go
// 每个会话两个文件
```

## 十一、架构决策记录

### ADR-001: Go 语言重写

**背景**: Python/Node.js 内存占用过高

**决策**: 使用 Go 1.25+ 重写

**结果**: 内存从 >100MB 降至 <10MB

### ADR-002: JSONL 存储

**背景**: 需要持久化会话历史，但 SQLite 过重

**决策**: 使用 append-only JSONL 文件

**结果**: 写入速度快，支持截断不删除，易于备份

### ADR-003: 消息总线架构

**背景**: 多通道并发处理需要解耦

**决策**: 引入 Channel-based Message Bus

**结果**: 组件松耦合，易于测试扩展

## 十二、风险与建议

### 12.1 架构风险评估

| 风险 | 等级 | 说明 |
|------|------|------|
| **配置文件热重载竞态** | 中 | 需确保重载时无请求丢失 |
| **MCP 连接故障扩散** | 中 | 单 MCP 故障不应影响整体 |
| **JSONL 文件膨胀** | 低 | 长期运行需定期归档 |
| **多平台测试覆盖** | 中 | 部分架构缺乏 CI 测试 |
