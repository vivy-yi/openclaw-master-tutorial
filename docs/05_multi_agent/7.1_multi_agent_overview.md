# 7.1 多 Agent 概述

## 本节目标
- 理解多 Agent 协作的原理
- 掌握多 Agent 系统架构
- 了解适用场景

---

## 7.1.1 什么是多 Agent

### 单 Agent vs 多 Agent

```
┌─────────────────────────────────────────────┐
│         单 Agent vs 多 Agent                 │
├─────────────────────────────────────────────┤
│                                             │
│  单 Agent                                    │
│  ├── 一个 Agent 处理所有任务                  │
│  ├── 简单高效                               │
│  └── 适合单一职责任务                        │
│                                             │
│  多 Agent                                    │
│  ├── 多个专业 Agent 协作                     │
│  ├── 分工明确                               │
│  └── 适合复杂任务                           │
│                                             │
└─────────────────────────────────────────────┘
```

### 多 Agent 优势

```
优势：

1. 专业分工
   ├── 每个 Agent 专注领域
   └── 深度优化

2. 可扩展性
   ├── 轻松添加新 Agent
   └── 灵活扩展

3. 容错性
   ├── 单点故障不影响全局
   └── 可替换 Agent

4. 成本优化
   ├── 按需调用不同模型
   └── 复杂任务分配给强模型
```

## 7.1.3 Agent 类型

### 按角色分类

```
Agent 角色类型：

1. 协调者 (Coordinator)
   ├── 任务分解
   ├── 流程控制
   └── 结果汇总
   例子：项目经理

2. 执行者 (Executor)
   ├── 具体执行
   ├── 工具调用
   └── 产出结果
   例子：开发人员

3. 审查者 (Reviewer)
   ├── 质量检查
   ├── 问题发现
   └── 改进建议
   例子：代码审查员

4. 规划者 (Planner)
   ├── 方案设计
   ├── 路径规划
   └── 策略制定
   例子：架构师
```

### 按模型分类

```
模型类型：

1. 强模型 Agent
   ├── GPT-4o / Claude 3.5
   ├── 复杂推理
   └── 重要决策

2. 平衡模型 Agent
   ├── GPT-4o-mini / Claude 3 Haiku
   ├── 日常任务
   └── 成本优化

3. 轻量模型 Agent
   ├── DeepSeek / GLM-4
   ├── 简单执行
   └── 批量处理
```

## 7.1.4 协作模式

### 模式分类

```
协作模式：

1. 串行模式
   A → B → C
   └── 顺序执行

2. 并行模式
   A → ┬ → D
   B ──┤
   C ──┘
   └── 同时执行，结果合并

3. 分支模式
   A → B (条件1)
   A → C (条件2)
   └── 根据条件选择

4. 循环模式
   A → B → A → B
   └── 迭代优化

5. 层级模式
   A → B → C
   └── 逐级传递
```

## 7.1.5 适用场景

### 场景示例

```
适合多 Agent：

1. 复杂项目开发
   ├── 需求分析 Agent
   ├── 开发 Agent
   ├── 测试 Agent
   └── 部署 Agent

2. 全方位客户服务
   ├── 接待 Agent
   ├── 问题分类 Agent
   ├── 处理 Agent
   └── 满意度回访

3. 内容创作
   ├── 选题 Agent
   ├── 写作 Agent
   ├── 配图 Agent
   └── 审核 Agent

4. 数据分析
   ├── 采集 Agent
   ├── 清洗 Agent
   ├── 分析 Agent
   └── 可视化 Agent

不适合多 Agent：
- 简单问答
- 单一任务
- 实时性要求极高
```

```
└─────────────────────────────────────────────────────────────────┘
```

### Session Target 选择

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| `isolated`（推荐） | 独立 session，完全隔离 | Cron 任务、后台处理 |
| `main` | 在主 session 中执行 | 需要共享上下文 |
| `current` | 在当前 session 执行 | 与当前对话协作 |

### 公告投递与 Telegram 群组限制

**问题**：在 Telegram 群组中触发 Subagent，任务完成后主 Agent 的公告会被静默丢弃，Gateway 日志显示：

```
Outbound not configured for channel: telegram
```

**根因**：Subagent completion announce 路径未配置 outbound。

**临时解决方案**：
1. 改用 **DM（私人对话）** 触发 Subagent
2. 监控 Gateway 日志：`openclaw gateway logs | grep subagent`
3. 关注官方修复：[Issue #56032](https://github.com/openclaw/openclaw/issues/56032)

## 7.1.8 多 Agent 最佳实践

```
✅ 推荐做法：

1. 始终配置 messages.queue.mode: "steer"
   → 保持对运行任务的控制能力

2. Subagent 用 isolated session
   → 隔离环境避免干扰主会话

3. 设置合理的 runTimeoutSeconds
   → 防止失控 agent 无限占用资源

4. Cron 任务配合 skipWhenIdle
   → 减少空闲时不必要的执行

5. Telegram 群组场景用 DM 中转
   → 绕过公告投递限制

❌ 避免做法：

1. 在高 heartbeat 频率时触发 Subagent
   → 可能触发 #56049 Heartbeat 风暴

2. 大量 Subagent 并行启动
   → 资源竞争，建议控制在 3-5 个以内

3. 跨群组共享同一 Agent
   → 上下文混淆，建议独立配置
```

## 7.1.9 常见问题

### Q1: Agent 之间如何通信

```
解决：
1. 共享上下文（通过 workspace 文件）
2. 结构化消息（JSON 格式）
3. 约定协议（通过 session send）
```

### Q2: Subagent 公告没有出现在群组里

```
原因：Telegram 群组公告路径未配置 outbound（#56032）

解决：
1. 改用 DM 触发 Subagent
2. 通过 /sessions history 查看执行结果
3. 监控日志排查
```

### Q3: Cron 删除后仍然收到推送

```
原因：#56045 Cron 删除后持续投递 bug

解决：
1. cron remove 后执行 openclaw gateway restart
2. 确认 job 确实已被删除
3. 观察是否停止
```
