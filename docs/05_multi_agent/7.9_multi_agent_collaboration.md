# 多 Agent 沟通与协作完全指南

## 本节目标
- 理解 OpenClaw 多 Agent 隔离架构
- 掌握 Agent 间通信机制（sessions_send、sessions_spawn）
- 学会配置 agentToAgent 跨 Agent 调用
- 理解广播组与消息路由的区别
- 设计高效的多 Agent 协作工作流

---

## 1. 多 Agent 隔离架构

### 什么是 Agent

在 OpenClaw 中，一个 **Agent** 是一个完全独立作用域的大脑，拥有自己的：

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                           OpenClaw Agent 架构                                   │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  每个 Agent 独立拥有：                                                         │
│                                                                                 │
│  ┌─────────────────────────────────────────────────────────────────────┐      │
│  │                         工作区 (Workspace)                            │      │
│  │  • 文件系统                                                           │      │
│  │  • AGENTS.md / SOUL.md / USER.md                                    │      │
│  │  • 本地笔记和知识库                                                   │      │
│  │  • 人设规则和身份定义                                                 │      │
│  └─────────────────────────────────────────────────────────────────────┘      │
│                                                                                 │
│  ┌─────────────────────────────────────────────────────────────────────┐      │
│  │                    状态目录 (agentDir)                              │      │
│  │  • 认证配置文件 (auth-profiles.json)                                  │      │
│  │  • 模型注册表                                                         │      │
│  │  • 每 Agent 配置                                                      │      │
│  └─────────────────────────────────────────────────────────────────────┘      │
│                                                                                 │
│  ┌─────────────────────────────────────────────────────────────────────┐      │
│  │                    会话存储 (sessions)                               │      │
│  │  • 聊天历史                                                           │      │
│  │  • 路由状态                                                           │      │
│  │  • 存储位置: ~/.openclaw/agents/<agentId>/sessions                  │      │
│  └─────────────────────────────────────────────────────────────────────┘      │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### 路径映射

| 类型 | 路径 |
|-----|------|
| 配置 | `~/.openclaw/openclaw.json` |
| 状态目录 | `~/.openclaw` |
| 工作区 | `~/.openclaw/workspace-<agentId>` |
| Agent 目录 | `~/.openclaw/agents/<agentId>/agent` |
| 会话 | `~/.openclaw/agents/<agentId>/sessions` |

## 2. Agent 间通信机制

OpenClaw 提供四个核心会话工具来实现 Agent 之间的通信：

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                          Agent 间通信工具                                        │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  sessions_list        →  列出会话                                             │
│  sessions_history     →  获取会话历史                                           │
│  sessions_send        →  向另一个会话发送消息（等待回复）                      │
│  sessions_spawn       →  生成子 Agent 运行（异步）                             │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### sessions_list - 列出会话

列出当前可访问的会话：

```json5
{
  // 参数
  kinds: ["main", "group", "cron"],  // 过滤器
  limit: 50,                          // 最大数量
  activeMinutes: 60,                  // 最近 N 分钟活跃
  messageLimit: 3                     // 包含最后 N 条消息
}
```

**返回格式：**
```json
[
  {
    "key": "agent:main:whatsapp:group:120363...@g.us",
    "kind": "group",
    "channel": "whatsapp",
    "displayName": "开发团队",
    "updatedAt": 1701234567890,
    "sessionId": "abc123",
    "model": "claude-sonnet-4-5",
    "messages": [...]
  }
]
```

### sessions_history - 获取会话历史

获取特定会话的消息历史：

```json5
{
  sessionKey: "agent:main:whatsapp:group:120363...@g.us",
  // 或使用 sessionId:
  // sessionId: "abc123"
  limit: 20,           // 最大消息数
  includeTools: false  // 是否包含工具消息
}
```

### sessions_send - 发送消息（同步）

向另一个 Agent 发送消息并等待回复：

```json5
{
  sessionKey: "agent:work:whatsapp:group:120363...@g.us",
  message: "请审查这段代码",
  timeoutSeconds: 120  // 0 = 即发即忘
}
```

**关键特性：**
- `timeoutSeconds = 0`：立即返回，不等待回复
- `timeoutSeconds > 0`：等待最多 N 秒获取回复
- 支持乒乓对话：最多 `session.agentToAgent.maxPingPongTurns` 轮（默认 5）

### sessions_spawn - 生成子 Agent（异步）

启动一个子 Agent 执行任务并异步返回结果：

```json5
{
  task: "分析这个 Python 项目的架构",
  label: "code-analysis",           // 可选标签
  agentId: "analyzer",               // 可选：指定 Agent
  model: "claude-opus-4-5",         // 可选：覆盖模型
  runTimeoutSeconds: 300,            // 可选：超时时间
  cleanup: "delete"                 // 可选：执行后删除会话
}
```

**返回：**
```json
{
  "status": "accepted",
  "runId": "run_abc123",
  "childSessionKey": "agent:analyzer:subagent:uuid"
}
```

## 3. agentToAgent 配置

### 启用跨 Agent 通信

默认情况下，Agent 间通信是**禁用**的。需要在配置中显式启用：

```json5
{
  tools: {
    agentToAgent: {
      enabled: true,           // 启用功能
      allow: ["main", "work", "analyzer"]  // 允许通信的 Agent 列表
    }
  }
}
```

### 配置项详解

| 配置项 | 类型 | 说明 |
|-------|------|------|
| `enabled` | boolean | 是否启用跨 Agent 通信 |
| `allow` | string[] | 允许通信的 Agent ID 列表 |
| `maxPingPongTurns` | number | 乒乓对话最大轮次 (0-5) |

### 乒乓对话配置

```json5
{
  session: {
    agentToAgent: {
      maxPingPongTurns: 3  // 默认 5
    }
  }
}
```

## 4. 协作模式详解

### 模式一：广播组（Broadcast Groups）

**适用场景**：同一消息触发多个 Agent 并行处理

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                           广播组工作模式                                        │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  用户消息                                                                         │
│      │                                                                       │
│      ▼                                                                       │
│  ┌─────────────────────────────────────────────────────────────────────┐      │
│  │                     广播组配置                                        │      │
│  │  "broadcast": {                                                     │      │
│  │    "120363xxx@g.us": ["code-reviewer", "security", "docs"]        │      │
│  │  }                                                                  │      │
│  └─────────────────────────────────────────────────────────────────────┘      │
│      │                                                                       │
│      ▼                                                                       │
│  所有 Agent 并行处理                                                         │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐                       │
│  │ CodeReviewer │  │  Security   │  │    Docs    │                       │
│  └─────────────┘  └─────────────┘  └─────────────┘                       │
│      │                                                                       │
│      └─────────────────────┬───────────────────────────────────────┘        │
│                            ▼                                                │
│                    所有回复合并发送                                           │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

**配置示例：**
```json5
{
  agents: {
    list: [
      { id: "code-reviewer", workspace: "~/agents/codereview" },
      { id: "security", workspace: "~/agents/security" },
      { id: "docs", workspace: "~/agents/docs" }
    ]
  },
  broadcast: {
    strategy: "parallel",  // 并行处理
    "120363403215116621@g.us": ["code-reviewer", "security", "docs"]
  }
}
```

### 模式二：串行协作（Sequential）

**适用场景**：任务有严格依赖，需要按顺序执行

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                          串行协作模式                                          │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  任务 A                                                                       │
│      │                                                                       │
│      ▼                                                                       │
│  Agent A 处理 ──结果──►                                                      │
│                            │                                                 │
│                            ▼                                                 │
│                      任务 B                                                   │
│                            │                                                 │
│                            ▼                                                 │
│                Agent B 处理 ──结果──►                                         │
│                                              │                               │
│                                              ▼                               │
│                                        最终结果                                │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

**实现方式：**
```json5
{
  broadcast: {
    strategy: "sequential",
    "+15551234567": ["researcher", "summarizer", "responder"]
  }
}
```

### 模式三：路由分发（Routing）

**适用场景**：根据消息内容或发送者分发到不同 Agent

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                           路由分发模式                                          │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  用户消息                                                                         │
│      │                                                                       │
│      ▼                                                                       │
│  ┌─────────────────────────────────────────────────────────────────────┐      │
│  │                     Binding 匹配                                     │      │
│  │  • peer (DM/群组 ID)                                               │      │
│  │  • channel (渠道)                                                   │      │
│  │  • guildId (Discord)                                               │      │
│  │  • accountId (账户)                                                 │      │
│  └─────────────────────────────────────────────────────────────────────┘      │
│      │                                                                       │
│      ▼                                                                       │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐                       │
│  │  Telegram   │  │  WhatsApp   │  │  Discord   │                       │
│  │    ↓       │  │    ↓       │  │    ↓       │                       │
│  │   main     │  │   work     │  │  coding    │                       │
│  └─────────────┘  └─────────────┘  └─────────────┘                       │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

**配置示例：**
```json5
{
  bindings: [
    // Telegram 路由到 main Agent
    { agentId: "main", match: { channel: "telegram" } },
    // WhatsApp 路由到 work Agent
    { agentId: "work", match: { channel: "whatsapp" } },
    // 特定 Discord 服务器路由到 coding Agent
    { agentId: "coding", match: { channel: "discord", guildId: "123456789" } }
  ]
}
```

### 模式四：主从协作（Master-Slave）

**适用场景**：主 Agent 协调多个子 Agent 完成复杂任务

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                          主从协作模式                                          │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│                         用户复杂任务                                            │
│                               │                                               │
│                               ▼                                               │
│  ┌─────────────────────────────────────────────────────────────────────┐      │
│  │                      主 Agent (main)                                 │      │
│  │  • 分析任务                                                         │      │
│  │  • 分解子任务                                                      │      │
│  │  • 协调子 Agent                                                    │      │
│  └─────────────────────────────────────────────────────────────────────┘      │
│                               │                                               │
│          ┌────────────────────┼────────────────────┐                       │
│          ▼                    ▼                    ▼                       │
│  ┌─────────────┐      ┌─────────────┐      ┌─────────────┐                 │
│  │  子 Agent A  │      │  子 Agent B  │      │  子 Agent C  │                 │
│  │ (执行任务)   │      │ (执行任务)   │      │ (执行任务)   │                 │
│  └──────┬──────┘      └──────┬──────┘      └──────┬──────┘                 │
│          │                    │                    │                         │
│          └────────────────────┼────────────────────┘                       │
│                               ▼                                               │
│  ┌─────────────────────────────────────────────────────────────────────┐      │
│  │                      主 Agent 汇总结果                               │      │
│  │  • 整合子结果                                                      │      │
│  │  • 返回最终答案                                                    │      │
│  └─────────────────────────────────────────────────────────────────────┘      │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

**实现方式：**
```json5
{
  // 启用 agentToAgent
  tools: {
    agentToAgent: {
      enabled: true,
      allow: ["main", "sub1", "sub2", "sub3"]
    }
  },
  agents: {
    list: [
      { id: "main", workspace: "~/.openclaw/workspace-main" },
      { id: "researcher", workspace: "~/.openclaw/workspace-researcher" },
      { id: "writer", workspace: "~/.openclaw/workspace-writer" },
      { id: "reviewer", workspace: "~/.openclaw/workspace-reviewer" }
    ]
  }
}
```

在主 Agent 的 AGENTS.md 中配置协作流程：
```markdown
# 任务处理流程

1. 收到复杂任务时，使用 sessions_spawn 生成子 Agent
2. 子 Agent 完成后，汇总结果
3. 返回最终答案
```

## 5. 完整配置示例

### 示例：多 Agent 客服系统

```json5
{
  agents: {
    list: [
      {
        id: "triage",
        name: "客服分流",
        workspace: "~/.openclaw/workspace-triage",
        model: "claude-sonnet-4-5"
      },
      {
        id: "sales",
        name: "销售助手",
        workspace: "~/.openclaw/workspace-sales",
        model: "claude-sonnet-4-5"
      },
      {
        id: "tech",
        name: "技术支持",
        workspace: "~/.openclaw/workspace-tech",
        model: "claude-opus-4-5"
      },
      {
        id: "billing",
        name: "账单管理",
        workspace: "~/.openclaw/workspace-billing"
      }
    ]
  },
  // 启用跨 Agent 通信

    agentToAgent: {
      enabled: true,

    { agentId: "triage", match: { channel: "whatsapp" } }

    whatsapp: {
      dmPolicy: "allowlist",
      allowFrom: ["+15551230001", "+15551230002"]
    }
  }
}
```

## 6. 安全与隔离

### 沙箱隔离

每个 Agent 可以配置独立的沙箱：

```json5
{
  agents: {
    list: [
      {
        id: "trusted",
        workspace: "~/.openclaw/workspace-trusted",
        sandbox: {
          mode: "off"  // 无沙箱
        }
      },
      {
        id: "untrusted",
        workspace: "~/.openclaw/workspace-untrusted",
        sandbox: {
          mode: "all",
          scope: "agent",
          docker: {
            setupCommand: "apt-get update && apt-get install -y git"
          }
        },
        tools: {
          allow: ["read", "exec"],
          deny: ["write", "edit", "browser"]
        }
      }
    ]
  }
}
```

### 会话可见性

沙箱隔离的 Agent 默认只能看到自己生成的会话：

```json5
{
  agents: {
    defaults: {
      sandbox: {
        sessionToolsVisibility: "spawned"  // 仅能看到生成的会话
        // 或 "all" - 可见所有会话
      }
    }
  }
}
```

## 8. 最佳实践

### 设计原则

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                        多 Agent 设计原则                                        │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  1. 单一职责                                                                    │
│     每个 Agent 只专注一个领域                                                   │
│     ✅ 正确: code-reviewer, security-auditor, docs-generator                  │
│     ❌ 错误: all-in-one-helper                                                 │
│                                                                                 │
│  2. 清晰接口                                                                    │
│     定义 Agent 之间的消息格式                                                   │
│     使用结构化提示词确保一致性                                                  │
│                                                                                 │
│  3. 最小权限                                                                    │
│     只授予 Agent 必要的工具权限                                                │
│     敏感操作需要多个 Agent 确认                                                │
│                                                                                 │
│  4. 故障隔离                                                                    │
│     一个 Agent 失败不影响其他                                                  │
│     使用 try-catch 保护跨 Agent 调用                                           │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### 性能优化

- 使用 `parallel` 策略代替 `sequential` 提高响应速度
- 为简单任务使用轻量模型（如 sonnet 而非 opus）
- 限制广播组中的 Agent 数量（建议 5-10 个）
- 合理设置 `timeoutSeconds` 避免长时间等待

---

## 本节小结

1. **Agent 隔离**：每个 Agent 拥有独立的工作区、认证和会话
2. **通信工具**：
   - `sessions_list`：列出会话
   - `sessions_history`：获取历史
   - `sessions_send`：同步消息
   - `sessions_spawn`：异步生成子 Agent
3. **agentToAgent**：需要显式启用才能跨 Agent 调用
4. **协作模式**：广播组、串行、路由、主从
5. **安全**：沙箱隔离 + 工具权限控制
