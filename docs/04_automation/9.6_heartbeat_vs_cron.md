# 9.6 Heartbeat vs Cron：如何选择合适的自动化机制

本节讲解 OpenClaw 的两种自动化机制——Heartbeat 和 Cron——帮助你根据场景选择合适的方式。

---

## 9.6.1 快速决策指南

| 场景 | 推荐方案 | 原因 |
|------|----------|------|
| 每 30 分钟检查收件箱 | Heartbeat | 可与其他检查批量处理，支持上下文 |
| 每天 9 点发送报告 | Cron (isolated) | 需要精确时间 |
| 监控日历 upcoming 事件 | Heartbeat | 适合周期性感知 |
| 运行每周深度分析 | Cron (isolated) | 独立任务，可使用不同模型 |
| 20 分钟后提醒 | Cron (main, `--at`) | 一次性精确计时 |
| 后台项目健康检查 | Heartbeat | 利用现有周期 |

---

## 9.6.2 Heartbeat：周期性感知

Heartbeat 在**主会话**中按固定间隔运行（默认：30 分钟）。其设计目的是让 Agent 检查事物并呈现重要内容。

### 适用场景

- **多个周期性检查**：一个 Heartbeat 批次处理收件箱、日历、天气、通知
- **上下文感知**：Agent 有完整的主会话上下文，可智能判断优先级
- **对话连续性**：Heartbeat 共享相同会话，记住最近对话
- **低开销监控**：一个 Heartbeat 替代多个轮询任务

### Heartbeat 优势

```
┌─────────────────────────────────────────────────────────────┐
│                     Heartbeat 优势                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 批量检查                                               │
│     └── 一次 Agent 回合可检查收件箱、日历、通知              │
│                                                             │
│  2. 减少 API 调用                                         │
│     └── 单个 Heartbeat 比 5 个独立 cron 任务更便宜           │
│                                                             │
│  3. 上下文感知                                             │
│     └── Agent 知道最近工作内容，可按优先级处理                │
│                                                             │
│  4. 智能抑制                                               │
│     └── 无需关注时回复 HEARTBEAT_OK，不发送消息             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 配置示例

```json5
{
  agents: {
    defaults: {
      heartbeat: {
        every: "30m",          // 间隔
        target: "last",        // 明确 alert 投递目标
        activeHours: {         // 可选：活跃时间段
          start: "08:00",
          end: "22:00"
        },
      },
    },
  },
}
```

### HEARTBEAT.md 示例

```markdown
# Heartbeat 检查清单

- 检查邮件中的紧急消息
- 查看接下来 2 小时的日历事件
- 如果后台任务完成，总结结果
- 如果空闲 8+ 小时，发送简短签到
```

Agent 在每次 Heartbeat 时读取此清单，一次性处理所有项目。

## 9.6.3 Cron：精确调度

Cron 任务在精确时间运行，可以在独立会话中运行而不影响主上下文。

### 适用场景

- **需要精确时间**："每周一 9:00 发送"（不是"大约 9 点"）
- **独立任务**：不需要对话上下文
- **不同模型/思考**：重分析可使用更强大模型
- **一次性提醒**："20 分钟后提醒"使用 `--at`
- **频繁任务**：会弄乱主会话历史的任务
- **外部触发**：应独立于 Agent 是否活跃的任务

### Cron 优势

```
┌─────────────────────────────────────────────────────────────┐
│                      Cron 优势                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 精确时间                                               │
│     └── 5 或 6 字段 Cron 表达式，支持时区                  │
│                                                             │
│  2. 内置负载分散                                           │
│     └── 重复整点任务自动分散在 0-5 分钟窗口                 │
│                                                             │
│  3. 独立控制                                               │
│     └── 可用 --stagger 或 --exact 覆盖                     │
│                                                             │
│  4. 会话隔离                                               │
│     └── 在 cron:<jobId> 中运行，不污染主历史               │
│                                                             │
│  5. 模型覆盖                                               │
│     └── 每个任务可使用不同模型                               │
│                                                             │
│  6. skipWhenIdle（v2026.3.24+）                            │
│     └── 空闲时跳过执行，节省资源                             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

> ⚠️ **v2026.3.24 Bug #56045**：Cron job 删除后可能持续投递。删除后需执行 `openclaw gateway restart`。

### Cron 示例：每日早间简报

```bash
openclaw cron add \
  --name "Morning briefing" \
  --cron "0 7 * * *" \
  --tz "America/New_York" \
  --session isolated \
  --message "Generate today's briefing: weather, calendar, top emails, news summary." \
  --model opus \
  --announce \
  --channel whatsapp \
  --to "+15551234567"
```

### Cron 示例：一次性提醒

```bash
openclaw cron add \
  --name "Meeting reminder" \
  --at "20m" \
  --session main \
  --system-event "Reminder: standup meeting starts in 10 minutes." \
  --wake now \
  --delete-after-run
```

## 9.6.4 决策流程

```
任务需要精确时间?
  是 → 使用 Cron
  否 → 继续...

任务需要与主会话隔离?
  是 → 使用 Cron (isolated)
  否 → 继续...

任务可以与其他周期性检查批量处理?
  是 → 使用 Heartbeat (添加到 HEARTBEAT.md)
  否 → 使用 Cron

是一次性提醒?
  是 → 使用 Cron + --at
  否 → 继续...

需要不同模型或思考级别?
  是 → 使用 Cron (isolated) + --model/--thinking
  否 → 使用 Heartbeat
```

## 9.6.5 组合使用：最佳实践

最有效的设置是**两者结合**：

### 1. Heartbeat 处理常规监控

```markdown
# HEARTBEAT.md (每 30 分钟检查)

- 扫描紧急邮件
- 查看接下来 2 小时日历
- 检查待处理任务
- 如果安静 8+ 小时，轻微签到
```

### 2. Cron 处理精确调度

```bash
# 每天早上 7 点简报
openclaw cron add --name "Morning brief" --cron "0 7 * * *" --session isolated --message "..." --announce

# 每周一项目回顾
openclaw cron add --name "Weekly review" --cron "0 9 * * 1" --session isolated --message "..." --model opus

# 一次性提醒
openclaw cron add --name "Call back" --at "2h" --session main --system-event "Call back the client" --wake now
```

## 9.6.6 主会话 vs 独立会话

| 特性 | Heartbeat | Cron (main) | Cron (isolated) |
|------|-----------|-------------|-----------------|
| 会话 | 主会话 | 主会话 | `cron:<jobId>` |
| 历史 | 共享 | 共享 | 每次全新 |
| 上下文 | 完整 | 完整 | 无（干净开始） |
| 模型 | 主会话模型 | 主会话模型 | 可覆盖 |
| 输出 | 非 HEARTBEAT_OK 时投递 | Heartbeat 提示 + 事件 | Announce 摘要 |

### 使用场景

**主会话 Cron**（`--session main`）：
- 希望提醒/事件出现在主会话上下文
- 希望 Agent 在下次 Heartbeat 时处理

**独立 Cron**（`--session isolated`）：
- 需要干净的起始状态
- 需要不同模型或思考设置
- 直接向渠道 Announce 摘要
- 不想弄乱主会话历史

## 9.6.7 Lobster：确定性工作流

Lobster 是**多步骤工具管道**的运行时，需要确定性执行和明确审批。

### 适用场景

- **多步骤自动化**：需要固定工具调用管道
- **审批门控**：副作用应暂停等待审批，然后恢复
- **可恢复运行**：继续暂停的工作流而不重新运行之前的步骤

### 与 Heartbeat 和 Cron 配合

- **Heartbeat/Cron** 决定何时运行
- **Lobster** 定义开始后执行哪些步骤

### Lobster 操作

```bash
# 触发 Lobster 工作流
lobster run --workflow my-workflow

# 恢复需要审批的工作流
lobster resume --token <token> --approve
```

## 9.6.8 成本考虑

| 机制 | 成本 |
|------|------|
| Heartbeat | 每 N 分钟一轮；随 HEARTBEAT.md 大小扩展 |
| Cron (main) | 添加事件到下次 Heartbeat（无独立轮次） |
| Cron (isolated) | 每任务完整 Agent 轮次；可使用便宜模型 |

### 成本优化建议

- 保持 `HEARTBEAT.md` 小巧，减少 token 开销
- 批量类似检查到 Heartbeat 而非多个 cron 任务
- 对 Heartbeat 使用 `target: "none"` 仅内部处理
- 对常规任务使用独立 cron + 便宜模型

---

## 本节小结

1. **Heartbeat**：适合周期性感知、批量检查、上下文感知
2. **Cron**：适合精确时间、独立任务、一次性提醒
3. **组合**：最佳实践是两者结合
4. **Lobster**：多步骤工作流，需要审批门控
5. **成本**：Heartbeat 最省，isolated Cron 最灵活
