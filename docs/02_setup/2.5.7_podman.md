## 2.5.7.1 为什么选择 Podman

| 特性 | Docker | Podman |
|------|--------|--------|
| 架构 | Docker Daemon（root 运行） | 无 Daemon（rootless 默认） |
| 安全性 | 需要 root 权限 | 普通用户即可运行 |
| systemd 集成 | 需要额外配置 | 原生支持 |
| 镜像格式 | Docker Image | Docker Image（兼容） + OCI Image |
| Socket | `docker.sock` | `podman.sock`（可兼容 docker.sock） |

**Podman 的核心优势**：

1. **Rootless 安全**：默认以普通用户运行容器，容器内 root 对应宿主机普通用户，零特权攻击面
2. **无 Daemon**：启动更快，Daemons 崩溃不会影响容器
3. **Docker 兼容**：`podman` 命令与 `docker` 高度兼容，大多数场景可直接替换

## 2.5.7.2 安装 Podman

### macOS

```bash
brew install podman

# 创建 Podman 虚拟机（Podman 在 macOS 上通过 Linux VM 运行）
podman machine init
podman machine start

# 验证
podman version
```

### Linux（Ubuntu/Debian）

```bash
# 推荐方式：apt 安装
sudo apt-get update
sudo apt-get install -y podman

# 或使用 RPM（适用于 Fedora/RHEL）
# sudo dnf install podman

# 验证
podman version
```

### Linux（Rootless 配置）

```bash
# 安装 slirp4netns 和 fuse-overlayfs（rootless 网络和存储驱动）
sudo apt-get install -y slirp4netns fuse-overlayfs

# 确认当前用户可使用 rootless Podman
podman unshare --usermap-uidmap=0:100000:65536 --usermap-gidmap=0:100000:65536

# 查看 rootless 状态
podman system connection list
```

## 2.5.7.3 Podman 部署 OpenClaw

### 方式一：podman run（快速启动）

```bash
# 基础运行（rootless，等效于 docker run）
podman run -d \
  --name openclaw \
  -p 18789:18789 \
  -v openclaw-data:/root/.openclaw \
  -e OPENCLAW_GATEWAY_URL=http://localhost:18789 \
  --restart unless-stopped \
  docker.io/openclaw/openclaw:latest

# 查看状态
podman ps --format "{{.Names}} {{.Status}} {{.Ports}}"
```

### 方式二：podman-compose（推荐，配置化管理）

```bash
# 安装 podman-compose
pip install podman-compose
# 或
brew install podman-compose

# 创建 docker-compose.yml（与 Docker 完全兼容）
cat > podman-compose.yml << 'EOF'
version: "3.8"

services:
  openclaw:
    image: docker.io/openclaw/openclaw:latest
    container_name: openclaw
    environment:
      - OPENCLAW_GATEWAY_URL=http://localhost:18789
      # v2026.3.24+ 推荐显式标注容器环境
      - OPENCLAW_CONTAINER=true
    volumes:
      - openclaw-data:/root/.openclaw
    ports:
      - "18789:18789"
    restart: unless-stopped

volumes:
  openclaw-data:
EOF

# 启动
podman-compose up -d

# 查看日志
podman-compose logs -f
```

### 方式三：Systemd 服务（长期运行，推荐 Linux 生产环境）

```bash
# 生成 systemd unit 文件
podman generate systemd --name openclaw --files

# 移动到 systemd 目录
mv container-openclaw.service ~/.config/containers/systemd/

# 重新加载 systemd
systemctl --user daemon-reload

# 启用开机自启
systemctl --user enable container-openclaw.service

# 启动服务
systemctl --user start container-openclaw.service

# 查看状态
systemctl --user status container-openclaw.service
```

## 2.5.7.4 Podman 与 Docker 关键差异

### Socket 兼容层（同时安装两者时）

```bash
# 如果机器上同时安装了 Docker 和 Podman
# Podman 可以模拟 Docker Socket，让 Docker 工具无缝使用 Podman

# 创建符号链接（可选，但有用）
sudo ln -sf /var/run/podman/podman.sock /var/run/docker.sock

# 验证 Docker CLI 可用 Podman
docker -H unix:///var/run/docker.sock ps
```

### Rootless Podman 的 UID/GID 映射

容器内进程以普通用户运行，UID/GID 通过 `slirp4netns` 映射到宿主机：

```bash
# 查看当前用户的 UID 映射范围
cat /etc/subuid   # 用户命名空间 UID
cat /etc/subgid   # 用户命名空间 GID

# 示例输出：
# user:100000:65536
# 表示 100000-165535 的 UID 映射给该用户
```

⚠️ **重要**：OpenClaw 的 `/root/.openclaw` 目录在容器内被映射到宿主机 UID。确保宿主机目录权限允许映射后的 UID 写入。

## 2.5.7.5 已知问题与 Workaround

### ⚠️ Issue #55388 — Podman + systemd 服务注册兼容性

**问题**：在某些 Linux 发行版（如 Fedora 38+、Ubuntu 22.04+）上，使用 `podman generate systemd` 生成的 unit 文件，配合 `systemctl --user enable` 时存在兼容性问题，导致服务无法正确启动。

**症状**：

```bash
systemctl --user status container-openclaw.service
# 输出：Unit cannot be started because it is blocked by condition
```

**诊断**：

```bash
# 检查 systemd 日志
journalctl --user -u container-openclaw.service -n 50

# 检查 Podman 版本（建议 >= 4.6.0）
podman version
```

**Workaround 方案一：使用 `podman start` 直接启动**

```bash
# 不依赖 systemd，直接用 podman 命令管理
podman start openclaw

# 设置自动重启
podman update --restart unless-stopped openclaw

# 开机自启：将启动命令加入 crontab @reboot
@reboot /usr/bin/podman start openclaw
```

**Workaround 方案二：更新 systemd unit 文件**

```bash
# 重新生成，添加 --new 参数
podman generate systemd --name openclaw --new --files

# 复制到正确位置
cp container-openclaw.service ~/.config/containers/systemd/

# 重新加载
systemctl --user daemon-reload
systemctl --user restart container-openclaw.service
```

**方案三：使用 Docker 替代**（最稳定）

如果 Podman systemd 问题持续影响服务稳定性，建议切换到 Docker：

```bash
# 停止 Podman 服务
podman stop openclaw
podman rm openclaw

# 切换到 Docker（docker-compose.yml 完全兼容）
docker-compose up -d
```

## 2.5.7.6 故障诊断

| 问题 | 诊断命令 | 解决方案 |
|------|---------|---------|
| `podman run` 报错 "cannot mount" | `podman unshare` 测试 | 安装 `slirp4netns fuse-overlayfs` |
| 容器启动后立即退出 | `podman logs openclaw` | 检查端口占用或权限问题 |
| 宿主机无法访问 18789 | `podman port openclaw` | 确认端口映射正确 |
| 数据持久化失败 | `ls -la ~/.local/share/containers/` | 检查 volume 挂载路径 |
| Rootless 模式下无网络 | `podman run --network=host ...` | 临时使用 host 网络 |

## 2.5.7.7 与 Docker 部署对比

| 维度 | Docker | Podman |
|------|--------|--------|
| **安装便利性** | ⭐⭐⭐⭐⭐ 成熟生态 | ⭐⭐⭐⭐ 快速崛起 |
| **Rootless 支持** | ⭐⭐ 需 Docker Daemon | ⭐⭐⭐⭐⭐ 原生支持 |
| **systemd 集成** | ⭐⭐⭐⭐ | ⭐⭐⭐（有已知问题） |
| **macOS 支持** | ⭐⭐⭐⭐⭐ 官方支持 | ⭐⭐⭐ VM 方式 |
| **生产稳定性** | ⭐⭐⭐⭐⭐ 广泛验证 | ⭐⭐⭐⭐ 持续改进 |

**建议**：

- **开发环境**：两者均可，Podman 更安全
- **生产环境 Linux**：优先 Docker；如需 rootless 安全隔离，用 Podman 但避免 systemd 方式，改用 `podman start` + crontab 自启
- **macOS**：Docker Desktop 稳定性更优

---

## 本节小结

1. Podman 提供 Docker 兼容的容器化方案，核心区别是无 Daemon rootless 运行
2. 部署命令与 Docker 高度兼容（`podman run`/`podman-compose`）
3. Systemd 集成存在已知兼容性问题（#55388），推荐使用 `podman start` + crontab 替代
4. Rootless 模式安全但需注意 UID/GID 映射与目录权限
