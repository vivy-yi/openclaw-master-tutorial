# 10.6 协议层与控制平面

本节全面讲解 OpenClaw 的协议层与控制平面机制，包括 Gateway 职责、连接协议、配对信任、事件幂等等核心概念。

---

## 10.6.1 控制平面职责与边界

在 OpenClaw 中，Gateway 不只是一个转发器。它必须在消息进入执行环前完成三类裁决：

### 三类裁决

| 裁决类型 | 核心问题 |
|----------|----------|
| **身份裁决** | 谁能连接？凭据是否有效？是否处于吊销或冷却状态？ |
| **归属裁决** | 消息应该路由到哪个智能体？哪个会话？避免串话与重复消费。 |
| **策略裁决** | 本次请求是否命中访问控制？群聊门控？工具治理？安全防护栏？ |

### 边界划分：控制平面与执行平面

```
控制平面：
├── 负责拒绝与降级
└── 不满足身份或策略的请求应尽早拒绝

执行平面：
├── 负责推理与工具
└── 只处理"已被裁决允许"的任务
```

**边界清晰的好处**：当出现故障时，能用观测信号判断是入口拒绝、路由异常、模型失败还是工具失败。

### 可观测与可审计

控制平面做的每一次关键决定，都应在日志与审计里留下证据：
- 允许或拒绝的原因
- 路由与会话键
- 配对与白名单是否命中
- 工具策略是否拦截

## 10.6.2 WebSocket 握手、鉴权与连接生命周期

### 握手阶段：身份与协议上下文必须明确

WebSocket 握手的工程意义不是建立一条通道，而是建立可验证的身份与协议上下文。

```
握手阶段必须完成：
├── 身份校验：令牌是否有效，是否与预期来源绑定
├── 协议协商：客户端能力与服务端能力是否一致
└── 上下文初始化：会话键与订阅范围如何确定
```

**最佳实践**：握手失败应尽早拒绝并记录拒绝原因。把失败拖到执行阶段，会把问题伪装成模型或工具异常。

### 生命周期：心跳、背压与关闭原因

长连接的难点在长期。生命周期需要三个可观测信号：

| 信号 | 作用 | 说明 |
|------|------|------|
| 心跳 | 发现半开连接与僵尸连接 | 定期 ping/pong |
| 背压 | 避免内存膨胀 | 限速、分级丢弃或断开 |
| 关闭原因 | 复盘"谁先断、为什么断" | 关闭码与原因必须落盘 |

### 重连与恢复：防重复订阅与防丢事件

```
1. 防重复订阅
   └── 重连后重复订阅同一流会导致重复处理与重复副作用

2. 防丢事件
   └── 断线期间错过的关键事件需要能恢复或至少能对账
```

**验收标准**：重连后系统状态可查询，且不会出现"同一输入触发两次写入"。

### 纵深鉴权：三层认证、限流与渠道健康监控

Gateway 实现了三层独立鉴权 + 多维限流 + 渠道健康监控的纵深架构。

**三层鉴权架构**：

| 层级 | 职责 | 细节 |
|------|------|------|
| **设备层** | 设备身份建立 | v2/v3 版本化载荷，含平台与设备族元数据，密码学签名 |
| **连接层** | 连接凭据解析 | 从配置/环境变量/密钥输入多源解析 |
| **用户层** | 用户身份验证 | 四种认证模式：`token`、`password`、`trusted-proxy`、`none` |

**认证链路执行顺序**：

```
解析客户端IP（代理感知）
    │
    ▼
检查限流器 → 尝试 Tailscale 认证 → 尝试令牌认证 → 尝试密码认证 → 尝试可信代理认证
    │
    ▼
记录失败或重置限流计数器
```

**WebSocket 两阶段认证**：握手时先验证共享密钥（Phase 1），再验证设备令牌（Phase 2），两者使用独立的限流作用域（`shared-secret` 与 `device-token`），互不干扰。

**多维限流配置**：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 最大尝试次数 | 10 | 滑动窗口内允许的最大失败次数 |
| 时间窗口 | 60,000ms（1 分钟） | 滑动窗口长度 |
| 锁定时长 | 300,000ms（5 分钟） | 触发锁定后的冷却期 |
| 回环豁免 | 默认开启 | 本地 CLI 永不被锁定 |

**渠道健康监控**：

| 健康状态 | 含义 |
|----------|------|
| `healthy` | 正常运行 |
| `not-running` | 渠道未启动 |
| `busy` | 智能体仍在执行中 |
| `stuck` | 智能体长时间无活动 |
| `disconnected` | WebSocket 已断开 |
| `stale-socket` | 半死连接：事件年龄 > 30 分钟 |
| `startup-connect-grace` | 启动宽限期（60 秒） |

## 10.6.3 事件幂等与一致性保障

### 事件语义：至少一次投递是默认假设

在多渠道、多重连、多重试的系统中，同一事件被投递多次并不罕见。

**导致重复的来源**：
```
├── 渠道回调重试
├── 网关重连重报
└── 消费者故障恢复后的重放
```

**设计原则**：系统设计应默认**至少一次**投递，并把"重复处理"设计成可接受行为。

### 幂等键：把重复处理变成无害

幂等键要满足三个性质：

| 性质 | 说明 |
|------|------|
| 稳定 | 同一业务意图在重试与重连中产生相同键 |
| 唯一 | 不同业务意图不会碰撞 |
| 有窗口 | 去重需要时间窗，匹配业务时效与回放需求 |

**强制要求**：对于写入类工具，幂等键应成为**强制入参或强制派生字段**。否则任何重试都会放大副作用。

### 乱序与提交点：把状态机写清楚

幂等只解决重复，不解决乱序。乱序会让旧指令覆盖新指令。

```
常见做法：
1. 为事件引入序列号或版本号
2. 在会话状态中记录 last_seq
3. 如果 seq <= last_seq，视为过期事件，直接丢弃或进入缓冲
4. 只有在跨过提交点后，才允许产生外部副作用
```

**提交点的意义**：提交点之前可安全终止，提交点之后必须走补偿或明确告知可能已产生副作用。

### 专项测试：重复、乱序与重试风暴

| 测试类型 | 测试内容 | 验收标准 |
|----------|----------|----------|
| **重复投递** | 同一事件投递两次 | 结果一致且无重复副作用 |
| **乱序投递** | 后发先到 | 系统拒绝或延迟处理过期事件 |
| **重试风暴** | 短时间大量失败重试 | 系统限速并保持可用 |

## 10.6.4 设备配对与本地信任链

设备配对用于把"某个真实设备或账号"与"系统中的受信入口"绑定起来。

### 配对的目标：身份锚点、权限边界与可撤销性

```
1. 身份锚点
   └── 稳定识别设备或账号，避免入口在共享设备或多用户环境中失去可追溯性

2. 权限边界
   └── 让受信入口与其它机制形成完整流程，避免低可信入口触发高风险能力

3. 可撤销性
   └── 当设备丢失、账号变更或合规要求变化时，可以快速撤销配对并阻断后续访问
```

### 配对流程：请求、审批、状态检查与撤销

```
┌──────────┐     发起配对      ┌──────────┐
│ 未配对   │ ───────────────► │ 待审批   │
└──────────┘                  └────┬─────┘
                                   │
                              approve  │
                                   ▼
┌──────────┐     revoke      ┌──────────┐
│ 已撤销   │ ◄────────────── │ 已生效   │
└──────────┘                  └──────────┘
```

**常用命令**：

```bash
# 列出待审批的配对请求
openclaw pairing list whatsapp

# 批准配对码并通知
openclaw pairing approve whatsapp <CODE> --notify

# 查看配对状态
openclaw pairing status whatsapp

# 撤销配对
openclaw pairing revoke whatsapp <CODE>
```

### 与渠道门控和路由联动

配对完成只意味着入口可被识别，不代表入口具备高权限。高权限应由确定性策略授予：

```
1. 渠道门控
   └── 私聊与群聊分开治理，群聊默认只在明确触发时响应

2. 路由绑定
   └── 对高确定性来源优先用绑定固定接管

3. 工具策略
   └── 默认拒绝高风险工具组，只在受控智能体与受控入口下允许
```

## 10.6.5 诊断命令

### 健康检查

```bash
# 健康检查，适合做自动化探测
openclaw health --json

# 状态总览，--deep 会触发更深的探测
openclaw status --deep

# 实时日志跟踪
openclaw logs --follow --json
```

### 日志分析

```bash
# 连接关闭原因分布
cat runtime.log | jq -r 'select(.type=="log") | .log | select(.event=="ws_close") | .close_reason' | sort | uniq -c | sort -nr | head
```

### 故障定界

```
如果 health 异常而 status 显示依赖不可用：
→ 优先回到认证与渠道状态

如果 status 正常但执行异常：
→ 优先看工具链路与模型回退
```

## 本节小结

1. **控制平面**：身份裁决、归属裁决、策略裁决
2. **连接管理**：握手鉴权、心跳背压、重连恢复
3. **三层鉴权**：设备层、连接层、用户层
4. **事件幂等**：幂等键、序列号、提交点
5. **设备配对**：身份锚点、权限边界、可撤销性
6. **限流架构**：四类作用域独立计数
