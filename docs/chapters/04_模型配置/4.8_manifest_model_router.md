# Manifest Model Router 完全指南

> Manifest 是什么：OpenClaw 的智能 LLM 路由插件，不是 Agent 配置文件。它坐落在 Agent 与 LLM 提供商之间，每次请求先经过多维评分算法（<2ms）判断复杂度，然后将请求分配到性价比最高的模型。

**适用版本**：manifest-model-router v6.0.2+，OpenClaw >= 2026.2.0

---

## 1. 核心概念

### 1.1 问题背景

OpenClaw 默认把所有请求发给同一个模型——简单问候也要调用 GPT-4o，浪费严重。

**Manifest 要解决的核心问题**：让 AI 请求自动路由到性价比最高的模型。

### 1.2 架构定位

```
OpenClaw Agent
      │
      ▼
┌─────────────────┐
│  manifest 插件   │  ← 路由决策层（<2ms）
│  (Provider 插件) │
└────────┬────────┘
         │
         ▼
   23 维复杂度评分
         │
         ▼
   四级 Tier 路由
         │
         ▼
   最优模型 + Dashboard
```

Manifest 是一个 **Provider 插件**，它注册为 `manifest` 提供商，提供 `manifest/auto` 模型。当 Agent 请求这个模型时，manifest 插件会根据请求复杂度自动选择最优的底层模型。

### 1.3 23 维评分算法（Multi-Dimension Scoring）

Manifest 的核心是一个轻量级复杂度评分算法，**耗时 <2ms**，评估三个维度：

| 维度 | 特征数量 | 示例 |
|------|---------|------|
| **关键词维度** | 14 个 | codePresence、mathSymbols、technicalTerms、filePaths、json Structures 等 |
| **结构维度** | 5 个 | tokenCount、messageCount、specialChars、hasSystemPrompt、multimodal 等 |
| **上下文维度** | 4 个 | followUp、ambiguity、formality、containsUrl 等 |

**评分结果**：-1 ~ +1 的浮点数，数字越大表示越复杂。

### 1.4 四级路由体系（Tier-Based Routing）

| Tier | 评分范围 | 适用场景 | 推荐模型 |
|------|---------|---------|---------|
| **Simple** | < -0.15 | 问候、基础问答 | GPT-3.5-Turbo、gemini-2.0-flash |
| **Standard** | -0.15 ~ 0.15 | 日常对话、翻译、总结 | GPT-4o-mini、claude-3-haiku |
| **Complex** | 0.15 ~ 0.4 | 代码调试、复杂分析 | GPT-4o、claude-3.5-sonnet |
| **Reasoning** | ≥ 0.4 | 高级数学、逻辑推导 | o1、claude-3-opus |

> 注意：这些 Tier 阈值是 Manifest 插件的默认行为，具体的阈值配置和模型映射由 Manifest 服务端控制。通过 `MANIFEST_API_KEY` 连接 Manifest 云服务时，这些规则由 Manifest 服务器定义。

└── openclaw.plugin.json ← 插件清单

### 2.2 Provider 注册机制（Provider Plugin）

Manifest 是一个 **Provider 插件**，不是 Runtime 插件。它通过 `provider-inject.ts` 向 OpenClaw 注册一个 `manifest` provider：

```typescript
// src/provider-inject.ts 核心逻辑
export const manifestProvider: ProviderPlugin = {
  id: 'manifest',
  name: 'Manifest',
  
  // manifest/auto 模型 → 复杂度评分 → 路由到最优模型
  models: {
    'auto': {
      endpoint: 'https://app.manifest.build', // Manifest 云端
      apiKey: process.env.MANIFEST_API_KEY,
      routing: 'tier-based',  // 启用四级路由
    }
  }
};
```

**关键点**：manifest 作为 provider 不自己做评分——它把请求转发给 Manifest 云服务，由云端完成 23 维评分和路由决策。

### 2.4 Agent 工具（Tools）

Manifest 为 Agent 提供了三个监控工具：

| 工具 | 用途 |
|------|------|
| `manifest_usage` | 查看当前使用量 |
| `manifest_costs` | 查看成本明细 |
| `manifest_health` | 健康检查 |

这些工具让 Agent 能够**自我感知**当前的模型消耗，便于做优化决策。

### 2.5 命令系统

`/manifest` 命令允许用户直接查询路由状态：

```bash

```

## 3. 安装与配置

### 3.1 安装（云端版本）

```bash
openclaw plugins install manifest-model-router
openclaw providers setup manifest-model-router
openclaw gateway restart
```

安装过程会引导输入 Manifest API Key（从 [app.manifest.build](https://app.manifest.build) 获取）。

### 3.2 环境变量配置

```bash
# CI/CD 场景
export MANIFEST_API_KEY=mnfst_...
```

### 3.3 本地部署版本

如果需要自托管（SQLite + 本地 Dashboard），安装完整包：

```bash
openclaw plugins install manifest
```

完整包包含本地路由引擎和 SQLite 存储，适合不想使用云端服务的用户。

## 4. 使用方法

### 4.1 设置默认模型

```yaml
# config.yaml
agents:
  defaults:
    model: "manifest/auto"
```

设置后，OpenClaw 所有请求都会经过 manifest 路由，选择最优模型。

### 4.2 指定特定 Agent

```yaml
agents:
  my-agent:
    model: "manifest/auto"
  simple-agent:
    model: "manifest/auto"  # 也会自动路由
```

### 4.3 查看路由状态

Agent 在日志中会显示路由决策：

```
[manifest] Tier: Standard → GPT-4o-mini (score: 0.08, <1ms)
[manifest] Tier: Complex → GPT-4o (score: 0.32, <1ms)
```

## 5. 工作原理详解

### 5.1 请求流程

```
1. Agent 发出请求（model: manifest/auto）
         │
         ▼
2. manifest provider 接收请求
         │
         ▼
3. 将请求转发至 Manifest 云服务
   (MANIFEST_API_KEY 认证)
         │
         ▼
4. Manifest 云端执行 23 维评分（<2ms）
         │
         ▼
5. 根据评分选择 Tier
         │
         ▼
6. 在 Tier 内选择最优模型
   (考虑：成本、延迟、可用性)
         │
         ▼
7. 执行 LLM 调用，返回结果
```

### 5.2 评分特征详解

**关键词维度（14 特征）**：

| 特征 | 含义 |
|------|------|
| `codePresence` | 是否包含代码块 |
| `mathSymbols` | 是否包含数学符号（∑, ∫, ≠） |
| `technicalTerms` | 技术术语密度 |
| `filePaths` | 是否包含文件路径 |
| `jsonStructures` | JSON/结构化数据密度 |
| `sqlKeywords` | SQL 关键词数量 |
| ... | ... |

**结构维度（5 特征）**：

| 特征 | 含义 |
|------|------|
| `tokenCount` | 请求 token 总数 |
| `messageCount` | 消息轮次 |
| `specialChars` | 特殊字符密度 |
| `hasSystemPrompt` | 是否含系统提示 |
| `multimodal` | 是否含图片/音频 |

**上下文维度（4 特征）**：

| 特征 | 含义 |
|------|------|
| `followUp` | 是否为跟进问题 |
| `ambiguity` | 歧义程度 |
| `formality` | 正式程度 |
| `containsUrl` | 是否含 URL |

### 5.3 模型切换策略

Manifest 支持 10+ 提供商：

| 提供商 | 模型覆盖 |
|--------|---------|
| OpenAI | GPT-3.5 / GPT-4 / o1 / o3 |
| Anthropic | Claude 3 / 3.5 / Opus |
| Google | Gemini 1.5 / 2.0 / Flash |
| DeepSeek | V3 / R1 |
| 本地模型 | Ollama / LM Studio |

每个 Tier 可以配置多个候选模型，按优先级尝试：

```
Complex Tier 路由示例：
  → GPT-4o (优先级1)
  → Claude-3.5-sonnet (优先级2，备用)
  → GPT-4o-mini (降级兜底)
```

## 6. 与 OpenClaw 插件系统的关系

### 6.1 Manifest 的插件类型

Manifest 是一个 **Provider 插件**，不是 Runtime 插件。这意味着：

| 对比维度 | Provider 插件 | Runtime 插件 |
|---------|-------------|-------------|
| 作用位置 | LLM 接口层 | Agent 决策层 |
| 加载时机 | 按需（配置引用时） | Gateway 启动时 |
| 配置位置 | `providers:` | `plugins:` |
| 示例 | manifest、openai、anthropic | acpx、browser、skill-workshop |

### 6.2 插件清单中的角色

```json
// openclaw.plugin.json
{
  "id": "manifest-model-router",
  "kind": "provider",        // 标记为 provider 类型
  "providers": ["manifest"], // 注册的 provider id
  "modelSupport": {          // 声明支持的模型
    "modelPrefixes": ["manifest/"]
  }
}
```

### 6.3 与 OpenClaw 模型别名的关系

OpenClaw 支持模型别名：

```yaml
agents:
  defaults:
    model: "minimax-m2.7"  # 会被解析为 minimax-portal/MiniMax-M2.7
```

Manifest 的 `manifest/auto` 是同样的机制——`manifest` 是 provider 前缀，`auto` 是模型名。当 Agent 请求 `manifest/auto` 时：

1. OpenClaw 识别 provider 前缀 `manifest/`
2. 找到 manifest provider 插件
3. 插件将请求转发给 Manifest 云服务
4. Manifest 决定使用哪个底层模型

## 7. 安全与隐私

### 7.1 凭证管理

Manifest 使用 `MANIFEST_API_KEY` 环境变量：

- **不写入** `config.yaml`（避免泄露）
- **日志遮蔽**：`sk-xxxx****ab12` 格式
- **可选**：CI 环境使用环境变量，生产环境使用交互式设置

### 7.2 隐私分析

Manifest 官方页面说明：

- 请求内容发送到 `app.manifest.build`
- API Key 仅用于认证
- 不存储请求内容（仅存储使用量统计）

### 7.3 安全扫描

ClawHub 对 Manifest 进行了安全扫描：

| 检查项 | 结果 |
|--------|------|
| VirusTotal | ✅ Benign |
| OpenClaw LLM 分析 | ✅ Benign（高置信度）|
| 静态分析 | ⚠️ 发现 3 个模式（需人工复核）|

静态分析发现的模式：
- `__tests__/build.test.ts:23`：测试中的 `eval()` 调用（测试代码，非运行时）
- `dist/index.js:2`：环境变量访问 + 网络发送（正常：API Key 发送到 manifest.build）
- `dist/index.js:11`：文件读取 + 网络发送（正常：配置文件读取）

**结论**：扫描发现的模式均属于正常用途，官方已提供详细的上下文解释。

## 8. 自托管 vs 云端

| 特性 | 云端版本 | 自托管版本 |
|------|---------|-----------|
| 安装命令 | `manifest-model-router` | `manifest` |
| 评分引擎 | Manifest 云服务 | 本地（SQLite） |
| Dashboard | app.manifest.build | 本地 localhost |
| API Key | 必须 | 可选 |
| 成本 | 按使用量收费 | 免费（本地运行） |
| 数据隐私 | 请求发送到第三方 | 数据留在本地 |
| 适用场景 | 小规模 / 不想运维 | 企业 / 高隐私需求 |

## 9. 故障排查

### 9.1 安装后模型不可用

```bash
# 检查插件是否加载
openclaw plugins list | grep manifest

# 检查 provider 配置
openclaw models list | grep manifest

# 重启 Gateway
openclaw gateway restart
```

### 9.2 API Key 无效

```bash
# 检查环境变量
echo $MANIFEST_API_KEY  # 应该输出 mnfst_... 开头

# 重新设置
openclaw providers setup manifest-model-router
```

### 9.3 路由不符合预期

Manifest 的路由逻辑由云端控制，如果路由结果不符合预期，可以：

1. 检查 Manifest Dashboard 的路由规则配置
2. 查看实际请求的评分日志
3. 反馈给 Manifest 社区
