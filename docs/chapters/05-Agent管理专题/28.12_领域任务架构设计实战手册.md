## 二、第一层：Multi-Agent 矩阵

### 2.1 三种 Loop 类型详解

#### 主从协调 Loop（Master-Slave）

**语义**：1 个协调者负责任务分解、分发、结果汇总。执行者之间不直接通信。

```
┌─────────────────────────────────────────────────────────────────┐
│                    主从协调 Loop 模式                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│                      ┌──────────────┐                            │
│                      │  协调者 Agent │                            │
│                      │  (Coordinator)│                            │
│                      └──────┬───────┘                            │
│                             │                                    │
│              ┌───────────────┼───────────────┐                   │
│              ↓               ↓               ↓                   │
│         ┌────────┐     ┌────────┐     ┌────────┐                │
│         │执行者A │     │执行者B │     │执行者C │                │
│         │ Agent  │     │ Agent  │     │ Agent  │                │
│         └────┬───┘     └────┬───┘     └────┬───┘                │
│              └───────────────┼───────────────┘                    │
│                             ↓                                    │
│                      ┌──────────────┐                            │
│                      │   结果汇总    │                            │
│                      └──────────────┘                            │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘

适用场景：
  ✅ 复杂任务需要统一规划（如：投研报告生成）
  ✅ 多个专业 Agent 需要协同但执行相互独立
  ✅ 需要一个"大脑"来协调全局

OpenClaw 实现：
  sessions_spawn（派生子 Agent）+ sessions_send（分发任务）+ 主 Agent 结果聚合
```

#### 链式 Loop（Chain）

**语义**：A → B → C，每个 Agent 的输出是下一个 Agent 的输入。前馈依赖，必须顺序执行。

```
┌─────────────────────────────────────────────────────────────────┐
│                      链式 Loop 模式                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌────────┐    数据    ┌────────┐    数据    ┌────────┐        │
│  │ Agent A│ ────────▶ │ Agent B│ ────────▶ │ Agent C│        │
│  │ (采集)  │    输出    │ (清洗)  │    输出    │ (分析)  │        │
│  └────────┘            └────────┘            └────────┘        │
│                                                                  │
│  特点：                                                           │
│  • 每步必须等上一步完成                                           │
│  • 数据在 Agent 间流动                                           │
│  • 任何一步失败，整条链需要决策：重试 / 跳过 / 中断               │
│                                                                  │
│  适用场景：                                                       │
│  ✅ ETL 流水线（采集→清洗→转换→加载）                             │
│  ✅ 多步骤推理（前提→分析→结论→建议）                             │
│  ✅ 内容流水线（选题→写作→审核→发布）                             │
│                                                                  │
│  OpenClaw 实现：                                                  │
│  sessions_send 接力，A 完成后发消息给 B，B 完成后发给 C            │
└─────────────────────────────────────────────────────────────────┘
```

#### 并行 Loop（Parallel）

**语义**：A ‖ B ‖ C，多个 Agent 同时执行，最后汇聚结果。独立同构任务的最佳选择。

```
┌─────────────────────────────────────────────────────────────────┐
│                      并行 Loop 模式                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│              ┌─────── A ───────┐                                 │
│              │  ┌──────────┐  │                                 │
│              │  │Agent A-1 │  │                                 │
│              │  │Agent A-2 │  │  并行执行                        │
│              │  │Agent A-3 │  │                                 │
│              │  └──────────┘  │                                 │
│              └─────── ↓ ───────┘                                 │
│                        ↓                                          │
│                   ┌────────┐                                      │
│                   │ 结果汇聚 │                                      │
│                   └────────┘                                      │
│                                                                  │
│  耗时对比：                                                       │
│  串行：T(A-1) + T(A-2) + T(A-3)                                  │
│  并行：max(T(A-1), T(A-2), T(A-3))  ← 节省 2/3 时间             │
│                                                                  │
│  适用场景：                                                       │
│  ✅ 多渠道数据并行采集（GitHub + Twitter + Reddit）                 │
│  ✅ 多角度分析并行（技术面 + 基本面 + 情绪面）                     │
│  ✅ 多版本生成并行（技术报告 + 通俗报告 + 视频脚本）                │
│                                                                  │
│  OpenClaw 实现：                                                  │
│  sessions_await（v2026.3.x 新增）— 原生支持并行等待和结果汇聚       │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 三种 Loop 的选择决策树

```
任务分析：输入是什么？

│
├─ 任务之间有数据依赖（输出→输入）？
│   └─ YES → 链式 Loop
│
├─ 任务是同一目标的多个独立方面/角度？
│   └─ YES → 并行 Loop
│
├─ 任务需要统一规划和协调？
│   └─ YES → 主从协调 Loop
│
└─ 混合场景？
    └─ YES → 多层嵌套
    示例：
      主从协调 Loop
        ├── 并行 Loop（多路采集）
        │     ├── A-1（GitHub 数据）
        │     ├── A-2（新闻舆情）
        │     └── A-3（市场数据）
        └── 链式 Loop（数据处理）
              ├── B-1（数据清洗）
              └── B-2（分析报告）
```

### 2.3 多层嵌套实战：晨间简报

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    晨间简报 Multi-Agent 矩阵                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  主从协调 Loop（mo-finance 协调者）                                          │
│  │                                                                           │
│  ├── 并行 Loop（同时采集三路数据）                                            │
│  │     ├── Agent-1：GitHub Trending 采集                                    │
│  │     ├── Agent-2：Twitter/Reddit 舆情采集                                 │
│  │     └── Agent-3：宏观数据抓取（PMI/CPI/利率）                            │
│  │     ↓                                                                     │
│  │     sessions_await → 三路结果汇聚                                         │
│  │                                                                           │
│  ├── 链式 Loop（数据处理流水线）                                             │
│  │     ├── B-1：数据清洗 + 结构化                                           │
│  │     ├── B-2：因子计算 + 指标评分                                         │
│  │     └── B-3：报告生成                                                   │
│  │                                                                           │
│  └── 主 Agent：汇总 + 格式化 + 推送                                          │
│                                                                             │
│  OpenClaw 实现：                                                            │
│  1. 主 Agent = mo-finance（协调者）                                         │
│  2. sessions_spawn 派生出三个采集 Agent（并行）                              │
│  3. sessions_await 等待三路完成                                             │
│  4. sessions_send 依次发送给清洗 Agent → 分析 Agent → 报告 Agent（链式）      │
│  5. 主 Agent 汇总结果，推送给用户                                            │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 三、第二层：Skills 三层架构

### 3.1 三层职责定义

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        Skills 三层架构                                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  L3 场景 Skills（领域封装层）                                                │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  定义：面向终端用户，封装完整业务流程，一键调用                          │   │
│  │  特征：输入/输出 有明确契约，不需要知道内部实现                          │   │

│  │                                                                       │   │
│  │  例：law-contract-review（合同审查）                                   │   │
│  │    输入：合同文本                                                       │   │
│  │    输出：风险条款清单 + 修改建议                                        │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                    ↓                                        │

│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  定义：可复用的子任务流水线，组合多个 L1 Skills 或 Tools                │   │
│  │  特征：被 L3 调用，也被 Agent 直接调用；原子化程度高                    │   │

│  │                                                                       │   │
│  │  例：agent-reach（社媒数据采集）                                       │   │
│  │    内部：web_search + 内容解析 + 结构化输出                             │   │
│  │    被调用：被 L3 股票舆情分析 Skill 调用                               │   │
│  │                                                                       │   │
│  │  例：data-cleaner（数据清洗流水线）                                    │   │
│  │    内部：去重 + 缺失值填充 + 格式标准化 + 异常值检测                    │   │
│  │    被调用：被所有需要数据的 L3 Skills 调用                             │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                    ↓                                        │
│  L1 AI 原生 Skills（底层能力层）                                             │

│  │  L1-1 AI 生成：                                                       │   │
```

### 3.2 三层调用关系实战：股票分析

以"贵州茅台七维度分析"为例，看三层 Skills 如何联动：

```
用户输入："分析贵州茅台"

═══════════════════════════════════════════════════════════════

L3 层调度：seven-dimension-analysis
  输入：贵州茅台（600519）
  决策：需要调用 L2 Skills × 7（每个维度一个）

  ┌─ 技术面 → L2: technical-analysis
  │     L2 内部：
  │       L1: extract_kpi（从 K线图提取均线/MACD/RSI）
  │       L1: classify（趋势分类：上升/下降/震荡）
  │       Tools: browser（抓取股票图表）
  │
  ├─ 基本面 → L2: fundamental-analysis
  │     L2 内部：
  │       L1: extract_kpi（从财报提取营收/净利/毛利率）
  │       L1: compare（与行业平均对比）
  │       Tools: web_fetch（抓取东方财富财报）
  │
  ├─ 宏观面 → L2: macro-analysis
  │     L2 内部：
  │       L1: extract_entities（提取 CPI/PMI/利率等指标）
  │       L1: sentiment（政策文本情感分析））
  │       Tools: web_search（宏观新闻搜索）
  │
  ├─ 情绪面 → L2: sentiment-analysis
  │     L2 内部：
  │       L1: sentiment（舆情情感分析）
  │       L1: classify（新闻分类：利好/利空/中性）
  │       Skills: agent-reach（Twitter/Reddit 帖子采集）
  │
  ├─ 行业面 → L2: industry-analysis
  │     L2 内部：
  │       L1: compare（茅台 vs 五粮液 vs 泸州老窖）
  │       L1: extract_kpi（行业集中度/增速）
  │       Tools: web_fetch（行业协会数据）
  │
  ├─ 风控面 → L2: risk-assessment
  │     L2 内部：
  │       L1: classify（风险等级：低/中/高）
  │       L1: generate_outline（风险点大纲）
  │       Tools: exec（本地风险计算脚本）
  │
  └─ 量化面 → L2: quantitative-factors
        L2 内部：
          L1: extract_entities（从因子库提取 PE/PB/ROE）
          L1: compare（当前 vs 历史分位）
          Tools: exec（量化因子计算）+ 内置数据库

═══════════════════════════════════════════════════════════════

L3 层聚合：seven-dimension-analysis
  7 个 L2 结果 → L1: generate_outline（汇总报告大纲）→ 最终报告
  输出：结构化七维度分析报告

```

### 3.3 Skills 的动态编排

Skills 不是静态配置，是**按需动态编排**的：

```json
// L3 Skill 调用时的完整声明
{
  "skill": "seven-dimension-analysis",
  "config": {
    "target": "贵州茅台",
    "invoke_skills": {
      "L3": [],                           // 本 Skill 不调用其他 L3
      "L2": [
        "technical-analysis",
        "fundamental-analysis", 
        "macro-analysis",
        "sentiment-analysis",
        "industry-analysis",
        "risk-assessment",
        "quantitative-factors"
      ],
      "L1": [
        "summarize",
        "extract_kpi",
        "sentiment",
        "classify"
      ]
    },
    "invoke_tools": {
      "L3_custom": ["wind_terminal_api"],   // 业务工具
      "L2_mcp": ["github", "browser"],       // MCP 工具
      "L1_core": ["read", "write", "exec"]  // 核心工具
    },
    "loop_type": {
      "outer": "parallel",    // 7 个维度并行
      "inner": "chain"        // 每个维度内部链式
    },
    "resources": {
      "model": {
        "coordinator": "claude-3.7-sonnet",
        "executors": "deepseek-v3"
      },
      "timeout_per_task": 120,
      "total_budget": 50000
    }
  }
}
```

### 4.2 三层工具的调用决策

```
工具选择优先级（从高到低）：

1. 优先用 L1 核心工具
   ✅ read 能解决就不用 L2/L3
   ✅ exec 能满足就不用包装
   ❌ 不要用 MCP server 调用 read

2. L2 协议工具用于：
   ✅ 需要版本管理/类型安全
   ✅ 需要跨系统集成（GitHub/文件搜索）
   ✅ 需要 MCP 原生的结构化输出

3. L3 业务工具用于：
   ✅ L1/L2 无法满足的专业需求
   ✅ 业务数据必须从专业系统获取
   ⚠️ 注意 API 成本和限流
```

### 4.3 工具与 Skills 的映射关系

```
Skills 描述"做什么"（意图层）
Tools 描述"怎么做"（执行层）

Skills 调用关系示例：

L3: seven-dimension-analysis
  │
  ├── L2: technical-analysis
  │     ├── Tools: browser（L3 业务）→ 抓取股票图表
  │     └── L1: extract_kpi → 从图表提取数据
  │
  ├── L2: fundamental-analysis
  │     ├── Tools: tushare_pro_api（L3 业务）→ 获取财报
  │     ├── Tools: web_fetch（L2 协议）→ 抓取公告
  │     └── L1: extract_kpi → 提取财务指标
  │
  └── L2: macro-analysis
        ├── Tools: web_search（L1 核心）→ 搜索宏观新闻
        ├── L1: sentiment → 政策情感分析
        └── L1: extract_entities → 提取宏观指标
```

### 5.2 Agent 三环驱动模型

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                      Agent 三环驱动模型                                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  环1：Agent 自身 Loop（自主驱动，内生）                                  │   │
│  │                                                                       │   │
│  │  触发源：Heartbeat 定时 / 条件阈值 / 自主决策                          │   │
│  │                                                                       │   │
│  │  循环体：                                                              │   │
│  │    1. 感知（Perceive）                                                │   │
│  │       → 检查定时任务是否到期                                            │   │
│  │       → 检查监控指标是否触发阈值                                        │   │
│  │       → 检查是否有等待中的协作请求                                       │   │
│  │                                                                       │   │
│  │    2. 思考（Think）                                                    │   │
│  │       → 判断需要做什么                                                 │   │
│  │       → 决定调用哪些 Skills/Tools                                      │   │
│  │       → 评估自身能力边界                                               │   │
│  │                                                                       │   │
│  │    3. 行动（Act）                                                     │   │
│  │       → 执行任务 / 发起协作 / 推送消息                                  │   │
│  │       → 记录结果到记忆                                                 │   │
│  │                                                                       │   │
│  │    4. 反思（Reflect）                                                 │   │
│  │       → 从结果中提取教训 → 更新 MEMORY.md                              │   │
│  │       → 判断是否需要主动发起测试/告警/会议                              │   │
│  │       → 判断是否需要学习新知识                                          │   │
│  │                                                                       │   │
│  │  示例（金融 Agent）：                                                  │   │
│  │    9:15 → 触发晨间简报（环1自主行动）                                   │   │
│  │    盘中  → 检测到茅台价格下跌 3% → 触发告警（环1主动告警）               │   │
│  │    15:00 → 触发收盘总结（环1自主行动）                                   │   │
│  │    收盘后 → 发起周度策略研讨会（环1主动会议）                           │   │
│  │    每周   → 从研报刊中学习新因子（环1自主学习）                          │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                    ↑                                        │
│                                    │ 触发                                    │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  环2：Task Loop（任务执行，外部触发）                                    │   │
│  │                                                                       │   │
│  │  触发源：用户请求 / 其他 Agent 协作请求 / 事件触发                       │   │
│  │                                                                       │   │
│  │  流程：                                                               │   │
│  │    用户："帮我分析苹果公司"                                            │   │
│  │       → 环2启动 → Task Loop                                            │   │
│  │       → L3 Skill 调度（seven-dimension-analysis）                      │   │
│  │       → 三层调用：L3 → L2 → L1 + Tools                                │   │
│  │       → 环1 自主记录到记忆                                              │   │
│  │       → 返回结果                                                        │   │
│  │                                                                       │   │
│  │  关键点：环2执行时，可能触发环1的学习和记录                              │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                    ↑                                        │
│                                    │ 协作请求                                │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  环3：Multi-Agent Loop（多 Agent 协作）                                  │   │
│  │                                                                       │   │
│  │  触发源：任务需要多 Agent 协同 / 环1发起协作请求                         │   │
│  │                                                                       │   │
│  │  模式：                                                               │   │
│  │    主从协调 Loop：mo-finance（协调）→ 派发任务给子 Agent                │   │
│  │    链式 Loop：A → sessions_send → B → sessions_send → C                │   │
│  │    并行 Loop：sessions_await → 多 Agent 同时执行 → 汇聚                 │   │
│  │                                                                       │   │
│  │  示例：                                                                │   │
│  │    mo-finance 协调者                                                  │   │
│  │       → 并行：mo-data（数据）× 3（GitHub/News/宏观）                   │   │
│  │       → 链式：数据 → 清洗 → 分析 → 报告                                │   │
│  │       → 结果推送 → 环1 记录                                            │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  三环关系：                                                                 │
│    环1 是"发动机" — 驱动自身持续运转                                         │
│    环2 是"执行器" — 响应外部任务                                            │
│    环3 是"放大器" — 将单 Agent 能力扩展到多 Agent 协作                       │
│    环1 可以触发环2（自主任务）和环3（协作任务）                               │
│    环2/环3 执行结果会回流到环1，驱动学习和记忆                                │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 5.3 主动探测（Proactive Beacons）的七种模式

#### 1. 🧪 主动测试（Self-Testing）

**触发条件**：定时自检（每小时/每天）/ 任务前预检 / 修复后验证

```
Agent 自主测试流程：

触发（定时或手动）
  │
  → 生成测试用例（基于历史失败模式）
  │
  → 执行测试（调用 Tools/Skills）
  │
  → 比对预期 vs 实际结果
  │
  ├── 正常 → 记录到 memory/YYYY-MM-DD.md，标注"自检通过"
  │
  └── 异常 → 触发 ⚠️ 主动告警（见下）
             → 写入 MEMORY.md（问题记录）
             → 尝试自我修复（如果能判断根因）

金融 Agent 示例：
  每交易日 9:00
    → 自检：尝试抓取茅台今日行情数据
    → 预期：有开盘价/成交量/当前价
    → 实际：返回完整数据 → 自检通过 ✅
    
  某次自检：
    → 预期：返回数据
    → 实际：API 返回 403 Forbidden
    → 告警：推送给运维 Agent "Wind API Token 过期"
    → 记录：MEMORY.md 更新 "Wind API 需每季度续期"
```

#### 2. ⚠️ 主动告警（Self-Alerting）

**触发条件**：监控指标超过阈值 / 异常模式识别 / 自主测试失败

```
告警分级：

P0 紧急（立即推送 + 立即处理）：
  • 系统完全不可用（Gateway down）
  • 严重数据泄露风险
  • 交易信号异常（资金安全）

P1 高（5 分钟内推送）：
  • API 连续失败 3 次
  • 监控指标突破硬止损线
  • 外部依赖（GitHub/Bloomberg）长时间无响应

P2 中（15 分钟内推送）：
  • Token 消耗超过日预算 80%
  • 非核心功能异常
  • 监控指标接近阈值

P3 低（次日汇总）：
  • 日志中出现偶发错误
  • 性能指标轻微下降
  • 非业务关键指标异常

告警推送路径：
  P0/P1 → 即时推送（Email/Telegram/电话）
  P2/P3 → 汇总推送（日报/周报）
  所有告警 → 写入 memory/YYYY-MM-DD.md
```

#### 3. 📅 主动会议（Self-Scheduling Meeting）

**触发条件**：定时复盘 / 策略调整期 / 异常事件后讨论

```
会议类型：

日度微复盘（每日收盘后 10 分钟）：
  • 参与者：Agent 自身（单 Agent）
  • 内容：今日任务完成情况 / 异常事件 / 明日关注点
  • 输出：写入 memory/YYYY-MM-DD.md

周度策略会（每周五收盘后）：
  • 参与者：mo-finance 主 Agent + 数据 Agent + 研究 Agent
  • 内容：本周策略执行回顾 / 下周市场展望 / 因子表现复盘
  • 输出：会议纪要 → 共享知识库 + MEMORY.md 更新

专题研讨会（按需）：
  • 触发条件：重大市场事件 / 新策略上线 / 持续亏损
  • 参与者：相关领域 Agent（可跨群）
  • 内容：深度分析 + 应对方案
  • 输出：决策记录 + 更新 Soul/Identity（如有策略调整）
```

#### 4. 🔍 主动探讨（Self-Initiated Research）

**触发条件**：新知识领域 / 不确定问题 / 前沿探索

```
探讨流程：

发现问题（Agent 不确定答案）
  │
  → 生成探讨请求（格式化问题描述）
  │
  → 确定参与探讨的 Agent（根据领域）
  │
  → sessions_send 发起探讨
  │
  → 收集各方观点
  │
  → 生成综合判断
  │
  → 写入 MEMORY.md（知识积累）

示例：
  mo-finance 发现：
    "最近 3 周茅台与沪深 300 相关性从 0.7 下降到 0.3，
     可能意味着市场风格转换，但样本量太小无法确定"
    
  主动探讨：
    → sessions_send → mo-data（数据 Agent）："分析过去 3 年
      茅台与沪深 300 相关性历史分布"
    → sessions_send → mo-yunying（内容 Agent）："搜集市场
      关于白酒板块近期表现的讨论"
    → 汇总 → 形成判断："风格切换可能性 65%，建议保持观察"
    → MEMORY.md：更新宏观分析框架中的相关性预警阈值
```

#### 5. 🔮 主动预测（Self-Predicting）

**触发条件**：周期性预测节点 / 重要事件前夕 / 趋势延续性判断

```
本质区别：
  告警 = 异常发生后告警（被动响应，已发生）
  预测 = 异常发生前预警（主动预防，未发生）

金融 Agent 实例：

mo-finance：
  触发：每周五 16:00
  动作：
    ① 分析过去 4 周茅台价格波动规律（均值/方差/趋势）
    ② 结合宏观日历（美联储议息/财报季/节假日）
    ③ 生成下周价格区间预测（±2σ 置信带）
    ④ 标注重要支撑位/压力位
    ⑤ 提前准备应对方案（"若向上突破压力位，追涨；若跌破支撑，减仓"）
  输出：
    → 写入 memory/YYYY-MM-DD.md（预测记录）
    → 若置信度 > 80% 且方向明确 → 提前 ⚠️ P2 预警

mo-weather：
  触发：每日 20:00
  动作：
    ① 分析未来 3 天气象数据
    ② 预测对业务的影响（如：暴雨 → 外卖订单 -40%，生鲜配送需求 +60%）
    ③ 提前调整运营资源配置
```

#### 6. 🔧 主动自愈（Self-Healing）

**触发条件**：检测到失败 / 异常返回 / 工具调用报错

```
本质区别：
  测试 = 预防性检查（正常时也做，定时）
  自愈 = 修复性行动（出问题才触发，自动响应）

自愈能力等级（L1 → L4 递进）：

  L1 重试（Retry）：
    最基础的自愈，相同参数重新执行一次
    例：web_fetch 返回 503 → 等 2 秒再试一次

  L2 切换（Switch）：
    主工具/主路径失败，切换到备用
    例：Wind API 报 403 → 切换到 tushare API
         tushare 也失败 → 切换到 eastmoney 数据

  L3 规避（Workaround）：
    绕过有问题的模块，用替代方案完成目标
    例：MCP server down → 改用 CLI 原生命令
         GitHub API 限流 → 从缓存读取历史数据

  L4 学习（Learn）：
    从失败中建立新的长效应对路径
    例：某 API 每月 15 日必限流 → MEMORY.md 记录 "每月 14 日前完成数据采集"

mo-finance 自愈实例：

  触发：定时自检发现 Wind API 返回 403 Unauthorized

  L1 重试：等待 3 秒重试 → 仍 403
  L2 切换：切换到 tushare_pro_api → 成功获取数据 ✅
  L3 规避：若 tushare 也失败 → 从本地缓存读取昨日数据 + 标注"数据可能滞后"

  结果处理：
    成功 → 记录 memory/YYYY-MM-DD.md "Wind API 切换至 tushare 完成"
    失败 → P1 告警推送给墨监 + "已尝试 L1/L2/L3 自愈均失败"
    任何情况 → MEMORY.md 更新："Wind API Token 需每季度续期，下次到期：XXXX-XX-XX"
```

#### 7. ⚡ 主动优化（Self-Optimizing）

**触发条件**：周期性评估（每周/每月/每季度）

```
本质区别：
  学习 = 积累更多知识（知道更多）
  优化 = 用更少资源做更好（效率提升）

评估维度：
  ① 准确率 — 上次预测/判断 vs 实际结果，偏差多大？
  ② 效率   — Token 消耗是否超出预算？响应速度是否可接受？
  ③ 覆盖率 — 哪些领域/市场/标的从未被分析过？
  ④ 稳定性 — 任务失败率是否在容忍范围内？
  ⑤ 自主性 — 有多少任务需要人工干预？

mo-finance 优化实例（每月末）：

  评估报告：
    准确率：本月价格趋势预测准确率 62%，较上月下降 5%
           → 根因分析：宏观因子权重过低
           → 优化动作：调高 CPI/PMI 在宏观面权重

    效率：Token 日均消耗 48000，超出预算 46000
           → 优化动作：减少不必要的 LLM 调用（数据采集不调用强模型）

    覆盖率：本月只覆盖白酒行业，化工/医药未覆盖
           → 优化动作：将化工/医药加入下月研究计划

    稳定性：晨间简报任务失败 2 次（API 限流）
           → 优化动作：增加 API 限流重试逻辑（L1 自愈增强）

  输出：
    → 优化记录写入 MEMORY.md
    → 若 Soul/Identity 有调整 → 更新对应文件
    → 下月执行计划调整
```

## 七、架构设计七步法（设计工作流）

```

  → 输入：领域任务描述

  → 方法：用自然语言描述流程 → 画出 DAG

第2步：Loop 类型决策
  → 输入：DAG 图
  → 输出：每个子任务分配 Loop 类型
  → 方法：参考"选择决策树"，独立并行，有依赖链式，统一协调主从

第3步：Agent 角色定义
  → 输入：子任务分配
  → 输出：Agent 角色矩阵（名称/专长/边界/协作关系）
  → 方法：按专长聚类，定义 Identity + Soul，标注调用关系

第4步：Skills 编排
  → 输入：Agent 专长
  → 输出：三层 Skills 调用链（L3→L2→L1）
  → 方法：先定义 L3 入口 Skill，再分解 L2，最后识别 L1

第5步：Tools 映射
  → 输入：Skills 所需能力
  → 输出：三层 Tools 映射（L3业务→L2协议→L1核心）
  → 方法：优先 L1，能用 L1 不引入 L2，有专业需求才用 L3

第6步：自主驱动配置
  → 输入：Agent 角色

  → 方法：设计环1自驱动（自检/告警/会议/探讨）

第7步：串讲验证
  → 输入：完整架构设计
  → 输出：端到端走读（用户说"分析茅台"→ 完整流程跑通）
  → 方法：用 ASCII 图走一遍每条路径，检查断点和循环
```
