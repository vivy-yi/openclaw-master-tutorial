## 1. 背景：垂直领域 Agent 的设计挑战

### 1.1 问题定义

构建一个垂直领域全能力 Agent（如 ai-interview 采访工作室）时，常见问题：

- **输出质量不稳定**：一次生成的结果参差不齐，需要人工返工
- **Context Window 饱和**：多会话后，上下文越来越长，Agent 开始遗忘关键信息
- **跨会话学习缺失**：每次会话从零开始，历史经验无法复用
- **Agent vs Tool 边界模糊**：把简单 Function 包装成 Agent，成本高且效率低
- **Cron 任务脆弱**：失败无 retry、无告警，凌晨故障无人知晓

### 1.2 解决思路

```
调研阶段：
  → 从网络获取 7 个 multi-agent design patterns（BenTerminal）
  → 从网络获取 H-MEM/MemGPT 分层记忆架构（TowardsAI）
  → 从网络获取 output validation harness 最佳实践

方法论阶段：
  → 提炼 8 个优化点（P0-1 到 P2-3）
  → 映射到现有 Skill 体系，找缺漏项
  → 新建缺失的 4 个 Skill

实现阶段：
  → 用优化后的 Skill 重写 ai-interview 设计
  → 验证 8 个优化点全部落地
```

## 2. 网络调研发现

### 2.1 Multi-Agent Design Patterns（BenTerminal）

来源：https://benterminal.com/en/posts/2025/agent-design-patterns/

**7 个核心模式**：

| 模式 | 描述 | 适用场景 |
|------|------|---------|
| **Supervisor-Mesh** | 主 Agent 并行调度多个 SubAgent，汇总结果 | 独立并行任务 |
| **Context-Isolated Subagents** | 每个 SubAgent 有独立上下文，不共享完整 history | 防止上下文爆炸 |
| **Hierarchical** | 多级 Supervisor，Supervisor 可调度 Supervisor | 复杂组织架构 |
| **Memory-Aware** | Agent 具备长期记忆能力 | 多会话场景 |
| **Tool-Integrated** | Agent 可调用外部工具扩展能力 | 开放世界任务 |
| **Self-Reflection** | Agent 产出后自我评审并修正 | 质量敏感任务 |
| **Planning-Aware** | Agent 在执行前显式规划步骤 | 复杂多步骤任务 |

### 2.2 Memory Architecture for Agentic AI Systems（TowardsAI）

来源：https://pub.towardsai.net/how-to-design-efficient-memory-architectures-for-agentic-ai-systems-81ed456bb74f

**H-MEM 分层记忆架构**：

| Layer | 功能 | TTL | 特点 |
|-------|------|-----|------|
| **Hot** | Working memory，当前任务上下文 | 会话级 | 高保真，低延迟 |
| **Mem** | Episodic，事件记录 | 30天 | 支持检索，有衰减 |
| **Store** | Semantic，长期知识 | 永久 | 提炼自 Episodic |
| **Summary** | 压缩摘要 | 永久 | 多次蒸馏的精华 |

**关键发现**：
- MemGPT 用"遗忘"模拟人类记忆曲线
- 纯向量检索不够，需要混合架构（向量 + 图谱）
- Context Window 是稀缺资源，必须分层管理

### 2.3 Output Validation Harness（Epsilla）

来源：Epsilla Blog + GitHub patterns

**12 个 Pattern 分类**：

| 类别 | Pattern 数 | 代表 |
|------|-----------|------|
| Memory | 4 | Persistent Instruction, Tiered Memory, Dream Consolidation, Progressive Compaction |
| Multi-Agent | 3 | Context-Isolated, Supervisor-Mesh, Tool Integration |
| Agentic | 5 | Self-Reflection, Planning, Loop-to-Harness |

**最值得借鉴的 3 个**：
1. **Progressive Compaction**：token 超过 60% 时渐进压缩，而非一次性截断
2. **Dream Consolidation**：空闲时自动去重 + 归档 + 重建索引
3. **Context-Isolated Subagents**：防止上下文污染

## 3. 8 个优化点详解

### 3.1 优化点全景图

```
P0 优先级（内容质量 + 风险控制）：
  ├── P0-1：Iterative Refinement（Generator→Critic→Quality Gate）
  └── P0-2：Human-in-the-Loop（Approval Gates）

P1 优先级（架构 + 成本）：
  ├── P1-1：Coordinator→Router（轻量模型路由）
  ├── P1-2：Forgetting 机制（遗忘 + 归档）
  └── P1-3：Cross-Session Learning（跨会话知识蒸馏）

P2 优先级（稳定性）：
  ├── P2-1：Agent-as-Tool（墨情/墨评降为 Tool）
  ├── P2-2：Cron retry + failureAlert
  └── P2-3：四种 Memory 分层
```

### 3.2 P0-1：Iterative Refinement

**问题**：一次生成质量不稳定，需要人工返工

**方案**：Generator→Critic→Quality Gate 三角色迭代

```
Generator → Critic → Quality Gate
              ↑
              └──── (不通过则回到 Generator)
```

**通过条件**：
- 高严重程度 = 0
- 中严重程度 ≤ 2
- 最大迭代 3 次

**实现**：新建 `iterative-refinement-harness` skill

### 3.3 P0-2：Human-in-the-Loop

**问题**：高风险操作（发布内容、敏感话题）缺乏人工审批

**方案**：Gate 0/2/3/4 四级审批门

| Gate | 触发时机 | 审批人 |
|------|---------|--------|
| Gate 0 | 采访开始前 | 用户 |
| Gate 2 | 内容发布前 | mo-law / 人工 |
| Gate 3 | 敏感话题出现 | 用户 |
| Gate 4 | 公开发布确认 | 用户 |

**实现**：`approval-gates-harness` skill（已存在，完整覆盖）

### 3.4 P1-1：Coordinator→Router

**问题**：主 Agent 用旗舰模型做路由决策，成本高但价值低

**方案**：主 Agent 降为轻量 Router，用 Haiku 级别模型

```yaml
# 降级前
主Agent: claude-sonnet  # 成本高

# 降级后
Router: claude-haiku    # 成本低 10x
```

**5Q 判断法**：
1. 是否需要历史上下文？否 → Tool
2. 是否需要自主规划？否 → Tool
3. 输出格式是否固定？是 → Tool
4. 是否需要自动重试？是 → Tool
5. 是否需要子任务分解？否 → Tool

**实现**：`agent-as-tool-harness` skill（新建）

### 3.5 P1-2：Forgetting 机制

**问题**：Context Window 饱和，Agent 开始遗忘关键信息但保留垃圾数据

**方案**：Weibull 衰减 + 重要性评分 + 自动归档

```python
def weibull_decay(t, lambda_=30, k=1.5):
    return math.exp(-(t / lambda_) ** k)
```

| 级别 | TTL 乘数 | 说明 |
|------|---------|------|
| critical | 永不 | 用户偏好、禁忌、安全边界 |
| high | ×2.0 | 核心观点、重要关系 |
| medium | ×1.0 | 日常交互 |
| low | ×0.5 | 临时查询，快速遗忘 |

**实现**：`forgetting-harness` skill（新建）

### 3.6 P1-3：Cross-Session Learning

**问题**：每次会话从零开始，历史经验无法复用

**方案**：三层知识蒸馏路径

```
Session → Episodic → Semantic → Procedural
 (Raw)     (事件)    (概念)    (规则)
```

**触发条件**：
- Episodic→Semantic：同主题 ≥3 次 或 置信度 ≥0.6
- Semantic→Procedural：验证 ≥10 次 + 置信度 ≥0.9

**实现**：`cross-session-harness` skill（新建）

### 3.7 P2-1：Agent-as-Tool

**问题**：墨情/墨评 是轻量无状态操作，但被包装成完整 Agent

**方案**：降级为 Tool，减少调度开销

| 角色 | 原类型 | 新类型 | 理由 |
|------|-------|-------|------|
| 墨情 | Agent | Tool | 情绪分析，无需历史上下文，输出固定 |
| 墨评 | Agent | Tool | 质量评分，无需历史上下文，输出固定 |
| 墨知 | Agent | Tool | 背景知识推送，查档即可，无状态 |

**实现**：`agent-as-tool-harness` skill（新建）

### 3.8 P2-2：Cron retry + failureAlert

**问题**：Cron 任务失败无 retry、无告警

**方案**：所有 Cron 任务配置重试 + 失败告警

```yaml
retry:
  max_attempts: 3
  backoff: exponential
  backoff_base: 2
  max_backoff: 60
failureAlert:
  channel: "telegram"
  to: "ai-interview"
  message: "daily_topics 失败，请检查日志"
```

**实现**：`cron-design-patterns` skill（已存在，完整覆盖）

### 3.9 P2-3：四种 Memory 分层

**问题**：所有记忆同一对待，导致重要信息被遗忘、垃圾数据占空间

**方案**：六层记忆架构（Hot/Warm/Cold/Shared/Vector/Graph）

```
TIER 1: HOT       — Working Memory（不过期，会话清理）
TIER 2: WARM      — Episodic（TTL: 7d, Weibull）
TIER 3: COLD      — Semantic（TTL: 30d, Weibull）
TIER 4: SHARED    — Procedural（永久，但需蒸馏）
TIER 5: VECTOR    — Archive（冷存储，可恢复）
TIER 6: GRAPH     — Identity（永不过期，critical）
```

**实现**：`memory-harness` skill（已存在，完整覆盖）

## 4. 4 个新 Skill 详解

### 4.1 Skill 总览

| Skill | 核心内容 | 行数 |
|-------|---------|------|
| `iterative-refinement-harness` | Generator→Critic→Quality Gate 循环 | ~250 |
| `forgetting-harness` | TTL + Weibull 衰减 + 归档策略 | ~300 |
| `cross-session-harness` | Episodic→Semantic→Procedural 蒸馏 | ~350 |
| `agent-as-tool-harness` | 5Q 判断法 + 降级/升级标准 | ~300 |

### 4.2 iterative-refinement-harness

**核心理念**：生成器生产，评审员挑刺，质量门放行

**三阶段循环**：

```
Generator → Critic → Quality Gate
              ↑
              └──── (FAIL 则回到 Generator)
```

**Critic 评审维度**：

| 维度 | 检查点 |
|------|--------|
| 准确性 | 事实错误、逻辑漏洞、数据伪造 |
| 完整性 | 遗漏关键信息、回答不全面 |
| 一致性 | 前后矛盾、与上下文冲突 |
| 可执行性 | 建议无法落地、步骤不可操作 |
| 风格 | 语气不当、目标受众不匹配 |
| 安全性 | 敏感内容、潜在有害建议 |

### 4.3 forgetting-harness

**核心理念**：Agent 必须有能力遗忘，不是所有记忆都同等重要

**遗忘策略**：

```yaml
ttl_policy:
  episodic:
    fresh_ttl: 7d
    decay_rate: 0.5
  semantic:
    fresh_ttl: 30d
    decay_rate: 0.7
  procedural:
    fresh_ttl: 90d
    decay_rate: 0.9
```

**遗忘执行流程**：

```
Every 6h:
  1. SCAN: 遍历所有记忆单元
  2. COMPUTE: 权重 = importance_score × weibull_decay
  3. CLASSIFY: active / watch / candidate
  4. ARCHIVE: candidate → 冷存储
  5. LOG: 记录遗忘事件
```

### 4.4 cross-session-harness

**核心理念**：每次会话结束后，Agent 应该变得更聪明

**三层蒸馏路径**：

```
Episode Record（30天TTL）
    ↓
Semantic Pattern（≥3次 OR 置信度≥0.6）
    ↓
Procedural Rule（验证≥10次 + 置信度≥0.9）
```

**Context 注入规则**：

```yaml
context_injection:
  max_semantic: 20    # 最多加载 20 个 semantic patterns
  max_procedural: 10  # 最多加载 10 个 procedural rules
  relevance_threshold: 0.5  # 只加载 relevance > 0.5 的
```

### 4.5 agent-as-tool-harness

**核心理念**：能用 Tool 解决的，不要用 Agent

**5Q 判断法**：

| Q | 否→Tool | 是→继续 |
|---|---------|--------|
| Q1: 需要历史上下文？ | Tool | Q2 |
| Q2: 需要自主规划？ | Tool | Q3 |
| Q3: 输出格式固定？ | Tool | Q4 |
| Q4: 需要自动重试？ | Tool | Q5 |
| Q5: 需要子任务分解？ | Tool | SubAgent |

**降级条件**：
- 上下文固化（行为稳定，不需要动态学习）
- 输出格式固定（schema 强制约束）
- 轻量路由（只做路由判断，不做执行）

**升级条件**：
- 需要记忆用户偏好
- 需要跨调用状态
- 需要与其他 Agent 协作

### 5.2 AGENTS.md 核心改动

**主 Agent 降为 Router**：

```
用户输入 → Router（Haiku 模型，< 1s）
    ↓
识别意图（enum 分类，不超过 5 种）
    ↓
派发对应 SubAgent
```

**Router 不做的事**：
- 不做嘉宾背景研究
- 不生成采访大纲
- 不分析情绪数据
- 不做内容合规判断

**Generator→Critic→Gate 示例（采访大纲）**：

```
Generator（墨纲）:
  输入：嘉宾信息 + 采访主题
  输出：初版大纲（含开场/核心/收尾）

Critic:
  - 逻辑：问题是否层层递进？
  - 覆盖度：背景/观点/反思三层都有吗？
  - 敏感度：职业伦理话题是否已铺垫？
  - 可执行性：每个问题嘉宾能回答吗？

Quality Gate:
  PASS：所有高/中严重问题已解决
  FAIL：打回 Generator 重新修订
  最大迭代：3 次
```

### 5.3 HARNESS.md 核心改动

**新增 2 个 Cron 任务**：

```yaml
# 每日记忆整合（凌晨 2:00）
daily_memory_consolidation:
  schedule: "0 2 * * *"
  retry.max_attempts: 1  # 失败不重试
  failureAlert: 开启

# 每周知识蒸馏（周日 3:00）
weekly_distillation:
  schedule: "0 3 * * 0"
  retry.max_attempts: 1
  failureAlert: 开启
```

**Gate 配置**：

```yaml
Gate 0: 用户确认（采访前）
Gate 2: mo-law 审批（发布前）
Gate 3: 用户决策（敏感话题）
Gate 4: 用户确认（公开发布）
```
