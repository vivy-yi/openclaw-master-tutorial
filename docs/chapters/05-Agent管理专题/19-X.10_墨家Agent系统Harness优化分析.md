## 二、用 Harness 思维审视现状

### 2.1 核心问题

> **当前系统缺乏：验证机制 + 反馈循环**

```
Harness = 约束 + 验证 + 迭代

当前系统：
  ✅ 有约束（Agent 角色定义）
  ❌ 缺验证（没有自动化验证）
  ❌ 缺迭代（没有反馈循环）
```

### 2.2 七类 Harness 对照检查

| Harness 类型 | 墨家系统现状 | 需要优化 |
|------------|-------------|---------|
| **ACP Agent Harness** | ✅ 有外部 Agent 集成能力 | 确认是否使用 |
| **Skill Harness** | ⚠️ 有 Skills，但缺少自动化测试 | 需要验证机制 |
| **CLI Harness** | ❌ 没有软件封装 | 按需引入 |
| **Superpowers** | ❌ 没有标准化工作流 | 需要流程标准化 |
| **Wiki Harness** | ✅ 有共享知识库 | 需要强化维护机制 |
| **Autoresearch** | ❌ 没有自主研究循环 | 按需引入 |
| **Coding Benchmark** | ❌ 没有自动化测试 | 需要质量验证 |

## 三、优化建议

### 3.1 优先级 P0：建立 Skills 验证机制

**问题**：Skills 质量无法保证，没有自动化测试。

**现状**：
```
mo-finance/skills/
├── autonomous-cycle.md      # 自主循环
├── central-bank-policy.md  # 央行政策
├── credit-market-monitoring.md  # 信用市场
├── equity-financing-monitoring.md  # 股权融资
├── geo-politics-event.md  # 地缘政治
├── ...
```

**优化方案**：参考 AntV Skill Harness，建立自动化评测。

```markdown
## mo-finance Skills 验证机制

### 1. 建立测试用例集
skills/
├── __tests__/
│   ├── equity-financing/
│   │   ├── case1_银行股分析.yaml
│   │   ├── case2_新能源分析.yaml
│   │   └── expected_output.yaml
│   ├── credit-market/
│   │   └── ...
│   └── workflow/
│       └── ...

### 2. 验证指标
- 格式正确性（YAML/Markdown 语法）
- 输出完整性（包含必需字段）
- 事实准确性（与权威数据对比）
- 一致性（跨文件引用）

### 3. 自动化验证脚本
scripts/
└── validate_skills.py
    - 检查所有 Skill 文件
    - 运行测试用例
    - 生成验证报告
```

### 3.2 优先级 P0：建立 Wiki/Lore 维护机制

**问题**：知识库缺乏自动维护，容易过时。

**现状**：
```
shared/knowledge/
├── facts/           # 事实库
├── decisions/       # 决策记录
└── entities/        # 实体库
```

**优化方案**：参考 llm-context-base，建立 Metadata 标准 + Lint 机制。

```markdown
## 知识库 Metadata 标准

所有知识文件必须包含：

---
type: fact|decision|entity
summary: 一句话描述
tags: [标签]
status: active|stale|archived
owner: 负责人
updated: YYYY-MM-DD
expires: YYYY-MM-DD  # 可选，过期自动提醒
related: [相关文件链接]
---

## Lint 规则

1. **过时检测**：超过 90 天未更新的 active 文件标记为 stale
2. **孤儿检测**：没有被引用的文件
3. **缺失检测**：缺少必需 Metadata 字段
4. **冲突检测**：同一事实的多处记录
```

### 3.3 优先级 P1：Superpowers 风格的工作流标准化

**问题**：Agent 工作流程不统一，质量参差不齐。

**现状**：
```
mo-finance 有复杂的 7 维度工作流
mo-yunying 有自己的运营流程
mo-law 有法律咨询流程
但缺乏统一的流程控制
```

**优化方案**：引入 Superpowers 的流程控制思想。

```markdown
## 标准工作流框架

### 1. 触发检查（参考 using-superpowers）
任何任务开始前必须：
- [ ] 确认任务类型
- [ ] 选择对应 Skill
- [ ] 遵循标准流程

### 2. 工作流类型

#### 分析类工作流
1. 收集信息（SEARCH / API 调用）
2. 整理事实（结构化输出）
3. 分析推理（多角度分析）
4. 验证结论（事实核查）
5. 输出建议（可操作）

#### 咨询类工作流
1. 理解需求（提问确认）
2. 查找依据（法规/案例）
3. 给出建议（结论先行）
4. 风险提示（明确说明）
5. 下一步指引（可执行）

### 3. 质量门禁
完成前必须：
- [ ] 事实有来源
- [ ] 结论有依据
- [ ] 风险有提示
- [ ] 建议可执行
```

### 3.4 优先级 P1：Self-Improving 机制强化

**问题**：纠正记录有了，但没有闭环。

**现状**：
```
shared/self-improving/
├── corrections.md    # 纠正记录
├── domains/           # 领域规则
├── heartbeat-rules.md  # 心跳规则
└── memory.md          # 记忆
```

**优化方案**：建立完整的反馈循环。

```markdown
## 自我改进闭环

### 1. 触发层
- 用户纠正 → corrections.md
- 重复成功 3 次 → 提升到 domain 规则
- 失败 → 降级机制

### 2. 执行层
- 新任务 → 检查 domain 规则
- 规则冲突 → 记录并询问
- 规则生效 → 更新状态

### 3. 验证层
- 每周检查 corrections 应用情况
- 清理已过时的规则
- 优化高频规则

### 4. 自动化脚本
scripts/
└── self_improvement_check.py
    - 统计纠正次数
    - 检查规则应用率
    - 生成改进报告
```

### 3.5 优先级 P2：Autoresearch 风格的知识积累

**问题**：知识积累依赖人工，缺乏自主性。

**现状**：
- 知识需要手动写入
- 没有主动学习和更新机制

**优化方案**：参考 Autoresearch，建立知识积累循环。

```markdown
## 知识自主积累循环

### 1. 触发条件
- 定时触发（每日/每周）
- 任务完成后触发
- 用户明确要求时触发

### 2. 积累流程
1. 检查活跃上下文
2. 提取新知识
3. 去重/更新现有知识
4. 验证一致性
5. 记录来源

### 3. 结果记录
knowledge/
└── _logs/
    └── accumulation_YYYY-MM-DD.md
```

### 3.6 优先级 P3：ACP Agent Harness 集成

**问题**：外部编码 Agent 能力没有被充分利用。

**现状**：
- 有 Claude Code / Codex 等外部 Agent
- 但没有标准化集成

**优化方案**：按需引入外部 Agent。

```markdown
## 外部 Agent 集成

### 使用场景
- 复杂代码生成 → Claude Code
- 批量数据处理 → Codex
- 特定工具调用 → 对应 Agent

### 集成方式
1. 通过 acp-router 自动路由
2. 显式调用（"用 Claude Code 生成..."）
3. 工作流嵌入（复杂任务自动调用）

### 注意事项
- 不要过度使用（成本考虑）
- 保留核心判断在墨家系统内
- 验证外部 Agent 输出
```

## 四、实施路线图

### 4.1 第一阶段（1-2 周）：基础建设

```
1. 建立 Skills Metadata 标准
   - 定义 Skill 文件必需字段
   - 编写 Lint 脚本

2. 强化知识库 Metadata
   - 为现有知识文件添加标准 Metadata
   - 编写过时检测脚本

3. 完善 corrections.md 闭环
   - 建立应用追踪机制
   - 每周生成改进报告
```

### 4.2 第二阶段（3-4 周）：流程标准化

```
1. 制定标准工作流框架
   - 分析类流程
   - 咨询类流程
   - 运营类流程

2. 引入质量门禁
   - 完成前检查清单
   - 事实核查机制

3. 强化 Self-Improving
   - 领域规则沉淀
   - 规则冲突处理
```

### 4.3 第三阶段（长期）：智能化

```
1. 知识积累自动化
   - 自主学习循环
   - 知识验证机制

2. Skills 质量评测
   - 自动化测试用例
   - 质量报告

3. 按需引入外部 Agent
   - 复杂任务自动化
   - 成本效益分析
```

## 五、一句话总结

> **墨家系统当前有 Harness 的"形"（多 Agent、分层），但缺 Harness 的"神"（验证机制 + 反馈循环）。优化的核心是建立：验证自动化 + 迭代闭环。**
