## 一、为什么需要 Knowledge Harness

### 1.1 传统知识管理的痛点

| 痛点 | 说明 |
|------|------|
| **知识孤岛** | 笔记散落在不同工具里，难以检索 |
| **维护负担** | 人工整理知识库耗时且容易遗忘 |
| **上下文丢失** | 搜索只返回片段，无法还原完整上下文 |
| **知识过时** | 过期的知识没人更新 |
| **RAG 的局限** | 每次查询都从原始文档重新生成，无法积累 |

### 1.2 Knowledge Harness 的核心理念

> **"让 AI 读取原始文档，自动构建和维护结构化知识库"**

```
传统方式：
  人类写笔记 → 人类维护 → 人类检索

Knowledge Harness：
  原始文档 → AI 自动提取 → 结构化 Wiki → AI 维护 → AI 检索
```

## 二、Karpathy 的 LLM Wiki 三层架构

### 2.1 三层结构

```
┌─────────────────────────────────────────────────────┐
│  Layer 1: 原始文档（raw/）                          │
│  - 不可变的源文档                                    │
│  - 文章、PDF、笔记、网页                             │
└─────────────────────────────────────────────────────┘
                    ↓ AI 读取
┌─────────────────────────────────────────────────────┐
│  Layer 2: LLM 维护的 Wiki（wiki/）                   │
│  - 结构化的知识页面                                  │
│  - AI 根据源文档自动维护                             │
└─────────────────────────────────────────────────────┘
                    ↓ AI 参考
┌─────────────────────────────────────────────────────┐
│  Layer 3: Schema Config（如 CLAUDE.md）              │
│  - 定义 Wiki 页面的格式规范                          │
│  - 保持一致性                                       │
└─────────────────────────────────────────────────────┘
```

### 2.2 核心价值

```
没有 Wiki 层：
  查询 → 搜索原始文档 → 片段拼接 → 质量不稳定
  
有 Wiki 层：
  查询 → Wiki 检索 → 结构化、交叉引用 → 质量稳定
```

### 2.3 关键洞察

**LLM 维护一个结构化的中间层，这样每次查询不需要从零开始。**

## 三、Metadata 标准：让 Wiki 可查询

### 3.1 为什么需要 Metadata

```
没有 Metadata：
  查询 → 加载所有文件 → 找到相关内容 → 慢且贵
  
有 Metadata：
  查询 → 扫描摘要和标签 → 直接加载相关文件 → 快且便宜
```

### 3.2 标准格式（YAML Frontmatter）

```yaml
---
type: knowledge
summary: 一句话描述，AI 用于判断相关性而不必读取完整文件
tags: [标签1, 标签2]
status: active
owner: 负责人
updated: 2026-04-07
related: "[相关文件](path/to/file.md)"
---

# 文件标题
```

### 3.3 字段说明

| 字段 | 用途 |
|------|------|
| **type** | 文件类型（knowledge、decision、project、how-to...）|
| **summary** | 一句话摘要，AI 扫描快速判断相关性 |
| **tags** | 标签，跨分类检索 |
| **status** | 状态（active/stale/draft/archived）|
| **owner** | 负责人 |
| **updated** | 更新日期，用于检测过时文件 |
| **related** | 相关链接，构建知识图谱 |

### 3.4 好处

- **快速路由**：AI 先扫描摘要，不加载完整文件
- **可扩展**：支持数百个文件而不降级
- **可维护**：Lint 系统自动标记过时文件

## 四、训练期适应机制

### 4.1 核心思想

**不需要预先设计知识库结构，系统会适应你的使用模式。**

```
传统方式（需要设计）：
  设计分类 → 创建文件夹 → 填充内容
  → 一半文件夹从未使用
  
训练期方式（自适应）：
  开始空 → AI 提问学习 → AI 建议结构 → 逐渐成型
```

### 4.2 三阶段

| 阶段 | 时间 | 行为 |
|------|------|------|
| **训练期** | 第 1-30 天 | AI 活跃提问，学习你的模式 |
| **冷却期** | 第 31-44 天 | AI 逐渐安静，~70% 减少建议 |
| **稳定期** | 第 45 天+ | AI 高效沉默，只在有问题时出声 |

### 4.3 训练期 AI 会做什么

- 问你关于工作方式的问题
- 当内容不适合时建议新目录
- 基于你的模式提议命名规范
- 在 `_config/config.md` 中记录学到的内容

## 五、代表项目详解

### 5.1 llm-context-base ⭐ 61

**Karpathy 模式的完整实现，开箱即用**

#### 目录结构

```
llm-context-base/
  _config/                    # 配置（大脑）
    config.md                 # 用户档案 + 训练控制器
    standard.md               # Metadata 标准

  _meta/                     # 系统指令（引擎）
    instructions/             # AI 操作指令
    templates/                # 文档模板

  _inbox/                    # 捕获区（7天 TTL）
  _output/                   # 生成产物
  _sources/                  # 原始文档（可选）
  
  1-Projects/                # 活跃项目
  2-Knowledge/               # 知识库
    HowTo/                   # 步骤指南
    Decisions/               # 决策记录
    References/              # 手册、规格
  3-Journal/                # 日记（私有）
  4-Private/                # 敏感内容
```

#### 核心特点

| 特点 | 说明 |
|------|------|
| **Metadata 标准** | 每个文件可查询 |
| **Inbox-first** | 7 天 TTL，先捕获后整理 |
| **训练期适应** | 30 天学习期 |
| **Lint 系统** | 检查过时文件、缺失 metadata |
| **多工具支持** | Claude Code/Cursor/Copilot/Windsurf |

#### 与 Karpathy 原始模式的区别

| 方面 | Karpathy 模式 | llm-context-base |
|------|--------------|-----------------|
| 源文档 | `raw/` 永久存档 | `_inbox/` 临时，7 天 TTL |
| Wiki 结构 | 预先设计 | 训练期自适应 |
| Schema 配置 | 单文件 | 拆分：profile + core + JIT |
| Metadata | 未指定 | 每个文件强制标准 |
| 查询路由 | 加载完整文件 | 摘要扫描优先 |

### 5.2 llm-wiki-agent ⭐ 2,241

**更侧重于从原始文档构建 Wiki，带知识图谱**

#### 目录结构

```
wiki/
├── index.md          # 所有页面的目录（每次 ingest 更新）
├── log.md            # 每次操作的追加记录
├── overview.md      # 跨所有源的活体综合
├── sources/          # 每个源文档的摘要页
├── entities/        # 人名、公司、项目（自动创建）
├── concepts/        # 想法、框架、方法（自动创建）
└── syntheses/       # 查询答案保存为 Wiki 页面

graph/
├── graph.json        # 持久化节点/边数据（SHA256 缓存）
└── graph.html       # 可视化图谱（vis.js）
```

#### 核心能力

| 能力 | 说明 |
|------|------|
| **自动实体页** | 每次新源文档引用时更新 |
| **自动概念页** | 跨所有源交叉引用 |
| **活体概览** | 每次 ingest 修订 |
| **矛盾标记** | ingest 时标记，不等到查询时 |
| **知识图谱** | graph.html 可视化 |
| **Lint 报告** | 孤儿页、断链、缺失实体 |

#### 支持的命令

```
ingest raw/papers/my-paper.md    # 摄入源文档
query: main themes?              # 从 Wiki 综合答案
lint                             # 发现问题
build graph                      # 构建图谱
```

#### vs RAG

| RAG | llm-wiki-agent |
|-----|---------------|
| 每次查询重新派生知识 | 编译一次，保持最新 |
| 原始块作为检索单元 | 结构化 Wiki 页面 |
| 无交叉引用 | 交叉引用已构建 |
| 矛盾在查询时出现（可能）| ingest 时标记 |
| 无积累 | 每个源使 Wiki 更丰富 |

### 5.3 Graphiti ⭐ 25K

**为 AI Agent 构建时序知识图谱**

#### 核心概念

```
Context Graph = 时序图谱
  - 实体（节点）：人、产品、政策、概念
  - 事实/关系（边）：带时序有效性窗口的三元组
  - Episodes（溯源）：原始摄取数据
```

#### 核心特点

| 特点 | 说明 |
|------|------|
| **时序事实管理** | 事实有有效期窗口，旧事实不会被删除 |
| **溯源** | 每个实体和关系追溯到原始 Episode |
| **增量构建** | 新数据立即整合，无需批量重算 |
| **混合检索** | 语义 + 关键词 + 图遍历 |
| **自主构建** | 从非结构化和结构化数据自主构建 |

#### vs GraphRAG

| 方面 | GraphRAG | Graphiti |
|------|----------|----------|
| 主要用途 | 静态文档摘要 | Agent 动态上下文 |
| 数据处理 | 批量导向 | 持续、增量更新 |
| 知识结构 | 实体簇和社区摘要 | 时序上下文图 |
| 检索方法 | 顺序 LLM 摘要 | 混合语义、关键词、图检索 |
| 适应性 | 低 | 高 |

## 六、如何设计自己的 Knowledge Harness

### 6.1 基础三步

```
Step 1：定义 Schema
  └─ Wiki 页面格式是什么？
  └─ Metadata 有哪些字段？
  
Step 2：建立摄取流程
  └─ 原始文档 → Wiki 页面的转换规则
  └─ 矛盾检测
  
Step 3：实现维护机制
  └─ 过时检测
  └─ 交叉引用更新
```

### 6.2 最小可行 Knowledge Harness

```markdown
# 最小结构
my-wiki/
├── _config.md          # Schema 配置
├── inbox/              # 待处理文档
└── wiki/              # 知识库
    ├── index.md       # 目录
    └── *.md           # 知识页面

# Schema 配置示例 (_config.md)
---
name: 我的知识库
version: 1.0
metadata:
  required:
    - type
    - summary
    - updated
  optional:
    - tags
    - related
page_types:
  - knowledge
  - decision
  - how-to
  - reference
---
```

### 6.3 自动化脚本

```python
#!/usr/bin/env python3
"""knowledge_harness.py - 最小知识管理 Harness"""

import os
import re
from datetime import datetime
from pathlib import Path

class KnowledgeHarness:
    def __init__(self, wiki_root):
        self.wiki_root = Path(wiki_root)
        self.inbox = self.wiki_root / "inbox"
        self.wiki = self.wiki_root / "wiki"
        
    def process_inbox(self):
        """处理收件箱中的文档"""
        for file in self.inbox.glob("*.md"):
            content = file.read_text()
            # 1. 提取关键信息
            summary = self.extract_summary(content)
            entities = self.extract_entities(content)
            
            # 2. 创建 Wiki 页面
            page = self.create_wiki_page(file.stem, content, summary, entities)
            
            # 3. 保存到 wiki/
            self.wiki.mkdir(exist_ok=True)
            (self.wiki / f"{file.stem}.md").write_text(page)
            
            # 4. 移动到已处理
            processed = self.wiki_root / "processed"
            processed.mkdir(exist_ok=True)
            file.rename(processed / file.name)
            
    def extract_summary(self, content):
        """提取摘要"""
        # 简化版：取前 100 字
        return content[:100] + "..."
    
    def extract_entities(self, content):
        """提取实体"""
        # 简化版：找大写开头的词
        return re.findall(r'\b[A-Z][a-z]+(?:\s+[A-Z][a-z]+)*\b', content)
    
    def create_wiki_page(self, name, content, summary, entities):
        """创建 Wiki 页面"""
        metadata = f"""---
type: knowledge
summary: {summary}
tags: []
status: active
updated: {datetime.now().strftime('%Y-%m-%d')}
entities: {entities}
---

# {name}

{content}

## 相关实体
{', '.join(entities)}
"""
        return metadata
    
    def lint(self):
        """检查 Wiki 健康状况"""
        issues = []
        for page in self.wiki.glob("*.md"):
            content = page.read_text()
            # 检查是否有时限标记
            if 'status: active' in content:
                # 检查更新日期
                updated = re.search(r'updated: (\d{4}-\d{2}-\d{2})', content)
                if updated:
                    date = datetime.strptime(updated.group(1), '%Y-%m-%d')
                    age = (datetime.now() - date).days
                    if age > 90:
                        issues.append(f"{page.name}: 超过 90 天未更新")
        return issues

if __name__ == "__main__":
    harness = KnowledgeHarness("./my-wiki")
    harness.process_inbox()
    issues = harness.lint()
    if issues:
        print("发现问题：")
        for issue in issues:
            print(f"  - {issue}")
```

## 八、与 RAG 的对比

| 维度 | RAG | Knowledge Harness |
|------|-----|-----------------|

| **适用场景** | 实时问答 | **知识积累** |
| **维护成本** | 低 | **中等** |
