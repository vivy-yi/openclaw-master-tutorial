## 二、核心哲学：图表应该「争论」

### 2.1 Display vs Argue

```
┌─────────────────────────────────────────────────────────────┐
│                    Display vs Argue                          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ❌ Display（展示）                                          │
│     └── 只是格式化的文字，用盒子标签内容                     │
│     └── 5 个相等的盒子 + 标签                                │
│     └── 卡片网格布局                                         │
│                                                              │
│  ✅ Argue（争论）                                            │
│     └── 形状结构本身传达概念                                 │
│     └── 每个概念用形状镜像其行为                             │
│     └── 视觉结构匹配概念结构                                 │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 Isomorphism Test（同构测试）

**核心问题**：如果移除所有文字，结构本身能传达概念吗？

```
❌ 不通过：移除文字后什么都没剩下
┌─────┐   ┌─────┐   ┌─────┐
│Input│ → │Proc │ → │Out  │
└─────┘   └─────┘   └─────┘

✅ 通过：结构本身就是含义
   ○ → ○ → ○
  ↙   ↘ ↙   ↘
  ○     ○     ○
 (扇出模式表示一个输入多个输出)
```

### 2.3 The Education Test（教育测试）

**问题**：这个图表能教别人一些具体的东西吗？还是只是贴标签？

| ❌ 不好（只是标签） | ✅ 好的（能教学） |
|-------------------|------------------|
| "API" 盒子 | 真实 API 端点 + 请求/响应示例 |
| "Database" 盒子 | 实际 Schema + 示例数据 |
| "Events" 标签 | 真实事件名称 + 时间线 |
| "UI" 矩形 | 模拟真实 UI 元素 |

## 三、深度评估：Simple vs Comprehensive

### 3.1 何时用简单图（Simple/Conceptual）

```
使用抽象形状，当：
• 解释心智模型或哲学
• 受众不需要技术细节
• 概念本身就是抽象（如「关注点分离」）
• ~30 秒解释完
```

### 3.2 何时用详细图（Comprehensive/Technical）

```
使用具体示例，当：
• 绘制真实系统、协议或架构
• 图表用于教学或解释（如 YouTube 视频）
• 受众需要理解实际看起来是什么样子
• 展示多个技术如何集成
• ~2-3 分钟教学内容
```

### 3.3 对比示例

| 简单图 | 详细图 |
|--------|--------|
| 通用标签："Input" → "Process" → "Output" | 具体：显示输入/输出的实际格式 |
| 命名盒子："API", "Database", "Client" | 命名盒子 + 实际请求/响应示例 |
| "Events" 或 "Messages" 标签 | 时间线 + 真实事件名称 |
| "UI" 或 "Dashboard" 矩形 | 模拟 UI 元素和内容 |

## 四、研究指令：技术图必须研究

### 4.1 为什么必须研究

```
❌ 不好：通用占位符
Protocol → Frontend

✅ 好：真实规格

  → CopilotKit renders via createA2UIMessageRenderer()
```

### 4.2 研究清单

绘制任何技术图之前：

1. **查找实际的 JSON/数据格式**
2. **找到真实的事件名称、方法名称、API 端点**
3. **理解各部分实际如何连接**
4. **使用真实术语，非通用占位符**

## 五、证据工件（Evidence Artifacts）

### 5.1 什么是证据工件

证据工件是**具体示例**，证明图表是准确的，帮助观众学习。技术图必须包含。

### 5.2 证据工件类型

| 工件类型 | 何时使用 | 如何渲染 |
|---------|---------|---------|
| **代码片段** | API、集成、实现细节 | 深色矩形 + 语法着色文字 |
| **数据/JSON 示例** | 数据格式、Schema、Payload | 深色矩形 + 绿色文字 |
| **事件/步骤序列** | 协议、工作流、生命周期 | 时间线模式（线 + 点 + 标签） |
| **UI 模拟** | 展示实际输出/结果 | 嵌套矩形模拟真实 UI |
| **真实输入内容** | 展示系统输入什么 | 矩形内显示示例内容 |
| **API/方法名称** | 真实函数调用、端点 | 使用文档中的真实名称 |

### 5.3 示例

**协议图**应包含：
- 规格中的真实事件名称（不只是 "Event 1", "Event 2"）
- 展示如何连接的代码片段
- 流式数据的实际样子

**数据转换管道**应包含：
- 示例输入数据（实际格式，非 "Input"）
- 示例输出数据（实际格式，非 "Output"）
- 相关中间状态

## 六、多层架构（Multi-Zoom Architecture）

### 6.1 三个层级

Comprehensive diagrams operate at **multiple zoom levels** simultaneously:

```
Level 1: 摘要流程
    简化的概览，-showing the full pipeline or process at a glance
    位置：通常在顶部或底部
    示例：Input → Processing → Output 或 Client → Server → Database

Level 2: 区块边界
    标记的区域，分组相关组件
    作用：创建视觉"房间"，帮助观众理解什么是一起的
    示例：按职责分组（Backend / Frontend）、按阶段（Setup / Execution / Cleanup）

Level 3: 区块内细节
    证据工件、代码片段、具体示例
    作用：教育价值所在
    示例：在 "Backend" 区块内，显示实际 API 响应格式，不只是标为 "API Response"
```

### 6.2 理想构成

```
┌─────────────────────────────────────────────────────────┐
│  Level 1: 摘要流程（简化概览）                            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐         │
│  │ Section A│    │ Section B│    │ Section C│         │
│  │          │    │          │    │          │         │
│  │ [证据]   │    │ [证据]   │    │ [证据]   │         │
│  │ [代码]   │    │ [JSON]   │    │ [示例]   │         │
│  └──────────┘    └──────────┘    └──────────┘         │
│        ↑              ↑              ↑                  │
│  Level 2: 区块边界（A / B / C 分组）                     │
│                                                         │
└─────────────────────────────────────────────────────────┘
                      ↑
         Level 3: 细节在区块内
```

## 七、视觉模式库

### 7.1 模式总览

| 模式 | 适用场景 |
|------|---------|
| **Fan-Out（扇出）** | 一对多、源、PRD、根本原因、中心枢纽 |
| **Convergence（汇聚）** | 聚合、漏斗、合成 |
| **Tree（树形）** | 层级、文件系统、组织架构、分类法 |
| **Spiral/Cycle（循环）** | 反馈循环、迭代过程、演进 |
| **Cloud（云形）** | 抽象状态、上下文、记忆、对话、心智模型 |
| **Assembly Line（流水线）** | 转换、处理、转换 |
| **Side-by-Side（并排）** | 对比、前后、选项、权衡 |
| **Gap/Break（间隔）** | 阶段变化、上下文重置、边界 |

### 7.2 Fan-Out（扇出）

```
        ○
       ↗
  □ → ○
       ↘
        ○
```
中心元素箭头辐射到多个目标。

### 7.3 Convergence（汇聚）

```
  ○ ↘
  ○ → □
  ○ ↗
```
多个输入通过箭头合并到单一输出。

### 7.4 Tree（树形）

```
  label
  ├── label
  │   ├── label
  │   └── label
  └── label
```
用 `line` 元素做树干和分支，free-floating 文字做标签（不需要盒子）。

### 7.5 Spiral/Cycle（循环）

```
  □ → □
  ↑     ↓
  □ ← □
```
序列元素，箭头返回起点。

### 7.6 Cloud（云形）

重叠的椭圆，不同大小。用于上下文、记忆、对话。

### 7.7 Assembly Line（流水线）

```
  ○○○ → [PROCESS] → □□□
  chaos              order
```
输入 → 处理框 → 输出，清晰的前后对比。

### 7.8 Lines as Structure

用 `line`（非箭头）作为主要结构元素：

```
Timeline:           Tree:
  ●─── Label 1        │
  │                   ├── item
  ●─── Label 2        │   ├── sub
  │                   │   └── sub
  ●─── Label 3        └── item
```

## 八、形状语义

### 8.1 形状选择

| 概念类型 | 形状 | 原因 |
|---------|------|------|
| 标签、描述、细节 | **无**（free-floating） | 字体创建层级 |
| 区块标题、注释 | **无**（free-floating） | 字号/粗细足够 |
| 时间线标记 | 小 `ellipse`（10-20px） | 视觉锚点，非容器 |
| 开始、触发、输入 | `ellipse` | 柔和、起源感 |
| 结束、输出、结果 | `ellipse` | 完成、目的地 |
| 决策、条件 | `diamond` | 经典决策符号 |
| 过程、动作、步骤 | `rectangle` | 包含的动作 |
| 抽象状态、上下文 | 重叠 `ellipse` | 模糊、云状 |
| 层级节点 | lines + text（无盒子） | 通过线创建结构 |

### 8.2 容器 vs Free-Floating 文字

| 何时用容器 | 何时用 Free-Floating |
|-----------|---------------------|
| 是区块焦点 | 是标签或描述 |
| 需要与其他元素视觉分组 | 是支持细节或元数据 |
| 箭头需要连接到它 | 描述附近内容 |
| 形状本身携带含义（决策菱形等） | 排版已创建足够层级 |
| 代表系统中一个独立的"东西" | 是区块标题、副标题或注释 |

**容器测试**：每个带盒子的元素问「这个能作为 free-floating 文字吗？」如果可以，去掉容器。

## 十、现代美学

### 10.1 Roughness（粗糙度）

- `roughness: 0` — 干净、锐利边缘。现代/技术图默认。
- `roughness: 1` — 手绘、有机感。头脑风暴/非正式图。

### 10.2 Stroke Width（线宽）

- `strokeWidth: 1` — 细、优雅。用于线条、分隔线、细微连接。
- `strokeWidth: 2` — 标准。用于形状和主要箭头。
- `strokeWidth: 3` — 粗。用于强调（主流程线、关键连接）。

### 10.3 Opacity（不透明度）

**始终使用 `opacity: 100`**。用颜色、大小、线宽创建层级，不用透明度。

### 10.4 小标记代替形状

用小点（10-20px ellipse）代替完整形状：
- 时间线标记
- 项目符号
- 连接节点
- free-floating 文字的视觉锚点

## 十一、布局原则

### 11.1 层级通过 Scale

| 层级 | 尺寸 | 用途 |
|------|------|------|
| Hero | 300×150 | 视觉锚点，最重要 |
| Primary | 180×90 | 主要元素 |
| Secondary | 120×60 | 次要元素 |
| Small | 60×40 | 小元素 |

### 11.2 Whitespace = 重要性

最重要的元素周围空白最多（200px+）。

### 11.3 Flow Direction

引导视线：通常左→右或上→下用于序列，径向用于中心辐射。

### 11.4 必须有连接

位置本身不显示关系。如果 A 关联 B，必须有箭头。

## 十三、验证循环（MANDATORY）

### 13.1 为什么必须渲染

**你无法仅从 JSON 判断图表好坏**。生成或编辑 Excalidraw JSON 后，**必须**渲染成 PNG、查看图片、修复问题——循环直到满意。这是核心工作流，不是最终检查。

### 13.2 循环流程

```
生成 JSON
    ↓
1. 渲染 & 查看 → 运行渲染脚本，然后 Read PNG
    ↓
2. 审计视觉 → 与原设计对比
    ↓
3. 检查缺陷 → 文本溢出、元素重叠、箭头错位等
    ↓
4. 修复 → 编辑 JSON
    ↓
5. 重新渲染 → 循环
    ↓
通过？→ 结束
否？→ 回到步骤 1
```

### 13.3 审计问题清单

**视觉缺陷：**
- 文本被容器裁剪或溢出
- 文本或形状与其他元素重叠
- 箭头穿过元素而非绕过
- 箭头落到错误元素或指向空处
- 标签漂浮不明确（未清晰锚定到描述对象）
- 应均匀间距的元素间距不均
- 某区块空白过多，相邻区块过于拥挤
- 渲染尺寸下文字太小
- 整体构图不平衡或偏移

### 13.4 常见修复

| 问题 | 修复 |
|------|------|
| 文本被裁剪 | 加宽容器 |
| 间距/对齐问题 | 调整 x/y 坐标 |
| 箭头穿过元素 | 在 arrow `points` 数组添加中间路径点 |
| 标签位置不对 | 移动到更靠近描述对象 |
| 视觉权重不平衡 | 调整元素大小 |

### 13.5 何时停止

循环结束当：
- 渲染图匹配规划步骤中的概念设计
- 无文本裁剪、溢出或不可读
- 箭头干净地连接到正确元素
- 间距一致，构图平衡
- 可以无顾虑地展示给别人

## 十四、质量检查清单

### 14.1 深度与证据（技术图首先检查）

```
□ 1. 研究完成：是否查阅了实际规格、格式、事件名称？
□ 2. 证据工件：有代码片段、JSON 示例或真实数据吗？
□ 3. 多层架构：有摘要流程 + 区块边界 + 细节吗？
□ 4. 具体胜于抽象：显示真实内容，非只标盒子？
□ 5. 教育价值：能学到具体东西吗？
```

### 14.2 概念性

```
□ 6. 同构：每个视觉结构镜像其概念的行为？
□ 7. 论点：图表展示了文字单独做不到的东西？
□ 8. 多样性：每个主要概念用不同视觉模式？
□ 9. 无统一容器：避免了卡片网格和等盒子？
```

### 14.3 容器纪律

```
□ 10. 最小容器：任何带盒元素可改为 free-floating 文字？
□ 11. 线作为结构：树/时间线模式用线+文字而非盒子？
□ 12. 排版层级：字号和颜色创建视觉层级（减少盒子需求）？
```

### 14.4 结构

```
□ 13. 连接：每个关系有箭头或线
□ 14. 流程：清晰的视线路径
□ 15. 层级：重要元素更大/更独立
```

### 14.5 技术

```
□ 16. 文字干净：`text` 只含可读文字
□ 17. 字体：`fontFamily: 3`
□ 18. 粗糙度：`roughness: 0`（除非手绘风格）
```
