## 漏洞描述

`matchArgPattern` 函数位于 `src/infra/exec-command-resolution.ts`，通过 `new RegExp(argPattern)` 编译用户提供的 `argPattern` 字符串，**没有任何 ReDoS 验证**。

**对比分析**：
- `buildArgPatternFromArgv` 自动生成的 `argPattern` 是**安全的**（使用字面量转义）
- 但遗留的手动编写 `argPattern`（来自 exec-approvals allowlist）可能包含**嵌套重复**（如 `(a+)+$`），导致灾难性回溯

**漏洞触发路径**：
1. 用户在 exec-approvals allowlist 中手写了一个含灾难性回溯的 `argPattern`（如 `(a+)+$`）
2. `matchAllowlist` 调用 `matchArgPattern(entry.argPattern, argv, platform)`
3. `matchArgPattern` 直接执行 `new RegExp(argPattern).test(argsString)`
4. 当输入匹配嵌套重复模式时，触发指数级回溯
5. **Node.js 事件循环被阻塞**，Gateway 无响应 → `crash-loop`

**受影响代码位置**：
```typescript
// src/infra/exec-command-resolution.ts:324
const regex = new RegExp(argPattern)  // ← 无 ReDoS 保护
```

**已有安全原语可用**：
- `src/security/safe-regex.ts:319` 中已有 `hasNestedRepetition` 和 `compileSafeRegex`
- 相邻的 approval-request filter 代码已使用 `compileSafeRegex` 来拒绝不安全的正则

## 漏洞验证

### 检查是否受影响

```bash
# 1. 搜索可能受影响的配置文件
grep -r "argPattern" ~/.openclaw/ 2>/dev/null

# 2. 检查 exec approval 相关日志
openclaw gateway logs | grep -i "exec\|approval\|argPattern" | head -50

# 3. 检查是否使用了手写的复杂 argPattern
grep -rE "\(\w*\+*\)\+\$" ~/.openclaw/ 2>/dev/null
```

### 临时缓解措施

1. **不要在 exec-approvals allowlist 中手写复杂的正则表达式**，尤其是含有嵌套量词（`+`, `*`, `{n,m}` 嵌套）的模式
2. 使用 `buildArgPatternFromArgv` 自动生成安全的 `argPattern`
3. 检查现有 allowlist 配置，搜索可能危险的嵌套重复模式

### 推荐的安全 argPattern 写法

```typescript
// ✅ 安全：使用 buildArgPatternFromArgv 自动生成
const argPattern = buildArgPatternFromArgv(['--flag', 'value'])

// ❌ 危险：手写嵌套重复正则
const dangerousPattern = '(a+)+$'  // 可导致 ReDoS
```
