# 11.2 多密钥治理：keys、keyId 与认证档案

## 本节目标

掌握多密钥治理机制，理解密钥轮换策略与应急响应流程。

---

## 11.2.1 为什么不能只配一把 key

### 单密钥问题

```
单密钥架构在生产环境通常会同时引爆三类问题：

1. 可用性问题
   └── 配额耗尽或被封禁时全量失败

2. 可治理性问题
   └── 无法分清哪些流量消耗了配额，难以对账

3. 可轮换性问题
   └── 紧急换 key 容易引入脏窗口，导致大量认证失败
```

## 11.2.2 多密钥配置与隔离策略

### 隔离维度

```
生产级多密钥治理的关键是能回答：
"哪把 key 服务哪些流量，出事时影响范围多大？"

建议按以下维度做隔离：

1. 按环境隔离
   ├── 开发、预发布与生产使用不同 key
   └── 避免测试流量消耗生产配额

2. 按智能体隔离
   ├── 高风险写操作智能体与低风险只读智能体使用不同 key
   └── 限制泄露后的爆炸半径

3. 按供应商隔离
   ├── 同一供应商的不同项目或不同计费账号分别配置
   └── 便于成本归因与配额管理
```

## 11.2.3 认证档案文件

### 档案位置

> 🆕 v2026.6.5 变更：Auth profiles 已迁移至 SQLite
> 相关 PR: #89102, #88585

```
SQLite 数据库路径：~/.openclaw/data/auth-profiles.db

遗留 JSON 路径：~/.openclaw/agents/<agentId>/agent/auth-profiles.json
（v2026.6.5 之前版本的存量路径，首次使用时会自动导入至 SQLite）
```

### SQLite 存储优势

| 对比项 | JSON 存储 | SQLite 存储 |
|--------|-----------|-------------|
| **并发安全** | ❌ 文件锁竞争 | ✅ ACID 事务 |
| **数据一致性** | ❌ 读写冲突 | ✅ 行级锁 |
| **查询能力** | ❌ 全量加载 | ✅ SQL 灵活查询 |
| **体积** | ❌ 重复冗余 | ✅ 索引压缩 |
| **备份** | ❌ 手动复制 | ✅ SQLite WAL 模式 |

### 数据表结构

```sql
-- auth_profiles 表
CREATE TABLE auth_profiles (
  id TEXT PRIMARY KEY,
  agent_id TEXT NOT NULL,
  provider TEXT NOT NULL,
  key_id TEXT,
  created_at INTEGER,
  updated_at INTEGER,
  trusted_pin INTEGER DEFAULT 0  -- 信任的 npm 插件安装记录
);

-- trusted_pins 表（新增，v2026.6.5）
CREATE TABLE trusted_pins (
  pin TEXT PRIMARY KEY,
  source TEXT NOT NULL,  -- 'npm' | 'github'
  fingerprint TEXT,
  created_at INTEGER
);

CREATE INDEX idx_agent_id ON auth_profiles(agent_id);
CREATE INDEX idx_provider ON auth_profiles(provider);
```

### 迁移说明

v2026.6.5 自动执行以下迁移：

1. **JSON → SQLite 转换**: 读取所有 `auth-profiles.json` 文件并导入
2. **Trusted Pins 增强**: npm 插件安装记录现在有 SHA256 fingerprint
3. **WAL 模式启用**: 支持并发读写而不阻塞

```bash
# 手动触发迁移（如需）
openclaw auth migrate --from json

# 导出 SQLite 为 JSON（备份用）
openclaw auth export --format json --output auth-backup.json

# 查看 auth 数据库状态
openclaw auth status --db
```

### 验证迁移成功

```bash
# 检查 SQLite 数据库存在
ls -la ~/.openclaw/data/auth-profiles.db

# 验证数据完整性
openclaw health --auth

# 查看 auth profiles 数量
sqlite3 ~/.openclaw/data/auth-profiles.db "SELECT COUNT(*) FROM auth_profiles;"
```

### 工程价值

```
认证档案的工程价值是可追溯：
事故复盘时可以回答"哪个智能体在故障窗口使用了哪把 key"
```

## 11.2.4 密钥轮换工作流

### 四步流程

```
1. 新增 secondary
   └── 在密钥系统与环境变量中准备新 key

2. 验证
   └── 用 models status --check 验证新 key 可用

3. 切换
   └── 修改 keyId 或相关档案指向，让主流量逐步切换

4. 吊销
   └── 观察一段时间后吊销旧 key，并保留审计记录
```

![安全配置]/assets/diagrams/12_security_profiles.png)



### 详细步骤

**第一步：生成新密钥**

1. 在密钥管理系统中生成新密钥
2. 将新密钥临时保存在安全位置，不直接写入配置文件
3. 为新密钥标记生成时间与预期弃用时间

**第二步：更新配置**

1. 在 OpenClaw 的 models.providers 配置中新增密钥条目
2. 通过环境变量注入新 key
3. 保持 keyId 仍指向旧密钥（默认优先级）

**第三步：验证**

```bash
openclaw models status --check
openclaw health --json
```

**第四步：灰度切换**

1. 修改 keyId 指向新密钥
2. 观察 5-15 分钟日志
3. 确保无异常认证失败

**第五步：吊销旧密钥**

1. 在供应商控制台吊销旧密钥
2. 从配置中移除旧密钥引用
3. 记录轮换审计日志

## 11.2.5 密钥泄露应急响应

### 第一步：立即吊销

```
1. 进入模型供应商控制台，立即吊销泄露的密钥
2. 检查供应商的使用记录，识别异常流量
3. 如需要，向供应商提交滥用举报
```

### 第二步：轮换依赖

```
1. 识别所有使用该密钥的 provider 与 agent
2. 按优先级逐个执行密钥轮换
3. 采用并行轮换加快恢复速度
```

### 第三步：审计日志

```bash
openclaw logs --follow --json --from "2026-03-09T10:00:00Z" --to "2026-03-09T12:00:00Z"
openclaw audit --agent <agentId> --timerange "2h"
```

### 第四步：加强监控

```
1. 临时停用所有智能体的主要功能，只保留只读工具
2. 部署额外的监控告警
3. 启用更严格的 Guardrail 与 Sandbox 沙箱限制
```

## 11.2.6 常见误区

### 三个误区

```
1. 把 key 明文写进配置文件或日志

2. 在故障窗口直接替换原 key 值而不保留回滚点

3. 切换后不做探针验证，导致把认证问题误判为模型质量问题
```

---

## 本节小结

1. **多密钥**：按环境、智能体、供应商隔离
2. **认证档案**：可追溯 key 使用记录
3. **轮换**：新增 → 验证 → 切换 → 吊销
4. **应急**：立即吊销 → 轮换 → 审计 → 加强监控
