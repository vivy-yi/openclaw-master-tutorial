## 背景：为什么需要 Operator Install Policy？

### 旧机制的问题

OpenClaw 早期使用 `dangerous-code` scanner 来检测潜在危险的插件代码，但该机制存在以下局限性：

| 问题 | 说明 |
|------|------|
| **检测粒度粗** | 仅基于关键词匹配，误报率高 |
| **无法理解语义** | 无法判断代码是否真正危险 |
| **规则维护成本高** | 需要持续更新关键词库 |
| **绕过风险** | 攻击者可使用等价表达式绕过检测 |

### 新机制的设计目标

Operator Install Policy 采用**白名单 + 权限控制**的思路：

1. **显式授权**: 管理员必须明确授权才能安装插件
2. **权限隔离**: 插件只能访问其声明的权限范围
3. **可审计**: 所有安装操作均有日志记录
4. **可回滚**: 支持撤销已安装的插件

## 核心概念

### 3.1 Install Policy（安装策略）

Install Policy 定义了插件安装时的授权行为：

```typescript
// 安装策略类型
type InstallPolicy =
  | 'deny-all'           // 拒绝所有安装（默认，最安全）
  | 'allow-listed'       // 仅允许白名单插件
  | 'prompt'             // 安装前询问管理员
  | 'allow-all'         // 允许所有安装（最不安全，仅开发环境使用）
```

### 3.2 Operator（操作员）

Operator 是被授权安装插件的主体：

```typescript
interface Operator {
  id: string              // 操作员唯一标识
  type: 'user' | 'agent' | 'system'
  trustLevel: 'none' | 'low' | 'medium' | 'high' | 'full'
  allowedSources: string[]  // 允许安装的插件源
}
```

### 3.3 Plugin Signature（插件签名）

每个插件现在都需要经过签名验证：

```typescript
interface PluginSignature {
  manifestHash: string    // 插件清单哈希
  publisherKey: string    // 发布者公钥
  signature: string        // 数字签名
  timestamp: number       // 签名时间戳
}
```

## 配置方法

### 4.1 查看当前安装策略

```bash
# 查看当前安装策略配置
openclaw config get install-policy

# 查看详细配置
openclaw config get operator-policy
```

### 4.2 设置安装策略

```bash
# 方式一：命令行设置（立即生效）
openclaw config set install-policy allow-listed

# 方式二：配置文件设置
# 在 ~/.openclaw/config.yaml 中添加：
# install-policy:
#   mode: allow-listed
#   allowed-plugins:
#     - @openclaw/skill-*
#     - approved-partner-*
```

### 4.3 添，白名单插件

```bash
# 添加插件到白名单
openclaw plugin allow @openclaw/skill-github

# 批量添加（支持通配符）
openclaw plugin allow "@openclaw/*"

# 查看当前白名单
openclaw plugin list-allowed
```

### 4.4 管理操作员权限

```bash
# 列出所有操作员
openclaw operator list

# 授权新操作员
openclaw operator authorize user:alice --trust-level=medium

# 撤销操作员权限
openclaw operator deauthorize user:alice

# 查看操作员安装历史
openclaw operator history user:alice
```

## 工作流程

### 5.1 插件安装流程（新版）

```
┌─────────────────────────────────────────────────────────────┐
│                     Operator Install Policy                  │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  1. 插件安装请求 → 验证 Install Policy                       │
│                  ↓                                           │
│  2. 检查 Operator 授权状态                                   │
│                  ↓                                           │
│  3. 验证 Plugin Signature（签名 + 哈希）                     │
│                  ↓                                           │
│  4. 检查权限范围（scopes）                                   │
│                  ↓                                           │
│  5. 记录审计日志                                             │
│                  ↓                                           │
│  6. 允许 / 拒绝安装                                          │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 信任级别与权限映射

| 信任级别 | 可安装来源 | 需要确认 | 权限范围 |
|----------|-----------|----------|----------|
| `none` | 无 | — | 无法安装任何插件 |
| `low` | 白名单插件 | 是 | 最小权限 |
| `medium` | 白名单 + 部分第三方 | 是 | 受限权限 |
| `high` | 所有已签名插件 | 部分 | 标准权限 |
| `full` | 所有插件（包括未签名） | 否 | 完全权限 |

## 与旧机制的对比

| 方面 | dangerous-code scanner | Operator Install Policy |
|------|------------------------|------------------------|
| **检测方式** | 关键词匹配（黑名单） | 白名单 + 签名验证 |
| **配置粒度** | 全局开关 | 精细化（操作员级别） |
| **安全性** | ⚠️ 中等 | ✅ 高 |
| **用户体验** | 烦琐（误报多） | 流畅（精确控制） |
| **可审计性** | 有限 | 完整审计日志 |
| **维护成本** | 高（需持续更新规则） | 低（自动签名验证） |

## 迁移指南

### 从 dangerous-code scanner 迁移

**Step 1**: 导出当前配置

```bash
# 导出当前 dangerous-code 配置（如果有）
openclaw config export --filter=dangerous-code > dangerous-code-backup.yaml
```

**Step 2**: 评估当前插件信任需求

```bash
# 查看已安装插件及其来源
openclaw plugin list --verbose

# 识别需要白名单的插件
openclaw plugin audit
```

**Step 3**: 配置新的安装策略

```bash
# 推荐配置（从最严格开始）
openclaw config set install-policy deny-all

# 逐步添加信任的插件到白名单
openclaw plugin allow @openclaw/skill-github
openclaw plugin allow @openclaw/skill-obsidian

# 逐步提升信任级别
openclaw operator authorize user:alice --trust-level=low
```

**Step 4**: 测试验证

```bash
# 测试安装流程
openclaw plugin install @openclaw/skill-github --dry-run

# 查看审计日志
openclaw logs --filter=install-policy
```

### 回滚方案

如果新机制出现问题，可以回滚到旧模式（不推荐长期使用）：

```bash
# 临时禁用 Operator Install Policy
openclaw config set install-policy allow-all
```

> ⚠️ **警告**: `allow-all` 模式会禁用所有安装检查，仅用于临时故障排查。

## 常见问题

### Q1: 如何处理未签名的第三方插件？

**答**: 有三种处理方式：

1. **联系插件开发者签名**: 请求开发者使用 OpenClaw 官方签名工具签名插件
2. **手动验证后添加到本地白名单**: 仅限确认为安全的插件
3. **使用 full trust 操作员安装**: 在隔离环境中测试后使用

### Q2: 安装策略会影响现有插件吗？

**答**: 不会。Operator Install Policy 仅影响**新安装**操作。已安装的插件不受影响。

### Q3: 如何批量迁移插件到白名单？

```bash
# 导出已安装插件列表
openclaw plugin list --format=name > installed-plugins.txt

# 批量添加到白名单
cat installed-plugins.txt | xargs -I {} openclaw plugin allow {}
```
