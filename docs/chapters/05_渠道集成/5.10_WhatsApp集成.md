# 8.6 WhatsApp 集成

本节讲解如何将 WhatsApp 与 OpenClaw 集成，使用 WhatsApp Web（Baileys）协议。

---

## 8.6.1 概述

WhatsApp 是全球最流行的即时通讯工具之一。OpenClaw 通过 WhatsApp Web（Baileys）实现连接。

```
★ Insight ─────────────────────────────────────
WhatsApp 集成特点：
1. Gateway 管理配对的会话
2. 需要 QR 码配对
3. 支持 DM 和群组
4. 建议使用独立号码
─────────────────────────────────────────────────
```

## 8.6.2 快速开始

### 步骤 1：配置访问策略

```json5
{
  channels: {
    whatsapp: {
      dmPolicy: "pairing",
      allowFrom: ["+15551234567"],
      groupPolicy: "allowlist",
      groupAllowFrom: ["+15551234567"],
    },
  },
}
```

### 步骤 2：QR 配对

```bash
# 登录 WhatsApp
openclaw channels login --channel whatsapp

# 指定账户
openclaw channels login --channel whatsapp --account work
```

### 步骤 3：启动 Gateway

```bash
openclaw gateway
```

### 步骤 4：批准配对请求

```bash
# 查看配对请求
openclaw pairing list whatsapp

# 批准配对
openclaw pairing approve whatsapp <CODE>
```

## 8.6.2 访问控制

### DM 策略

```json5
{
  channels: {
    whatsapp: {
      dmPolicy: "pairing",  // pairing | allowlist | open | disabled
      allowFrom: ["+15551234567", "+15559876543"],
    },
  },
}
```

| 策略 | 说明 |
|------|------|
| `pairing` | 需要配对批准 |
| `allowlist` | 仅允许白名单号码 |
| `open` | 开放（需设置 allowFrom: ["*"]） |
| `disabled` | 禁用 DM |

### 群组策略

```json5
{
  channels: {
    whatsapp: {
      groupPolicy: "allowlist",  // open | allowlist | disabled
      groupAllowFrom: ["+15551234567"],
    },
  },
}
```

## 8.6.5 故障排除

### 常见问题

**Q: QR 码无法扫描**

```
解决：
1. 检查网络连接
2. 确保 WhatsApp Web 已退出其他设备
3. 重新生成 QR 码
```

**Q: 消息发送失败**

```
解决：
1. 检查配对是否过期
2. 重新登录：openclaw channels login --channel whatsapp
3. 查看日志：openclaw logs
```

**Q: 配对请求超时**

```
说明：配对请求 1 小时后过期
解决：重新发起登录请求
```

## 8.6.6 Active Connection Liveness（主动连接保活）

> **来源**: GitHub Issue [#93406](https://github.com/openclaw/openclaw/issues/93406)
> **类型**: Feature（提案阶段）
> **优先级**: P2 📝
> **状态**: Open

### 问题背景

WhatsApp 连接健康判断在**被动信号**模式下存在盲区：

| 信号 | 更新时机 | 盲区 |
|------|---------|------|
| `lastTransportActivityAt` | 任何 ws frame（含 keepalive） | keepalive 不能证明消息可达 |
| `lastInboundAt` | 真实消息到达时 | "无消息"无法区分"安静账户"和"死Inbound路径" |

**Zombie 连接现象**：
- Connection 报告 `healthy`
- Transport 报告 `healthy`
- 但 3+ 小时无真实消息到达
- Periodic reconnect 反而重置了 watchdog 时钟

### 根因分析

现有健康检测依赖**被动信号**：

```
Agent 发送消息 → 收到 ACK → lastTransportActivityAt 更新 ✅
  但 ACK 只能证明服务器收到，不能证明对端收到

长期无消息 → keepalive 维持 ws 连接 → lastTransportActivityAt 继续更新
  但消息实际无法到达对方
```

### 提案方案：sendPresenceUpdate 主动探测

```jsonc
// 配置项（提案中）
channels:
  whatsapp:
    liveness:
      enabled: true
      intervalMs: 30000        // 无消息 30 秒后开始探测
      probes: 3               // 最多发 3 次
      timeoutMs: 5000          // 每次探测超时
```

**工作原理**：

```
无消息 idle 窗口
    ↓
发送 sendPresenceUpdate() 主动探测
    ↓
┌──────────────────┐     ┌─────────────────┐
│ N 次无响应        │     │ 收到 presence   │
│ → forceClose    │     │ → 保持连接      │
│ → 重新连接       │     │                 │
└──────────────────┘     └─────────────────┘
```

**安全约束**：
- 主动探测本身也在消耗 WhatsApp 服务器资源
- 太频繁会触发 WhatsApp rate limit
- 建议 `intervalMs >= 30000`，`probes <= 3`

### 当前 Workaround

在该 Feature 实现前，可通过以下方式监控 WhatsApp 连接健康：

```bash
# 检查 WhatsApp 连接状态
gateway status --channel whatsapp

# 查看最后收到的消息时间
openclaw sessions list --channel whatsapp --long | grep lastInbound

# 手动重连（如发现 zombie 连接）
openclaw channels logout --channel whatsapp
openclaw channels login --channel whatsapp
```

**Zombie 连接识别特征**：
- `lastInboundAt` 超过 2 小时无更新
- `lastTransportActivityAt` 仍在更新（keepalive 活跃）
- 发送消息返回成功，但对方永远收不到
