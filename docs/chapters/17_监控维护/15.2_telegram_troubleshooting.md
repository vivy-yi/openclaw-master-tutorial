## 一、Telegram ENETDOWN 瞬态网络故障（已修复）

### 1.1 问题描述

| 项目 | 说明 |
|------|------|
| Issue | [#86762](https://github.com/openclaw/openclaw/pull/86762) |
| 状态 | ✅ **已修复** - PR 已通过 ClawSweeper 验证，等待 maintainer 合入 |
| 严重程度 | 🟠 P2 → ✅ 已修复 |

### 1.2 故障现象

```
Telegram provider intermittently stalls despite stable direct Bot API connectivity
```

Telegram 渠道在直接 Bot API 连通性正常的情况下出现间歇性卡顿。

### 1.3 修复方案

PR #86762 将 ENETDOWN 网络错误视为**瞬态网络故障**处理：

```typescript
// 修复前：ENETDOWN 被视为永久错误
// 修复后：将 ENETDOWN 标记为可重试的瞬态错误
const TRANSIENT_NETWORK_ERRORS = [
  'ENETDOWN',
  'ETIMEDOUT',
  'ECONNRESET',
  // ... 其他瞬态错误
];

function isTransientError(error: Error): boolean {
  return TRANSIENT_NETWORK_ERRORS.includes(error.code);
}
```

### 1.4 临时缓解措施

在 PR 合入前，可使用以下临时方案：

```bash
# 重启网关清除 ENETDOWN 状态
openclaw gateway restart

# 或使用 --force 标志强制重启
openclaw gateway restart --force
```

## 三、Docker/WSL2 事件循环饥饿（影响 Telegram）

### 3.1 问题描述

| 项目 | 说明 |
|------|------|
| Issue | [#86752](https://github.com/openclaw/openclaw/issues/86752) |
| 严重程度 | 🔴 P1 |
| 标签 | `impact:message-loss` `impact:crash-loop` |

### 3.2 故障现象

```
2026.5.22 Docker/WSL2 gateway event-loop starvation:
- 284s provider-auth prewarm
- slow Telegram turn
- local RPC timeouts
```

### 3.3 影响范围

- Docker 部署的 OpenClaw 网关
- WSL2 环境下的 OpenClaw
- Telegram 消息延迟（最高可达 284 秒）
- 本地 RPC 调用超时

### 3.4 临时解决方案

```bash
# 方案1：升级到最新 Beta 版本
openclaw upgrade --channel beta

# 方案2：切换到主机网络模式
# 编辑 docker-compose.yml
services:
  openclaw:
    network_mode: host

# 方案3：增加事件循环超时配置
# 在 openclaw.yaml 中添加
gateway:
  eventLoopTimeout: 30000  # 30秒

# 方案4：禁用提供商认证预热
# 在 openclaw.yaml 中添加
provider:
  authPrewarm: false
```

### 3.5 已知受影响配置

| 环境 | 版本 | Telegram 影响 |
|------|------|---------------|
| Docker (Linux) | v2026.5.22 | 🔴 事件循环饥饿 |
| Docker (WSL2) | v2026.5.22 | 🔴 事件循环饥饿 + 284s 认证 |
| 直接部署 | v2026.5.22 | ⚠️ 可能受影响 |

## 四、Telegram 消息丢失防护

### 4.1 启用消息确认

```yaml
# openclaw.yaml
telegram:
  deliveryConfirmation: true
  retryOnFailure: true
  maxRetries: 3
```

### 4.2 监控配置

```bash
# 启用 Telegram 专门监控
openclaw monitor --channel telegram --interval 30s

# 检查消息队列状态
openclaw queue status --channel telegram
```

### 4.3 日志分析

```bash
# 提取 Telegram 相关错误
openclaw logs | grep -E "telegram|Telegram|message.*loss" | tail -100

# 检查消息ID序列
openclaw logs | grep -E "message_id|update_id" | tail -50
```

## 九、新增文档

### Telegram Polling Silent Crash Loop（#93375）

> ⚠️ **重要**：如果你的 Telegram polling 在网络瞬时故障后每 10-15 分钟重启一次，且无错误日志，请参考专项文档：

📄 **[Telegram-polling-crash-loop.md](./故障排除/Telegram/Telegram-polling-crash-loop.md)** — 完整日志分析 + workaround + 自动化检测脚本

**关键特征**：
- Exit code 1 但无错误日志（silent）
- Health monitor 每 10-15 分钟重启一次
- 网络本身正常（curl api.telegram.org 返回 200）
- **唯一恢复方式**：`systemctl restart openclaw-gateway.service`
