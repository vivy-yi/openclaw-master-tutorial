### 15.4.1 #84203 — Windows authStatus 冷启动延迟 (10-24s)

**问题识别时间**: 2026-05-19

**基本信息**：

| 项目 | 值 |
|------|---|
| Issue # | [#84203](https://github.com/openclaw/openclaw/issues/84203) |
| 标签 | P1, regression, clawsweeper:no-new-fix-pr, clawsweeper:needs-maintainer-review |
| 影响版本 | v2026.5.18 + Codex 0.131.0 |
| 影响系统 | Windows 11 + npm-global + Codex 用户 |

**问题本质**：

v2026.5.18 引入 **auth cache TTL 硬编码为 60s**。每次新的 CLI 调用（`openclaw agent`）打开新的 WS 连接，触发完整的 cold auth check，耗时 10-24s。

```
Cold path: WS new connection → models.authStatus (10-24s) → 60s cache expires → next call cold again
Warm path (cached=true): ~267-300ms
```

**影响范围**：

| 配置 | 影响 | 严重度 |
|------|------|--------|
| Windows 11 + npm-global + Codex | 每次 CLI 调用 27-45s | 🔴 严重 |
| Linux/macOS + npm-global + Codex | 每次 CLI 调用延迟更高 | 🔴 严重 |
| Windows 11 + non-Codex | 待确认 | 🟠 中 |
| Docker 安装 | ❌ 不受影响 | 🟢 无 |

**Workaround**：

```bash
# ✅ 方案一：保持 Gateway 持续运行（避免 CLI cold start）
# Gateway 本身有 cache，CLI 冷启动问题只影响 "openclaw agent" 短调用
# 将短任务改为通过持续 Gateway session 处理

# ✅ 方案二：Linux/macOS 开发环境
# 如有 Linux 或 macOS 机器，迁移 Codex 开发到此环境

# ❌ 方案三：降级 v2026.5.17 — 不推荐，v2026.5.18 有 200+ 修复

# ⚠️ 方案四：等待官方 fix（clawsweeper:needs-maintainer-review）
```

**技术根因**：

```javascript
// 根因文件（推测）：
// server-methods-BAy3NbTS.js 中
CACHE_TTL_MS = 6e4  // 60s 硬编码，无法通过配置覆盖

// 问题：CLI 每次新 WS 连接 → auth cache miss → 10-24s cold auth
// Gateway 内部调用：cache hit → ~300ms
// CLI 外部调用：cache miss（60s TTL 内新 WS 连接仍 miss）
```

**修复状态**: 🟡 clawsweeper:needs-maintainer-review（等待官方修复）

### 15.4.2 #84193 — Auto-compaction 持有 JSONL 写锁导致 Discord 消息失败

**问题识别时间**: 2026-05-19

**基本信息**：

| 项目 | 值 |
|------|---|
| Issue # | [#84193](https://github.com/openclaw/openclaw/issues/84193) |
| 标签 | P1, clawsweeper:needs-live-repro, impact:session-state, impact:message-loss |
| 影响版本 | v2026.5.18 |
| 影响系统 | Discord + Anthropic Opus + 大文件会话用户 |

**问题本质**：

v2026.5.18 中，Agent 使用 Opus 模型在 Discord 完成大文件生成任务后，post-run auto-compaction 触发，但 compaction 路径超时后**没有释放 session JSONL 写锁**，导致：

```
后续所有 Discord 请求：
→ 等待 session file lock 60000ms
→ SessionWriteLockTimeoutError
→ Session 无法恢复，只能 Gateway 重启
```

**触发路径**：

```
Opus 大任务完成
  → post-run auto-compaction 启动（同一 session 的 .jsonl）
    → compaction 路径超时
      → 持有写锁未释放
        → 后续所有 channel turn 等待锁 → 60000ms timeout → 失败
```

**受影响系统**：

| 系统 | 影响 | 原因 |
|------|------|------|
| Discord + Opus | 🔴 严重 | Auto-compaction 持有锁 |
| Discord + 其他模型 | 🟡 可能受影响 | 待确认 |
| Telegram | 🟢 正常 | 不使用 session JSONL |
| 飞书 | 🟢 正常 | 不同存储路径 |

**Workaround**：

```bash
# ✅ 方案一：监控 + 自动 Gateway 重启
# 适合 Discord heavy 用户
while true; do
  if openclaw gateway status | grep -q "running"; then
    # 检查 Discord 消息是否正常
    openclaw gateway logs 2>/dev/null | grep -q "SessionWriteLockTimeoutError" && \
      openclaw gateway restart && echo "$(date): Restarted due to lock timeout"
  fi
  sleep 60
done

# ✅ 方案二：避免大文件会话在 Discord 直接生成
# Opus 大任务 → 使用 isolated session + 结果发布到 Discord

# ✅ 方案三：使用 Telegram 替代 Discord（不受此 bug 影响）

# ❌ 方案四：降级 — 放弃 200+ fixes，不推荐
```

**修复状态**: 🟡 clawsweeper:needs-live-repro（等待用户提供复现步骤）

### 15.4.3 #83778 — Cron + Tools 投递失败

**问题识别时间**: 2026-05-13（持续跟踪）

**基本信息**：

| 项目 | 值 |
|------|---|
| Issue # | [#83778](https://github.com/openclaw/openclaw/issues/83778) |
| 标签 | P1, clawsweeper:needs-info, impact:message-loss |
| 影响版本 | v2026.5.18 |

**问题本质**：

启用 tools 的 Cron job 执行后，Agent 尝试通过 message tool 发送结果，但 message tool 失败时（如 WhatsApp group 无权限），announce delivery 降级为 DM，导致用户以为群组收到消息，实际发到了 DM。

```
Cron job with tools enabled
  → Agent 收到 tools in context
    → 模型尝试通过 message tool 发送结果
      → message tool 失败（如 WhatsApp group 无权限）
        → announce delivery 降级为 DM
          → 静默成功（cron status: ok, delivered: true）
            → 用户以为群组收到，实际发到了 DM
```

**Workaround**：

```bash
# ✅ 方案一：禁用 tools
# 在 cron 配置中使用 --tools "" 禁用工具
openclaw cron add "my-task" --tools ""

# ✅ 方案二：移除 tools 配置
# 检查并移除 .openclaw.yaml 中的 tools 配置

# ✅ 方案三：在主 session 处理
# 将需要 tools 的任务改为主 session 处理，而非 cron
```

**修复状态**: 🟡 clawsweeper:needs-info（等待用户提供更多信息）

### 15.4.4 #84201 — gateway install PATH 缺少 openclaw 二进制目录

**问题识别时间**: 2026-05-19

**基本信息**：

| 项目 | 值 |
|------|---|
| Issue # | [#84201](https://github.com/openclaw/openclaw/issues/84201) |
| 标签 | P2, clawsweeper:queueable-fix, clawsweeper:source-repro |
| 影响系统 | npm-global 安装用户 |

**问题本质**：

`openclaw gateway install` 生成的服务环境文件中，PATH 硬编码为 homebrew + 系统路径，**不包含 npm-global 安装目录**（如 `~/.npm-global/bin/`）。

**Workaround**：

```bash
# ✅ 手动修复 service-env 文件
# 编辑 ~/.openclaw/service-env/ai.openclaw.gateway.env
# 在 PATH 行添加 npm-global bin 目录：

export PATH='/home/user/.npm-global/bin:/opt/homebrew/opt/node/bin:/opt/homebrew/bin:...'

# ✅ 或者在配置中添加 openclaw binary path
openclaw config set gateway.service.env.PATH "/home/$(whoami)/.npm-global/bin:$PATH"

# ✅ 重启 Gateway 使配置生效
openclaw gateway restart
```

**修复状态**: 🟡 clawsweeper:queueable-fix（计划修复）

### 15.4.5 #84204 — Fallback 链被 EmbeddedAttemptSessionTakeoverError 耗尽

**问题识别时间**: 2026-05-19

**基本信息**：

| 项目 | 值 |
|------|---|
| Issue # | [#84204](https://github.com/openclaw/openclaw/issues/84204) |
| 标签 | P2, impact:session-state, impact:auth-provider |

**问题本质**：

Subagent turn 中，session `.jsonl` 并发写入（来自 hook 或父 agent）触发 `EmbeddedAttemptSessionTakeoverError`，但 fallback 链遍历所有配置的 provider **仍然遇到同样的 session file race**，导致每次请求都失败。

```
触发：subagent LLM turn 中 → 其他 writer 修改 session .jsonl
  → embedded prompt lock fingerprint check 失败
    → 三个 provider (Opus → GPT-5.5 → Gemini) 全部遇到相同错误
      → 只有 Gemini 3 Flash 偶然赢得 race
```

**Workaround**：

```bash
# ✅ 方案一：禁用 subagent session 并发写入插件
# 如果使用 memory-episodic 插件，暂时禁用 before_prompt_build / agent_end hooks

# ✅ 方案二：使用单一 provider（避免 fallback）
openclaw config set models.default "google/gemini-3-flash-preview"

# ✅ 方案三：subagent 使用 isolated session（减少并发）
```

**修复状态**: 🟡 官方已知，待修复

### 15.4.6 #83782 — Skill Registry extraDirs 不传播

**问题识别时间**: 2026-05-13（持续跟踪）

**基本信息**：

| 项目 | 值 |
|------|---|
| Issue # | [#83782](https://github.com/openclaw/openclaw/issues/83782) |
| 标签 | P2, clawsweeper:queueable-fix |

**根因**：

```
修改 skills.load.extraDirs
  → 写入 config.yaml
    → 新 session 使用新 snapshot ✅
      → 现有 session 不触发 snapshot 重建 ❌
        → 现有 session 仍用旧的 skills snapshot
```

**Workaround**：

| Workaround | 有效性 |
|------------|--------|
| `openclaw agents restart` | ✅ 有效（重建所有 session） |
| 开启新 session 验证 | ✅ 有效（新建 session 用新 snapshot） |
| 写入 workspace openclaw.yaml | ✅ 持久化有效 |

**修复状态**: 🟡 clawsweeper:queueable-fix（计划修复）
