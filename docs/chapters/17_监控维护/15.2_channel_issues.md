## 15.2.1 排查检查项

### 1. 渠道令牌与权限检查

渠道令牌与权限是否有效：

```bash
# 查看渠道状态
openclaw channels status

# 测试渠道连接
openclaw channels test telegram
openclaw channels test feishu
```

**典型报错**：

```
[Telegram] Polling error: 401 Unauthorized
[Feishu] API error: invalid app_id
```

**原因**：
- Token 过期
- 权限不足
- App ID/Secret 错误

### 2. 配对关系检查

配对关系是否正确，是否存在未审批的配对请求：

```bash
# 查看配对列表
openclaw pairing list

# 审批配对请求
openclaw pairing approve <device_id>
```

**典型报错**：

```
[Gateway] Connection rejected: device 9f8a not paired
```

**原因**：
- 设备未配对
- 配对请求未批准

### 3. 路由配置检查

路由是否命中目标 Agent，会话归属是否稳定：

```bash
# 查看路由规则
openclaw routes list

# 查看当前会话
openclaw sessions list
```

### 4. 渠道日志检查

渠道状态与日志是否一致：

```bash
# 查看渠道详细日志
openclaw logs --channel telegram --limit 100

# 查看实时日志
openclaw logs --follow
```

## 15.2.2 常见渠道问题

### Telegram 问题

#### 问题 1：Bot 无法加入群组

```
错误：Bot is not a member of the group chat
```

**排查**：
1. 检查 Bot 是否已添加到群组
2. 检查 Bot 是否有管理员权限
3. 检查群组设置是否允许 Bot

#### 问题 2：消息发送失败

```
错误：Bad Request: chat not found
```

**排查**：
1. 确认 chat_id 正确
2. 检查 Bot 是否有发送消息权限
3. 确认 Bot 与用户有对话

#### 问题 3：Webhook 设置失败

```
错误：Webhook validation failed
```

**排查**：
1. 确认 Webhook URL 可访问
2. 检查 SSL 证书有效
3. 确认 Token 正确

### 飞书问题

#### 问题 1：消息收不到

```
错误：Webhook call failed: verification token mismatch
```

**排查**：
1. 检查 Verification Token 正确
2. 确认 Webhook URL 可公网访问
3. 检查事件订阅是否启用

#### 问题 2：权限不足

```
错误：permission denied
```

**排查**：
1. 检查应用权限配置
2. 确认已发布应用
3. 重新安装应用到群

## 15.2.3 Discord 问题

### 问题 1：WebSocket stale-socket 导致 Gateway 崩溃（#56274）

**现象**：
- Discord Bot 完全崩溃，Gateway 无法响应
- WebSocket 连接出现 stale-socket 状态
- 大量 uncaught exception 循环
- 无自动重连

**排查**：

```bash
# 检查 Gateway 进程是否存活
ps aux | grep openclaw

# 查看 Discord 相关崩溃日志
openclaw gateway logs | grep -i discord | grep -i "error\|exception\|crash"
```

**根因**：Discord WebSocket 处理 stale socket 时未做防御性检查。

**Workaround**：

1. **监控 + 自动重启**：

```bash
# 写一个监控脚本，检测到崩溃自动重启
#!/bin/bash
while true; do
  if ! openclaw gateway status | grep -q "running"; then
    echo "$(date): Gateway down, restarting..." >> /var/log/openclaw-watchdog.log
    openclaw gateway restart
  fi
  sleep 30
done
```

2. **关注 PR #56287**：[Fix: Discord can not find function withFileLockadd](https://github.com/openclaw/openclaw/pull/56287)，合入后可解决

**状态**：活跃 Bug，建议优先监控。

## 15.2.4 Telegram 问题

### 问题 1：Telegram Bot 启动时报 404 Not Found（#55387）

**现象**：
- Gateway 日志中出现 `404 Not Found` 错误
- Bot 无法连接 Telegram API，消息收发完全失效
- 日志示例：

```
[Telegram] getUpdates error: 404 Not Found
[Polling] Telegram polling failed: HTTP 404
error: Telegram API error: not Found
```

**系统性排查步骤**：

> 按以下顺序逐一排查，每步完成后重新测试。

**Step 1：验证 Bot Token 有效性**

```bash
# 直接用 Token 调用 Telegram API，查看响应
curl -s "https://api.telegram.org/bot<YOUR_TOKEN>/getMe"
```

- 返回 `{"ok":true,...}` → Token 有效，跳至 Step 2
- 返回 `{"ok":false,"error_code":404,"description":"Not Found"}` → Token 无效或 Bot 被删除

**常见原因**：
- Token 填错（多见于复制粘贴时多了空格或换行）
- Bot 被 BotFather 删除或封禁
- Token 已通过 BotFather 的 `/revoke` 命令吊销
- 使用了其他 Bot 的 Token

**修复方法**：
1. 打开 Telegram 搜索 **@BotFather**
2. 发送 `/mybots` 查看你的 Bot
3. 如 Bot 存在但 Token 失效，发送 `/token` 重新获取
4. 如 Bot 不存在，需通过 `/newbot` 重新创建

---

**Step 2：检查 Webhook 配置冲突**

如果配置了 `webhookUrl`，Telegram 会禁用 Long Polling，导致 404：

```bash
# 确认 Webhook 状态
curl -s "https://api.telegram.org/bot<YOUR_TOKEN>/getWebhookInfo"
```

- 如果 `url` 字段有值（设置了 Webhook），Long Polling 会被 Telegram 自动禁用
- 如果 `url` 为空但 `has_custom_certificate` 有值，也可能残留冲突

**修复方法**（二选一）：

```bash
# 方案 A：删除 Webhook，切回 Long Polling
curl -X POST "https://api.telegram.org/bot<YOUR_TOKEN>/deleteWebhook"
```

```bash
# 方案 B：确认 Webhook URL 正确配置（公网可达 + HTTPS）
# 在 openclaw 配置中确保 webhookUrl 填写正确
```

---

**Step 3：检查网络与代理设置**

在中国大陆等地区，直接访问 Telegram API 会超时或 404：

```bash
# 测试 Telegram API 是否可达
curl -v --max-time 10 "https://api.telegram.org/bot<YOUR_TOKEN>/getMe"
```

**常见原因**：
- 服务器/开发机无法直连 Telegram API（网络封锁）
- 使用了代理但代理本身无法访问 Telegram
- 代理配置错误（写了 HTTP 代理但 Bot 用的是 SOCKS）

**修复方法**：

```bash
# 确认系统代理设置（HTTP_PROXY / HTTPS_PROXY / ALL_PROXY）
echo $HTTP_PROXY
echo $HTTPS_PROXY
echo $ALL_PROXY

# 如使用代理软件（如 Clash），确保 Telegram 走代理
# OpenClaw 会读取环境变量中的代理设置
export HTTPS_PROXY="socks5://127.0.0.1:7890"
```

> ⚠️ 注意：OpenClaw 优先读取 `channels.telegram.proxyUrl` 配置，如果配置了但代理无效，同样会 404。检查配置：

```json
{
  "channels": {
    "telegram": {
      "enabled": true,
      "botToken": "your-token",
      "proxyUrl": "socks5://127.0.0.1:7890"  // 确认此项有效
    }
  }
}
```

---

**Step 4：检查 Bot 是否被 BotFather 封禁或限制**

Bot 被限制后 API 仍返回 200 但无响应，或返回 404：

1. 在 Telegram 搜索 **@BotFather**
2. 发送 `/mybots`
3. 点击你的 Bot → 查看是否有 "This bot is blocked by user" 或类似提示

---

**Step 5：重新初始化 Bot**

如果以上均正常但问题持续，尝试重建：

```bash
# 1. 先删除 Webhook
curl -X POST "https://api.telegram.org/bot<YOUR_TOKEN>/deleteWebhook"

# 2. 停止 OpenClaw Gateway
openclaw gateway stop

# 3. 清除 Telegram 相关缓存
rm -rf ~/.openclaw/telegram/

# 4. 重启 Gateway
openclaw gateway start

# 5. 观察日志确认 Bot 正常连接
openclaw gateway logs | grep -i telegram
```

---

**快速诊断命令汇总**：

```bash
# 1. 验证 Token
curl -s "https://api.telegram.org/bot<YOUR_TOKEN>/getMe"

# 2. 查看 Webhook 状态
curl -s "https://api.telegram.org/bot<YOUR_TOKEN>/getWebhookInfo"

# 3. 删除 Webhook（切回 Long Polling）
curl -X POST "https://api.telegram.org/bot<YOUR_TOKEN>/deleteWebhook"

# 4. 测试 API 连通性
curl -v --max-time 10 "https://api.telegram.org/bot<YOUR_TOKEN>/getMe"

# 5. 查看 OpenClaw Telegram 日志
openclaw gateway logs | grep -i telegram
```

**关联 Issue**：[GitHub #55387](https://github.com/openclaw/openclaw/issues/55387) - Telegram 404 Not Found on startup

### 🔴 Bug #82722 — Heartbeat 打断 Streaming

**问题**：Heartbeat 消息打断 streaming 响应，导致输出不完整或截断。

**影响**：长文本输出被截断，Agent 回复不完整。

**根因**：Heartbeat 与 streaming 响应共享同一通道，优先级处理不当。

**Workaround**：
1. **关闭 Heartbeat**：在配置中暂时禁用
```json
{
  "gateway": {
    "heartbeat": {
      "enabled": false
    }
  }
}
```
2. **使用 Webhook 模式**：切换到 webhook delivery 模式

**状态**：🟠 OPEN | [Issue #82722](https://github.com/openclaw/openclaw/issues/82722)

```bash
# ✅ 方案一：使用 Telegram DM 而非群组
# DM 模式下 message tool 非必须，群组模式下必须显式调用

# ✅ 方案二：确保 Agent 总是调用 message tool
# 在 prompt 中强调 "always use message tool to respond"

# ✅ 方案三：使用 v2026.5.17-1（等待修复合入）

# ❌ 方案四：降级至 v2026.5.16 — 不推荐（放弃 200+ fixes）
```

### 🔴 Bug #84316 — Telegram 群组 TTS 语音从未送达

**基本信息**：

| 项目 | 值 |
|------|---|

| 影响版本 | v2026.5.12+ |
| 影响场景 | Telegram 群组 + TTS 语音消息 |

**问题本质**：

Telegram 群组中，Agent 生成的 TTS 语音消息**从未实际送达**，日志显示 "sent" 但用户从未收到。

```
Agent 生成语音回复
  → TTS 转换完成
    → Telegram API 调用 "send voice" ✅
      → 日志显示 sent ✅
        → 用户从未收到 ❌
```

**Workaround**：

```bash
# ✅ 方案一：使用文本回复而非 TTS

{
  "tts": {
    "enabled": false
  }
```
