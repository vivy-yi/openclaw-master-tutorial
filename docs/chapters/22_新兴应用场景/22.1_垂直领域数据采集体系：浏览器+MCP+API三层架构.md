## 22.1.0 概述：为什么浏览器是垂直领域开发的核心

垂直领域 Agent 的本质是**让 AI 可靠地获取并操作特定网站/平台的数据**。而浏览器是这座桥梁的核心——无论是读取网页内容、还是操作网站界面，浏览器都是 AI 与互联网服务交互的最通用接口。

```
                    垂直领域 Agent 的数据获取问题

  ┌─────────────────────────────────────────────────────┐
  │                   垂直领域 Agent                      │
  │              （金融、医疗、电商、社交...）               │
  └──────────────────────┬────────────────────────────┘
                         │ 需要什么
                         ▼
            ┌────────────────────────────┐
            │     可靠的数据获取能力        │
            ├────────────────────────────┤
            │  • 读取网页/内容数据         │
            │  • 操作网站界面（登录/查询）  │
            │  • 调用平台自有 API/MCP     │
            │  • 抓取结构化数据           │
            │  • 自动化工作流             │
            └──────────┬─────────────────┘
                       │
                       ▼
        ┌──────────────────────────────┐
        │    三层数据获取架构            │
        ├──────────────────────────────┤
        │ 第1层：OpenClaw 内置浏览器     │  ← 通用界面操作
        │ 第2层：外部 MCP/CLI/反检测浏览器│  ← 平台专项
        │ 第3层：平台自有 API/服务        │  ← 高质量数据
        └──────────────────────────────┘
```

### 三层架构的选型决策树

```
需要获取数据？
    │
    ├─► 公开数据/静态页面 ──► 第3层：Jina Reader / Firecrawl
    │
    ├─► 平台提供官方 API ──► 第3层：官方 API（最准确、最稳定）
    │
    ├─► 需要登录/Cookie ──► 第2层：Camoufox（反检测浏览器）
    │                        或平台自有 MCP/CLI
    │
    ├─► 需要 AI 视觉理解页面 ──► 第1层：OpenClaw 浏览器 + AI 截图分析
    │
    └─► 需要复杂交互流程 ──► 第1层 + 第2层组合
        （例：先浏览器登录 → 再 API 获取数据）
```

| Kind | 功能 | 关键参数 |
|------|------|---------|
| `click` | 点击元素 | `ref`, `doubleClick`, `button`(left/right/middle), `modifiers` |
| `type` | 输入文本（逐字输入，模拟人类打字） | `ref`, `text`, `submit`, `slowly`, `selector` |
| `fill` | 批量填写多字段表单 | `fields[]`（每个含 `ref` + `value`） |

#### Snapshot 的三种格式

| Format | 生成方式 | 用途 |
|--------|---------|------|
| `aria` | `snapshotAriaViaPlaywright()` | 获取原生 ARIA 树（含所有 ref） |
| `ai`（默认） | `snapshotAiViaPlaywright()` | AI 多模态理解（视觉 + 结构） |
| `role` | `buildRoleSnapshotFromAiSnapshot()` | 交互元素快照（带可操作 ref） |

#### Snapshot 的参数详解

```javascript
browser(action: "snapshot", targetId: "ABC123", {
  format: "aria",       // "aria" | "ai" | "role"
  mode: "efficient",    // 高效模式：自动设置 maxChars=4000, compact=true, depth=6
  maxChars: 80000,      // AI 格式最大字符数（默认 80000）
  limit: 200,           // DOM 节点数量上限
  refs: "aria",         // "aria" | "role"（ref 生成格式）
  interactive: true,    // 仅返回可交互元素
  compact: true,        // 紧凑输出，减少无用结构
  depth: 6,             // DOM 树深度限制
  selector: ".main",    // 仅返回匹配选择器的子树
  frame: "iframe",      // 仅返回指定 frame
  labels: true          // 显示元素标签
})
```

### 1.2 高效模式（Efficient Mode）≈ Computer-Use

OpenClaw 没有单独内置"computer-use"模式，但通过 `mode="efficient"` 参数实现等效能力：

```javascript
// 高效模式 = AI 自主循环的标准配置
browser(action: "snapshot", targetId: "ABC123", mode: "efficient")

// 等效于手动设置以下全部参数：
// format="ai", maxChars=4000, compact=true, depth=6, interactive=true
```

**高效模式的工作流**：

```
AI 视觉感知（snapshot） → 决策下一步 → act 操作 → wait 等待响应 → 循环
```

| 维度 | 标准模式 | 高效模式（efficient） |
|------|---------|---------------------|
| maxChars | 80000 | 4000（减少 token 消耗）|
| compact | false | true |
| depth | 无限制 | 6 |
| interactive | 全部元素 | 仅可交互元素 |
| 适用场景 | 页面分析/存档 | AI 自主决策循环 |
| token 消耗 | 高 | 低（约 50%）|

### 1.3 多标签页管理

```javascript
// 列出所有标签页
browser(action: "tabs")

// 打开新标签页
browser(action: "open", url: "https://example.com", profile: "openclaw")
// → 返回 { targetId, title, url, type }

// 聚焦指定标签页
browser(action: "focusTab", targetId: "ABC123")

// 关闭标签页
browser(action: "closeTab", targetId: "ABC123")
// 或

// 在指定标签页导航

```

### 1.4 完整操作示例

```javascript
// 场景：完成 GitHub 登录并获取通知

// Step 1: 打开 GitHub
browser(action: "open", url: "https://github.com", profile: "openclaw")
// → { targetId: "ABC123", url: "https://github.com/", title: "GitHub" }

// Step 2: AI 理解页面，找到登录按钮
browser(action: "snapshot", targetId: "ABC123", mode: "efficient")
// → AI 看到登录链接 ref: "ax-login-link"

// Step 3: 点击登录
browser(action: "act", targetId: "ABC123", kind: "click", ref: "ax-login-link")

// Step 4: 等待登录页加载完成
browser(action: "act", targetId: "ABC123", kind: "wait",
        loadState: "networkidle", timeoutMs: 30000)

// Step 5: 填写用户名密码（batch 批量操作）
browser(action: "act", targetId: "ABC123", kind: "batch", actions: [
  { kind: "fill", fields: [
    { ref: "ax-username", value: "myaccount" },
    { ref: "ax-password", value: "mypassword" }
  ]},
  { kind: "press", key: "Enter" }
])

// Step 6: 等待跳转，再理解页面
browser(action: "act", targetId: "ABC123", kind: "wait", loadState: "networkidle")
browser(action: "snapshot", targetId: "ABC123", mode: "efficient")
```

### 1.5 SSRF 防护配置（源码级理解）

#### 配置结构

```json
{
  "browser": {
    "ssrfPolicy": {
      "dangerouslyAllowPrivateNetwork": true,   // 默认 true（信任网络模式）
      "allowedHostnames": ["localhost", "metadata.internal"],  // 精确匹配
      "hostnameAllowlist": ["*.example.com"]   // 支持 *.xxx 泛域名
    }
  }
}
```

#### 向后兼容

源码支持旧配置字段自动迁移：

```javascript
// allowPrivateNetwork → dangerouslyAllowPrivateNetwork（自动转换）
if ("allowPrivateNetwork" in rawSsrFPolicy) {
    dangerouslyAllowPrivateNetwork = allowPrivateNetwork === true
    delete nextSsrFPolicy.allowPrivateNetwork  // 删除旧字段
}
```

#### 代理环境变量冲突的判断逻辑（源码）

```javascript
// cdp.helpers-Cs33yWzE.js 中的判断
if (hasProxyEnvConfigured() && !isPrivateNetworkAllowedByPolicy(opts.ssrfPolicy)) {
    throw new InvalidBrowserNavigationUrlError(
        "Navigation blocked: strict browser SSRF policy cannot be enforced while env proxy variables are set"
    )
}

// 判断条件：hasProxyEnvConfigured() && dangerouslyAllowPrivateNetwork !== true
// hasProxyEnvConfigured() 检测 HTTP_PROXY / HTTPS_PROXY / http_proxy / https_proxy 环境变量
```

#### 解决方案

```json
// openclaw.json
{
  "browser": {
    "enabled": true,
    "headless": false,
    "defaultProfile": "openclaw",
    "ssrfPolicy": {
      "dangerouslyAllowPrivateNetwork": true    // 有代理时必须设为 true
    }
  }
}
```

> ⚠️ 仅适用于本地开发机。共享/生产环境请评估风险后使用。

## 22.1.3 第三层：平台自有 API / 服务（深度调研）

平台官方 API 永远是数据获取的最优选择——**最准确、最稳定、最不容易被封**。

### 3.1 平台官方 API vs 其他方式

| 维度 | 官方 API | MCP/CLI | 浏览器 |
|------|---------|---------|--------|
| 数据准确性 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| 稳定性 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐ |
| 反封禁 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐ |
| 覆盖范围 | 限官方提供 | 社区补充 | 任意网站 |
| 成本 | 免费/付费 | 免费 | 免费 |
| 认证复杂度 | 高（需申请） | 低 | 无/中 |

### 3.2 金融数据平台 API（深度分析）

#### Tushare（⭐最主流，活跃维护）

| 维度 | 数据 |
|------|------|
| **定位** | 国内最主流的量化金融数据平台，Beta已升级至2026-03-10版本 |
| **认证** | 注册 → 申请 token（免费注册即有基础权限） |
| **积分机制** | 核心制度：数据权限与积分挂钩。初始2000分，高频数据需5000/10000/20000分 |
| **覆盖** | 股票/期货/期权/基金/指数/港股/美股；舆情/资金流向/因子数据 |
| **成本** | 基础功能免费；专业数据（Level-2/盈利预测）需付费或高积分 |
| **接入** | `pip install tushare`，几行代码即可调用 |

#### AKShare（⭐完全免费MIT协议）

| 维度 | 数据 |
|------|------|
| **定位** | 纯开源免费金融数据库 |
| **认证** | 无需注册、无token，直接调用 |
| **覆盖** | 股票/期货/期权/基金/债券/黄金/宏观；舆情/加密货币 |
| **数据质量** | 中等（依赖各源网站稳定性） |
| **成本** | 完全免费（MIT开源协议） |

#### 聚宽 / 东方财富 Choice

| 平台 | 定位 | 成本 | 特色 |
|------|------|------|------|
| **聚宽** | 量化云平台（回测+实盘） | ¥299/月起 | 回测引擎是核心竞争力 |
| **东方财富 Choice** | 专业机构级 | 万元/年 | 研报PDF/盈利预测/ESG/另类数据 |

**选型建议**：学习/原型 → AKShare；A股量化研究 → Tushare；回测+实盘闭环 → 聚宽；机构研究 → Choice

### 3.3 社交媒体 API（深度分析）

#### 微博开放 API（最开放 ✅）

| 维度 | 数据 |
|------|------|
| **认证** | OAuth 2.0，需要开发者账号 |
| **申请难度** | 中等（1-3个工作日审核） |
| **数据范围** | 微博内容/评论/用户信息/关系链；写入：发布/评论 |
| **限制** | 部分高级数据（热门话题原始数据）需商业授权 |
| **成本** | 基础读取免费；部分商业数据需付费 |

#### 微信公众号 API（受限 ⚠️）

> ⚠️ 重要限制：不能通过 API 直接获取任意公众号的文章内容（需公众号授权）

| 维度 | 数据 |
|------|------|
| **认证** | 微信公众平台账号 + AppID/AppSecret |
| **能力** | 素材管理/消息接收/自动回复/用户管理/菜单管理 |
| **限制** | 只能获取**自身公众号**的粉丝消息和素材；无法爬取其他公众号 |
| **成本** | 免费（微信官方） |

#### 知乎 / 小红书（难接入 ❌）

| 平台 | 申请难度 | 说明 |
|------|----------|------|
| **知乎 API** | ⭐⭐⭐ 高 | 2024年后审查严格，需实际产品/网站，申请成功率低 |
| **小红书** | ⭐⭐⭐ 高 | 仅对企业开发者开放（个人无法申请），需营业执照 |

### 3.4 GitHub API（深度分析）

#### REST API v3

| 维度 | 数据 |
|------|------|
| **未认证限制** | 60次/小时 |
| **PAT 认证** | 5000次/小时 |
| **搜索 API** | 30次/分钟（独立限制） |
| **覆盖** | 190+ 端点：Repo/Issues/PR/代码搜索/用户/Actions |
| **稳定性** | 极高，GitHub 官方 SLA |
| **接入** | 直接 HTTP 调用，文档清晰 |

#### GraphQL API v4

| 维度 | 数据 |
|------|------|
| **速率限制** | 5000点/小时（每字段有不同点值） |
| **优势** | 一次请求获取多维数据（减少请求数，避免 Over-fetching）|
| **适用** | 跨多个数据维度聚合（如一次性获取 PR+review+CI 状态）|
| **学习曲线** | 中等（需理解 GraphQL schema 和查询语法）|

**选型建议**：简单独立调用 → REST；复杂数据聚合 → GraphQL；两者可混用

#### Exa AI Search（语义搜索 ⭐）

## 22.1.5 垂直领域数据采集架构设计

### 5.1 架构设计原则

```
垂直领域数据采集的核心矛盾：
  "最准确的数据 = 官方 API，但官方 API 往往需要资质/付费/申请"
  "最便宜的数据 = 爬虫，但爬虫有法律风险和封禁风险"
  "最可靠的数据 = 浏览器操作，但速度慢且消耗高"

解决思路：建立三层降级机制
  第1优先 → 官方 API（高质量、稳定）
  第2优先 → 平台 MCP/CLI（社区维护、平衡）
  第3优先 → 浏览器自动化（兜底、通用）
```

### 5.2 典型垂直领域采集架构

#### 金融领域（股票/基金/宏观）

```
数据源                    工具层                  OpenClaw 层
─────────────────────────────────────────────────────────────
股票行情（实时）   →   Tushare / AKShare      →  mo-finance subagent
财务报表          →   东方财富 Choice API     →  mo-finance subagent
宏观数据          →   国家统计局 API          →  mo-finance subagent
财经新闻          →   Jina Reader / Firecrawl →  mo-finance subagent
券商研报（需登录）→   Camoufox                →  mo-finance subagent
社交舆情          →   xurl / 微博 API         →  mo-finance subagent
```

#### 内容运营领域（社交媒体/视频/图文）

```
数据源                    工具层                  OpenClaw 层
─────────────────────────────────────────────────────────────
热点搜索          →   Tavily / Exa MCP        →  mo-yunying subagent
竞品内容          →   平台官方 API            →  mo-yunying subagent
视频内容          →   yt-dlp                  →  mo-yunying subagent
社交媒体          →   xurl / xhs-cli          →  mo-yunying subagent
公众号文章        →   Camoufox + Jina Reader →  mo-yunying subagent
评论区/用户反馈   →   平台 API / 浏览器       →  mo-yunying subagent
```

## 22.1.6 实战工作流示例

### 示例 1：监控竞品公众号内容更新

```
目标：每天早上 9 点抓取目标公众号的最新文章，分析标题和摘要

工具选型：
  Step 1: Camoufox 登录微信公众平台，抓取文章列表
          （无法通过公开 API 访问）

  Step 2: Jina Reader 提取文章正文
          （弥补 Camoufox 的内容提取能力）

  Step 3: OpenClaw Skill 格式化输出 + 推送 Telegram 群

OpenClaw Cron 配置：
  schedule: "0 9 * * *"  tz: "Asia/Shanghai"
  payload: agentTurn message: "执行 mo-yunying 竞品监控工作流"
  delivery: announce → Telegram 群: mo-yunying
```

### 示例 2：金融舆情实时监控

```
目标：监控 Twitter/X 上特定股票相关的讨论热度

工具选型：
  Step 1: xurl 搜索 + 读取推文内容
          （官方 API v2，支持搜索）

  Step 2: Tavily AI 搜索补充
          （全网搜索补充 Twitter 外的讨论）

  Step 3: OpenClaw 分析情感 + 提取关键信息

  Step 4: 触发阈值（如：1 小时内讨论量 > X）→ 推送告警
```

### 示例 3：电商竞品价格追踪

```
目标：追踪京东/淘宝竞品价格变化

工具选型：
  京东 → 京东开放平台 API（需商家资质）
  淘宝 → 淘宝开放平台 API（需商家资质）

  备选（无商家资质）：
  → Camoufox 定期抓取页面（反检测浏览器）
  → 价格变化 → 推送告警

注意：爬虫抓取电商数据需遵守 robots.txt 和平台服务条款
```

## 22.1.8 合规与安全注意事项

> ⚠️ 在进行数据采集前，请务必了解并遵守以下原则

### 法律合规

| 原则 | 说明 |
|------|------|
| **遵守 robots.txt** | 允许的路径才采集 |
| **不破解付费墙** | 付费内容不能通过漏洞获取 |
| **不侵犯著作权** | 采集内容仅供个人/研究使用，发布需授权 |
| **不对抗反爬措施** | 明确的反爬标识不要绕过（法律风险）|
| **遵守平台服务条款** | 每个平台有不同的 API 使用政策 |

### 技术安全

| 原则 | 说明 |
|------|------|
| **登录态安全管理** | Cookie/Token 不要硬编码，用环境变量或密钥管理 |
| **最小权限** | API Key 只申请需要的权限范围 |
| **轮询间隔** | 不要高频请求，给目标服务器留出资源 |

### OpenClaw 安全配置

```json
{
  "tools": {
    "fs": {
      "workspaceOnly": true
    },
    "exec": {
      "safeBins": ["curl", "python3", "gh", "yt-dlp"],
      "allowlist": ["git", "python3", "curl"]
    }
  }
}
```

## 本章小结

1. **三层架构是垂直领域数据采集的核心方法论**
   - 第1层：OpenClaw 内置浏览器（通用界面操作 + AI 视觉理解）
   - 第2层：外部 MCP/CLI/反检测浏览器（平台专项能力）
   - 第3层：平台官方 API（最准确、最稳定）

2. **选型决策树**
   - 公开数据 → API/MCP → 浏览器（降级）
   - 需登录数据 → 反检测浏览器（Camoufox）为主
   - 复杂长流程 → OpenClaw 浏览器 act 序列
   - 需要 AI 判断 → Computer-Use 模式

3. **Agent Reach / OpenCLI 是快速启动的好帮手**
   - 每个平台提供当前最优工具 + 替代方案
   - 不满意可以随时替换单个工具

4. **组合使用才是最优解**
   - 很少有一个工具能完成完整任务
   - 浏览器 + API、MCP + CLI 组合是常见模式

5. **合规永远是前提**
   - 了解目标平台的法律法规
   - 技术能力越强，责任越大
