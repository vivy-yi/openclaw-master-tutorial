## 2. 各层详解

### Layer 1: 每日日志 (Daily Logs)

**位置**: `memory/YYYY-MM-DD.md`

**内容**:
- 完成任务清单
- 重要讨论摘要
- 待处理事项
- 明日计划

**触发**: 每日21:00自动生成

```markdown
# 2026-04-06 日志

## 完成任务
- [x] 配置墨自Cron任务
- [x] 修复预测追踪delivery.to

## 重要讨论
- 用户要求修复Cron任务sessionKey问题
- mo-fortune技能开发完成

## 待处理
- [ ] 测试mo-fortune完整流程
- [ ] 启用Chrome remote debugging

## 明日计划
- 继续完善mo-fortune知识库
```

### Layer 2: 长期记忆 (MEMORY.md)

**位置**: `workspace/MEMORY.md`

**内容**:
- 精选的日志精华
- 用户偏好和规则
- 重要决策记录
- 系统配置要点

**规则**:
- 日志超过200行时清理并提取精华
- 每周蒸馏一次（7条日志→3-5条精华）

```markdown
# MEMORY.md - 长期记忆

## 用户偏好
- 不要先问，直接执行
- 使用OpenClaw原生工具
- 健康报告发送到私聊

## 重要规则
- Cron任务配置: sessionKey + delivery.to + payload身份必须匹配
- 文件删除用trash不用rm

## 系统配置
- Gateway监听: ws://127.0.0.1:18789
- 代理配置: 移除env.vars中的代理
```

### Layer 3: 纠正记录 (Corrections)

**位置**: `~/.openclaw/shared/self-improving/corrections.md`

**格式**:
```yaml
- id: [uuid]
  type: [correction|preference|pattern]
  content: [内容]
  source: [来源]
  date: [日期]
  applied: [次数]
```

**规则**:
- correction: 用户明确纠正 → 直接记录
- preference: 用户偏好 → 出现2次记录
- pattern: 行为模式 → 出现3次提升到MEMORY.md

```markdown
# Corrections Log

- id: 001
  type: preference
  content: 不要先问，直接执行任务
  source: 用户纠正
  date: 2026-04-05
  applied: 1

- id: 002
  type: pattern
  content: 使用OpenClaw原生工具
  source: 用户纠正
  date: 2026-04-05
  applied: 1
```

### Layer 4: 共享知识库 (Shared Knowledge)

**位置**: `~/.openclaw/shared/`

**目录结构**:
```
shared/
├── consensus/     # 共识协议
│   ├── agent-group-binding.md
│   ├── triggers.md
│   └── protocols.md
├── knowledge/     # 知识
│   └── self-improving.md
├── context/       # 上下文
├── memory/        # 集体记忆
└── sync/         # 同步状态
```

**规则**:
- ✅ 影响≥2个Agent的决策
- ✅ 重大架构/配置变更
- ✅ 用户明确要求
- ❌ 禁止：日常对话、私有内部讨论

### Layer 5: 向量数据库 (LanceDB)

**位置**: `memories/lancedb/`

**用途**:
- 语义搜索记忆
- 跨会话上下文检索
- 长期知识积累

**触发**: 需要语义搜索时启用

### Layer 6: 外部知识库

**位置**: Obsidian / Notion / GitHub

**用途**:
- 备份重要记忆
- 跨设备同步
- 外部查阅分享

## 3. 关键规则

### 3.1 写入时机

| 情况 | 写入位置 | 备注 |
|------|----------|------|
| 每日日志 | `memory/YYYY-MM-DD.md` | 自动生成 |
| 提炼精华 | `MEMORY.md` | 200行时清理 |
| 用户纠正 | `corrections.md` | 立即记录 |
| 同一纠正≥3次 | 提升到`MEMORY.md` | 蒸馏 |
| 跨Agent知识 | `shared/` | 影响≥2个Agent |

### 3.2 记忆召回优先级

```
当前会话
    ↓
HEARTBEAT检查
    ↓
MEMORY.md (长期记忆)
    ↓
corrections.md (纠正记录)
    ↓
shared/ (共享知识)
    ↓
memory/ (每日日志)
    ↓
向量数据库 (可选)
```

### 3.3 自我优化触发条件

- **被动触发**: 用户纠正 → 立即记录
- **主动触发**: 每日检查 corrections.md
  - 统计各type出现次数
  - 同一type≥3次 → 提升到MEMORY.md
  - 分析Pattern趋势

## 4. Cron任务中的记忆配置

### 4.1 sessionKey与delivery.to必须匹配

**错误配置**:
```yaml
sessionKey: agent:main:cron:...
payload: "你是一个法律助手"
delivery.to: -5207101318  # 法律群
```

**正确配置**:
```yaml
# 方式1: 保持isolated，但payload身份正确
sessionTarget: "isolated"
payload: "你是mo-law，执行法律任务"
delivery.to: -5207101318

# 方式2: 在目标Agent的workspace中创建任务
sessionTarget: "mo-law"
payload: "法律任务"
delivery.to: -5207101318
```

### 4.2 修改Cron任务时检查清单

```
□ delivery.to - 消息发送到哪
□ sessionTarget - 任务在哪个session运行
□ payload.message中的Agent身份 - 必须与目标群匹配
```

## 6. 最佳实践

### 6.1 写入习惯

- **立即记录**: 重要决策、用户偏好、错误教训
- **定期蒸馏**: 每周7条日志→3-5条精华
- **不过度**: 不需要记录每句话，只记重要内容

### 6.2 召回习惯

- **每次对话**: 检查MEMORY.md关键配置
- **每日heartbeat**: 检查corrections.md新模式
- **遇到错误**: 回溯每日日志定位问题

### 6.3 遗忘策略

- MEMORY.md超过200行才清理
- corrections中同一type<3次的不提升
- 简单任务不提取，复杂任务提取3-5条

## 7. 常见问题

### Q1: 什么时候用shared/什么时候用MEMORY.md?

**shared/**: 影响多个Agent的配置、协议、知识
**MEMORY.md**: 单个Agent的长期记忆、用户偏好

### Q2: corrections.md和MEMORY.md的区别?

**corrections.md**: 原始纠正记录，用于分析模式
**MEMORY.md**: 蒸馏后的精华，已确认的规则

### Q3: 向量数据库什么时候用?

当需要语义搜索、跨会话回忆时启用。简单场景不需要。
## 8. 搜索工具选用原则 ⚠️

### 优先级排序（正确顺序）

```
┌─────────────────────────────────────────────────────────┐
│                   搜索工具优先级                          │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  1. 平台原生API  ✅ 优先                               │
│     - Twitter/GitHub/知乎/微博官方API                    │
│     - 精确、无配额、官方支持                             │
│                                                          │
│  2. browser.open()  ✅ 其次                            │
│     - 页面抓取、登录态、交互                            │
│     - 灵活、不消耗API配额                               │
│                                                          │
│  3. web_search (Tavily)  ⚠️ 最后兜底                │
│     - 无特定数据源时使用                                │
│     - 有配额限制，不滥用                                │
└─────────────────────────────────────────────────────────┘
```

### 为什么这样排序？

| 工具 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **平台API** | 精确、无配额、官方支持 | 需要配置 | 有官方API的场景 |
| **browser** | 灵活、通用、不耗配额 | 需要页面加载 | 大部分场景 |
| **Tavily** | 快速、结构化 | 有限额、易耗尽 | 无API时的兜底 |

### 设计原则

```markdown
1. 搜索信息时：
   - 优先使用平台原生API (Twitter/GitHub/知乎等)
   - 其次 browser.open(url)
   - 最后才考虑 web_search

2. 遇到限流/失败时：
   - 自动降级：API → browser → web_search → 本地缓存
   - 记录失败原因到 corrections.md

3. 批量搜索时：
   - 优先使用平台API批量接口
   - 避免连续调用 web_search
```

### 示例工作流

```yaml
# 正确的搜索流程
steps:
  1. 尝试平台原生API
     ↓ 有API
  2. 使用API获取数据
     ↓ 无API或失败
  3. browser.open(url)
     ↓ 成功
  4. browser.snapshot() + 提取
     ↓ 不可访问
  5. web_search(query) 作为兜底
     ↓ 有配额
  6. 使用搜索结果
     ↓ 配额不足
  7. 返回"搜索服务暂时不可用"
```
