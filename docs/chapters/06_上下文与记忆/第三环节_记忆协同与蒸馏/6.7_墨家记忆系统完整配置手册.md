# 墨家记忆系统完整配置手册

> 墨家 Agent 系统的记忆架构是分层协同设计：上下文压缩 → 候选召回 → 主记忆 → 知识库，每层各司其职，共同支撑 Agent 的持续记忆能力。

**整理时间**: 2026-04-25
**适用版本**: OpenClaw 2026.3+
**维护者**: 墨家 Agent 系统
**章节**: 第1-7章（架构+配置+协同+学习）| 第8章（降级策略）| 第9章（协同场景）

---

```
│  │   afterTurn           → auto-capture            │   │

│  │   Phase2             → 记忆提取                 │   │

┌─────────────────────────▼───────────────────────────────┐

│  状态: ✅ loaded                                     │
└─────────────────────────┬───────────────────────────────┘
                          │
┌─────────────────────────▼───────────────────────────────┐
│              Layer 2: 主记忆层                          │
│  ┌─────────────────────────────────────────────────┐   │
│  │          memory-lancedb-pro                       │   │

│  │   Rerank: jina-reranker-v3 (cross-encoder)      │   │

│  状态: ✅ loaded (vault ready)                        │
```

### 1.2 各层职责对比

| 层级 | 模块 | 核心职责 | 数据类型 | 持久化 |
|------|------|---------|---------|--------|
| Layer 0 | openviking | 对话压缩 + 上下文组装 | 会话历史、对话片段 | ✅ 归档 |
| Layer 1 | active-memory | LLM 前召回候选 | 检索结果 | ❌ 临时 |
| Layer 2 | memory-lancedb-pro | 混合检索 + 智能提取 | 6类结构化记忆 | ✅ 持久 |
| Layer 3 | memory-wiki | 外部知识库 | Obsidian 笔记 | ✅ 持久 |

### 1.3 记忆流向

```
会话结束
    │
    ▼
openviking Phase2 提取
    │ 生成结构化记忆摘要
    ▼
memory-lancedb-pro 存储
    │ 智能分类 (6类)
    │ 混合检索索引 (向量+BM25)
    │ 自优化钩子记录
    ▼
下次会话
    │
    ▼
active-memory 召回候选
    │ 检索相关记忆
    ▼
openviking 组装上下文
    │ auto-recall → 组装 prompt
    ▼
LLM (MiniMax-M2.7) 生成回复
    │
    ▼
memory-wiki 补充外部知识（如需要）
    │ Obsidian vault 检索
    ▼
返回用户
```

### 2.1 openviking（上下文引擎）

**Slot**: `contextEngine`
**状态**: ✅ loaded
**版本**: 2026.4.34

#### 配置

```json
{
  "plugins": {
    "entries": {
      "openviking": {
        "enabled": true,
        "config": {
          "mode": "local",
          "configPath": "/Users/d/.openviking/ov.conf",
          "port": 1933
        }
      }
    }
  },
  "plugins": {
    "slots": {
      "contextEngine": "openviking"
    }
  }
}
```

#### ov.conf 配置

```json
{
  "storage": {
    "workspace": "/Users/d/.openviking/workspace"
  },
  "log": {
    "level": "INFO",
    "output": "stdout"
  },
  "embedding": {
    "dense": {
      "api_base": "https://api.jina.ai/v1",
      "api_key": "jina_...",
      "provider": "jina",
      "dimension": 1024,
      "model": "jina-embeddings-v5-text-small"
    },
    "max_concurrent": 10
  },
  "vlm": {
    "api_base": "https://api.minimaxi.com/v1",
    "api_key": "MiniMax API Key",
    "provider": "openai",
    "model": "MiniMax-M2.7",
    "max_concurrent": 100
  }
}
```

#### 核心能力

| 能力 | 说明 |
|------|------|
| `before_prompt_build=auto-recall` | 检索相关历史会话组装到 prompt |
| `afterTurn=auto-capture` | 每个回合结束后自动捕获上下文 |
| `assemble=archive+active` | 归档历史 + 保留活跃上下文 |
| `Phase2` | VLM 分析会话 → 生成结构化记忆摘要 |

#### 安装命令

```bash
openclaw plugins install clawhub:@openclaw/openviking
openclaw openviking setup
# 选择 local 模式
openclaw gateway restart
```

```json
            "dimensions": 1024,
            "normalized": true
          },
          "llm": {
            "model": "minimax-portal/MiniMax-M2.7",
            "baseURL": "https://api.minimax.chat/v1"
          },
          "smartExtraction": true,
          "autoRecall": true,
          "autoRecallMinLength": 10,
          "retrieval": {
            "mode": "hybrid",
            "rerank": "cross-encoder",
            "rerankApiKey": "jina_...",
            "rerankModel": "jina-reranker-v3",
            "rerankEndpoint": "https://api.jina.ai/v1/rerank",
            "rerankProvider": "jina",
            "candidatePoolSize": 20
          },
          "sessionStrategy": "memoryReflection",
          "selfImprovement": {
            "enabled": true
          }
        }
      }
    }
  },
  "plugins": {
    "slots": {
      "memory": "memory-lancedb-pro"
    }
  }
}
```

#### 核心能力

| 能力 | 说明 |
|------|------|
| **混合检索** | 向量70% + BM25 30%，兼顾语义和关键词 |
| **Rerank** | jina-reranker-v3 cross-encoder 重排序 |
| **Smart Extraction** | MiniMax-M2.7 分析会话 → 6类结构化记忆 |
| **6类记忆** | fact / preference / entity / decision / reflection / other |
| **自优化** | selfImprovement 钩子，记录学习到 .learnings |
| **Noise Bank** | 噪声数据隔离，不影响检索质量 |

#### 6类记忆说明

| 类型 | 说明 | 示例 |
|------|------|------|
| fact | 客观事实 | "用户住在上海" |
| preference | 偏好 | "用户喜欢麻辣口味" |
| entity | 实体 | "A公司是供应商" |
| decision | 决策 | "选择了方案B" |
| reflection | 反思 | "上次开会效率低" |
| other | 其他 | 临时性重要信息 |

## 3. 模块协同机制

### 3.1 完整协同流程

```
会话开始
    │
    ├─ ① active-memory ─────────────────────────────┐
    │   触发时机: LLM 调用前                       │
    │   操作: 检索 memory-lancedb-pro             │
    │   输出: 相关记忆候选列表                     │
    │   注入: → LLM prompt                       │
    │                                                  │
    ├─ ② memory-wiki ────────────────────────────┤
    │   触发时机: 需要外部知识时                   │
    │   操作: 搜索 Obsidian vault                  │
    │   输出: 相关笔记片段                         │
    │   注入: → LLM prompt                       │
    │                                                  │
    └─ ③ openviking ──────────────────────────────┘
        触发时机: 每次 LLM 调用前后
        操作: 组装上下文 (archive + active)
        Phase2: 提取结构化记忆 → memory-lancedb-pro
        │
        ▼
    LLM (MiniMax-M2.7) 生成回复
        │
        ▼
    会话结束
        │
        ▼
    openviking Phase2 记忆提取
        │
        ▼
    memory-lancedb-pro Smart Extraction
        │
        ├─→ 6类分类 (fact/preference/entity/...)
        ├─→ 混合检索索引 (向量+BM25)
        ├─→ 自优化钩子记录 (.learnings)
        └─→ 归档存储
```

### 3.2 各模块数据交换

```
openviking
  输出 → memory-lancedb-pro (存储提取的记忆)
  输出 → active-memory (历史上下文)

active-memory
  输入 ← memory-lancedb-pro (检索)
  输出 → LLM prompt

memory-wiki
  输出 → LLM prompt (补充知识)

memory-lancedb-pro
  输入 ← openviking (Smart Extraction)
  输入 ← active-memory (检索源)
  输出 → active-memory (检索结果)
```

### 3.3 slot 冲突说明

| Slot | 只能有一个 | 当前占用 |
|------|-----------|---------|
| `contextEngine` | ✅ | openviking |
| `memory` | ✅ | memory-lancedb-pro |
| `wiki` | ✅ | memory-wiki |

**重要**: 同一 slot 只能有一个插件。openviking 取代了 lossless-claw，memory-lancedb-pro 取代了 memory-core 和 memory-lancedb。

## 4. 自主学习机制

### 4.1 自优化钩子（selfImprovement）

memory-lancedb-pro 内置自优化机制，持续学习和改进。

#### 工作原理

```
每次重要交互后
    │
    ▼
selfImprovement 钩子触发
    │
    ├─→ 记录到 .learnings/
    │      corrections.md (纠正)
    │      patterns.md (模式)
    │      errors.md (错误)
    │
    ├─→ 模式识别
    │      3次相同行为 → 提升为长期模式
    │
    └─→ 知识蒸馏
           有价值的 → 蒸馏到 MEMORY.md
```

#### .learnings 目录结构

```
~/.openclaw/.learnings/
├── corrections.md    # 用户纠正记录
├── patterns.md      # 识别到的行为模式
├── errors.md        # 错误记录
└── distillations/   # 蒸馏出的知识
```

#### 自优化触发条件

| 类型 | 触发条件 | 目标文件 |
|------|---------|---------|
| 纠正 (correction) | 用户直接纠正 | corrections.md |
| 模式 (pattern) | 3次相同行为 | patterns.md → MEMORY.md |
| 错误 (error) | 操作失败 | errors.md |
| 最佳实践 (best_practice) | 发现更好方式 | patterns.md |

### 4.2 Smart Extraction（智能记忆提取）

#### Phase2 流程

```
会话片段 (最新对话)
    │
    ▼
openViking Phase2
    │
    ▼
VLM (MiniMax-M2.7) 分析
    │
    ├─→ 识别关键事实 (fact)
    ├─→ 提取用户偏好 (preference)
    ├─→ 记录实体关系 (entity)
    ├─→ 总结决策要点 (decision)
    └─→ 生成反思洞察 (reflection)
    │
    ▼
memory-lancedb-pro 存储
    │
    ├─→ 向量索引 (embedding)
    ├─→ BM25 索引 (关键词)
    ├─→ 元数据 (时间/来源/置信度)
    └─→ 衰减权重 (Weibull)
```

#### 衰减机制（Weibull）

记忆不是静态存储，而是根据使用频率和时效性动态调整权重。

```
权重 = f(访问频率, 时间, 重要性)
    │
    ├─→ 高频访问 → 权重增强
    ├─→ 长时间未访问 → 权重衰减
    └─→ 重要决策 → 权重衰减更慢
```

### 4.3 知识蒸馏流程

```
定期 (cron: 每日 01:30)
    │
    ▼
读取 .learnings/
    │
    ▼
LLM 蒸馏
    │ 识别值得长期保留的知识
    │ 去除噪声和临时信息
    │
    ▼
更新 MEMORY.md
    │ 长期记忆文件
    │
    ▼
清理 .learnings/
    │ 归档已蒸馏的记录
```

### 4.4 自主学习配置

```json
{
  "selfImprovement": {
    "enabled": true,
    "triggers": {
      "correction": true,
      "pattern": true,
      "error": true,
      "best_practice": true
    },
    "promotionThreshold": 3,
    "distillationCron": "0 30 1 * * *",
    "destinations": {
      "shortTerm": ".learnings",
      "longTerm": "MEMORY.md"
    }
  }
}
```

## 6. 故障排除

### 6.1 记忆检索不准确

**检查项**:
1. embedding 是否正常 → 测试 Jina API
2. 混合检索比例 → 调整 vector/bm25 ratio
3. Rerank 是否启用 → 检查 rerank 配置

### 6.2 Smart Extraction 不生效

**检查项**:
1. VLM 是否可达 → `curl api.minimaxi.com/v1/chat/completions`
2. Phase2 是否启用 → 检查 openviking 日志
3. MiniMax API Key 是否有效 → 检查余额

### 6.3 memory-wiki 检索无结果

**检查项**:
1. vault 路径是否正确 → `wiki_status`
2. Obsidian vault 是否存在笔记
3. vaultMode 是否为 unsafe-local

## 8. 降级策略

### 8.1 降级优先级链

```
请求记忆
    │
    ▼
┌─ active-memory ───────────────────────────┐
│  尝试检索 memory-lancedb-pro             │
│  超时/失败 → 降级到 memory-core           │
│  memory-core 也失败 → 降级到 memory-wiki  │
│  wiki 失败 → 返回空（不影响主流程）         │
└──────────────────────────────────────────┘
```

### 8.2 各模块降级路径

| 模块 | 失败时降级到 | 降级触发条件 |
|------|-------------|-------------|
| **openviking** | lossless-claw | openviking 未安装/崩溃 |
| **openviking Phase2** | 跳过，不影响主流程 | VLM 调用失败 |
| **memory-lancedb-pro** | memory-core | embedding/VLM 失败 |
| **memory-lancedb-pro rerank** | 纯向量检索 | jina API 失败 |
| **active-memory** | 跳过，内存召回 | 子Agent崩溃 |
| **memory-wiki** | 跳过，返回空 | vault 不可读 |

### 8.3 具体降级场景

#### 场景A：Jina API 失败

```
memory-lancedb-pro embedding 调用失败
    │
    ▼
降级策略触发
    │
    ├─→ 禁用 rerank（返回原始混合分数）
    ├─→ 禁用 BM25（纯向量检索）
    └─→ 降级到 memory-core（如果也失败）
    │
    ▼
最终降级：返回空列表（不影响LLM回复）
```

#### 场景B：MiniMax API 失败

```
openviking Phase2 / Smart Extraction 调用失败
    │
    ▼
降级策略触发
    │
    ├─→ Phase2 跳过（不提取记忆）
    ├─→ 会话继续正常进行
    └─→ 仅影响本次会话的"记忆写入"
    │
    ▼
读取记忆：memory-lancedb-pro 仍然可用
```

#### 场景C：Obsidian vault 不可读

```
memory-wiki 访问 vault 失败
    │
    ▼
降级策略触发
    │
    ├─→ 返回空（不影响主流程）
    └─→ 不阻塞 LLM 回复
    │
    ▼
不影响其他三个记忆层
```

### 8.4 降级配置

```json
{
  "plugins": {
    "entries": {
      "memory-lancedb-pro": {
        "config": {
          "degradation": {
            "onEmbeddingFailure": "skip-rerank",
            "onLLMFailure": "skip-extraction",
            "onDBFailure": "fallback-memory-core"
          }
        }
      }
    }
  }
}
```

### 8.5 降级设计原则

| 原则 | 说明 |
|------|------|
| **fail-safe** | 任何模块失败不影响 LLM 主流程 |
| **graceful** | 降级后仍能提供有限服务 |
| **fast-fail** | 快速降级，不阻塞用户 |
| **recoverable** | 上游恢复后自动恢复 |
