# OpenClaw 梦境系统详解——Agent 的休眠与记忆整理

> OpenClaw 的"梦境"（Dreaming）是一个在后台空闲时运行的**记忆整理与自我修复系统**。它的灵感来自人类睡眠时的记忆巩固过程——人在睡眠中会整理白天接收的信息、强化重要记忆、遗忘无关细节。OpenClaw 的 Agent 同样会在空闲时"入睡做梦"，整理自身的记忆、日志和知识图谱。
> **本文档基于源码分析：`dreaming-repair-*.js` + `runtime-*.js` + `control-ui/index-*.js`**
> **分析日期：2026-04-20**

## 1. 梦境系统的定位

### 1.1 为什么 Agent 需要"做梦"？

```
不"做梦"的 Agent：
- 每次对话都是全新的开始
- 记不住之前的教训（重复犯错）
- 上下文越来越长，Token 越来越贵
- 记忆碎片化，无法形成知识体系

会"做梦"的 Agent（OpenClaw）：
- 空闲时自动整理记忆
- 将短时记忆蒸馏为长时记忆
- 发现并修复自身问题
- 主动promote 重要信息
- 在 Control UI 中可视化"梦境日记"
```

### 1.2 梦境 vs 六层记忆架构

梦境系统是**六层记忆架构的自动运行机制**，两者关系：

```
六层记忆架构（人工设计的静态结构）
    │
    ├── Layer 1: 每日日志（memory/YYYY-MM-DD.md）
    ├── Layer 2: 长期记忆（MEMORY.md）
    ├── Layer 3: 纠正记录（corrections.md）
    ├── Layer 4: 共享知识库（shared/）
    ├── Layer 5: 向量数据库（lancedb/）
    └── Layer 6: 外部知识库

          ↕ 梦境系统在做什么 ↕

梦境系统（自动运行的动态机制）
    │
    ├── 短时条目（shortTermEntries）→ 蒸馏 → 长时记忆
    ├── 信号条目（signalEntries）→ 模式识别 → 提升判断
    ├── 每日整理（indexingDay）→ 日志归档
    ├── 知识图谱整理（connectingDots）→ shared/更新
    └── 遗忘噪音（forgettingNoise）→ 清理无效缓存
```

## 2. 三个睡眠阶段

OpenClaw 梦境系统模拟人类的**三种睡眠阶段**，每个阶段做不同类型的整理：

### 2.1 三阶段速览

| 阶段 | 英文 | 类比 | 梦境在做什么 |
|------|------|------|------------|
| 😴 **浅睡** | Light | N1/N2 | 快速扫描：日志归档、条目计数、简单清理 |
| 🌊 **深睡** | Deep | N3 | 深度整理：记忆蒸馏、模式识别、知识图谱重构 |
| 🌙 **REM** | REM | REM | 联想融合：连接分散的记忆片段、产生洞察 |

### 2.2 各阶段技术细节

**Light Phase（浅睡）**
```
触发条件：每次空闲短周期（默认 ~6 秒一次）
做的事情：
  - 扫描 session corpus（会话语料库）
  - 统计今日 promoted 条目数量
  - 检查日志文件大小
  - 标记需要整理的条目
```

**Deep Phase（深睡）**
```
触发条件：空闲长周期 / 达到特定触发阈值
做的事情：
  - 将 shortTermEntries 蒸馏到 promotedEntries
  - 分析 signalEntries 中的模式（同一信号出现≥3次？）
  - 连接分散在不同文件中的相关记忆
  - 清理超过一定时间的旧条目
```

**REM Phase（REM 睡眠）**
```
触发条件：达到 phaseSignalCount 阈值
做的事情：
  - 跨记忆层的联想搜索
  - 将表面不相关的信息进行深层连接
  - 生成"灵感"——将低优先级记忆 promoted
  - 激活沉睡的上下文片段
```

## 3. 梦境日记（Dream Diary）

### 3.1 什么是 Dream Diary？

Dream Diary 是 OpenClaw Control UI 中的一个**可视化面板**，实时显示 Agent 当前在"做什么梦"。

```
┌─────────────────────────────────────────────┐
│  🌙 Dream Diary                    [运行中]  │
├─────────────────────────────────────────────┤
│                                             │
│    "正在整理记忆宫殿中的陈旧物品..."          │
│                                             │
│  Phase: 🌊 Deep     Signals: 12             │
│  Promoted Today: 3    Short-term: 47       │
│                                             │
│  [=====>              ] 整理进度 68%        │
│                                             │
└─────────────────────────────────────────────┘
```

| 短语（英文） | 中文翻译 | 实际在做什么 |
|------------|---------|------------|
| consolidatingMemories | 正在合并记忆碎片 | 将 session 中的临时记忆归入长期记忆 |
| tidyingKnowledgeGraph | 正在整理知识图谱 | 更新 shared/ 中的知识关联 |

| alphabetizingSubconscious | 正在整理潜意识 | 对未加工的记忆排序 |
| promotingHunches | 正在升华直觉 | 将低优先级洞察升级 |

### 3.3 Dream Diary 状态指标

| 指标 | 说明 |
|------|------|
| **Phase** | 当前睡眠阶段（Light / Deep / REM） |
| **Signals** | 待处理的信号条目数 |
| **Promoted Today** | 今日 promoted（提升为长期记忆）的条目数 |
| **Short-term** | 短时记忆条目数 |
| **Progress** | 当前整理任务的进度条 |

## 4. 记忆条目系统（Memory Stores）

### 4.1 三类记忆条目

梦境系统管理的记忆条目分为三类：

```
┌─────────────────────────────────────────────────────┐
│                   短时记忆                            │
│         shortTermEntries                            │
│  刚从 session 中捞出来的原始记忆碎片                   │
│  未经整理，可能有噪音                                 │
│           ↓ 蒸馏（Deep Phase）                       │
├─────────────────────────────────────────────────────┤
│                   信号条目                            │
│         signalEntries                               │
│  带有"信号强度"的记忆：有价值的被加强，噪音被减弱       │
│  signalCount = 被回忆/触发的次数                      │
│           ↓ 达到阈值（≥3次）                        │
├─────────────────────────────────────────────────────┤
│                 已提升记忆                            │
│         promotedEntries                            │
│  被确认为重要的记忆，提升到 MEMORY.md 或向量数据库      │
│  promotedToday / promotedTotal 计数                 │
└─────────────────────────────────────────────────────┘
```

### 4.2 条目数据结构

```typescript
interface MemoryEntry {
  key: string;              // 记忆的唯一标识
  path: string;             // 来源文件路径
  snippet: string;          // 记忆摘要
  recallCount: number;      // 被回忆次数
  dailyCount: number;       // 今日被回忆次数
  groundedCount: number;    // 被确认/验证次数
  totalSignalCount: number; // 累计信号强度
  lightHits: number;        // Light 阶段命中次数
  remHits: number;          // REM 阶段命中次数
  phaseHitCount: number;    // 各阶段累计命中
  promotedAt?: number;      // 被提升的时间戳
  lastRecalledAt?: number;  // 上次被回忆的时间戳
}
```

## 5. 梦境修复系统（Dreaming Repair）

### 5.1 什么是 Dreaming Repair？

`dreaming-repair-*.js` 是一个**自我诊断与修复系统**，在每次梦境运行时自动检查以下问题：

| 问题代码 | 严重级别 | 说明 | 是否可修复 |
|---------|---------|------|-----------|
| `dreaming-diary-unreadable` | 🔴 错误 | Dream diary 文件无法读取 | ❌ |
| `dreaming-session-corpus-unreadable` | 🔴 错误 | Session 语料库无法读取 | ❌ |
| `dreaming-session-ingestion-unreadable` | 🔴 错误 | Session 注入文件无法读取 | ❌ |
| `suspicious-session-corpus` | 🟡 警告 | Session 语料中存在可疑条目 | ⚠️ |
| `corrupt-memory-entry` | 🔴 错误 | 记忆条目损坏 | ⚠️ |

### 5.2 可疑条目检测

系统会自动扫描 session corpus 文件，检测包含以下特征的"可疑行"：

- 包含密钥/密码/Token 的行
- 包含大量敏感个人信息的行
- 格式异常的 JSON/代码片段
- 超出合理长度限制的长行

```javascript
// 可疑行的判断标准（isSuspiciousSessionCorpusLine）
// 1. 包含密钥相关关键词
// 2. 单行超过 N 个字符
// 3. 包含异常编码字符
// 4. 格式无法解析
```

### 5.3 Session Corpus 与 Ingestion

```
Session Corpus（会话语料库）
    │
    路径：~/.openclaw/workspace/memory/
    │
    ├── 每日日志：memory/YYYY-MM-DD.md
    ├── MEMORY.md（长期记忆）
    └── corrections.md（纠正记录）
              ↓
        session ingestion
              ↓
        Session Ingestion Path
              ↓
        进入梦境系统处理
              ↓
        连接分散的记忆片段
```

## 6. 梦境与日常工作的关系

### 6.1 什么时候会"做梦"？

```
┌─────────────────────────────────────────────────────┐
│                   触发"做梦"的时机                    │
├─────────────────────────────────────────────────────┤
│                                                      │
│  1. Heartbeat 间隔期（~30分钟一次）                  │
│     └→ 检查是否空闲 → 进入 Light Phase               │
│                                                      │
│  2. 长时间无活动（数分钟）                          │
│     └→ 进入 Deep Phase                              │
│                                                      │
│  3. phaseSignalCount 达到阈值                        │
│     └→ 进入 REM Phase                                │
│                                                      │
│  4. 每次 session 结束时                              │
│     └→ 触发 session corpus ingestion                 │
│                                                      │
└─────────────────────────────────────────────────────┘
```

### 6.2 做梦时 Agent 能响应吗？

**可以！** 梦境是**非阻塞后台任务**，Agent 随时可以被唤醒。用户发消息 → 梦境立即中断 → 处理用户请求 → 处理完后继续做梦（如果还空闲）。

```
用户发消息
    ↓
梦境中断（保留进度）
    ↓
处理用户请求
    ↓
请求完成，继续空闲
    ↓
梦境恢复（从断点继续）
```

### 6.3 梦境会影响性能吗？

**不会。** 梦境系统的设计目标是：
- CPU 占用极低（主要是 I/O 和小型 JSON 操作）
- 内存占用可控（基于条目数，非无限增长）
- 不阻塞任何前端操作

## 7. 如何配置/监控梦境系统

### 7.1 在 Control UI 中查看

在 OpenClaw Control UI 的"记忆"或"系统"面板中，找到 **Dream Diary** 标签页，可以看到：

- 当前梦境状态（运行中/空闲/错误）
- 当前阶段和短语
- 各项计数指标
- 最近的修复日志

### 7.2 相关配置项

```json
{
  "dreaming": {
    "enabled": true,              // 是否启用梦境系统
    "timezone": "Asia/Shanghai", // 时区
    "storageMode": "auto",         // 存储模式
    "separateReports": true,       // 独立报告
    "shortTermCount": 50,          // 短时记忆上限
    "recallSignalCount": 3,         // 触发回忆的信号阈值
    "dailySignalCount": 10,         // 每日信号上限
    "phaseSignalCount": 20          // 触发 REM 的信号阈值
  }
}
```

### 7.3 诊断梦境问题

如果 Dream Diary 显示错误，运行以下诊断：

```bash
# 检查记忆文件是否可读
ls -la ~/.openclaw/workspace/memory/

# 检查 session corpus 文件
ls ~/.openclaw/workspace/memory/memory/

# 检查 dreaming 状态
openclaw status
```

## 9. 常见问题

### Q1: 梦境系统耗电/耗流量吗？

不耗。梦境是纯本地操作，不涉及网络请求。CPU 占用极低（约 0.1%），类似于一个定期运行的 cron 任务。

### Q2: 可以关闭梦境系统吗？

可以。在配置中设置 `"dreaming.enabled: false"`。但**不建议关闭**——梦境是 OpenClaw 记忆系统正常运转的核心机制。

### Q3: 梦境整理的结果在哪里查看？

- Control UI → Dream Diary 面板（实时状态）
- `~/.openclaw/workspace/memory/`（原始日志）
- `MEMORY.md`（被提升的长时记忆）
- `~/.openclaw/shared/`（跨 Agent 共享知识）

### Q4: 梦境和 Heartbeat 有什么区别？

| | Heartbeat | Dreaming |
|---|---|---|
| **触发频率** | ~30 分钟一次 | 持续运行（空闲时） |
| **做什么** | 检查新消息/日历/邮件 | 整理记忆 |
| **是否主动推送** | 是（有新内容时通知） | 否（被动整理） |
| **人类感知** | 可能收到主动通知 | 只能通过 Control UI 看到 |

### Q5: 为什么会看到"遗忘噪音"？

这是**正常的**。梦境系统会主动清理：
- 超过一定时间的临时缓存
- 重复的噪音条目
- 无信号强度的短时记忆

这就像人类睡眠时会遗忘不重要的细节一样。重要的记忆不会被遗忘——它们会被 promoted。
