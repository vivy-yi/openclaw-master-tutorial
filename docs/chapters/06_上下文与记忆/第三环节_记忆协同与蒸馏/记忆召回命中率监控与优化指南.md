## 1. 召回失败原因深度分析

### 1.1 召回全链路

```
用户输入
    │
    ▼
┌─ Query 解析 ────────────────────────────────────┐
│  口语化 → 关键词提取 → 向量化                   │
└─────────────────────┬───────────────────────────┘
                      │
                      ▼
┌─ 多层召回 ──────────────────────────────────────┐
│ Layer 0: openviking → 历史会话片段               │
│ Layer 1: active-memory → 候选记忆                │
│ Layer 2: memory-lancedb-pro → 结构化记忆（主战场） │
│ Layer 3: memory-wiki → Obsidian 笔记             │
└─────────────────────┬───────────────────────────┘
                      │
                      ▼
┌─ 结果聚合 ────────────────────────────────────┐
│  排序/Rerank → 去重 → 注入prompt → LLM回复   │
└───────────────────────────────────────────────┘
```

### 1.2 召回失败的7个根因

| # | 根因 | 所属层级 | 表现 |
|---|------|---------|------|
| 1 | **Query 与记忆不是同一语义空间** | Query解析 | "我上次说的那个" → 匹配不到任何记忆 |
| 2 | **Embedding 模型不适配中文/专业术语** | Layer 2 | "墨家" 匹配到 "墨水" |
| 3 | **分块(Chunking)策略不当** | Layer 2 | 跨段落语义被切断 |
| 4 | **记忆未存入（Smart Extraction失败）** | Layer 2 | 根本没存，当然查不到 |
| 5 | **Weibull 衰减过度** | Layer 2 | 老记忆被错误衰减 |
| 6 | **跨 Agent 记忆隔离** | 全局 | mo-finance 的记忆 mo-law 看不到 |
| 7 | **Rerank 候选池过小** | Layer 2 | 初期排序错误无法纠正 |

### 1.3 各根因详解

#### 根因1: Query 语义空间不匹配

```
问题: 用户口语 vs 存储时的语义
  用户说: "我之前问过的那个"
  存储时: "用户询问了XXX"
  → 向量空间不同，无法匹配
```

**解决方案**:
- Query 扩展：在检索前用 LLM 将口语 query 扩展为多个语义变体
- Synonym expansion：添加同义词/近义词
- Hyponym expansion：添加上下位词

#### 根因2: Embedding 不适配中文/专业术语

```
问题: jina-embeddings-v5 对中文和专业术语支持有限
  存储: "墨家Agent系统"
  检索: "Multi-Agent Framework"
  → 两个完全相关的概念，向量距离远
```

**解决方案**:
- 使用中文优化的 embedding（text-embedding-3-small 中文版）
- 使用混合检索（BM25 弥补向量不足）
- Rerank 阶段纠正向量召回错误

#### 根因3: Chunking 策略不当

```
问题: 语义完整的段落被粗暴切分
  原文: "墨家系统分为4层...第1层是...第2层是..."
  切分后:
    Chunk1: "墨家系统分为4层"
    Chunk2: "第1层是..."
  → "第2层是什么" → 匹配 Chunk1 而不是 Chunk2
```

**解决方案**:
- 基于语义切分（Sentence boundaries）
- 保留段落开头/结尾的上下文（Overlap）
- 标题+内容 作为一个 chunk

#### 根因4: Smart Extraction 未执行

```
问题: 对话结束了，但记忆没存进去
原因:
  - VLM (MiniMax) 调用失败
  - 会话太短（< extractMinMessages=2）
  - 内容被 noise bank 过滤
  - 衰减后被清理
```

#### 根因5: Weibull 衰减过度

```
问题: 有价值的记忆被过早衰减
配置:
  timeDecayHalfLifeDays: 60
  reinforcementFactor: 0.5
  → 访问频率低的记忆权重快速衰减
```

#### 根因6: 跨 Agent 记忆隔离

```
问题: mo-finance 的记忆 mo-law 看不到
原因:
  - 每个 Agent 有独立的 memory-lancedb-pro 实例
  - shared/knowledge/ 需要显式写入/读取
```

#### 根因7: Rerank 候选池过小

```
问题: candidatePoolSize: 20 → 初始排序错了20个以内无法纠正
配置:
  rerankCandidatePoolSize: 20
  minScore: 0.3 → 低于0.3直接过滤
  → 高价值记忆在池外，直接被过滤
```

## 2. 召回质量评估体系

### 2.1 核心指标

| 指标 | 定义 | 理想值 | 当前可测 |
|------|------|--------|---------|
| **Recall@K** | Top-K 结果中命中的比例 | K=5 时 >80% | ⚠️ 需自建 |
| **MRR** | 首个命中结果的排名倒数均值 | >0.6 | ⚠️ 需自建 |
| **NDCG** | 归一化折损累计增益 | >0.7 | ⚠️ 需自建 |
| **Hit Rate** | 至少命中1条的比例 | >90% | ⚠️ 需自建 |
| **Latency** | 召回耗时 | <200ms | ✅ 可测 |
| **Coverage** | 有记忆可召回的query比例 | >95% | ⚠️ 需自建 |

### 2.2 各层指标

```
Layer 0 (openviking):
  - 会话召回命中率 = 有历史会话被召回的比例
  - auto-recall 触发次数
  - 召回片段与query相关性评分

Layer 1 (active-memory):
  - 子Agent召回耗时
  - 召回结果注入率 = 成功注入prompt的比例
  - 召回质量评分

Layer 2 (memory-lancedb-pro):
  - 检索耗时（向量+BM25+Rerank分解）
  - 召回结果数分布
  - Smart Extraction 成功率
  - 衰减率 = 单位时间记忆权重下降比例
  - 候选命中率 = candidatePool命中率
  - 无结果率 = 检索返回空的比例

Layer 3 (memory-wiki):
  - wiki检索耗时
  - 命中笔记数分布
```

### 2.3 监控实现方案

#### 方案A: 自建监控（推荐）

在 memory-lancedb-pro 的 retrieval 逻辑中注入监控：

```javascript
// retrieval-monitor.js
class RetrievalMonitor {
  async monitor(query, result, latency) {
    const record = {
      query,
      resultCount: result.length,
      topHit: result[0]?.score || 0,
      latency,
      timestamp: Date.now(),
      hit: await this.checkIfUseful(result, query) // LLM判断
    };
    await this.writeMetrics(record);
  }

  async checkIfUseful(results, query) {
    // LLM 判断召回结果是否真的有用
    const verdict = await llm.invoke({
      prompt: `Query: ${query}\nResults: ${JSON.stringify(results)}\n是否有用?`
    });
    return verdict.includes('有用');
  }
}
```

#### 方案B: 日志分析

通过 OpenClaw 日志分析召回：

```bash
# 提取检索日志
openclaw gateway logs | grep "memory-lancedb-pro.*retrieval" | jq '{
  query: .query,
  resultCount: .results | length,
  latency: .latencyMs,
  topScore: .results[0].score
}'
```

#### 方案C: 人工反馈回路

```
用户说"你没记得吗？"
    │
    ▼
触发 recall_negative_feedback 事件
    │
    ▼
记录:
  - query: "之前说的那个"
  - recallResults: [当时召回了什么]
  - userCorrection: [用户实际指的是什么记忆]
    │
    ▼
分析:
  - 如果 recallResults 为空 → 存储失败
  - 如果结果不对 → embedding/分块/Rerank问题
    │
    ▼
优化对应配置
```

## 4. 各层优化策略

### 4.1 Layer 0 (openviking) 优化

#### 问题: auto-recall 匹配错误会话

```
优化策略:
  1. 会话指纹 + 关键词双重匹配
  2. 时间衰减：近期会话权重更高
  3. 相关性阈值：低于阈值不召回
```

**配置优化**:
```json
{
  "plugins": {
    "entries": {
      "openviking": {
        "config": {
          "autoRecall": {
            "threshold": 0.6,      // 提高阈值
            "maxSessions": 5,        // 限制会话数
            "recencyWeight": 0.3,    // 近期权重
            "keywordBoost": true      // 关键词加权
          }
        }
      }
    }
  }
}
```

#### 问题: Phase2 提取失败

```
优化策略:
  1. 增加 VLM 重试次数
  2. 降低 extractMinMessages（当前=2）
  3. 监控 Phase2 成功率
```

### 4.2 Layer 1 (active-memory) 优化

#### 问题: 子Agent召回质量不稳定

```
优化策略:
  1. 固定召回 prompt 模板
  2. 控制召回数量（不超过3条）
  3. 结果质量评分
```

**配置优化**:
```json
{
  "plugins": {
    "entries": {
      "active-memory": {
        "enabled": true,
        "config": {
          "maxRecallResults": 3,
          "recallThreshold": 0.7,
          "fallbackOnEmpty": true,
          "subagentPromptTemplate": "从记忆库中检索与以下问题相关的记忆，返回最相关的3条：\n{query}\n只返回相关内容，不返回其他。"
        }
      }
    }
  }
}
```

### 4.3 Layer 2 (memory-lancedb-pro) 优化

这是**最核心的优化层**，所有参数都集中在这里。

#### 4.3.1 Embedding 优化

```json
{
  "embedding": {
    "model": "jina-embeddings-v4",  // 升级到图文混合
    "chunking": {
      "strategy": "semantic",        // 语义切分
      "chunkSize": 500,              // 每块字符数
      "overlap": 100                  // 重叠保持上下文
    }
  }
}
```

#### 4.3.2 Hybrid 检索权重优化

```json
{
  "retrieval": {
    "vectorWeight": 0.7,            // 向量权重
    "bm25Weight": 0.3,              // 关键词权重
    "minScore": 0.3,                // 最低分阈值（降低避免过滤好结果）
    "recencyWeight": 0.1,           // 时间权重
    "recencyHalfLifeDays": 30        // 时间衰减半衰期
  }
}
```

**调参建议**:
| 场景 | vectorWeight | bm25Weight | minScore |
|------|-------------|-------------|---------|
| 专业术语多 | 0.5 | 0.5 | 0.25 |
| 口语化query | 0.6 | 0.4 | 0.3 |
| 精确匹配优先 | 0.4 | 0.6 | 0.35 |

#### 4.3.3 Rerank 优化

```json
{
  "retrieval": {
    "rerank": "cross-encoder",
    "rerankModel": "jina-reranker-v3",
    "candidatePoolSize": 50,          // 从20提高到50
    "topK": 10,                     // 最终返回10条
    "rerankWeight": 0.8              // Rerank结果权重
  }
}
```

#### 4.3.4 Smart Extraction 优化

```json
{
  "smartExtraction": {
    "enabled": true,
    "extractMinMessages": 1,          // 从2降到1
    "extractMaxChars": 8000,
    "noiseThreshold": 0.2,             // 提高噪声阈值
    "memoryTypes": ["fact", "preference", "entity", "decision", "reflection", "other"]
  }
}
```

#### 4.3.5 Weibull 衰减优化

```json
{
  "retrieval": {
    "timeDecayHalfLifeDays": 90,      // 从60提高到90
    "reinforcementFactor": 0.7,       // 从0.5提高到0.7（衰减更慢）
    "maxHalfLifeMultiplier": 5            // 从3提高到5
  }
}
```

## 5. 召回优化实践流程

### 5.1 发现问题

```
用户反馈: "你没记得我说过XX吗？"
    │
    ▼
查询 recall_negative_feedback 事件
    │
    ├─→ 有记录但为空 → 存储失败 → 检查Smart Extraction
    ├─→ 有记录但不匹配 → 检索质量问题 → 检查embedding/rerank
    └─→ 无反馈但体感差 → 主动探测 → A/B测试
```

### 5.2 诊断流程

```
Step 1: 检查是否有记忆被存储
  → 查询 memory_chunk_count 指标
  → 如果为0 → Smart Extraction 问题

Step 2: 检查检索是否返回空
  → 查询 recall_empty_rate
  → 如果>30% → 向量质量问题 or BM25问题

Step 3: 检查候选池命中率
  → candidatePool 命中率
  → 如果低 → candidatePoolSize 需要扩大

Step 4: 检查 Rerank 质量
  → 对比 Rerank 前后的 NDCG
  → 如果无改善 → Rerank 模型问题

Step 5: 检查衰减
  → memory_decay_rate
  → 如果过高 → Weibull 参数过激
```

### 5.3 A/B 测试框架

```javascript
// recall-ab-test.js
const variants = [
  {
    name: 'baseline',
    config: { vectorWeight: 0.7, bm25Weight: 0.3 }
  },
  {
    name: 'more-bm25',
    config: { vectorWeight: 0.5, bm25Weight: 0.5 }
  },
  {
    name: 'higher-rerank',
    config: { vectorWeight: 0.6, bm25Weight: 0.4, rerankWeight: 0.9 }
  }
];

async function runTest(variant, queries) {
  const results = [];
  for (const query of queries) {
    const result = await recall(query, variant.config);
    const hit = await evaluateHit(result, query);
    results.push({ variant: variant.name, query, result, hit });
  }
  return aggregateMetrics(results);
}
```

## 6. 召回质量看板（轻量版）

如果不想搭建完整的 Prometheus+Grafana，可以用日志+脚本实现轻量监控：

```bash
#!/bin/bash
# recall-monitor.sh

LOG_FILE="/tmp/recall-monitor.log"
ALERT_THRESHOLD=0.3

# 提取最近1小时的召回空率
EMPTY_RATE=$(grep "memory-lancedb-pro.*retrieval" "$LOG_FILE" \
  | jq 'select(.resultCount == 0)' \
  | wc -l)

TOTAL=$(grep "memory-lancedb-pro.*retrieval" "$LOG_FILE" | wc -l)

RATE=$(echo "scale=2; $EMPTY_RATE / $TOTAL" | bc)

if (( $(echo "$RATE > $ALERT_THRESHOLD" | bc -l) )); then
  echo "⚠️ 召回空率警告: $RATE (阈值: $ALERT_THRESHOLD)"
  # 触发告警
fi
```

## 7. 常见问题

### Q1: recall_hit_rate 如何定义"命中"？
**A**: 需要人工标注或用户反馈。可以先用 LLM 判断："检索结果是否回答了 query？"

### Q2: candidatePoolSize 越大越好吗？
**A**: 不是。越大计算成本越高，建议 30-50 之间，配合 minScore 过滤。

### Q3: 多 Agent 记忆如何共享？
**A**: 写入 shared/knowledge/decisions/ 目录，各 Agent 可读取。或者统一 memory slot 到共享的 lancedb 实例。

### Q4: 监控本身会影响性能吗？
**A**: 会。建议采样监控（10%请求），不影响主流程。
