# OpenViking 完整教程

> OpenViking 是字节跳动火山引擎开源的上下文数据库，采用文件系统范式统一管理 memory、resources、skills。
> 本文档聚焦架构设计、L0/L1/L2 分层原理、Phase2 提取算法，以及与 OpenClaw 的集成。

## 2. 文件系统范式架构

### 2.1 核心概念

OpenViking 将上下文组织为**虚拟文件系统**，让 AI 可以像浏览本地文件一样访问记忆：

```
┌─────────────────────────────────────────────────────────────┐
│                   OpenViking 虚拟文件系统                     │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  /openviking/                                                │
│  ├── memory/                   ← 对话记忆                    │
│  │   ├── sessions/                                          │
│  │   │   ├── session_2026_04_27/                           │
│  │   │   │   ├── L0_raw/        ← 原始消息                  │
│  │   │   │   ├── L1_summary/   ← 第一层摘要                 │
│  │   │   │   └── L2_global/    ← 全局摘要                   │
│  │   │   └── session_2026_04_26/                           │
│  │   └── long_term/            ← 长期记忆                   │
│  │       ├── facts/                                         │
│  │       ├── preferences/                                    │
│  │       └── patterns/                                       │
│  │                                                          │
│  ├── resources/                 ← 外部资源                   │
│  │   ├── documents/                                        │
│  │   └── databases/                                          │
│  │                                                          │
│  └── skills/                   ← 技能定义                   │
│      ├── extraction/                                        │
│      └── reasoning/                                         │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 文件系统 vs 传统 KV 存储

```
┌─────────────────────────────────────────────────────────────┐
│            文件系统范式 vs 传统 KV 存储                       │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  传统 KV 存储:                                               │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  key1 → value1                                       │   │
│  │  key2 → value2                                       │   │
│  │  key3 → value3                                       │   │
│  │  ...                                                 │   │
│  │                                                      │   │
│  │  问题:                                              │   │
│  │  - 所有数据无结构混在一起                            │   │
│  │  - 检索只能暴力扫描                                  │   │
│  │  - 无法按"类型"筛选                                 │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  文件系统范式:                                               │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  memory/sessions/.../L0_raw/msg_001                 │   │
│  │  memory/sessions/.../L1_summary/sum_001            │   │
│  │  memory/long_term/facts/user_location               │   │
│  │  resources/documents/paper_001.pdf                  │   │
│  │                                                      │   │
│  │  优势:                                              │   │
│  │  - 自然分层结构                                      │   │
│  │  - 可用路径前缀筛选                                  │   │
│  │  - 目录即分类                                        │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

```
│      │                                                      │

│      📝 包含: 创建时间、消息数、token 统计等                 │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 3. L0/L1/L2 三层架构

### 3.1 三层概述

```
┌─────────────────────────────────────────────────────────────┐
│                   L0/L1/L2 三层架构                          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│                        用户消息                               │
│                           │                                 │
│                           ▼                                 │
│  ┌───────────────────────────────────────────────────────┐  │
│  │                    L0: 原始层                          │  │
│  │  ┌─────────────────────────────────────────────────┐  │  │
│  │  │  msg_001: "用户: 帮我点个外卖"                   │  │  │
│  │  │  msg_002: "助手: 好的，想吃什么？"               │  │  │
│  │  │  msg_003: "用户: 麻辣烫"                        │  │  │
│  │  │  ...                                            │  │  │
│  │  └─────────────────────────────────────────────────┘  │  │
│  │  特点: 完整保留，可展开                               │  │
│  └───────────────────────────────────────────────────────┘  │
│                           │                                 │
│                           │ 触发条件:                       │
│                           │ - L0 超过阈值                   │
│                           │ - 时间到达                      │
│                           ▼                                 │
│  ┌───────────────────────────────────────────────────────┐  │
│  │                   L1: 摘要层                          │  │
│  │  ┌─────────────────────────────────────────────────┐  │  │
│  │  │  sum_001: "用户要点外卖，已确认麻辣烫"           │  │  │
│  │  │  sum_002: "用户询问天气"                         │  │  │
│  │  └─────────────────────────────────────────────────┘  │  │
│  │  特点: 压缩率 ~80%，保留关键信息                     │  │
│  └───────────────────────────────────────────────────────┘  │
│                           │                                 │
│                           │ 触发条件:                       │
│                           │ - L1 积累足够多                 │
│                           │ - 需要全局理解                  │
│                           ▼                                 │
│  ┌───────────────────────────────────────────────────────┐  │
│  │                   L2: 全局层                           │  │
│  │  ┌─────────────────────────────────────────────────┐  │  │
│  │  │  global: "今日用户主要讨论: 外卖、天气、工作"     │  │  │
│  │  └─────────────────────────────────────────────────┘  │  │
│  │  特点: 会话级别摘要，跨 L1 聚合                      │  │
│  └───────────────────────────────────────────────────────┘  │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 各层详细说明

```
┌─────────────────────────────────────────────────────────────┐
│                   各层详细对比                               │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  L0: 原始层 (Raw Layer)                            │   │
│  │                                                      │   │
│  │  内容: 原始消息，完整保留                            │   │
│  │  格式: msg_{timestamp}_{role}.msg                   │   │
│  │  触发: 新消息直接写入                                │   │
│  │  保留: 始终保留，不删除                             │   │
│  │  检索: 可通过路径直接访问                            │   │
│  │                                                      │   │
│  │  示例:                                              │   │
│  │  /openviking/memory/sessions/s_001/L0_raw/         │   │
│  │    msg_1714204800_user.msg                          │   │
│  │    msg_1714204860_assistant.msg                     │   │
│  │                                                      │   │
│  │  替代: 无（核心数据，必须保留）                       │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  L1: 摘要层 (Summary Layer)                         │   │
│  │                                                      │   │
│  │  内容: 片段摘要，每 N 条消息生成一个                 │   │
│  │  格式: chunk_{index}_summary.summary                 │   │
│  │  触发: L0 消息数达到阈值（默认 20 条）              │   │
│  │  模型: 可配置（默认 NVIDIA VLM）                    │   │
│  │  压缩率: ~80%（20 条 → 4 条摘要）                   │   │
│  │                                                      │   │
│  │  示例:                                              │   │
│  │  /openviking/memory/sessions/s_001/L1_summary/      │   │
│  │    chunk_001.summary  ← msg_001-020 的摘要          │   │
│  │    chunk_002.summary  ← msg_021-040 的摘要         │   │
│  │                                                      │   │
│  │  触发条件:                                          │   │
│  │  - 自动: L0 达到 20 条                              │   │
│  │  - 手动: 调用 compress API                         │   │
│  │                                                      │   │
│  │  替代: 无（核心分层机制）                            │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  L2: 全局层 (Global Layer)                           │   │
│  │                                                      │   │
│  │  内容: 会话全局摘要，跨 L1 聚合                       │   │
│  │  格式: session_{id}_global.summary                   │   │
│  │  触发: 会话结束时 / 手动触发                         │   │
│  │  用途: 跨会话理解、快速上下文重建                    │   │
│  │                                                      │   │
│  │  示例:                                              │   │
│  │  /openviking/memory/sessions/s_001/L2_global/        │   │
│  │    session_s001_global.summary                       │   │
│  │                                                      │   │
│  │  用途场景:                                          │   │
│  │  - 新会话开始时快速了解用户                         │   │
│  │  - 跨会话追踪用户偏好                                │   │
│  │  - 长期记忆整合                                      │   │
│  │                                                      │   │
│  │  替代: 无（最高层抽象）                              │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 3.3 分层加载流程

```
┌─────────────────────────────────────────────────────────────┐
│                   分层加载流程                               │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  用户查询: "我上次点什么外卖？"                                │
│                                                              │
│  Step 1: L2 全局搜索                                         │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  搜索范围: L2_global/                               │   │
│  │  结果: session_s001_global.summary                   │   │
│  │  内容: "今日用户主要讨论: 外卖、天气、工作"           │   │
│  │                                                      │   │
│  │  匹配度: 中等                                       │   │
│  │  原因: L2 只有全局摘要，不够详细                     │   │
│  └─────────────────────────────────────────────────────┘   │
│                    │                                        │
│                    ▼ 匹配度不够，继续                        │
│                                                              │
│  Step 2: L1 摘要搜索                                         │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  搜索范围: L1_summary/                               │   │
│  │  结果: chunk_003.summary                             │   │
│  │  内容: "用户要点外卖，点了麻辣烫"                    │   │
│  │                                                      │   │
│  │  匹配度: 高                                         │   │
│  │  原因: L1 有片段级摘要                               │   │
│  └─────────────────────────────────────────────────────┘   │
│                    │                                        │
│                    ▼ 匹配度足够，停止                        │
│                                                              │
│  Step 3: 返回结果                                            │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  返回: L1 摘要内容                                   │   │
│  │  "用户要点外卖，点了麻辣烫"                          │   │
│  │                                                      │   │
│  │  如果需要更多细节:                                   │   │
│  │  可进一步展开 L0 原始消息                            │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  🔑 优化效果:                                               │
│  - 不需要加载所有 L0 原始消息                                │
│  - token 消耗大幅减少                                       │
│  - 响应速度提升                                             │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

```
│  │  频繁访问的 L1/L2 缓存在内存                        │   │
│  │  L0 保持磁盘存储                                    │   │
│  │                                                      │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

## 4. Phase2 提取算法

### 4.1 Phase2 提取概述

Phase2 提取是 OpenViking 的**自动知识提取**功能，从对话中自动识别和提取结构化信息：

```
┌─────────────────────────────────────────────────────────────┐
│                   Phase2 提取流程                           │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  输入: 对话消息列表                                          │
│                                                              │
│  Phase 1: 消息处理 (已在 L0 完成)                            │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  - 消息解析                                         │   │
│  │  - 时间戳记录                                       │   │
│  │  - 角色识别 (user/assistant/system)                │   │
│  └─────────────────────────────────────────────────────┘   │
│                    │                                        │
│                    ▼                                        │
│  Phase 2: 结构化提取 ⭐                                      │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                      │   │
│  │  2.1 候选识别                                        │   │
│  │      LLM 分析对话，识别值得记住的内容                 │   │
│  │                    │                                │   │
│  │                    ▼                                │   │
│  │  2.2 类型分类                                        │   │
│  │      判断是哪种类型的知识                           │   │
│  │      - fact (事实)                                  │   │
│  │      - preference (偏好)                            │   │
│  │      - pattern (模式)                               │   │
│  │      - decision (决策)                             │   │
│  │      - entity (实体)                               │   │
```

### 4.3 置信度评分

```
┌─────────────────────────────────────────────────────────────┐
│                   置信度评分机制                             │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  评分维度:                                                   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                      │   │
│  │  清晰度 (clarity)                                    │   │
│  │  - 内容是否明确，无歧义                              │   │
│  │  - 权重: 30%                                        │   │
│  │                                                      │   │
│  │  一致性 (consistency)                                │   │
│  │  - 与已知信息是否一致                                │   │
│  │  - 权重: 25%                                        │   │
│  │                                                      │   │
│  │  频率 (frequency)                                   │   │
│  │  - 同样的信息出现了多少次                            │   │
│  │  - 权重: 20%                                        │   │
│  │                                                      │   │
│  │  来源可靠性 (source_reliability)                    │   │
│  │  - 用户直接说的 > 推测的                             │   │
│  │  - 权重: 25%                                        │   │
│  │                                                      │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  评分公式:                                                   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                                                      │   │
│  │  score = 0.3×clarity + 0.25×consistency              │   │
│  │        + 0.2×frequency + 0.25×source                │   │
│  │                                                      │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
│  评分阈值:                                                   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  score >= 0.8 → 立即写入 long_term                  │   │
│  │  score >= 0.5 → pending 队列，等待强化               │   │
│  │  score < 0.5 → 丢弃                                 │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```
