## 二、记忆分层架构

### 2.1 四层架构

```
Layer0 (contextEngine)
  └─ openviking: 对话压缩 + 上下文组装 + Phase2 提取

Layer1 (before_agent_start)
  └─ active-memory: LLM 前记忆注入子Agent

Layer2 (memory slot)
  └─ memory-lancedb-pro: 混合检索 + Rerank + 智能提取 + 自优化

Layer3 (wiki slot)
  └─ memory-wiki (Obsidian vault): 本地知识库持久化
```

### 2.2 各层职责

| 层 | 模块 | 职责 | 数据流 |
|---|------|------|--------|
| L0 | openviking | 对话压缩，上下文组装，Phase2提取 | 输入原始对话 → 输出压缩上下文 |
| L1 | active-memory | 记忆检索与注入 | LanceDB → relevant-memories 块 |
| L2 | memory-lancedb-pro | 向量存储，混合检索，生命周期管理 | 写入/检索 LanceDB |
| L3 | memory-wiki | Obsidian 本地知识库 | 归档长期知识 |

### 2.3 生命周期（Tier 系统）

```
Peripheral ⟷ Working ⟷ Core

- Peripheral: 低频/低重要性，自动衰减
- Working: 中等访问频率，稳定运行
- Core: 高频/高重要性，长久保留

触发条件：
- 访问频率高 → 晋升 Core
- 长时间无访问 → 衰减 → 降级 Peripheral
- 重要性高 → 衰减更慢（reinforcement）
```

### 2.4 记忆分类（6类别）

| `profile` | Agent/用户身份特征 | 身份、角色、能力边界 |
| `preferences` | 用户偏好和行为习惯 | 喜欢麻辣、优先私聊 |
| `entities` | 实体和对象 | 特定人名、公司、产品 |

### 3.2 多 Agent 记忆隔离

## 四、记忆治理规则

### 4.1 通用治理规则

**写入时过滤（Noise Filter）：**
- Agent 拒绝回复、Meta 问题、寒暄不写入
- 过短内容（<10 chars）不写入
- 重复内容不写入

**检索时过滤（Retrieval）：**
- `minScore: 0.3` 硬截断
- `filterNoise: true` 过滤低质量结果
- BM25 weight: 0.3（轻量文本检索权重）

**生命周期维护：**
- Weibull 衰减：复合分数 = 时效 × 频率 × 重要性
- `timeDecayHalfLifeDays: 60`（默认60天半衰）
- `recencyWeight: 0.1`（近期偏差权重）

### 4.2 删除/清理规则

**什么时候需要主动清理：**

| 情况 | 处理方式 |
|------|----------|
| 记忆内容错误/过时 | `memory_forget` 或 LanceDB 直接删除 |
| 噪音记忆（系统元数据误写入） | 直接删除 |
| 大量积压（>100条） | `memory-pro upgrade` 批量处理 |
| 全部清空 | LanceDB 数据文件删除 + 重启 |

**删除方法优先级：**

1. **Tool 级别**：`memory_forget(memoryId="xxx")`
   - 适用于单条删除，需要知道 ID
   
2. **CLI 级别**：`openclaw memory-pro delete <id>`
   - 适用于批量删除
   
3. **Python 级别**：直接操作 LanceDB
   - 适用于批量清理、强制删除
   ```python
   import lancedb
   db = lancedb.connect('~/.openclaw/memory/lancedb-pro')
   table = db.open_table('memories')
   table.delete('id = "xxx"')  # 单条
   table.delete('category = "entity"')  # 批量
   ```

4. **物理删除**：删除整个 LanceDB 目录
   - 适用于完全重建
   - ⚠️ 会丢失所有记忆，需谨慎

### 4.3 治理 Cron 任务

| Cron | 时间 | 功能 |
|------|------|------|
| 墨自-每日整理 | 21:00 | 清理 corrections.md、整理 MEMORY.md |
| 墨自-自我监控 | 09:00, 18:00 | 检查 self-improving 系统 |
| 墨家-自我改进维护 | 09:00 | 评估 pattern 提升、更新 heartbeat-state |

### 4.4 记忆与 corrections.md 的关系

```
用户纠正 → corrections.md → 【≥3次相同type】→ MEMORY.md
                              ↓
                         持续应用
```

**corrections.md 只记录真正的学习内容：**
- ✅ 用户偏好（"不要先问"）
- ✅ 行为规则（"搜索优先级"）

## 六、问题记忆诊断与处理

### 6.1 常见问题

| 问题 | 症状 | 处理方式 |
|------|------|----------|
| relevant-memories 注入了无关记忆 | 聊 Cron 出现 exec SIGKILL 旧账 | 删除 LanceDB 中对应条目 |
| corrections.md 过于臃肿 | 大量系统元数据混入 | 清理 + 保留真正偏好 |
| 记忆检索返回错误条目 | 语义匹配不精准 | 提高 minScore / filterNoise |
| LanceDB 数据损坏 | 查询报错 "all columns must have same length" | `table.optimize()` 或重建 |
| 记忆持久化失效 | 新 session 无法召回旧记忆 | 检查 LanceDB 目录权限 |

### 6.2 relevant-memories 触发机制

**注入时机：** SessionStart 时
**注入内容：** 语义检索匹配的记忆（最多 3 条）
**匹配方式：** 向量相似度 + BM25 + Rerank

**为什么会注入无关记忆：**
- 向量检索只做相似度匹配，不理解"有没有用"
- 噪音记忆（如 exec SIGKILL 系统日志）被错误分类为 `entity` 而被注入

**解决方案：**
1. 删除问题记忆（LanceDB 直接删除）
2. 提高 `minScore` 阈值（减少噪音）
3. 清理 corrections.md 和 daily notes 中的系统元数据

### 6.3 记忆治理检查清单

```
□ 检查 LanceDB 是否有过期/错误记忆
□ 检查 corrections.md 是否混入系统元数据
□ 检查 daily notes 是否包含无关 exec 日志
□ 检查各 Agent memory 目录是否正常写入
□ 检查 cron daily-log 是否正常（避免 no route）
□ 检查 relevant-memories 注入是否正常
```

## 八、总结

**记忆系统是一把双刃剑：**
- ✅ 让 Agent 跨越 session 保持连续性
- ✅ 自动积累经验（corrections → patterns → MEMORY.md）
- ⚠️ 噪音记忆会污染 relevant-memories
- ⚠️ 系统元数据误写入会混淆检索

**治理核心原则：**
1. **精准优于多多**：少而准的记忆 > 多而噪的记忆
2. **分类清晰**：preference/fact/decision/entity 分开存储
3. **自动 + 手动**：自动生命周期 + 手动纠正
4. **scope 隔离**：私有记忆不泄露，共享记忆明确边界

**维护节奏：**
- 每日：检查 relevant-memories 是否正常
- 每周：检查 corrections.md 是否需要整理
- 每月：检查 LanceDB 是否有过期记忆需要清理
