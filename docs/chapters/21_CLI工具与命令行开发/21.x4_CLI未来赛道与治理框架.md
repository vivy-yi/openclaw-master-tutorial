## 21.x4.2 CLI 治理思维框架

### 什么是 CLI 治理？

CLI 治理 = **在多领域、多 Agent、多任务场景下，对 CLI 工具的：访问控制、操作审计、编排协同、质量保证、策略执行，进行系统化管理。**

```
无治理的 CLI 生态：
  Agent A → cli_A（随便用）
  Agent B → cli_B（随便用）
  问题：数据泄露、权限混乱、操作不可追溯

有治理的 CLI 生态：
  ┌─────────────────────────────────────────┐
  │              治理层（Governance）         │
  │  ┌─────────┐  ┌──────────┐  ┌────────┐  │
  │  │访问控制  │  │操作审计   │  │策略引擎│  │
  │  └─────────┘  └──────────┘  └────────┘  │
  │  ┌─────────┐  ┌──────────┐  ┌────────┐  │
  │  │数据隔离  │  │编排协调   │  │质量管控│  │
  │  └─────────┘  └──────────┘  └────────┘  │
  └─────────────────────────────────────────┘
          ↑            ↑            ↑
    ┌─────┴────┐ ┌─────┴────┐ ┌─────┴────┐
    │ Agent A  │ │ Agent B  │ │ Agent C  │
    └─────┬────┘ └─────┬────┘ └─────┬────┘
          ↓            ↓            ↓
    ┌─────────────────────────────────┐
    │         工具舱（Tool Arsenal）    │
    │  金融CLI  法律CLI  生活CLI  运营CLI │
    └─────────────────────────────────┘
```

### CLI 治理四维度

| 维度 | 问题 | 解法 |
|------|------|------|
| **访问控制** | 谁可以用哪个 CLI？数据权限怎么划？ | 身份认证 + 范围限定 |
| **操作审计** | 哪个 Agent 什么时候执行了什么？ | 日志 + 可信执行记录 |
| **编排协同** | 多 CLI 怎么协同？依赖关系怎么处理？ | 工作流引擎 + DAG 编排 |
| **策略合规** | 什么能执行？什么不能执行？ | 白名单/黑名单 + 策略引擎 |

## 21.x4.3 案例：金融助手 CLI 治理体系（全流程详解）

> 这是一个**贯穿全节**的案例：从需求到架构到执行，展示治理思维如何让金融领域 CLI 真正发挥作用。

### 第一步：金融助手需要哪些 CLI 工具？

```
金融助手工具舱：
├── 📰 市场数据层
│   ├── tushare / akshare    → A股/宏观数据
│   ├── yfinance (Yahoo Finance) → 美股数据
│   ├── eastmoney-finance-news → 财经新闻
│   └── xurl / mcporter      → 实时新闻聚合
│
├── 📊 分析建模层
│   ├── quantitatif-factors  → 多因子模型
│   ├── portfolio-optimizer   → 组合优化
│   ├── stock-analysis        → 个股/指数分析
│   └── macro-strategy        → 宏观策略
│
├── ⚠️ 风控合规层
│   ├── risk-management       → 风险指标计算
│   ├── insurance-analysis    → 保险配置分析
│   ├── tax-optimization      → 税务优化
│   └── compliance-framework  → 合规框架
│
├── 📋 报告生成层
│   ├── report-generation     → 自动化报告
│   ├── pdf generation        → PDF 导出
│   └── notify (OpenClaw)     → 推送通知
│
└── 📁 记录管理层
    ├── memory (OpenClaw)     → 记忆存储
    ├── notion / obsidian     → 笔记归档
    └── MEMORY.md            → 长期记忆
```

### 第二步：定义治理策略（Policy Layer）

```
金融助手策略配置（policy.json）：
{
  "access_control": {
    "mo-finance": {
      "allowed_tools": ["tushare", "yfinance", "eastmoney", "xurl", "mcporter"],
      "allowed_scope": ["public_data", "user_portfolio"],
      "denied_operations": ["withdraw", "trade_execution", "transfer"]
    },
    "mo-law": {
      "allowed_tools": ["compliance-framework", "law-investment"],
      "allowed_scope": ["regulation_check"]
    }
  },
  "audit": {
    "log_all_executions": true,
    "require_approval_for": ["trade_execution", "transfer", "withdraw"],
    "retention_days": 90
  },
  "data_governance": {
    "pii_fields": ["account_id", "balance", "transaction_history"],
    "mask_in_logs": true,
    "cross_border_check": true
  }
}
```

### 第三步：编排工作流（Orchestration Layer）

**场景：用户问"帮我分析一下腾讯目前的投资价值"**

```
用户输入（自然语言）
        ↓
   AI 意图解析
        ↓
   任务分解（DAG）：
  ┌────────────┐
  │ 获取股价   │ ← tushare (A股数据)
  │ 获取财务   │ ← tushare (财务报表)
  │ 获取新闻   │ ← eastmoney + xurl
  │ 行业对比   │ ← stock-analysis
  │ 宏观环境   │ ← macro-strategy
  │ 风险评估   │ ← risk-management
  │ 估值建模   │ ← quantitative-factors
  │ 生成报告   │ ← report + notify
  └────────────┘
        ↓
   并行执行（无依赖）：
   tushare + eastmoney + xurl 可同时执行
        ↓
   依赖收敛（等待上游完成）：
   stock-analysis 等 tushare 结果
   macro-strategy 等 xurl 结果
        ↓
   综合分析（合并结果）
        ↓
   生成报告 → 推送通知 → 记忆存档
```

### 第四步：访问控制执行

```
场景：mo-finance Agent 想执行「卖出股票」操作

  mo-finance.exec("stock sell --symbol 00700 --quantity 1000")
                    ↓
          治理层拦截检查
                    ↓
    ┌─ denied_operations 检查 ─┐
    │ "sell" in denied_operations  → TRUE
    └──────────────────────────────┘
                    ↓
          返回：操作被拒绝 ❌
    原因：该操作需要二次审批
    提示：请通过合规流程申请
```

### 第五步：操作审计记录

```
每次 CLI 执行，自动写入审计日志：

{
  "timestamp": "2026-04-20T18:30:00+08:00",
  "agent": "mo-finance",
  "tool": "tushare",
  "command": "tushare get_financials --symbol 00700 --period annual",
  "data_accessed": ["stock_price", "financial_statements"],
  "result": "success",
  "duration_ms": 234,
  "pii_exposed": false,
  "compliance_check": "passed"
}
```

### 第六步：多 Agent 协作（跨领域）

```
场景：持仓中出现「北新建材」，需要税务优化建议

  mo-finance → 检测到持仓变化
        ↓
  触发跨 Agent 协作：
  ┌──────────────────────────────────┐
  │  mo-finance（发起）              │
  │  → mo-law（tax-optimization）    │
  │  携带数据：持仓明细、买入成本      │
  └──────────────────────────────────┘
        ↓
  mo-law 分析：
  计算：利得税、持有期节税建议
        ↓
  mo-law → mo-finance（返回建议）
  建议：持有满1年可节税 XX元
        ↓
  mo-finance 更新 MEMORY.md
  记录：北新建材建议持有至 [具体日期]
```

## 21.x4.4 治理框架落地：OpenClaw 中的实现

> 以下是墨家 Agent 系统中 CLI 治理的实际落地方式。

### 1. 访问控制（OpenClaw Exec Approvals）

```bash
# 设置 mo-finance 的 exec 白名单
openclaw approvals allowlist mo-finance --tools "tushare,yfinance,xurl,mcporter,stock-analysis"

# 审批高风险操作
openclaw approvals set policy.json
```

### 2. 操作审计（OpenClaw Logs）

```bash
# 查看 mo-finance 的执行历史
openclaw exec-logs --agent mo-finance --since 24h

# 查看所有被拒绝的操作
openclaw exec-logs --status denied --since 7d
```

### 3. 编排协同（OpenClaw Cron + Sessions）

```bash
# 每日持仓分析（Cron 触发）
cron add \
  --schedule "0 9 * * 1-5" \
  --agent mo-finance \
  --task "执行持仓分析工作流"

# 跨 Agent 协作（sessions_send）
sessions_send mo-finance "请调用 mo-law 进行税务分析"
```

### 4. 数据治理（OpenClaw Memory 隔离）

```
墨家各 Agent 的 memory 目录天然隔离：
  mo-finance/  → 持仓数据、分析结果
  mo-law/      → 合规记录、法律意见
  mo-yunying/  → 内容运营数据
  跨域访问 → 必须通过 shared/knowledge/ 显式共享
```

## 21.x4.6 各领域 CLI 治理特征对比

| 维度 | 金融 | 法律 | 内容运营 | 研发 |
|------|------|------|---------|------|
| **核心诉求** | 合规 + 风控 | 准确性 + 隐私 | 效率 + 时效 | 质量 + 安全 |
| **审计要求** | ⭐⭐⭐ 极高 | ⭐⭐⭐ 极高 | ⭐⭐ 中 | ⭐⭐⭐ 高 |
| **访问控制** | 持仓/账户级 | 案件级 | 平台级 | 仓库级 |
| **高危操作** | 交易/转账/提现 | 删除证据/泄露 | 发布/删除内容 | rm/force push |
| **审批流** | 多级审批 | 律师确认 | 运营确认 | PR Review |
| **数据隔离** | 账户数据 PII | 案件隐私 | 竞品数据 | 源码 |
| **报告需求** | 每日风控报告 | 法律意见书 | 运营日报/周报 | 代码质量报告 |
