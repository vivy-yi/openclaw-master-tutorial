## 零、架构总览：两种能力，一个系统

```
┌─────────────────────────────────────────────────────────────────────┐
│                         OpenClaw 主控系统                              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  ┌─────────────────┐          ┌──────────────────────────────────┐ │
│  │   Think 模式      │          │        ACP Harness 系统            │ │
│  │   控制自身推理深度  │          │   连接外部编码 Agent              │ │
│  └─────────────────┘          └──────────────────────────────────┘ │
│                                                                      │
│  • 推理开关（on/off）      │  • ACP Router 自动路由               │
```

**本质区别：**
- **Think 模式** = 控制 OpenClaw 自己的思考方式

两者可以同时使用：开启 Think 模式的同时，通过 ACP 调用外部 Harness。

## 一、Think 模式：控制 OpenClaw 的推理深度

### 1.1 什么是 Think 模式

Think 模式是 OpenClaw 内置的**推理思考开关**，让模型在回答前进行更深入的多步思考。

```
Think OFF（默认）：
  用户问题 → 直接回答 → 输出

Think ON：
  用户问题 → 推理步骤 → 推理步骤 → 推理步骤 → 最终回答 → 输出
```

### 1.2 如何控制

| 操作 | 命令/方式 |
|------|---------|
| **开启** | `/reasoning` 或 `/think on` |
| **关闭** | `/reasoning off` 或 `/think off` |
| **查看状态** | `/status` |
| **当前状态** | 右上角显示 `🧠 Think: on/off` |

**运行时参数**（高级）：

```bash
# 通过 cron 创建带 thinking 的任务
{
  "payload": {
    "kind": "agentTurn",
    "message": "分析这段代码",
    "thinking": "on"       # 开启深度推理
  }
}

# sessions_spawn 时指定
{
  "task": "重构这段代码",
  "runtime": "subagent",
  "thinking": "verbose"   # verbose 比 on 更详细
}
```

### 1.3 三种推理级别

| 级别 | 参数值 | 说明 |
|------|--------|------|
| **关闭** | `off` | 无额外推理，直接回答 |
| **开启** | `on` | 基础多步推理 |
| **详细** | `verbose` | 最详细的推理过程 |

### 1.4 适用场景

| 场景 | 推荐设置 | 原因 |
|------|---------|------|
| 简单问答/日常对话 | Think OFF | 快速响应，无需推理 |
| 代码调试/复杂逻辑 | Think ON | 需要多步推理 |
| 架构设计/方案分析 | Think ON | 深度思考避免遗漏 |
| 写作/创意内容 | Think OFF | 模型自身能力足够 |
| 长文本总结 | Think ON | 确保总结全面 |

### 1.5 与模型自带推理的区别

| 维度 | OpenClaw Think 模式 | 模型内置推理 |
|------|--------------------|-------------|
| **控制方式** | OpenClaw 层开关 | 模型自身能力 |
| **生效范围** | 当前 OpenClaw 会话 | 取决于模型 |
| **兼容性** | 通用 | 仅部分模型支持 |

> 注意：部分模型（如 GPT-4o）自带 CoT 推理，不受 OpenClaw Think 开关影响。

## 二、ACP Harness 系统：让外部编码 Agent 为你工作

### 2.1 什么是 ACP Harness

**ACP（Agent Client Protocol）** 是一种让 OpenClaw 与外部编码 Agent 对话的协议。

外部 Agent（统称 **Harness）** 是专门的编码助手，通过 ACP 接入 OpenClaw 后，可以：
- 接收 OpenClaw 转发的任务
- 在持久化线程中保持上下文
- 利用各自特色能力完成任务

```
┌──────────────┐      ACP 协议       ┌─────────────────┐
│   OpenClaw   │ ←────────────────→  │  Claude Code    │
│   （主控）   │                     │  （Harness）    │
│              │                     │                 │
│  • 协调任务   │                     │  • 专业编码      │
│  • 聚合结果   │                     │  • 深度推理      │
│  • 展示输出   │                     │  • 工具生态      │
└──────────────┘                     └─────────────────┘
```

### 2.2 支持的 Harness 列表

| Harness | Agent ID | 特色能力 | 安装方式 |
|---------|----------|---------|---------|
| **Claude Code** | `claude` | 官方 Claude Agent，深度推理 | `npx -y @zed-industries/claude-agent-acp@0.21.0` |
| **Codex** | `codex` | OpenAI 编码专家 | `npx @zed-industries/codex-acp@^0.9.5` |
| **Cursor Agent** | `cursor` | Cursor IDE 集成 | `cursor-agent acp` |
| **Copilot** | `copilot` | GitHub 代码补全 | `copilot --acp --stdio` |
| **Pi** | `pi` | 友好对话型编码助手 | `npx pi-acp@^0.0.22` |
| **Gemini CLI** | `gemini` | Google Gemini 编码 | `gemini --acp` |
| **Qwen Code** | `qwen` | 阿里通义编码 | `qwen --acp` |
| **Kiro CLI** | `kiro` | 国产 AI 编码 | `kiro-cli acp` |
| **Kimi CLI** | `kimi` | 月之暗面编码 | `kimi acp` |
| **Factory Droid** | `droid` | 企业级自动化 | `droid exec --output-format acp` |
| **iFlow** | `iflow` | 实验性国产 Agent | `iflow --experimental-acp` |
| **OpenCode** | `opencode` | 开源编码 Agent | `npx -y opencode-ai acp` |
| **KiloCode** | `kilocode` | 轻量编码 | `npx -y @kilocode/cli acp` |
| **OpenClaw ACP** | `openclaw` | OpenClaw 自身（自循环）| `openclaw acp` |

### 2.3 核心概念：两种调用路径

OpenClaw 调用 Harness 有**两条路径**：

| 路径 | 说明 | 何时使用 |
|------|------|---------|
| **OpenClaw ACP Runtime** | 通过 `sessions_spawn` 管理线程生命周期 | 默认首选，持久化对话 |
| **Direct acpx** | 直接执行 `acpx` CLI 命令 | ACP 不可用 / 单次任务 |

**推荐原则**：
- 优先使用 **ACP Runtime**（会话管理更完善）
- 备用 **Direct acpx**（ACP 不可用时的降级方案）

## 三、acp-router：自动路由技能

### 3.1 什么是 acp-router

`acp-router` 是 OpenClaw 内置的 **Skill**，负责：
1. 检测用户是否想使用外部 Harness
2. 选择合适的调用路径
3. 路由到对应 Agent

它**自动触发**，不需要用户手动调用。

### 3.2 触发条件

当用户说（或类似）这些时，自动触发：

| 用户说法 | 触发行为 |
|---------|---------|
| "用 Claude Code 帮我..." | 路由到 Claude Code |
| "spawn 一个 codex 线程" | 创建 Codex 持久化会话 |
| "继续之前的 harness 工作" | 恢复已有会话 |
| "用 cursor 重构这段代码" | 路由到 Cursor Agent |
| "relay 到 gemini" | 中转指令到 Gemini |

### 3.3 路由决策流程

```
用户请求
    │
    ▼
┌───────────────────────┐
│  acp-router 拦截      │
│  检测 Harness 意图    │
└───────────────────────┘
    │
    ▼
┌───────────────────────┐
│  判断路径              │
│                       │
│  有 ACP 运行时 ──→ 路径1│
│  (默认优先)            │
│                       │
│  ACP 不可用 ────→ 路径2│
│  (降级方案)            │
└───────────────────────┘
```

## 四、使用详解：ACP Runtime 路径

### 4.1 基本调用

**通过 sessions_spawn（推荐）：**

```json
{
  "task": "重构这段代码，使其更易读",
  "runtime": "acp",
  "agentId": "claude",
  "thread": true,
  "mode": "session"
}
```

**等效的用户说法：**
> "用 Claude Code 重构这段代码"

### 4.2 线程管理

ACP Runtime 支持**持久化线程会话**：

| 操作 | 说明 |
|------|------|
| **创建线程** | 首次调用时自动创建 |
| **继续对话** | 后续调用使用相同 `sessionKey` |
| **线程命名** | 格式：`oc-<harness>-<conversationId>` |

```json
// 继续已有线程
{
  "task": "再添加单元测试",
  "runtime": "acp",
  "agentId": "claude",
  "thread": true,
  "mode": "session"
}
```

### 4.3 Agent ID 映射速查

| 用户说 | agentId 参数 |
|--------|------------|
| Claude Code / claude | `claude` |
| Codex / codex | `codex` |
| Cursor | `cursor` |
| Copilot / GitHub Copilot | `copilot` |
| Pi | `pi` |
| Gemini | `gemini` |
| Qwen | `qwen` |
| Kiro | `kiro` |
| Kimi | `kimi` |
| OpenClaw | `openclaw` |

### 4.4 完整示例

**场景：让 Claude Code 审查代码**

```
用户："用 Claude Code 帮我审查 src/auth/login.ts"
    ↓
OpenClaw 检测到 harness 请求
    ↓
acp-router 路由到 Claude Code ACP Runtime
    ↓
sessions_spawn 创建 Claude Code 会话
    ↓
Claude Code 执行代码审查
    ↓
结果返回 OpenClaw，展示给用户
```

## 五、使用详解：Direct acpx 路径

### 5.1 何时使用

当以下情况时，使用 Direct acpx：

- ACP Runtime 不可用或出错
- 用户明确要求直接执行
- 单次快速任务，不需要会话保持

### 5.2 基本命令

**持久化会话（创建 + 发送）：**

```bash
# 创建会话（如不存在）并发送任务
acpx codex sessions show oc-codex-<convId> \
  || acpx codex sessions new --name oc-codex-<convId>

acpx codex -s oc-codex-<convId> \
  --cwd /path/to/workspace \
  --format quiet "重构 auth/login.ts 使其更易读"
```

**一次性执行：**

```bash
acpx codex exec \
  --cwd /path/to/workspace \
  --format quiet "重构 auth/login.ts 使其更易读"
```

**取消进行中的任务：**

```bash
acpx codex cancel -s oc-codex-<convId>
```

**关闭会话：**

```bash
acpx codex sessions close oc-codex-<convId>
```

### 5.3 acpx 命令速查

| 命令 | 用途 |
|------|------|
| `acpx <agent> sessions new --name <name>` | 创建新会话 |
| `acpx <agent> sessions show <name>` | 查看会话 |
| `acpx <agent> sessions list` | 列出所有会话 |
| `acpx <agent> sessions close <name>` | 关闭会话 |
| `acpx <agent> -s <name> -- <prompt>` | 发送消息 |
| `acpx <agent> exec -- <prompt>` | 一次性执行 |
| `acpx <agent> cancel -s <name>` | 取消任务 |

## 六、Think + ACP 组合使用

### 6.1 何时组合

在以下场景，组合使用效果最佳：

| 场景 | Think 模式 | ACP Harness | 效果 |
|------|-----------|-------------|------|
| 复杂代码架构设计 | Think ON | Claude Code | OpenClaw 深度分析 → 交给 Claude Code 执行 |
| 调试多步骤 Bug | Think ON | Codex | OpenClaw 定位 → Codex 修改 |
| 快速代码片段 | OFF | Claude Code | 直接交给 Harness |
| 需要外部工具的编码 | ON | Cursor | OpenClaw 协调 → Cursor 执行 |

### 6.2 组合示例

**场景：深度代码重构**

```
用户："深度重构 src/core/ 目录，用 Claude Code，需要先分析依赖关系"

    ① OpenClaw Think ON
       → 分析模块依赖，确定重构顺序

    ② OpenClaw 通过 ACP Runtime 调用 Claude Code
       → Claude Code 执行重构

    ③ OpenClaw 聚合结果，展示给用户
```

```json
{
  "task": "重构 src/core/ 目录，先分析依赖关系，再按顺序重构",
  "runtime": "acp",
  "agentId": "claude",
  "thread": true,
  "mode": "session",
  "model": "claude-sonnet-4-20250514"
}
```

## 七、Think 模式与 ACP 的核心区别

| 维度 | Think 模式 | ACP Harness |
|------|-----------|-------------|
| **控制对象** | OpenClaw 自身 | 外部 Agent |
| **能力来源** | OpenClaw 的模型 | Harness 的模型 |
| **典型用途** | 深度分析、协调 | 专业编码 |
| **响应速度** | 即时 | 取决于 Harness |
| **工具生态** | OpenClaw 工具 | Harness 工具（如 Git/IDE）|
| **会话持久性** | 当前会话 | 可跨会话保持 |

## 八、故障处理

### 8.1 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| `acpx: command not found` | acpx 未安装 | 重新安装 acpx，重启 Gateway |
| `NO_SESSION` | 会话不存在 | 重新创建会话 |
| `queue busy` | Harness 队列满 | 等待或用 `--no-wait` |
| 线程无响应 | ACP Runtime 问题 | 切换到 Direct acpx |

### 8.2 降级策略

```
ACP Runtime 不可用
    │
    ├─→ 自动尝试修复 acpx 安装
    │       │
    │       └─→ 重启 Gateway
    │
    └─→ 降级到 Direct acpx
            │
            └─→ 直接执行命令
```
