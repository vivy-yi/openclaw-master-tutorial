## 核心认知：为什么需要"全面思考"

在 AI Agent 时代，"让 AI 获取网页内容"是一个**多维度权衡**的复杂决策，而非简单的"哪个最好"。

没有"全功能工具"——每个工具都在以下维度上做权衡：

| 维度 | 权衡内容 |
|------|---------|
| **Token 消耗** | 零 Token（命令式）vs 高 Token（LLM 决策） |
| **反检测能力** | 无 vs 可选 vs 强制 |
| **灵活性** | 确定性操作 vs AI 自主决策 |
| **开源/闭源** | 完全开源 vs 核心开源服务付费 vs 商业闭源 |
| **API Key** | 无需 vs 免费额度 vs 必须付费 |
| **维护活跃度** | 活跃 vs 较慢 vs 停止维护 |

**正确思路**：根据具体场景的多维约束，选择最匹配的组合。

```

├─────────────────────────────────────────────────────────────────────────────────────┤
│                                                                                     │
│  Q1: 需要什么类型的输出？                                                             │
│      │                                                                               │
│      ├─ 搜索结果列表 → Q2a                                                            │
│      │                                                                               │
│      ├─ 结构化内容（Markdown/JSON）→ Q2b                                              │
│      │                                                                               │
│      └─ 浏览器交互操作 → Q2c                                                          │
│                                                                                     │
│  ─────────────────────────────────────────────────────────────────────────────────  │
│                                                                                     │
│  Q2a: [搜索场景] 需要什么级别的搜索？                                                 │
│      │                                                                               │
│      ├─ 简单快速原型 → DuckDuckGo MCP / 官方 Fetch                                   │
│      ├─ 高质量语义搜索 → Exa MCP                                                     │
│      ├─ 实时+AI摘要 → Tavily MCP                                                    │
│      └─ 完全免费+无需Key → Open WebSearch MCP                                        │
│                                                                                     │
│  Q2b: [内容抓取场景] 需要什么级别的处理？                                            │
│      │                                                                               │
│      ├─ 简单静态页面 → 官方 Fetch / Jina Reader                                      │
│      ├─ JS渲染页面 → Firecrawl / Crawl4AI                                           │
│      ├─ 强反爬页面 → Camoufox                                                        │
│      └─ 需要AI结构化提取 → Firecrawl (Extract) / browser-use                        │
│                                                                                     │
│  Q2c: [浏览器交互场景] 需要什么级别的自动化？                                        │
│      │                                                                               │
│      ├─ 确定性操作（点击/填表/截图）→ OpenClaw内置 / Playwright MCP / OpenCLI        │
│      ├─ AI自主决策多步骤 → browser-use                                               │
│      └─ 需要复用登录态 → OpenCLI (Chrome扩展) / BB-Browser                           │
│                                                                                     │
│  ─────────────────────────────────────────────────────────────────────────────────  │
│                                                                                     │
│  Q3: 是否有以下硬性约束？                                                             │
│      │                                                                               │
│      ├─ 必须开源/自托管？                                                            │
│      │     ├─ 是 → browser-use(开源) / Playwright MCP / Crawl4AI / Exa MCP          │
│      │     └─ 否 → 均可                                                               │
│      │                                                                               │
│      ├─ 无法获取 API Key？                                                           │
│      │     ├─ 是 → Open WebSearch / DuckDuckGo / Playwright MCP / OpenCLI           │
│      │     └─ 否 → 均可                                                               │
│      │                                                                               │
│      ├─ Token 消耗敏感？（成本优先）                                                 │
│      │     ├─ 是 → OpenClaw act / OpenCLI / Agent Reach（零 Token）                  │
│      │     └─ 否 → browser-use（高 Token 但省开发时间）                              │
│      │                                                                               │
│      └─ 需要高强度反爬？（WAF/Cloudflare/Datadome）                                  │
│            ├─ 是 → Camoufox / Camoufox MCP                                           │
│            └─ 否 → 普通浏览器/MCP即可                                                │
│                                                                                     │
│  ─────────────────────────────────────────────────────────────────────────────────  │
│                                                                                     │
│  Q4: 数据规模是什么级别？                                                             │
│      │                                                                               │
│      ├─ 小规模（<100 URL/天）→ 任意工具                                               │
│      ├─ 中规模（100-10000）→ Firecrawl(云端) / Crawl4AI(自托管)                     │
│      └─ 大规模（>10000）→ Crawl4AI(自托管免费) / Apify(现成爬虫)                     │
│                                                                                     │
└─────────────────────────────────────────────────────────────────────────────────────┘
```

## 3. 按场景选型推荐

### 3.1 快速原型测试

| 场景 | 首选 | 备选 | 理由 |
|------|------|------|------|
| 只想验证一个URL能否抓取 | **官方 Fetch** | Jina Reader | 零配置，`uvx mcp-server-fetch` 即可 |
| 想快速搜索一个关键词 | **DuckDuckGo MCP** | Open WebSearch | 无需任何凭证 |
| 想看看某个页面结构 | **OpenClaw snapshot** | — | 多模态AI视觉理解 |

### 3.2 生产级数据采集

| 场景 | 首选 | 备选 | 理由 |
|------|------|------|------|
| RAG pipeline 摄入 | **Firecrawl** | Crawl4AI | 直接输出Markdown，支持JS渲染 |
| 大规模爬取（免费）| **Crawl4AI自托管** | — | 完全免费，异步高性能 |
| 需要AI结构化提取 | **Firecrawl Extract** | browser-use | Schema-based提取 |
| 定期执行确定性任务 | **OpenClaw Cron + act** | OpenCLI | 零Token，可靠定时 |

### 3.3 需要登录态/反爬

| 场景 | 首选 | 备选 | 理由 |
|------|------|------|------|
| 强反爬（WAF/Cloudflare）| **Camoufox** | — | C++层指纹，完全不可检测 |
| 微信公众号 | **Camoufox** | — | 反检测+JS渲染首选 |
| 小红书/B站/知乎 | **OpenCLI** | BB-Browser | 90+适配器，登录态复用 |
| Twitter/X | **xurl** | — | OAuth2比Cookie更稳 |
| 只需简单登录 | **OpenCLI** | Playwright MCP | 确定性强，成本低 |

### 3.4 AI Agent 自主操作

| 场景 | 首选 | 备选 | 理由 |
|------|------|------|------|
| 复杂多步骤表单任务 | **browser-use** | — | LLM自主决策，适应网页变化 |
| 已知网站的重复操作 | **OpenCLI** | Agent Reach | 确定性命令，零Token |
| 需要接入MCP生态 | **Playwright MCP** | — | 标准MCP协议，生态丰富 |
| 企业SaaS自动化 | **browser-use Cloud** | Apify MCP | 1000+集成，Captcha解决 |

### 3.5 搜索场景

| 场景 | 首选 | 备选 | 理由 |
|------|------|------|------|
| 免费语义搜索 | **Exa MCP** | — | 1000次/月，语义理解强 |

| 免Key快速测试 | **Open WebSearch** | DuckDuckGo | 多引擎聚合，无需凭证 |
| 代码/技术搜索 | **Exa MCP** | Tavily MCP | 专注技术内容 |

### 3.6 视频/多媒体

## 5. 组合方案：没有"全功能工具"

### 5.1 推荐的组合策略

```
┌─────────────────────────────────────────────────────────────────┐
│                    完整数据采集能力组合                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Layer 1: 搜索能力                                              │
│     Exa MCP（语义搜索） + Tavily MCP（实时搜索）                  │
│                                                                  │
│  Layer 2: 爬取能力                                              │
│     Firecrawl（高质量）/ Crawl4AI（免费）/ Jina（轻量）           │
│                                                                  │
│  Layer 3: 反检测能力                                            │
│     Camoufox（强反爬）/ OpenCLI（登录态复用）                     │
│                                                                  │
│  Layer 4: 交互能力                                              │
│     OpenClaw内置（确定性）/ browser-use（AI自主）                 │
│                                                                  │
│  Layer 5: 协议接入                                              │
│     Playwright MCP / MCP-Chrome（标准MCP生态）                   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 5.2 典型场景组合

| 场景 | 组合方案 |
|------|---------|
| **个人知识库 RAG** | Exa MCP（搜索） + Firecrawl（爬取） |
| **竞品监控定时任务** | OpenClaw Cron + OpenCLI（确定性） |
| **社交媒体数据采集** | xurl（Twitter）+ OpenCLI（B站/小红书） |
| **金融数据采集** | Tushare + Camoufox（财报抓取） |
| **复杂自动化工作流** | browser-use + Firecrawl + OpenCLI |

## 7. 决策流程速查卡

```
当你需要采集数据时，按以下顺序问自己：

1️⃣ 输出是什么？
   - 搜索结果 → 2a
   - 内容（Markdown/JSON）→ 2b  
   - 需要点击/填表 → 2c

2️⃣ 有硬性约束吗？
   - 必须开源 → 排除闭源工具
   - 无API Key → 排除需Key工具
   - Token敏感 → 选零Token工具
   - 强反爬 → Camoufox系列

3️⃣ 规模多大？
   - 小规模 → 任意
   - 中等 → Firecrawl云端
   - 大规模 → Crawl4AI自托管

4️⃣ 查表：场景→工具推荐
   - 参考第3节

5️⃣ 考虑成本
   - 参考第4节
```

## 8. 常见误区

| 误区 | 正确认知 |
|------|---------|
| "browser-use 最好" | Token消耗高，简单场景浪费 |
| "必须用付费工具" | 大量免费工具可满足需求 |
| "MCP一定更好" | MCP是协议层，不改变底层能力 |
| "反检测越强越好" | 简单网站用普通工具即可 |
| "一个工具解决所有问题" | 组合是常态，无全功能工具 |
