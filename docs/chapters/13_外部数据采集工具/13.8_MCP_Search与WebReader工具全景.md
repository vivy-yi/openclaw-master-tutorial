# 11.8 MCP Search 与 Web Reader 工具全景（调研报告）

> 📅 调研日期：2026-04-23
> 🎯 适合读者：想了解 MCP 生态中搜索/网页读取工具的开发者
> ⏱️ 阅读时间：约 20 分钟
> 📋 调研方式：GitHub API + npm registry，数据截至 2026-04-23

---

## 调研背景

在 MCP（Model Context Protocol）生态中，"让 AI Agent 获取网页内容"是一个核心需求。MCP 将外部能力以标准化工具形式暴露给 AI，但目前**没有"全功能 MCP"**——每个 MCP 都是单一能力，没有一个同时具备"语义搜索 + JS渲染 + 反检测 + 交互操作"。

---

## 工具全景图

### 分类概览

```
┌─────────────────────────────────────────────────────────────────┐
│           MCP Search / Web Reader 工具分类                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  🔍 搜索类    → Exa / Tavily / DuckDuckGo / Open WebSearch     │
│     特点：输入关键词，返回搜索结果列表                           │
│                                                                  │
│  📄 内容读取类  → 官方 Fetch / Fetcher / Crawl4AI MCP          │
│     特点：输入 URL，返回页面内容（HTML/Markdown）                │
│                                                                  │
│  🕷️ 爬虫类    → Firecrawl MCP / Apify MCP / Crawl4AI MCP      │
│     特点：输入 URL/关键词，返回结构化内容，支持 JS 渲染           │
│                                                                  │
│  🌐 浏览器控制类 → Playwright MCP / Camoufox MCP / Steel        │
│     特点：可导航/点击/填表/截图，支持交互操作                    │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 1. 官方 MCP 服务器

### 官方 Fetch（⭐ 84k 父仓库）

| 属性 | 值 |
|------|-----|
| **Stars** | ⭐ 84,329（modelcontextprotocol/servers）|
| **最后更新** | 2026-04-17（非常活跃）|
| **语言** | TypeScript |
| **安装** | `uvx mcp-server-fetch` 或 `pip install mcp-server-fetch` |

**核心能力：**
- `fetch` 工具：输入 URL，返回 Markdown 格式内容
- 支持 `max_length`（最大字符数）和 `start_index`（分页读取）
- 支持 `raw` 模式（获取原始 HTML）

```json
{
  "mcpServers": {
    "fetch": {
      "command": "uvx",
      "args": ["mcp-server-fetch"]
    }
  }
}
```

**局限：** 无 JS 渲染、无反检测、不支持搜索，仅适合简单公开页面。


**适用场景：** RAG pipeline、需要语义理解的搜索、代码/文章/新闻搜索。

|------|------|

**与 Firecrawl API 的关系：** MCP Server 只是 Firecrawl API 的 MCP 协议包装，底层能力相同。

## 6. OpenClaw 集成方式

### 6.1 通过 mcporter（当前系统已装）

mcporter 可以管理多个 MCP Server，是当前系统内接入 MCP 的推荐方式：

```bash
# 列出当前 MCP Server
mcporter list

# 安装 MCP Server
mcporter add tavily-mcp 'npx -y tavily-mcp'
mcporter add exa-mcp 'npx -y exa-mcp-server'
mcporter add firecrawl-mcp 'npx -y @firecrawl/mcp-server'
```

### 6.2 通过 OpenClaw 原生配置

在 `openclaw.json` 中直接配置 MCP：

```json
{
  "mcpServers": {
    "fetch": {
      "command": "uvx",
      "args": ["mcp-server-fetch"]
    },
    "tavily": {
      "command": "npx",
      "args": ["-y", "tavily-mcp"],
      "env": {
        "TAVILY_API_KEY": "tvly-xxxx"
      }
    },
    "exa": {
      "command": "npx",
      "args": ["-y", "exa-mcp-server"],
      "env": {
        "EXA_API_KEY": "exa-xxxx"
      }
    }
  }
}
```

### 6.3 OpenCLI 方式

OpenCLI 自带部分 MCP 能力，且通过 Chrome 扩展复用登录态：

```bash
# OpenCLI 的 Exa 搜索
opencli list  # 查看可用命令

# 通过 mcporter 调用 Exa
mcporter call 'exa.web_search_exa(query: "搜索词", numResults: 5)'
```

## 7. 选型决策

### 7.1 按需求场景

```
需要让 AI Agent 获取网页内容？

├─ OpenClaw 内置工具？
│    ├─ Cron定时任务 / 确定性act序列 → OpenClaw 内置浏览器
│    └─ 快速安装全网能力 → Agent Reach（脚手架）
│
├─ 只需要搜索？
│    ├─ 免费 + 无 Key → Open WebSearch / DuckDuckGo MCP
│    ├─ 高质量语义搜索 → Exa MCP
│    └─ 实时 + AI 摘要 → Tavily MCP
│
├─ 需要抓取具体 URL？
│    ├─ 简单页面 → 官方 Fetch（零配置）
│    ├─ JS 重度页面 → Firecrawl MCP / Crawl4AI MCP
│    └─ 高强度反爬 → Camoufox MCP
│
├─ 需要交互操作（点击/填表）？
│    ├─ 高可靠性确定性 → Playwright MCP / OpenCLI
│    ├─ AI 自主决策多步骤 → browser-use（见 11.3）
│    └─ 需要复用登录态 → OpenCLI（Chrome扩展）
│
└─ 需要大规模/多平台？
     ├─ 快速接入现成爬虫 → Apify MCP
     └─ 完全免费自托管 → Crawl4AI MCP
```

### 7.2 当前系统推荐配置

| 需求 | 推荐方案 | 理由 |
|------|---------|------|
| **全网语义搜索** | Exa MCP（通过 mcporter ✅已装）| 免费 1000次/月，无需 Key |
| **AI 友好结构化提取** | Firecrawl MCP | 质量最高，支持多种格式 |
| **完全免费爬虫** | Crawl4AI MCP | 开源自托管，无限并发 |
| **快速原型测试** | 官方 Fetch | 零配置，`uvx` 即可 |
| **绕过强反爬** | Camoufox MCP | C++ 层指纹，完全不可检测 |

## 8. 重要发现

### 8.1 没有"全功能工具"，组合是常态

无论是 MCP 还是独立工具，没有一个同时具备：
- ✅ 语义/实时搜索
- ✅ JS 渲染
- ✅ 反检测
- ✅ 交互操作

**OpenClaw + browser-use + MCP 组合才是完整方案：**
- 确定性任务 / Cron → OpenClaw 内置浏览器（最低成本）
- AI 自主决策 → browser-use（最高灵活性）
- 搜索 → Exa MCP / Tavily MCP
- 爬取 → Firecrawl MCP / Crawl4AI MCP
- 交互 → Playwright MCP / OpenCLI

### 8.2 Token 消耗是分层的主要原因

| Token 消耗 | 工具 | 原因 |
|-----------|------|------|
| **零** | OpenClaw act / OpenCLI / Agent Reach | 命令式执行，无需 LLM |
| **低** | Playwright MCP / Fetch / Crawl4AI | 固定工具调用 |
| **中** | Exa / Tavily / Firecrawl MCP | API 调用 + 简单处理 |
| **高** | browser-use | 每步操作需 LLM 决策 |

### 8.3 API Key 是主要门槛

| 无需 Key | 需要 Key（免费额度）|
|---------|-------------------|
| OpenClaw 内置浏览器 | Exa（1000次/月）|
| browser-use（本地）| Tavily（1000次/月）|
| 官方 Fetch | Firecrawl（自部署免费）|
| DuckDuckGo MCP | Apify（按量付费）|
| Open WebSearch | |
| Playwright MCP | |
| OpenCLI | |
| Agent Reach | |
| Crawl4AI MCP（自部署）| |

### 8.4 开源 vs 闭源

- **完全开源**：OpenClaw 内置 / browser-use / 官方 Fetch / DuckDuckGo / Open WebSearch / Playwright MCP / Crawl4AI MCP
- **核心开源，服务付费**：Firecrawl / Exa / Tavily
- **商业闭源**：Apify
