# 六层诊断优化 Skill

> 墨家领域助手六层架构诊断与优化工具

## 概述

通过对话方式唤起Skill，对墨家系统下的领域助手进行**六层架构诊断**，找出短板并提供优化建议。

**核心方法论**：
1. Agent能力层 - 推理/记忆/工具/角色设定
2. 外部知识层 - API > browser > web_search
3. Agent协作层 - @触发跨群协作
4. 领域专业层 - Level3决策支持
5. 合规风控层 - 硬性/软性边界
6. 用户体验层 - 一致性/主动性/透明度

## 触发条件

**对话触发**：
- "诊断[助手名]"
- "分析[助手名]"
- "六层分析"
- "优化[助手名]"
- "检查[助手名]"

**示例**：
```
用户: 诊断mo-finance
用户: 分析mo-law
用户: 六层分析mo-yunying
```

## 六层诊断框架

### 第1层：Agent能力层诊断

**检查项**：

| 检查项 | 说明 | 评分 |
|--------|------|------|
| 角色设定 | SOUL.md是否清晰定义人格/语气/边界 | 0-20 |
| 自主性 | 主动行动 vs 被动响应是否平衡 | 0-20 |
| 工具集成 | MCP/Tool是否完备 | 0-20 |
| 记忆能力 | 六层记忆是否正确配置 | 0-20 |
| 推理能力 | 逻辑推导是否清晰 | 0-20 |

**问题识别**：
```
❌ 角色模糊：语气时而专业时而随意
❌ 自主性差：每次都要用户明确指令
❌ 工具缺失：需要调用API但未配置
```

### 第2层：外部知识层诊断

**检查项**：

| 检查项 | 说明 | 评分 |
|--------|------|------|
| 平台API | 是否有配置平台原生API | 0-25 |
| browser使用 | 是否优先使用browser而非web_search | 0-25 |
| 知识库 | 内部知识库是否完备 | 0-25 |
| 搜索策略 | 是否遵循API>browser>web_search优先级 | 0-25 |

**搜索优先级规范**：
```yaml
1. 平台原生API  ✅ 优先
   - Twitter/GitHub/知乎等官方API
   
2. browser.open()  ✅ 其次
   - 页面抓取、登录态
   
3. web_search (Tavily)  ⚠️ 最后兜底
   - 有配额限制
```

### 第3层：Agent协作层诊断

**检查项**：

| 检查项 | 说明 | 评分 |
|--------|------|------|
| 群绑定 | 是否有独立Telegram群 | 0-20 |
| @触发 | @目标Agent是否能触发协作 | 0-20 |
| sessions_send | 是否配置跨域调用 | 0-20 |
| 协作链路 | 是否形成正确的协作链路 | 0-20 |
| 跨域调用 | 是否引用墨家共享知识库协议 | 0-20 |

**协作链路示意**：
```
用户 → mo-finance (金融问题)
           ↓ 需要法律
        mo-law 协作
           ↓
        mo-finance 整合 → 用户
```

### 第4层：领域专业层诊断

**检查项**：

| 检查项 | 说明 | 评分 |
|--------|------|------|
| 信息聚合 | Level1: 能否聚合信息 | 0-15 |
| 分析解读 | Level2: 能否分析解读 | 0-15 |
| 决策支持 | Level3: 能否提供决策支持 | 0-20 |
| 差异化 | 是否构建了"护城河" | 0-25 |
| 专业深度 |  vs 泛泛而谈 | 0-25 |

**专业深度三层次**：
```markdown
Level 1: 信息聚合
  "今天的财经新闻有哪些"
  
Level 2: 分析解读  
  "这些新闻对市场有什么影响"
  
Level 3: 决策支持
  "基于这些信息，我应该如何调整投资组合"
```

### 第5层：合规风控层诊断

**检查项**：

| 检查项 | 说明 | 评分 |
|--------|------|------|
| 硬性边界 | 是否明确"绝对不能做" | 0-30 |
| 软性边界 | 是否明确"不建议做" | 0-30 |
| 风控机制 | 触发风险时是否有打断/提示 | 0-40 |

**合规边界模板**：
```yaml
硬性边界 (绝对不能碰):
  - 不提供投资建议 (≠分析)
  - 不提供医疗建议 (≠健康)
  - 不提供法律诊断 (≠法律信息)
  - 不承诺结果

软性边界 (不建议但可提示):
  - 不过度乐观/悲观
  - 不代替用户做决定
  - 不隐瞒风险
```

### 第6层：用户体验层诊断

**检查项**：

| 检查项 | 说明 | 评分 |
|--------|------|------|
| 一致性 | 回复风格是否统一 | 0-25 |
| 主动性 | 是否主动提供价值 | 0-25 |
| 透明度 | 是否告知能力边界和置信度 | 0-25 |
| 情感连接 | 是否有温度而非冷冰冰 | 0-25 |

**体验维度说明**：
```
一致性：记忆跨会话延续，"记得上次我们讨论过..."
主动性：不打扰但有价值，如主动预警
透明度：知道Agent能做什么/不能做什么
情感连接：理解用户处境，有温度
```

## 优化执行

### 优先级排序

| 优先级 | 问题类型 | 说明 |
|--------|----------|------|
| 🔴 紧急 | 硬性边界缺失 | 可能导致合规风险 |
| 🟠 重要 | 协作链路断裂 | 影响跨域服务能力 |
| 🟡 改进 | 体验优化 | 长期能力建设 |
