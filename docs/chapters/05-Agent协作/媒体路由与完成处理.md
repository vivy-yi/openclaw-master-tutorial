## 一、功能概述

### 1.1 背景问题

当 Agent 发起媒体生成请求（图片/音频/视频）后，生成的媒体完成响应（completion response）未正确路由至发起请求的 Agent，而是路由到了原始 Agent，导致：

| 问题 | 影响 |
|------|------|
| 媒体响应路由错误 | 请求方 Agent 无法收到生成结果 |
| Agent 协作中断 | 需要媒体作为输入的后续步骤无法执行 |
| 用户体验下降 | 媒体生成完成后无明确反馈 |

### 1.2 解决方案

修复媒体完成响应路由至 **requester agent**（请求方 Agent），确保：

1. 媒体生成结果正确传递至发起请求的 Agent
2. Agent 协作流程完整闭环
3. 多渠道可见性证明正确路由

## 二、技术实现

### 2.1 核心变更

| 变更项 | 说明 |
|--------|------|
| **修复目标** | 媒体完成响应（generated media completion） |
| **路由方向** | → 请求方 Agent（requester agent） |
| **涉及标签** | `agents`, `telegram-visible-proof` |
| **提交者** | steipete (maintainer) |
| **优先级** | 高 (XL size, maintainer) |

### 2.2 修复范围

```
媒体生成流程:
┌──────────────┐     请求      ┌──────────────┐
│ Requester    │ ──────────→  │ Media Gen    │
│ Agent        │              │ Service       │
└──────────────┘              └──────────────┘
       ↑                              │
       │         响应                  │
       └──────────────────────────────┘
              (修复后: 正确路由)
```

## 三、使用场景

### 3.1 Agent 协作中的媒体生成

当父 Agent 调用子 Agent 生成媒体时：

```
父 Agent (Requester)
  │
  ├─→ 调用子 Agent 生成图片
  │
  │    子 Agent 调用媒体生成服务
  │
  └─← 接收完成的媒体响应 ← (修复后)
```

### 3.2 多渠道消息发布

支持在 Telegram 等渠道发布带有媒体附件的消息：

```
Requester Agent
  │
  ├─→ 生成宣传图片
  │
  ├─→ 生成文案
  │
  └─→ 通过 Telegram 发送（含媒体）
```

## 四、相关配置

### 4.1 启用媒体路由

媒体路由功能默认启用，无需额外配置。

### 4.2 日志查看

```bash
# 查看媒体路由日志
tail -f ~/.openclaw/logs/openclaw.log | grep -i "media"

# 查看 Agent 协作日志
tail -f ~/.openclaw/logs/openclaw.log | grep -i "agent"
```

## 五、故障排除

### 5.1 媒体响应未收到

| 检查项 | 操作 |
|--------|------|
| Agent 日志 | 检查 `~/.openclaw/logs/` 目录 |
| 媒体生成状态 | 确认媒体服务正常运行 |
| 网络连接 | 确认渠道连接正常 |

### 5.2 路由异常

```bash
# 重启 Gateway
openclaw gateway restart

# 检查 Agent 状态
openclaw agents list
```
