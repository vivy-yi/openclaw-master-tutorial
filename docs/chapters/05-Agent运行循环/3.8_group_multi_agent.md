## 3.8.1 群组消息概述

### 群组聊天行为

OpenClaw 在不同平台上对群组聊天的处理是一致的：WhatsApp、Telegram、Discord、Slack、Signal、iMessage、Microsoft Teams、Zalo。

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                           支持的群组平台                                       │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  即时通讯                                                                   │
│  ├── WhatsApp      ✅ 完全支持                                               │
│  ├── Telegram     ✅ 完全支持                                               │
│  ├── Discord      ✅ 完全支持                                               │
│  ├── Slack        ✅ 完全支持                                               │
│  ├── Signal       ✅ 完全支持                                               │
│  ├── iMessage     ✅ 完全支持                                               │
│  ├── Microsoft Teams  ✅ 完全支持                                         │
│  └── Zalo         ✅ 完全支持                                             │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### 核心概念

```
OpenClaw "生活" 在你自己的消息账户中。
没有独立的 WhatsApp 机器人用户。
如果**你**在群组中，OpenClaw 可以看到该群组并在那里回复。
```

## 3.8.2 群组策略配置

### groupPolicy 选项

| 值 | 说明 |
|---|------|
| `disabled` | 禁用群组回复 |
| `allowlist` | 仅允许白名单群组 |
| `open` | 允许所有群组 |

### 配置示例

```json
{
  "channels": {
    "whatsapp": {
      "groupPolicy": "allowlist",
      "groups": {
        "120363403215116621@g.us": {
          "allow": true
        }
      }
    },
    "telegram": {
      "groupPolicy": "disabled"
    }
  }
}
```

### 快速参考表

| 目标 | 配置 |
|------|------|
| 允许所有群组，但只在 @提及 时回复 | `groups: { "*": { requireMention: true } }` |
| 禁用所有群组回复 | `groupPolicy: "disabled"` |
| 仅特定群组 | `groups: { "<group-id>": { ... } }` |
| 只有你能触发群组回复 | `groupPolicy: "allowlist"`, `groupAllowFrom: ["+1555..."]` |

## 3.8.4 广播组 (Broadcast Groups)

### 什么是广播组

广播组使多个 Agent 能够同时处理和响应同一条消息。这允许你创建专门的 Agent 团队，在一个 WhatsApp 群组或 DM 中协同工作。

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                          广播组工作方式                                       │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  当前范围: WhatsApp only (Web 渠道)                                        │
│                                                                                 │
│  评估时机: 频道白名单和群组激活规则之后                                      │
│                                                                                 │
│  在 WhatsApp 群组中，这意味着广播发生在 OpenClaw 通常会回复时               │
│  (例如: 提及后，取决于你的群组设置)                                         │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### 使用场景

#### 1. 专门的 Agent 团队

```
群组: "开发团队"
Agent:
  - CodeReviewer (审查代码片段)
  - DocumentationBot (生成文档)
  - SecurityAuditor (检查漏洞)
  - TestGenerator (建议测试用例)
```

#### 2. 多语言支持

```
群组: "国际支持"
Agent:
  - Agent_EN (英语回复)
  - Agent_DE (德语回复)
  - Agent_ES (西班牙语回复)
```

#### 3. 质量保证工作流

```
群组: "客户支持"
Agent:
  - SupportAgent (提供答案)
  - QAAgent (审查质量，仅在发现问题时间复)
```

#### 4. 任务自动化

```
群组: "项目管理"
Agent:
  - TaskTracker (更新任务数据库)
  - TimeLogger (记录时间)
  - ReportGenerator (创建摘要)
```

## 3.8.5 广播组配置

### 基本配置

添加顶级 `broadcast` 部分（与 `bindings` 相邻）：

```json
{
  "broadcast": {
    "120363403215116621@g.us": ["alfred", "baerbel", "assistant3"]
  }
}
```

- 群组聊天: group JID (例如 `120363403215116621@g.us`)
- DM: E.164 电话号码 (例如 `+15551234567`)

**结果:** 当 OpenClaw 在此聊天中回复时，它将运行所有三个 Agent。

### 处理策略

#### Parallel (并行) - 默认

所有 Agent 同时处理：

```json
{
  "broadcast": {
    "strategy": "parallel",
    "120363403215116621@g.us": ["alfred", "baerbel"]
  }
}
```

#### Sequential (顺序)

Agent 按顺序处理（一个等待前一个完成）：

```json
{
  "broadcast": {
    "strategy": "sequential",
    "120363403215116621@g.us": ["alfred", "baerbel"]
  }
}
```

### 完整示例

```json
{
  "agents": {
    "list": [
      {
        "id": "code-reviewer",
        "name": "Code Reviewer",
        "workspace": "/path/to/code-reviewer",
        "sandbox": { "mode": "all" }
      },
      {
        "id": "security-auditor",
        "name": "Security Auditor",
        "workspace": "/path/to/security-auditor",
        "sandbox": { "mode": "all" }
      },
      {
        "id": "docs-generator",
        "name": "Documentation Generator",
        "workspace": "/path/to/docs-generator",
        "sandbox": { "mode": "all" }
      }
    ]
  },
  "broadcast": {
    "strategy": "parallel",
    "120363403215116621@g.us": ["code-reviewer", "security-auditor", "docs-generator"],
    "120363424282127706@g.us": ["support-en", "support-de"],
    "+15555550123": ["assistant", "logger"]
  }
}
```

## 3.8.6 会话隔离

### 隔离机制

广播组中的每个 Agent 保持完全独立：

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                          会话隔离架构                                       │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  每个 Agent 独立拥有：                                                      │
│  ├── Session keys (会话密钥)                                                │
│  │   agent:alfred:whatsapp:group:120363...                              │
│  │   vs                                                                    │
│  │   agent:baerbel:whatsapp:group:120363...                             │
│  │                                                                     │
│  ├── Conversation history (对话历史)                                       │
│  │   Agent 看不到其他 Agent 的消息                                        │
│  │                                                                     │
│  ├── Workspace (工作空间)                                                │
│  │   独立沙箱（如果配置）                                                │
│  │                                                                     │
│  ├── Tool access (工具访问)                                               │
│  │   不同的 allow/deny 列表                                            │
│  │                                                                     │
│  └── Memory/context (记忆/上下文)                                        │
│      独立的 IDENTITY.md, SOUL.md 等                                     │
│                                                                                 │
│  📌 群组上下文缓冲区 (最近群组消息) 是共享的                             │
│     所有广播 Agent 在触发时看到相同的上下文                               │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### 会话密钥格式

```
Agent: alfred 的上下文:
Session: agent:alfred:whatsapp:group:120363403215116621@g.us
History: [用户消息, alfred 之前的回复]
Workspace: ~/.openclaw/workspace-alfred/
Tools: read, write, exec

Agent: baerbel 的上下文:
Session: agent:baerbel:whatsapp:group:120363403215116621@g.us
History: [用户消息, baerbel 之前的回复]
Workspace: ~/.openclaw/workspace-baerbel/
Tools: read only
```

## 3.8.7 最佳实践

### 1. 保持 Agent 专注

```json
{
  "broadcast": {
    "DEV_GROUP": ["formatter", "linter", "tester"]
  }
}
```

✅ **推荐:** 每个 Agent 有一个工作
❌ **避免:** 一个通用的"开发助手"Agent

### 2. 使用描述性名称

```json
{
  "agents": {
    "security-scanner": { "name": "Security Scanner" },
    "code-formatter": { "name": "Code Formatter" },
    "test-generator": { "name": "Test Generator" }
  }
}
```

### 3. 配置不同的工具访问

```json
{
  "agents": {
    "reviewer": {
      "tools": { "allow": ["read", "exec"] } // 只读
    },
    "fixer": {
      "tools": { "allow": ["read", "write", "edit", "exec"] } // 读写
    }
  }
}
```

### 4. 监控性能

```
- 使用 "strategy": "parallel" (默认) 以获得速度
- 限制每个广播组 5-10 个 Agent
- 为简单的 Agent 使用更快的模型
```

### 5. 优雅处理失败

```
消息 → [Agent A ✓, Agent B ✗ 错误, Agent C ✓]
结果: Agent A 和 C 回复，Agent B 记录错误
```

## 3.8.8 兼容性

### 支持的渠道

| 渠道 | 状态 |
|------|------|
| WhatsApp | ✅ 已实现 |
| Telegram | 🚧 计划中 |
| Discord | 🚧 计划中 |
| Slack | 🚧 计划中 |

### 与路由的兼容性

```json
{
  "bindings": [
    {
      "match": { "channel": "whatsapp", "peer": { "kind": "group", "id": "GROUP_A" } },
      "agentId": "alfred"
    }
  ],
  "broadcast": {
    "GROUP_B": ["agent1", "agent2"]
  }
}
```

- `GROUP_A`: 只有 alfred 回复（正常路由）
- `GROUP_B`: agent1 **和** agent2 回复（广播）

**优先级:** `broadcast` 优先于 `bindings`。

## 3.8.9 常见问题

### Q1: Agent 没有响应

```bash
# 检查:
1. Agent ID 存在于 agents.list 中
2. Peer ID 格式正确（例如 120363403215116621@g.us）
3. Agent 不在拒绝列表中

# 调试:
tail -f ~/.openclaw/logs/gateway.log | grep broadcast
```

### Q2: 只有一个 Agent 回复

**原因:** Peer ID 可能在 `bindings` 但不在 `broadcast`。

**修复:** 添加到广播配置或从 bindings 中移除。

### Q3: 性能问题

```
如果多个 Agent 很慢:
- 减少每组 Agent 数量
- 使用更轻量的模型 (sonnet 而非 opus)
- 检查沙箱启动时间
```
