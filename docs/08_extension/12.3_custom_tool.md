# 12.3 自定义工具：把副作用关进可控边界

自定义工具的难点不在"写一个函数"，而在"如何把副作用做成可控、可审计、可回滚的工程能力"。

---

## 12.3.1 自定义工具的最小原则：输入可验证，输出可回放

将工具纳入系统之前，应先明确三项契约：

### 三项核心契约

```
1. 输入约束
   └── 参数必须可验证，避免把"自由文本"直接作为高风险工具参数

2. 输出约束
   └── 输出必须结构化，避免回注导致上下文膨胀

3. 副作用边界
   └── 写操作必须具备幂等或对账路径，失败时必须给出人工核查步骤
```

这些约束不依赖模型能力，属于工具设计的确定性底线。

---

## 12.3.2 把工具接入工具策略：分层规则与拒绝优先

工具治理应落在工具策略层。官方文档给出了工具的允许、拒绝与按渠道/群组分层限制机制，并强调**拒绝规则优先**。

### 配置示例

```json5
{
  tools: {
    profile: 'coding',
    deny: ['group:runtime', 'write', 'edit', 'apply_patch'],
  },
  channels: {
    webchat: {
      groups: {
        '*': {
          tools: { deny: ['group:runtime'] },
        },
      },
    },
    ops: {
      groups: {
        '*': {
          tools: { deny: ['group:runtime'] },
          toolsBySender: {
            'admin_user': { alsoAllow: ['group:fs'] },
          },
        },
      },
    },
  },
}
```

### 验收标准

当某个请求触发了写工具但被拒绝时，日志应给出明确拒绝原因，便于追溯是策略拒绝还是参数不合法。

---

## 12.3.3 高风险工具的隔离执行：多智能体沙箱工具

对于涉及代码执行、文件写入或系统操作的高风险工具，建议配合多智能体沙箱工具将执行域隔离出来。

### 沙箱工具认证配置

认证配置文件位于：`~/.openclaw/agents/<agentId>/agent/auth-profiles.json`

### "少数智能体拥有写能力"模式

```
大多数入口智能体：
└── 只允许读工具与诊断命令

少数运维智能体：
└── 在明确入口与审计前提下允许沙箱写工具
```

这与入口绑定与路由策略配合使用，可以把越权风险收敛到可计算范围。

---

## 12.3.4 验收与排障：先看策略加载，再看单次调用链

自定义工具上线后的验收建议固定为两步：

### 两步验收法

```
1. 配置验收
   └── 用 status --deep 确认 tools.deny 与 channels.*.groups.*.tools 限制已加载

2. 行为验收
   └── 用结构化日志观察工具调用与拒绝事件，确保拒绝规则生效
```

### 诊断命令

```bash
openclaw status --deep
openclaw logs --follow --json
```

### 常见问题排查

| 问题类型 | 排查方向 |
|----------|----------|
| 工具被误允许 | 优先回到工具策略检查 deny 与渠道/群组级策略限制 |
| 工具执行失败 | 优先区分外部依赖故障与参数不合法，再决定是否允许有界重试 |

---

## 本节小结

1. **三项契约**：输入可验证、输出可回放、副作用边界清晰
2. **分层规则**：工具策略的允许、拒绝与按渠道/群组分层
3. **拒绝优先**：deny 规则优先于 allow 规则
4. **沙箱隔离**：高风险工具使用沙箱执行域
5. **两步验收**：配置验收 + 行为验收
