18-18 知识蒸馏到 Skills ​

学习目标: 理解知识蒸馏的概念、掌握从对话/轨迹/流程中提取可复用 Skills 的方法论 预计用时: 30 分钟 前置要求: 熟悉 Skills 基础概念（第18章）

18.18.1 什么是"知识蒸馏" ​

定义 ​

知识蒸馏（Knowledge Distillation）是将 AI Agent 在执行任务过程中积累的隐式经验转化为显式的、可复用的结构化 Skills 的过程。

类比：炼金术
原料：对话碎片、执行轨迹、错误记录
产出：精炼的技能模块（Skill）

三种待蒸馏对象 ​

对象定义来源蒸馏难度
知识 (Knowledge)事实、判断标准、偏好规则对话中零散透露的上下文、多次验证的信息⭐⭐ 中
流程 (Process)多步骤操作序列、决策分支重复任务的共同执行路径⭐⭐⭐ 高
轨迹 (Trace)历史行为模式、错误恢复路径sessions 历史、cron 执行日志⭐⭐⭐⭐ 很难

蒸馏触发条件 ​

满足任一条件即应启动蒸馏：

  • ✅ 同一错误被纠正 ≥3 次
  • ✅ 同一任务被执行 ≥3 次且流程相似
  • ✅ 用户明确说"记住……"
  • ✅ 发现了有价值的错误恢复路径
  • ✅ 新建工作流超过 5 步

18.18.2 知识蒸馏 — 从隐式到显式 ​

核心问题 ​

"这个 Agent 是如何知道 X 的？" → 追溯到显式文档

提炼路径 ​

对话/执行历史
    ↓ 模式识别
重复出现的判断/偏好
    ↓ 显式化
SKILL.md 中的规则 或 references/ 中的文档

知识类型与对应技能结构 ​

知识类型技能存放位置示例
事实性 (Facts)references/"张总喜欢 BLUF 格式" → references/people.md
判断规则 (Rules)SKILL.md 正文"凡是涉及 rm -rf 必须确认" → SKILL.md 安全规则
偏好模式 (Preferences)corrections.md 类文件"用户说'不要'就是永久规则" → 持久化偏好库
领域知识 (Domain)references/金融/法律行业知识 → 分域 references

提炼检查清单 ​

  • [ ] 这个信息在对话中出现过 ≥2 次吗？
  • [ ] 这是用户的明确偏好还是偶然表述？
  • [ ] 将它写入 Skill vs 临时查询，哪个更高效？

18.18.3 流程蒸馏 — 从经验到剧本 ​

核心问题 ​

"做这件事的标准步骤是什么？" → 建立可重复的工作流

提炼路径 ​

多个相似任务的执行轨迹
    ↓ 比对共性，提取变量
抽象工作流（保留关键决策点）
    ↓ 参数化
SKILL.md 工作流 + 必要的 scripts/

流程标准化三层次 ​

Level 1: 文字指导（高自由度）
  → SKILL.md 中用自然语言描述流程
  → 适用于：判断空间大、上下文敏感的流程

Level 2: 伪代码/脚本框架（中等自由度）
  → SKILL.md 中含参数化脚本模板
  → 适用于：有标准模式但允许参数变化的流程

Level 3: 确定性脚本（低自由度）
  → scripts/ 中包含完整可执行脚本
  → 适用于：易出错、多步骤、必须精确执行的流程

流程识别信号 ​

  • ✅ 同一任务被执行 ≥3 次且步骤相似
  • ✅ 用户说"按上次的方式做"
  • ✅ 任务有清晰的输入→处理→输出结构

18.18.4 轨迹蒸馏 — 从历史到模式 ​

核心问题 ​

"Agent 在这件事上通常怎么做的？" → 从执行日志中反推决策逻辑

提炼路径 ​

sessions 历史 / cron 执行日志
    ↓ 事件序列分析
常见行为序列 + 异常恢复路径
    ↓ 归纳为规则
SKILL.md 的决策树 / 错误处理章节

轨迹挖掘技术 ​

技术输入输出
频繁序列挖掘工具调用序列常见执行模式
错误归因分析错误日志 + 恢复操作错误处理知识库
决策点提取对话分支点SKILL.md 决策树
时间模式分析cron 执行时间/结果HEARTBEAT.md 规则

轨迹提炼检查清单 ​

  • [ ] 这个轨迹有 ≥3 次重复吗？
  • [ ] 轨迹中有错误恢复路径吗？（高价值！）
  • [ ] 轨迹暴露了 SKILL.md 中缺失的安全规则吗？

18.18.5 蒸馏方法论框架 — 六步法 ​

Step 1: 数据收集 ​

触发信号:

bash
# 对话历史
sessions_history(sessionKey, limit=50)

# Cron 执行记录
~/.openclaw/cron/jobs.json

# 共享上下文
~/.openclaw/shared/context/recent/

# 每日记忆日志
memory/YYYY-MM-DD.md

Step 2: 模式分析 ​

维度问什么问题产出
频率这件事多久发生一次？优先级判断
变异性每次做法有何不同？确定自由度层次
依赖性依赖哪些工具/文件/变量？确定 scripts/ 需求
边界什么情况下不能这样做？安全规则

Step 3: 知识编码 ​

编码原则:

  1. 先写判断规则，后写执行步骤 — 规则比步骤更难获取
  2. 用具体示例锚定抽象规则 — "当 X 时用 Y" 优于 "视情况处理"
  3. 标注置信度 — [高置信度] [待验证]
  4. 区分知识与流程 — 知识放 references，流程放 SKILL.md

Step 4: Skill 生成 ​

bash
# 已有技能迭代：
# 1. 找到现有最接近的 Skill
# 2. 在其 references/ 下新增专题文件
# 3. 或在 SKILL.md 中扩展章节

# 全新技能：
# 1. 初始化框架
# 2. 填入 SKILL.md（核心）
# 3. 建立 references/（知识库）
# 4. 编写 scripts/（确定性代码）
# 5. 包装并测试

Step 5: 验证回路 ​

验证类型执行方式通过标准
功能验证用 Skill 执行真实任务产出正确
边界验证在不安全场景测试正确拒绝
效率验证对比 Skill 使用前后的 token 消耗降低 ≥30%
回归验证用历史轨迹重放行为一致

Step 6: 持续蒸馏 ​

markdown
# Skill 生命周期
初创期:  高自由度文字指令，观察执行情况
稳定期:  提炼为 scripts/ 脚本，覆盖高频路径
成熟期:  建立 corrections.md，处理边缘情况

18.18.6 实操案例 ​

案例 1: 知识蒸馏 — "用户偏好" ​

来源: 对话中多次透露"禁止用 rm -rf"

对话历史 → 识别重复安全规则 → 写入 skill 安全章节

产出 SKILL.md:

markdown
## 安全边界
- 禁止对用户数据执行 `rm -rf`
- 删除文件一律用 `trash` 或 `mv ~/.Trash/`
- 执行 destructive command 前必须确认

案例 2: 流程蒸馏 — "每日调研工作流" ​

来源: 连续多日执行的 Cron 任务（采集→调研→分析→推送）

比对4天的执行日志 → 提取共同流程框架 → 参数化变体步骤

产出 SKILL.md:

markdown
# Tutorial Master Workflow

## 每日工作流
1. 22:00 触发 GitHub 数据采集 (`daily-sync.sh`)
2. 23:35 触发深度调研 (`deep-research.sh`)
3. 23:50 缺口扫描 (`gap-scan.sh`)
4. 00:05 文档更新 (`doc-update.sh`)
5. 00:10 推送通知 (`notify.sh`)

## 参数
- SEARCH_KEYWORDS: 每日调研关键词
- RESEARCH_DEPTH: 浅/深

案例 3: 轨迹蒸馏 — "错误恢复模式" ​

来源: Agent 多次遇到 npm install 失败并成功恢复

错误日志序列 → 提取恢复路径 → 写入 troubleshooting 章节

产出 SKILL.md:

markdown
## 常见错误恢复

| 错误 | 恢复命令 |
|------|---------|
| npm install 失败 | `npm cache clean --force && npm install` |
| 端口冲突 | `openclaw config set gateway.port 18790` |
| JSON配置损坏 | `python3 -m json.tool ~/.openclaw/openclaw.json` |

18.18.8 核心原则总结 ​

  1. 宁可不蒸馏，不要硬凑 — 只有重复出现 ≥3 次的内容才值得蒸馏
  2. 知识 > 流程 > 轨迹 — 知识最容易蒸馏，优先从知识开始
  3. 渐进式披露 — SKILL.md 只放触发器和核心指令，细节放 references/
  4. 蒸馏是持续过程 — Skill 交付不是终点，错误纠正本身就是蒸馏
