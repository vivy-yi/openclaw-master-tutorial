## 12.2.1 插件是什么：长驻进程内的 TypeScript 模块

插件与技能（Skills）有本质区别：

```
┌─────────────────────────────────────────────────────┐
│           插件 vs 技能                               │
├─────────────────────────────────────────────────────┤
│                                                     │
│  技能 (Skills)                                    │
│  ├── 纯文本提示思路                               │
│  ├── 无需代码                                     │
│  └── 通过提示词扩展能力                            │
│                                                     │
│  插件 (Plugins)                                   │
│  ├── 运行在 Gateway 进程内的 TypeScript 模块        │
│  ├── 可以挂载自定义 Agent Tools                   │
│  ├── 注册全新聊天频道                             │
│  ├── 实现后台轮询服务                             │
│  └── 可接管系统的 RPC/HTTP handler               │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### Manifest 文件

每个插件必须提供 `openclaw.plugin.json` (Manifest) 声明文件：

- 包含配置 JSON Schema
- 包含 uiHints 用于 Control UI 表单渲染
- 系统在渲染表单或校验配置时，**不会执行第三方插件底层代码**

### 工程化建议

```
插件应该被当作严肃的代码级系统依赖对待：

1. 插件版本进入版本控制与发布流程
2. 插件的 openclaw.plugin.json 严格约束参数
3. 不合规清单在 Gateway 启动期直接拒载
4. 插件的工具与副作用被工具策略与白名单彻底约束
```

## 12.2.2 配置与启用：entries、enabled、config 与 load.paths

### 插件启用配置

```json5
{
  plugins: {
    enabled: true,
    allow: ['my-plugin'],           // 白名单：仅允许列出的插件加载
    deny: ['untrusted-plugin'],     // 黑名单：deny 优先于 allow
    load: {
      paths: ['~/Projects/my-plugin'], // 本地开发中的插件目录
    },
    entries: {
      'my-plugin': {
        enabled: true,
        config: {
          // 插件自定义配置
        }
      },
    },
  },
}
```

### 安全建议

为降低供应链风险，建议配合白名单机制只允许经过审计的插件生效。

## 12.2.3 安全边界：插件白名单与工具策略协同

### 双层收敛策略

```
1. 先用 plugins.allow 收敛可加载插件集合
2. 再用工具策略把插件提供的工具按风险分组
3. 在默认档案中拒绝高风险组
```

### 实际案例：Jira 插件灰度上线

**阶段一：白名单准入**

```json5
{
  plugins: {
    allow: ['jira-sync'],
    entries: { 'jira-sync': { enabled: true } },
  },
  tools: {
    deny: ['jira.create*', 'jira.update*', 'jira.delete*'],
  },
}
```

**阶段二：小范围放开**

```json5
{
  channels: {
    telegram: {
      groups: {
        '*': {
          toolsBySender: {
            'admin_id_001': { alsoAllow: ['jira.create*'] },
          },
        },
      },
    },
  },
}
```

**阶段三：全量放开**

移除 `tools.deny` 中的 Jira 写入限制，保留审计日志与幂等键。

## 12.2.4 插件加载生命周期

### 完整流程

```
1. 发现
   └── 从 bundled、global、workspace、config 四个源扫描候选插件

2. 校验
   └── 读取 openclaw.plugin.json 清单，验证配置是否符合 JSON Schema

3. 安全加载
   └── 通过 jiti（TypeScript/JS 运行时编译器）加载入口文件，执行边界检查

4. 注册
   └── 同步调用插件的 register(api) 函数
   └── 插件通过 API 注册工具、钩子、渠道、服务、HTTP 路由等

5. 激活
   └── 所有插件注册完成后，构建 PluginRegistry，初始化全局 Hook Runner

6. 运行
   └── 在每个钩子点，系统通过 Hook Runner 调度已注册的处理器
```

## 12.2.5 验收与排障

### 诊断命令

```bash
# 列出所有插件
openclaw plugins list

# 查看插件详情
openclaw plugins info my-plugin

# 插件诊断
openclaw plugins doctor

# 深度状态检查
openclaw status --deep
```

### 日志分析

```bash
# 实时日志跟踪
openclaw logs --follow --json
```
