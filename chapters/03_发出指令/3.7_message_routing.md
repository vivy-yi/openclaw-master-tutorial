## 本节目标
- 理解 OpenClaw 消息路由的核心概念

## 3.7.1 消息路由概述

### 核心组件

OpenClaw 消息路由是 Gateway 的核心功能，负责将来自不同渠道的消息准确路由到对应的 Agent。

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                           消息路由架构                                          │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│                    OpenClaw routes replies                                      │
│                    back to the channel                                          │
│                    where a message came from                                     │
│                                                                                 │
│   用户侧                                                                    │
│       │                                                                    │
│       ▼                                                                    │
│   ┌─────────────┐     ┌─────────────┐     ┌─────────────┐                  │
│   │  Telegram   │     │   Discord   │     │  WhatsApp   │   ...         │
```

### 核心术语

| 术语 | 说明 |
|-----|------|
| **Channel** | 渠道类型：`whatsapp`, `telegram`, `discord`, `slack`, `signal`, `imessage`, `webchat` |
| **AccountId** | 每个渠道的账户实例（支持多账户时） |
| **AgentId** | 隔离的工作空间 + 会话存储（"大脑"） |
| **SessionKey** | 用于存储上下文和控制并发的桶密钥 |

## 3.7.2 Session Key 格式

### 直接消息 (DM)

```
直接消息默认使用主会话：
agent:<agentId>:<mainKey>  →  默认: agent:main:main
```

### 群组和频道

```
群组: agent:<agentId>:<channel>:group:<id>
频道: agent:<agentId>:<channel>:channel:<id>
```

### 线程

```
Slack/Discord 线程: 在基础密钥上追加 :thread:<threadId>
Telegram 论坛话题: 在群组密钥中嵌入 :topic:<topicId>
```

### 示例

```
agent:main:telegram:group:-1001234567890:topic:42
agent:main:discord:channel:123456:thread:987654
```

## 3.7.3 路由优先级规则

### 优先级顺序

OpenClaw 按以下顺序选择 Agent：

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                          路由优先级顺序                                        │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  高优先级 ←─────────────────────────────────────→ 低优先级                  │
│                                                                                 │
│  ┌─────────────────────────────────────────────────────────────────────────┐   │
│  │  1. peer.kind + peer.id        → 精确匹配 DM/群组 ID               │   │
│  │  2. parentPeer                 → 线程继承                          │   │
│  │  3. guildId + roles            → Discord 服务器 + 角色              │   │
│  │  4. guildId                    → Discord 服务器                      │   │
│  │  5. teamId                     → Slack 工作区                       │   │
│  │  6. accountId                   → 账户匹配                          │   │
│  │  7. channel                    → 渠道级别 (任何账户)                │   │
│  │  8. default                    → 默认 Agent                        │   │
│  └─────────────────────────────────────────────────────────────────────────┘   │
│                                                                                 │
│  当 binding 包含多个匹配字段时，必须所有字段都匹配才生效                      │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

![消息处理](./.assets/diagrams/104_message_handling.png)



![消息格式](./.assets/diagrams/80_message_format.png)



![会话上下文](./.assets/diagrams/08_session_context.png)



### 匹配规则详解

| 优先级 | 字段 | 说明 | 示例 |
|-------|------|------|------|
| 1 | `peer.kind + peer.id` | 精确匹配 DM/群组 ID | WhatsApp: `+15551230001` |
| 2 | `parentPeer` | 线程继承 | Discord 线程 |
| 3 | `guildId + roles` | Discord 服务器+角色 | Discord 服务器+角色 |
| 4 | `guildId` | Discord 服务器 | Discord 服务器 ID |
| 5 | `teamId` | Slack 工作区 | Slack 团队 ID |
| 6 | `accountId` | 账户匹配 | 多账户区分 |
| 7 | `channel` | 渠道级别 | `accountId: "*"` |
| 8 | `default` | 默认 Agent | `agents.list[].default` |

## 3.7.4 Binding 配置

### 基本配置

```json
{
  "bindings": [
    {
      "agentId": "main",
      "match": {
        "channel": "telegram"
      }
    }
  ]
}
```

### 多维度配置

```json
{
  "bindings": [
    {
      "agentId": "opus",
      "match": {
        "channel": "whatsapp",
        "peer": {
          "kind": "direct",
          "id": "+15551230001"
        }
      }
    },
    {
      "agentId": "chat",
      "match": {
        "channel": "whatsapp"
      }
    }
  ]
}
```

### Discord 角色路由

```json
{
  "bindings": [
    {
      "agentId": "dev",
      "match": {
        "channel": "discord",
        "guildId": "123456789012345678",
        "roles": ["developer", "admin"]
      }
    }
  ]
}
```

### Slack 团队路由

```json
{
  "bindings": [
    {
      "agentId": "work",
      "match": {
        "channel": "slack",
        "teamId": "T1234567890"
      }
    }
  ]
}
```

## 3.7.5 dmScope 配置

### 会话分桶策略

`dmScope` 只控制私聊会话如何分桶，**不决定**消息去哪个 Agent。

| 值 | 说明 | 适用场景 |
|---|------|---------|
| `main` | 所有私聊共用主会话 | 单用户 |
| `per-peer` | 按发送者隔离 | 多用户收件箱 |
| `per-channel-peer` | 按渠道+发送者隔离 | 多渠道多用户 |
| `per-account-channel-peer` | 按账号+渠道+发送者隔离 | 多账号多渠道多用户 |

```json
{
  "session": {
    "dmScope": "per-channel-peer"
  }
}
```

### 主会话路由固定

当 `dmScope` 为 `main` 时，为防止非所有者 DM 覆盖会话的 `lastRoute`，OpenClaw 会从 `allowFrom` 推断固定所有者：

- `allowFrom` 只有一个非通配符条目
- 该条目可以规范化为该渠道的具体发送者 ID
- 入站 DM 发送者不匹配固定所有者

## 3.7.6 多渠道多账户配置

### 多 Discord Bot 配置

```json
{
  "agents": {
    "list": [
      { "id": "main", "workspace": "~/.openclaw/workspace-main" },
      { "id": "coding", "workspace": "~/.openclaw/workspace-coding" }
    ]
  },
  "bindings": [
    { "agentId": "main", "match": { "channel": "discord", "accountId": "default" } },
    { "agentId": "coding", "match": { "channel": "discord", "accountId": "coding" } }
  ],
  "channels": {
    "discord": {
      "accounts": {
        "default": { "token": "DISCORD_BOT_TOKEN_MAIN" },
        "coding": { "token": "DISCORD_BOT_TOKEN_CODING" }
      }
    }
  }
}
```

### 多 Telegram Bot 配置

```json
{
  "bindings": [
    { "agentId": "main", "match": { "channel": "telegram", "accountId": "default" } },
    { "agentId": "alerts", "match": { "channel": "telegram", "accountId": "alerts" } }
  ],
  "channels": {
    "telegram": {
      "accounts": {
        "default": { "botToken": "123456:ABC..." },
        "alerts": { "botToken": "987654:XYZ..." }
      }
    }
  }
}
```

### 默认账户配置

```json
{
  "channels": {
    "whatsapp": {
      "defaultAccount": "personal",
      "accounts": {
        "personal": { ... },
        "biz": { ... }
      }
    }
  }
}
```

> ⚠️ **重要**：在多账户设置中，当配置两个或更多账户时，务必设置明确的默认账户（`defaultAccount` 或 `accounts.default`）。否则，回退路由可能会选择第一个规范化的账户 ID。

## 3.7.7 广播组 (Broadcast Groups)

### 功能说明

广播组允许在 OpenClaw 通常会回复的同一对等体上运行**多个 Agent**（例如：WhatsApp 群组中，提及/激活门控后）。

### 配置示例

```json
{
  "broadcast": {
    "strategy": "parallel",
    "120363403215116621@g.us": ["alfred", "baerbel"],
    "+15555550123": ["support", "logger"]
  }
}
```

### 工作方式

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                          广播组工作流程                                       │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  用户在群组中 @提及 Agent                                                      │
│           │                                                                    │
│           ▼                                                                    │
│  Gateway 接收消息                                                            │
│           │                                                                    │
│           ▼                                                                    │
│  检查 broadcast 配置                                                           │
│           │                                                                    │
│           ▼                                                                    │
│  ┌─────────────────────────────────────────────────────────────────────┐    │
│  │  匹配到广播组: ["alfred", "baerbel"]                                │    │
│  │  ↓                                                                   │    │
│  │  并行运行多个 Agent                                                  │    │
│  │  ┌─────────────┐  ┌─────────────┐                                 │    │
│  │  │   alfred    │  │  baerbel    │  ← 同时处理请求                 │    │
│  │  └─────────────┘  └─────────────┘                                 │    │
│  │  ↓                                                                   │    │
│  │  合并结果                                                           │    │
│  └─────────────────────────────────────────────────────────────────────┘    │
│           │                                                                    │
│           ▼                                                                    │
│  返回合并后的响应                                                            │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘
```

## 3.7.9 管理命令

### 查看 Bindings

```bash
# 查看所有 bindings
openclaw bindings list

# 查看特定渠道的 bindings
openclaw bindings list --channel telegram

# 详细查看
openclaw bindings info <binding-id>
```

### 管理 Bindings

```bash
# 添加 binding
openclaw bindings add telegram:main work

# 移除 binding
openclaw bindings remove <binding-id>

# 测试路由
openclaw gateway call message.route --params '{"source":"telegram","peer":"123456789"}'
```

### 调试命令

```bash
# 查看路由日志
openclaw logs | grep routing

# 查看会话状态
openclaw sessions list

# 查看 Gateway 状态
openclaw gateway status
```

## 3.7.10 常见问题

### Q1: 消息没有路由到正确的 Agent

```bash
# 检查 bindings 配置
openclaw bindings list

# 检查匹配顺序
# 确认 peer/guildId 等字段是否正确

# 测试路由
openclaw gateway call message.route --params '{"source":"telegram","peer":"123456789","channel":"telegram"}'
```

### Q2: 多账户路由混乱

```bash
# 检查 accountId 配置
openclaw config get channels.telegram.accounts

# 确认 bindings 中的 accountId 匹配
# 建议使用明确的 accountId 匹配
# 设置 defaultAccount 避免回退路由问题
```

### Q3: Discord 角色路由不生效

```bash
# 检查 guildId 是否正确
# 需要使用 Discord 服务器 ID，不是频道 ID

# 检查角色名称/ID
# 确认角色配置正确
# 所有字段必须全部匹配才能生效
```

## 本节小结

1. **消息路由**：Gateway 核心功能，消息回复返回原渠道
2. **Session Key**：按 DM/群组/频道/线程区分会话
3. **Binding 规则**：8 级优先级，精确匹配优先
4. **dmScope**：控制私聊会话分桶策略
5. **广播组**：同一消息触发多个 Agent 并行处理
