# AI Agent 路由策略完整指南

> 路由是 Agent 系统的灵魂——让正确的 Agent 处理正确的任务
> 日期：2026-03-27

---

## 1. 路由机制概述

### 1.1 什么是路由？

**路由（Routing）** = 根据一定规则，将用户请求分发到最合适的 Agent 或工作流的过程。

```
用户请求
    ↓
路由引擎 (Router)
    ↓
匹配规则 → 选择 Agent
    ↓
Agent 处理 → 返回结果
```

### 1.2 路由 vs 绑定

| 概念 | 说明 | OpenClaw 实现 |
|------|------|---------------|
| **Binding** | 通道级路由（哪个群绑定哪个Agent） | `bindings` 配置 |
| **Routing** | 意图级路由（什么意图分发给哪个Agent） | SOUL.md/AGENTS.md 规则 |
| **Workflow** | 流程级路由（复杂任务的多Agent流水线） | 状态机/流程编排 |

## 2. 八大路由机制详解

### 2.1 @直接路由

**原理**：用户显式 @指定 Agent，消息直接路由到该 Agent。

```
用户: @分数分析师 我的分数能上什么大学？
    ↓
路由: 匹配到 @分数分析师
    ↓
Agent: 分数分析师 处理
```

**适用场景**：
- 多个专业 Agent 并存
- 用户明确知道需要哪个 Agent
- 高频专业咨询

**OpenClaw 配置**：
```json5
{
  bindings: [
    // Telegram群中 @分数分析师 → 路由到对应Agent
    { agentId: "gaokao-score", match: { channel: "telegram", chatId: "-5154981764" } },
  ]
}
```

**SOUL.md 规则示例**：
```markdown
## 路由规则

当用户 @分数分析师 时，调用 gaokao-score Agent 处理。

当用户 @学校推荐师 时，调用 gaokao-school Agent 处理。
```

---

### 2.2 意图识别路由

**原理**：LLM 理解用户意图后，智能分发到合适的 Agent。

```
用户: 帮我看看能上什么大学
    ↓
LLM 理解: 意图 = 高考志愿咨询
    ↓
路由: → 学校推荐师 Agent
    ↓
Agent: 返回大学列表
```

**适用场景**：
- 用户表达自然语言
- 无明确 Agent 引用
- 上下文模糊

**SOUL.md 规则示例**：
```markdown
## 意图理解流程

当用户描述高考相关需求（但没有@指定Agent）时：
1. 分析用户核心意图
2. 匹配最合适的 Agent
3. 转发请求并返回结果

意图分类：
- "能上什么大学" / "报考" / "志愿" → 学校推荐师
- "选什么专业" / "专业推荐" / "就业方向" → 专业匹配师
- "多少分" / "录取线" / "位次" → 分数分析师
```

---

### 2.3 关键词/模式匹配路由

**原理**：命中特定关键词 → 触发对应 Agent。

```
用户: "我的位次是1000名"
    ↓
关键词匹配: "位次" 命中
    ↓
→ 分数分析师 Agent
```

**适用场景**：
- 高频标准问题
- 明确术语/概念
- 快速简单分发

**SOUL.md 规则示例**：
```markdown
## 关键词路由表

| 关键词 | 目标Agent | 说明 |
|--------|-----------|------|
| 位次 / 排名 / 分数段 | 分数分析师 | 分数换算、位次查询 |
| 学校 / 大学 / 报考 | 学校推荐师 | 院校推荐、录取概率 |
| 专业 / 方向 / 就业 | 专业匹配师 | 专业分析、职业规划 |
| 志愿 / 填报 / 录取 | 志愿规划师 | 志愿填报策略 |
| 政策 / 规则 / 改革 | 政策研究员 | 高考政策解读 |
```

**正则模式匹配**：
```markdown
## 正则路由规则

当消息匹配以下正则时：
- `\d+分` → 分数分析师（提取分数）
- `\d+名` → 分数分析师（提取位次）
- `(清华|北大|复旦|交大)` → 学校推荐师（目标院校）
- `(计算机|医学|金融|法律)` → 专业匹配师（专业选择）
```

### 2.4 上下文路由

**原理**：根据对话历史状态，决定继续保持当前 Agent 还是切换。

```
用户: "志愿填报有什么技巧？" (首次)
    ↓ 匹配关键词 "志愿" → 志愿规划师
    ↓
Agent: 志愿规划师 处理
    ↓
用户: "那平行志愿呢？" (追问)
    ↓
上下文识别: 仍处于"志愿咨询"状态
    ↓
→ 继续保持 志愿规划师 处理
```

**适用场景**：
- 连续追问
- 多轮深度对话
- 复杂问题的渐进式解答

**SOUL.md 规则示例**：
```markdown
## 上下文路由规则

### 保持当前 Agent 的条件
- 用户追问与当前 Agent 领域相关
- 对话轮次 < 5 且意图未变
- 用户未明确切换

### 切换 Agent 的条件
- 用户明确 @另一个Agent
- 意图发生明显转变（如从"查分数"转到"问专业"）
- 超过 10 轮无结果

### 状态标记
当 Agent 处理时，在 context 中记录：
- current_domain: "高考/学校推荐"
- depth: 3
- agent_id: "gaokao-school"
```

### 2.5 工作流编排（Pipeline 路由）

**原理**：复杂任务触发多 Agent 流水线，串联处理。

```
高考规划任务
    ↓
分数分析师 (获取分数) → 学校推荐师 (匹配院校) → 专业匹配师 (推荐专业) → 志愿规划师 (生成方案)
    ↓
职业评估师 (职业规划) → 最终报告
```

**适用场景**：
- 复杂多步骤任务
- 需要多专业知识
- 标准化流程

**SOUL.md 工作流定义**：
```markdown
## 高考规划工作流

当用户说"帮我规划高考志愿"时，执行以下流水线：

### Step 1: 分数分析
- Agent: 分数分析师
- 输入: 用户分数/位次
- 输出: 分数定位、可报院校范围

### Step 2: 院校推荐
- Agent: 学校推荐师
- 输入: Step 1 结果 + 用户偏好
- 输出: 冲/稳/保院校列表

### Step 3: 专业匹配
- Agent: 专业匹配师
- 输入: 用户兴趣 + 分数范围
- 输出: 推荐专业及就业前景

### Step 4: 志愿生成
- Agent: 志愿规划师
- 输入: Step 2+3 结果
- 输出: 志愿填报方案

### Step 5: 职业规划
- Agent: 职业评估师
- 输入: 推荐专业
- 输出: 职业发展路径

### 最终输出
整合以上结果，生成《高考志愿规划报告》
```

**状态机表示**：
```markdown
## 高考规划状态机

States: IDLE → SCORE_INPUT → SCHOOL_MATCH → MAJOR_MATCH → PLAN_GENERATE → COMPLETE

Transitions:
- IDLE + "规划志愿" → SCORE_INPUT
- SCORE_INPUT + "提供分数" → SCHOOL_MATCH
- SCHOOL_MATCH + "确认学校" → MAJOR_MATCH
- MAJOR_MATCH + "确认专业" → PLAN_GENERATE
- PLAN_GENERATE + "生成方案" → COMPLETE
- COMPLETE + "新任务" → IDLE
```

### 2.6 Fallback 路由（兜底机制）

**原理**：无任何规则匹配时，交给默认 Agent 或人工处理。

```
用户: "今天天气怎么样？" (非高考问题)
    ↓
路由: 无关键词匹配
    ↓
Fallback: main Agent 处理
    ↓
main: 调用天气 skill → 返回天气
```

**适用场景**：
- 跨领域问题
- 意图模糊
- Agent 无法处理

**SOUL.md 规则示例**：
```markdown
## Fallback 路由规则

### 兜底顺序
1. main Agent（通用能力）
2. shenghuo（生活服务）
3. 转人工（记录日志，人工处理）

### 触发条件
- 无任何关键词匹配
- 当前 Agent 返回 "无法处理"
- LLM 判断超出能力范围

### 日志记录
每次触发 Fallback 时：
- 记录用户请求
- 记录未匹配原因
- 用于优化路由规则
```

### 2.7 时序路由（时间/事件触发）

**原理**：根据时间段或特定事件触发专属路由。

```
日常时期 (1月-5月)
    ↓
通用高考咨询 → 学校推荐师

高考出分期 (6月-7月)
    ↓
高峰期工作流 → 分数分析师 + 志愿规划师（优先级提升）

录取期 (8月)
    ↓
录取查询 → 录取追踪 Agent
```

**适用场景**：
- 季节性需求变化
- 高考出分/填报关键期
- 促销活动期

**SOUL.md 规则示例**：
```markdown
## 时序路由规则

### 常规期（1月-5月）
- 路由到: 通用高考咨询
- 优先级: 均衡

### 出分期（6月10日-6月25日）
- 触发条件: 高考出分
- 路由到: 分数分析师（高优）
- 特殊处理: 并行处理 + 限流

### 填报期（6月25日-7月5日）
- 触发条件: 志愿填报开始
- 路由到: 志愿规划师（高优）
- 特殊处理: 实时预警 + 截止提醒

### 录取期（7月-8月）
- 触发条件: 录取开始
- 路由到: 录取追踪 Agent
- 特殊处理: 录取结果查询
```

**Cron 触发配置**：
```json
{
  "name": "高考高峰期路由切换",
  "schedule": { "kind": "cron", "expr": "0 0 10 6 *", "tz": "Asia/Shanghai" },
  "action": "updateRouting",
  "payload": { "mode": "high_priority" }
}
```

### 2.8 优先级路由（紧急任务专属通道）

**原理**：紧急/高优任务走快速通道，优先处理。

```
填报截止前24小时
    ↓
紧急路由: 志愿评估师（优先）
    ↓
快速响应 + 预警通知
```

**适用场景**：
- 截止时间敏感
- 紧急问题
- VIP 用户

**SOUL.md 规则示例**：
```markdown
## 优先级路由规则

### P0 - 紧急（立即处理）
- 触发: 填报截止前24小时
- 路由: 志愿规划师（专属快速通道）
- SLA: 30秒内响应

### P1 - 高优（优先处理）
- 触发: 分数异常/录取异常
- 路由: 分数分析师
- SLA: 5分钟内响应

### P2 - 常规（标准处理）
- 触发: 普通咨询
- 路由: 按规则匹配
- SLA: 30分钟内响应

### VIP 通道
- 触发: VIP 用户
- 路由: 专属 Agent + 人工辅助
- SLA: 绿色通道
```

## 3. 路由规则配置位置

### 3.1 Binding 层（通道路由）

`openclaw.json` 中的 `bindings` 配置：

```json5
{
  bindings: [
    // 群绑定
    { agentId: "gaokao-service", match: { channel: "telegram", chatId: "-5154981764" } },
    { agentId: "shenghuo", match: { channel: "telegram", chatId: "-5153278498" } },
    { agentId: "mo-finance", match: { channel: "telegram", chatId: "-1003633299525" } },
    
    // DM 绑定
    { agentId: "main", match: { channel: "telegram", chatType: "direct" } },
  ]
}
```

### 3.2 SOUL.md 层（意图路由）

在业务 Agent 的 `SOUL.md` 中定义：

```markdown
## 路由规则

### 关键词路由
- "分数" / "位次" → 分数分析师
- "学校" / "大学" → 学校推荐师

### 意图理解
当无关键词匹配时，LLM 判断意图并分发。
```

### 3.3 AGENTS.md 层（行为路由）

```markdown
## 路由行为

### 转发条件
- 明确 @指定其他Agent
- 命中关键词路由表
- 意图识别匹配

### 转发格式
当需要转发时：
1. 说明转发原因
2. 调用目标Agent
3. 整合结果返回
```

## 4. 路由决策流程图

```
用户消息
    ↓
┌─ @指定Agent？ ─Yes→ 直接路由到该Agent
│
└─No→
    ┌─ 关键词匹配？ ─Yes→ 路由到对应Agent
    │
    └─No→
        ┌─ 上下文保持？ ─Yes→ 保持当前Agent
        │
        └─No→
            ┌─ 意图识别 ─Yes→ 路由到最佳Agent
            │
            └─No→
                ┌─ 时间/事件触发？ ─Yes→ 触发对应工作流
                │
                └─No→
                    Fallback → main / 转人工
```

## 6. 路由优化建议

### 6.1 日志记录
```markdown
每次路由决策时记录：
- 用户ID
- 原始请求
- 匹配规则
- 路由结果
- 处理时长
- 是否成功
```

### 6.2 效果分析
```markdown
定期分析：
- 各Agent处理量分布
- 路由命中率
- Fallback触发率
- 用户满意度
```

### 6.3 持续迭代
```markdown
根据日志优化：
1. 新增高频意图的路由规则
2. 合并低效路由
3. 调整优先级
4. 补充 Fallback 场景
```

## 7. 总结

| 路由机制 | 触发条件 | 优先级 | 适用场景 |
|---------|---------|--------|---------|
| @直接路由 | 用户@指定 | P0 | 明确需求 |
| 关键词匹配 | 命中关键词 | P1 | 高频标准问题 |
| 上下文路由 | 对话历史状态 | P2 | 连续追问 |
| 意图识别 | LLM理解 | P2 | 自然语言 |
| 工作流编排 | 复杂任务 | P1 | 多步骤流程 |
| 时序路由 | 时间/事件 | P1 | 季节性需求 |
| 优先级路由 | 紧急/高优 | P0 | 截止提醒 |
| Fallback | 无匹配 | P3 | 兜底处理 |

**最佳实践**：
1. 优先 @直接路由（用户明确）
2. 其次关键词匹配（高频场景）
3. 再意图识别（自然语言）
4. 最后 Fallback（兜底）
