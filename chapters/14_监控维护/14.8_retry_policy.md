# 10.8 重试策略

本节讲解 OpenClaw 的重试策略机制，包括重试目标、默认配置、各渠道行为以及如何自定义重试规则。

---

## 10.8.1 重试策略目标

### 三大目标

```
┌─────────────────────────────────────────────────────────────┐
│                     重试策略核心目标                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 按请求重试                                             │
│     └── 每个 HTTP 请求独立重试，非整个多步骤流程             │
│                                                             │
│  2. 保持顺序                                               │
│     └── 只重试当前步骤，保持执行顺序                         │
│                                                             │
│  3. 避免重复                                               │
│     └── 不重复执行非幂等操作                                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 10.8.2 默认配置

### 全局默认值

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 尝试次数 | 3 | 最大重试次数 |
| 最大延迟 | 30000ms | 延迟上限 |
| 抖动 | 0.1 (10%) | 随机抖动因子 |

### 渠道特定默认值

| 渠道 | 最小延迟 |
|------|----------|
| Telegram | 400ms |
| Discord | 500ms |

## 10.8.3 各渠道重试行为

### Discord

```
重试条件：仅限速率限制错误 (HTTP 429)
延迟策略：优先使用 Discord 的 retry_after，否则指数退避
```

### Telegram

```
重试条件：
├── 429 (速率限制)
├── timeout (超时)
├── connect/reset/closed (连接问题)
└── temporarily unavailable (临时不可用)

延迟策略：优先使用 retry_after，否则指数退避

不重试：
└── Markdown 解析错误 → 回退到纯文本
```

## 10.8.4 配置示例

### 完整配置

```json5
{
  channels: {
    telegram: {
      retry: {
        attempts: 3,
        minDelayMs: 400,
        maxDelayMs: 30000,
        jitter: 0.1,
      },
    },
    discord: {
      retry: {
        attempts: 3,
        minDelayMs: 500,
        maxDelayMs: 30000,
        jitter: 0.1,
      },
    },
  },
}
```

### 简化配置

```json5
{
  channels: {
    telegram: {
      retry: {
        attempts: 5,
      },
    },
  },
}
```

## 10.8.5 重试行为详解

### 什么会重试

```
每个请求独立重试：
├── 消息发送
├── 媒体上传
├── 表情反应
├── 投票创建
└── 贴纸发送
```

### 什么不会重试

```
复合流程不重试已完成的步骤：
└── 多步骤操作中，成功执行的步骤不会重做
```

---

## 10.8.6 指数退避示例

```
重试延迟计算公式：
delay = min(cap, base * (2 ^ attempt)) * (1 + jitter)

示例（base=1000ms, cap=30000, jitter=0.1）：
├── 第1次重试：~2000ms (1000 * 2^1)
├── 第2次重试：~4000ms (1000 * 2^2)
├── 第3次重试：~8000ms (1000 * 2^3)
└── 第4次重试：~30000ms (达到上限)
```

## 本节小结

1. **目标**：按请求重试、保持顺序、避免重复
2. **默认**：3 次尝试、30s 上限、10% 抖动
3. **渠道差异**：Discord 只重试 429，Telegram 重试多种错误
4. **配置**：可按渠道自定义重试参数
