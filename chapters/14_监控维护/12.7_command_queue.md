# 10.7 命令队列与消息处理

本节讲解 OpenClaw 的命令队列机制和消息处理策略，包括队列模式、并发控制、消息合并等核心概念。

---

## 10.7.1 为什么需要命令队列

### 核心问题

- 自动回复运行成本高（LLM 调用），多条消息同时到达时可能发生碰撞
- 串行化处理可以避免争夺共享资源（会话文件、日志、CLI stdin）
- 减少上游速率限制的可能性

### 队列的设计目标

```
┌─────────────────────────────────────────────────────────────┐
│                     命令队列核心目标                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 串行化处理                                             │
│     └── 防止多个 Agent 运行发生冲突                          │
│                                                             │
│  2. 会话隔离                                               │
│     └── 不同会话可以安全并行                                  │
│                                                             │
│  3. 资源保护                                               │
│     └── 避免争夺共享资源                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 10.7.2 队列工作机制

### 泳道感知 FIFO 队列

- 每个泳道按 FIFO 顺序处理，可配置并发上限
- 默认未配置的泳道并发为 1；`main` 默认为 4，子 Agent 默认为 8
- 按**会话键**入队 (`session:<key>`)，保证每个会话只有 一个活跃运行

### 并发控制

```
全局泳道（默认 main）：
└── 并行度由 agents.defaults.maxConcurrent 控制

其他泳道：
├── cron - 定时任务
├── subagent - 子 Agent
└── ... - 自定义泳道
```

![消息队列](./.assets/diagrams/27_message_queue.png)



### 消息处理流程

```
1. 消息入队 → 按会话键进入对应泳道
2. 等待并发槽位 → 超过并发上限则排队等待
3. 开始执行 → 占用会话锁
4. 执行完成 → 释放锁，其他消息可继续
```

## 10.7.3 队列模式详解

### 五种队列模式

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| `steer` | 立即注入当前运行（取消后续工具调用） | 需要即时响应（**推荐**） |
| `followup` | 入队，等待当前运行结束后处理 | 连续对话 |
| `collect` | 合并所有排队消息为**单个** followup（默认） | 批量处理 |
| `steer-backlog` | 立即注入 + 保留消息用于 followup | 即时+后续 |
| `interrupt` | 中止当前运行，处理最新消息 | 紧急插队 |

> ⚠️ **v2026.3.24 重要变更**：该版本将 `mode` 默认值改为 `collect`，导致 `/stop` 和 `/queue interrupt` 无法中断正在执行的任务。**强烈建议手动配置为 `steer`**（见下方 Workaround）。

### ⚠️ 默认模式导致 /stop 失效（v2026.3.24 Bug #56044）

**问题现象**：发送 `/stop` 或 `/queue interrupt` 后，Agent 仍在继续运行，用户消息被静默排队。

**根因**：v2026.3.24 将默认模式改为 `collect`，该模式会批量收集消息，吞掉中断信号。

**Workaround**：在 `openclaw.json` 中显式设置 `steer` 模式：

```json5
{
  messages: {
    queue: {
      mode: "steer",   // ✅ 改为 steer 以恢复 /stop 中断能力
      debounceMs: 1000,
      cap: 20,
      drop: "summarize",
      byChannel: { discord: "collect" },
    },
  },
}
```

配置后重启 Gateway：

```bash
openclaw gateway restart
```

验证配置：

```bash
openclaw config get messages.queue.mode
# 应输出: steer
```

**已知影响**：目前仅有 `steer` 模式能正确响应 `/stop` 中断，其他模式在 v2026.3.24 下均失效。官方修复跟踪：[Issue #56044](https://github.com/openclaw/openclaw/issues/56044)

### 配置示例

```json5
{
  messages: {
    queue: {
      mode: "steer",   // ✅ 推荐显式配置，避免版本更新后行为变化
      debounceMs: 1000,
      cap: 20,
      drop: "summarize",
      byChannel: { discord: "collect" },
    },
  },
}
```

### 队列选项

| 选项 | 说明 | 默认值 |
|------|------|--------|
| `debounceMs` | 静默等待时间，防止"继续，继续" | 1000ms |
| `cap` | 每会话最大排队消息数 | 20 |
| `drop` | 溢出策略：`old`/`new`/`summarize` | summarize |

**Summarize 策略**：保留被丢弃消息的简短摘要，注入为合成的 followup 提示。

## 10.7.4 会话级覆盖

### Slash 命令

```bash
# 设置队列模式
/queue collect
/queue steer
/queue followup

# 带选项
/queue collect debounce:2s cap:25 drop:summarize

# 重置为默认
/queue default
/queue reset
```

## 10.7.5 诊断与排障

### 查看队列状态

```bash
# 启用详细日志
openclaw logs --verbose

# 查找排队信息
# "queued for …ms" 表示排队等待时间
```

### 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 命令看起来卡住 | 队列满或并发受限 | 启用 verbose 日志 |
| 响应延迟高 | 消息堆积 | 检查 debounceMs 配置 |
| 重复响应 | steer-backlog 模式 | 改用 collect/steer |

## 本节小结

1. **队列目的**：串行化处理、避免资源冲突
2. **泳道机制**：按会话隔离，支持并行
3. **五种模式**：steer/followup/collect/steer-backlog/interrupt
4. **队列选项**：debounceMs、cap、drop 策略
