14.1 多渠道智能客服系统 ​

场景概述 ​

小型企业需要同时管理 WhatsApp、Instagram DM、Gmail 和 Google Reviews，但客户期望 24/7 即时响应，而雇佣客服人员成本高昂。本案例展示如何构建一个统一的 AI 客服平台。

核心功能 ​

  • 统一收件箱：WhatsApp Business、Instagram DM、Gmail、Google Reviews 集中管理
  • AI 自动回复：自动处理常见问题、预约请求
  • 人工转接：复杂问题转人工处理或标记待审核
  • 测试模式：演示系统而不影响真实客户
  • 业务上下文：基于服务、定价、政策训练

真实企业案例 ​

某餐饮企业部署后，响应时间从 4+ 小时缩短至 2 分钟内，80% 的咨询实现自动处理。

所需技能 ​

  • WhatsApp Business API 集成
  • Instagram Graph API（通过 Meta Business）
  • Gmail CLI（gog）
  • Google Business Profile API
  • AGENTS.md 中的消息路由逻辑

实施步骤 ​

1. 连接渠道 ​

通过 OpenClaw 配置连接各渠道：

![日报案例]/assets/diagrams/72_case_daily_report.png)

![业务指标]/assets/diagrams/121_business_metrics.png)

json
{
  "channels": {
    "whatsapp": {
      "enabled": true,
      "type": "business",
      "provider": "360dialog"
    },
    "instagram": {
      "enabled": true,
      "type": "messenger",
      "business_id": "${INSTAGRAM_BUSINESS_ID}"
    },
    "gmail": {
      "enabled": true,
      "type": "email",
      "oauth": {
        "client_id": "${GMAIL_CLIENT_ID}",
        "client_secret": "${GMAIL_CLIENT_SECRET}"
      }
    }
  }
}

2. 创建业务知识库 ​

创建业务知识库包含：

  • 服务项目和定价
  • 营业时间和位置
  • FAQ 响应模板
  • 转接触发条件（如投诉、退款请求）

3. 配置路由逻辑 ​

在 AGENTS.md 中配置路由规则：

markdown
## 客服模式

收到客户消息时：

1. 识别渠道（WhatsApp/Instagram/Email/Review）
2. 检查是否为测试模式
3. 意图分类：
   - FAQ → 从知识库回复
   - 预约 → 检查可用时间，确认预订
   - 投诉 → 标记人工审核，确认收到
   - 评价 → 感谢反馈，解决疑虑

回复风格：
- 友好、专业、简洁
- 匹配客户语言（中文/英文/其他）
- 不虚构知识库外的信息
- 落款注明商家名称

测试模式：
- 回复前添加 [TEST] 前缀
- 记录但不发送到真实渠道

4. 设置心跳检查 ​

配置定时检查监控响应：

markdown
## 心跳：客服检查

每 30 分钟：
- 检查未回复消息（超过 5 分钟）
- 如果响应队列积压则告警
- 记录每日响应指标

关键洞察 ​

  • 语言检测很重要：自动检测并用客户语言回复
  • 测试模式必不可少：客户需要先看到效果再上线
  • 转接规则：明确定义升级触发条件，避免 AI 越权
  • 回复模板：敏感话题（退款、投诉）使用预批准模板

14.2 个人第二大脑 ​

场景概述 ​

你有很多想法、有趣的链接、想读的书，但没有好的系统来记录它们。Notion 变得太复杂，Apple Notes 积累了上万条未读笔记。你需要一个像给朋友发短信一样简单的系统。

核心功能 ​

  • 零摩擦捕获：通过 Telegram、iMessage 或 Discord 发送任何内容
  • 永久记忆：OpenClaw 内置记忆系统永久存储所有内容
  • 可搜索 UI：自定义 Next.js 仪表板，随时浏览所有记忆
  • 全局搜索：跨所有记忆、文档和任务搜索
  • 无需整理：无需文件夹、标签、复杂组织

痛点解决 ​

每个笔记应用最终都会变得繁琐。你停止使用它，因为整理的摩擦高于遗忘的摩擦。核心洞察是：捕获应该像发短信一样简单，检索应该像搜索一样简单。

所需技能 ​

  • Telegram、iMessage 或 Discord 集成
  • Next.js（OpenClaw 为你构建，无需编码）

实施步骤 ​

1. 连接消息平台 ​

确保 OpenClaw 连接到首选消息平台（Telegram、Discord 等）。

2. 开始使用 ​

立即开始使用——只需向你的机器人发送任何想记住的内容：

Hey，提醒我读 "Designing Data-Intensive Applications"
保存这个链接：https://example.com/interesting-article
记住：John 推荐了 5 街上的那家餐厅

3. 构建可搜索 UI ​

通过提示 OpenClaw 构建可搜索 UI：

我想要构建一个第二大脑系统，可以回顾我们所有的笔记、
对话和记忆。请用 Next.js 构建。

包括：
- 所有记忆和对话的可搜索列表
- 跨所有内容的全局搜索（Cmd+K）
- 按日期和类型筛选的能力
- 简洁、最小化 UI

4. 访问仪表板 ​

OpenClaw 将为你构建和部署整个 Next.js 应用。访问它提供的 URL，你就有了一个第二大脑仪表板。

5. 持续使用 ​

以后无论何时想到什么——在手机上、在会议上、睡前——只需发消息给你的机器人。需要查找时回到仪表板。

关键洞察 ​

  • 零摩擦捕获的力量：无需打开应用、选择文件夹或添加标签，只需发消息
  • 记忆系统是累积的：它会记住你告诉它的每一件事，随着时间推移越来越强大
  • 可以在手机上发消息，它在你的电脑上构建东西：界面就是对话

14.3 自动化开发助手 ​

场景概述 ​

软件开发团队需要一个全天候的 AI 编程助手，能够进行代码审查、bug 定位、技术文档生成和命令辅助。

核心功能 ​

  • 代码生成：根据需求生成高质量代码
  • 代码审查：自动审查代码规范、安全漏洞、性能问题
  • Bug 定位：分析错误日志，定位问题根源
  • 技术文档：自动生成 API 文档、README
  • 命令辅助：提供正确的终端命令

所需技能 ​

  • Claude 3.5 Sonnet（编程能力最强）
  • 代码相关工具（read、write、edit、exec、grep）
  • 代码审查 Skill

实施步骤 ​

1. 配置编程模型 ​

yaml
models:
  coder:
    provider: anthropic
    model: claude-3-5-sonnet-20241022
    temperature: 0.3
    max_tokens: 8192

2. 配置代码工具 ​

yaml
tools:
  enabled:
    - read
    - write
    - edit
    - exec
    - grep
    - glob

  exec:
    allowed_commands:
      - npm
      - yarn
      - git
      - docker
      - docker-compose
      - go
      - cargo

    denied_commands:
      - rm -rf /
      - sudo
      - chown

3. 配置代码审查 Skill ​

yaml
skill: code_review
description: 专业的代码审查技能

capabilities:
  - 代码规范检查（ESLint、Prettier）
  - 安全漏洞扫描
  - 性能问题识别
  - 代码可维护性评估
  - 修复建议生成

prompt: |
  你是一个资深代码审查专家，具有 10 年开发经验。
  审查标准：
  - 代码规范：命名、格式、注释
  - 安全检查：SQL 注入、XSS、敏感信息
  - 性能：循环优化、缓存使用、数据库查询
  - 可维护性：代码复杂度、重复代码、依赖管理

4. 配置工作流 ​

yaml
workflows:
  code_review:
    trigger: pull_request

    stages:
      - name: scan
        tool: security_scan
        description: 安全扫描

      - name: review
        agent: code_reviewer
        description: 代码审查

      - name: comment
        tool: github_comment
        description: 添加审查评论

使用示例 ​

代码审查 ​

用户：审查这段代码

→ 读取代码文件
→ 分析代码结构
→ 检查安全漏洞
→ 评估性能问题
← 返回审查报告

审查报告：
✅ 代码规范通过
⚠️ 3 个潜在问题
💡 5 条优化建议

Bug 定位 ​

用户：帮我看看这个错误

错误日志：
TypeError: Cannot read property 'foo' of undefined
at Object.<anonymous> (/app/index.js:10:15)

→ 分析错误堆栈
→ 定位问题代码
→ 分析原因
← 返回分析结果

分析：
- 问题：访问了 undefined 对象的属性
- 位置：index.js 第 10 行
- 原因：bar 可能为 undefined
- 建议：添加空值检查

14.4 知识库 RAG 系统 ​

场景概述 ​

企业需要构建一个基于私有文档的知识库问答系统，让 AI 能够基于企业文档回答用户问题。

核心功能 ​

  • 文档上传：支持 PDF、Word、Markdown 等格式
  • 智能分块：自动将文档分割为语义完整的小块
  • 向量存储：使用向量数据库存储文档嵌入
  • 语义检索：基于语义相似度检索相关内容
  • 问答生成：基于检索内容生成准确答案

真实企业案例 ​

某咨询公司将历史项目文档导入系统后，客户咨询响应时间缩短 70%，知识复用率提升 50%。

所需技能 ​

  • 文档处理工具（PDF 解析、OCR）
  • 向量数据库（Pinecone、Weaviate、Qdrant）
  • RAG 技能

实施步骤 ​

1. 配置向量存储 ​

json
{
  "memory": {
    "vector": {
      "enabled": true,
      "provider": "qdrant",
      "url": "http://localhost:6333",
      "collection": "knowledge_base",
      "dimensions": 1536
    }
  }
}

2. 配置文档处理 ​

yaml
documents:
  supported_formats:
    - pdf
    - docx
    - md
    - txt

  chunking:
    method: semantic
    overlap: 100
    max_size: 1000

3. 创建知识库 ​

markdown
## 知识库构建流程

1. 上传文档到知识库目录
2. 系统自动：
   - 解析文档格式
   - 提取文本内容
   - 语义分块
   - 生成向量
   - 存储到向量数据库

3. 知识库就绪，可以问答

4. 问答配置 ​

yaml
rag:
  enabled: true
  top_k: 5
  similarity_threshold: 0.7

  prompt: |
    你是一个专业的知识库问答助手。
    基于以下参考资料回答用户问题。

    参考资料：
    {{context}}

    回答要求：
    - 只基于参考资料回答
    - 如不确定，坦诚说明
    - 引用来源

使用示例 ​

用户：我们的退换货政策是什么？

→ 检索知识库
→ 找到相关文档
→ 提取关键信息
← 返回答案

根据我们的退换货政策：
- 商品未使用可 7 天内退货
- 质量问题 30 天内换货
- 退货需保留完整包装

详情请参考：退换货政策文档.pdf

14.5 多 Agent 团队协作 ​

场景概述 ​

需要构建一个多 Agent 协作团队，分别负责需求分析、代码开发、代码审查、测试等不同阶段。

Agent 团队配置 ​

架构师 Agent ​

yaml
architect:
  name: 架构师
  model: gpt-4o
  role: 技术架构设计

  responsibilities:
    - 系统架构设计
    - 技术选型
    - 代码审查
    - 性能优化

  skills:
    - system_design
    - tech_selection

开发 Agent ​

yaml
developer:
  name: 开发工程师
  model: claude-3-5-sonnet
  role: 代码实现

  responsibilities:
    - 代码实现
    - Bug 修复
    - 单元测试

  tools:
    - read
    - write
    - edit
    - exec

审查 Agent ​

yaml
reviewer:
  name: 代码审查员
  model: claude-3-5-sonnet
  role: 质量把关

  responsibilities:
    - 代码规范检查
    - 安全漏洞扫描
    - 改进建议

  tools:
    - read
    - grep

测试 Agent ​

yaml
tester:
  name: 测试工程师
  model: gpt-4o-mini
  role: 质量验证

  responsibilities:
    - 测试用例生成
    - 执行测试
    - 报告缺陷

工作流配置 ​

yaml
workflow: code_development

stages:
  - agent: architect
    task: design
    output: architecture_doc
    prompt: 设计一个电商系统架构

  - agent: developer
    task: implement
    input: architecture_doc
    output: code

  - agent: reviewer
    task: review
    input: code
    output: review_report

  - agent: developer
    task: fix
    input: review_report
    output: fixed_code
    condition: review_report.has_issues

  - agent: tester
    task: test
    input: fixed_code
    output: test_report

需求：构建一个用户管理系统
