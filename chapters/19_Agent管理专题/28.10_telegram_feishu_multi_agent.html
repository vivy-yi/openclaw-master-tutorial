一、现状分析 ​

二、Telegram 群组多 Agent 方案 ​

方案一：Binding 路由（基础方案） ​

原理：一个群组绑定到一个主 Agent，通过主 Agent 调用其他 Agent

┌─────────────────────────────────────────────────────────────────────────────────┐
│                        Telegram Binding 路由方案                                 │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  用户消息 → 群组 → Binding → 主 Agent → 汇总结果 → 回复群组                  │
│                                                                                 │
│  特点：                                                                       │
│  • 群里只看到一个 Agent 回复                                                  │
│  • 主 Agent 可以调用 sessions_spawn/sessions_send 调用其他 Agent               │
│  • 所有渠道支持                                                               │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

配置示例：

json5
{
  agents: {
    list: [
      {
        id: "main",
        name: "主助手",
        workspace: "~/.openclaw/workspace-main"
      },
      {
        id: "coder",
        name: "代码专家",
        workspace: "~/.openclaw/workspace-coder"
      },
      {
        id: "docs",
        name: "文档助手",
        workspace: "~/.openclaw/workspace-docs"
      }
    ]
  },
  // 启用跨 Agent 通信
  tools: {
    agentToAgent: {
      enabled: true,
      allow: ["main", "coder", "docs"]
    }
  },
  channels: {
    telegram: {
      enabled: true,
      botToken: "YOUR_BOT_TOKEN",
      groups: {
        "-1001234567890": {
          allow: true,
          requireMention: true
        }
      }
    }
  },
  bindings: [
    {
      agentId: "main",
      match: {
        channel: "telegram",
        peer: {
          kind: "group",
          id: "-1001234567890"
        }
      }
    }
  ]
}

在 AGENTS.md 中配置主 Agent 行为：

markdown
# 主 Agent 工作规则

## 任务分发逻辑

当用户发送消息时，根据内容判断调用哪个 Agent：

1. 包含代码/编程/调试 → 使用 sessions_spawn 调用 coder Agent
2. 包含文档/写作/总结 → 使用 sessions_spawn 调用 docs Agent
3. 其他问题 → 自行处理

## 调用示例

使用 sessions_spawn 生成子 Agent：
- task: "请审查这段代码：\n<code>"
- agentId: "coder"
- timeout: 120

方案二：论坛话题隔离（进阶方案） ​

原理：利用 Telegram 的论坛话题（Topics）功能，每个话题绑定到不同 Agent

┌─────────────────────────────────────────────────────────────────────────────────┐
│                     Telegram 论坛话题隔离方案                                    │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  群组                                                                         │
│  ├── 通用话题 (Thread ID: 1) → Agent: main                                  │
│  ├── 代码话题 (Thread ID: 2) → Agent: coder                                 │
│  ├── 文档话题 (Thread ID: 3) → Agent: docs                                  │
│  └── 运营话题 (Thread ID: 4) → Agent: ops                                   │
│                                                                                 │
│  特点：                                                                       │
│  • 不同话题 → 不同 Agent                                                       │
│  • 群里可以看到多个 Agent 回复                                                │
│  • Telegram 特有功能                                                          │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

配置示例：

json5
{
  agents: {
    list: [
      { id: "main", workspace: "~/.openclaw/workspace-main" },
      { id: "coder", workspace: "~/.openclaw/workspace-coder" },
      { id: "docs", workspace: "~/.openclaw/workspace-docs" }
    ]
  },
  channels: {
    telegram: {
      enabled: true,
      botToken: "YOUR_BOT_TOKEN",
      groups: {
        "-1001234567890": {
          allow: true,
          requireMention: false,
          topics: {
            // 通用话题 - 主 Agent
            "1": {
              requireMention: false,
              systemPrompt: "你是通用助手，处理日常问题"
            },
            // 代码话题 - 编码 Agent
            "2": {
              requireMention: false,
              systemPrompt: "你是编程专家，专注于代码问题和调试"
            },
            // 文档话题 - 文档 Agent
            "3": {
              requireMention: false,
              systemPrompt: "你是文档专家，专注于技术文档撰写"
            }
          }
        }
      }
    }
  }
}

注意：话题 ID 可以通过查看群组日志获取，或在飞书开放平台查看。

三、飞书群组多 Agent 方案 ​

飞书群组多 Agent 支持情况 ​

根据官方文档，飞书群组支持：

  • ✅ Binding 路由（按群组/用户分流）
  • ✅ 多账户支持
  • ✅ 群组话题（replyInThread）
  • 🚧 广播组（计划中）

方案一：Binding 路由（飞书标准方案） ​

原理：通过 bindings 配置，根据群组 ID 或用户 ID 路由到不同 Agent

┌─────────────────────────────────────────────────────────────────────────────────┐
│                         飞书 Binding 路由方案                                    │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  群组 A (oc_xxx) → Agent: main  → 处理日常对话                               │
│  群组 B (oc_yyy) → Agent: coder → 处理技术问题                               │
│  群组 C (oc_zzz) → Agent: ops  → 处理运营问题                               │
│                                                                                 │
│  私聊 (ou_xxx)  → Agent: main  → 默认路由到主 Agent                         │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

配置示例：

json5
{
  agents: {
    list: [
      {
        id: "main",
        name: "主助手",
        workspace: "~/.openclaw/workspace-main"
      },
      {
        id: "coder",
        name: "技术专家",
        workspace: "~/.openclaw/workspace-coder"
      },
      {
        id: "support",
        name: "客服助手",
        workspace: "~/.openclaw/workspace-support"
      }
    ]
  },
  channels: {
    feishu: {
      enabled: true,
      dmPolicy: "pairing",
      groupPolicy: "allowlist",
      groupAllowFrom: ["oc_group1", "oc_group2", "oc_group3"],
      groups: {
        "oc_group1": { allow: true, requireMention: false },
        "oc_group2": { allow: true, requireMention: false },
        "oc_group3": { allow: true, requireMention: false }
      }
    }
  },
  bindings: [
    // 技术群 → coder Agent
    {
      agentId: "coder",
      match: {
        channel: "feishu",
        peer: { kind: "group", id: "oc_group1" }
      }
    },
    // 客服群 → support Agent
    {
      agentId: "support",
      match: {
        channel: "feishu",
        peer: { kind: "group", id: "oc_group2" }
      }
    },
    // 其他群 → main Agent
    {
      agentId: "main",
      match: {
        channel: "feishu",
        peer: { kind: "group", id: "oc_group3" }
      }
    }
  ]
}

方案二：用户分流（按人路由） ​

原理：根据发送消息的用户 ID 路由到不同 Agent

┌─────────────────────────────────────────────────────────────────────────────────┐
│                         飞书用户分流方案                                        │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  用户 A (ou_aaa) → Agent: main  → 管理员                                    │
│  用户 B (ou_bbb) → Agent: coder → 开发团队                                   │
│  用户 C (ou_ccc) → Agent: ops  → 运营团队                                     │
│  其他用户      → Agent: main  → 默认                                        │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

配置示例：

json5
{
  channels: {
    feishu: {
      enabled: true,
      dmPolicy: "allowlist",
      allowFrom: ["ou_admin", "ou_dev1", "ou_dev2", "ou_ops1"]
    }
  },
  bindings: [
    // 开发用户 → coder Agent
    {
      agentId: "coder",
      match: {
        channel: "feishu",
        peer: { kind: "dm", id: "ou_dev1" }
      }
    },
    {
      agentId: "coder",
      match: {
        channel: "feishu",
        peer: { kind: "dm", id: "ou_dev2" }
      }
    },
    // 运营用户 → ops Agent
    {
      agentId: "ops",
      match: {
        channel: "feishu",
        peer: { kind: "dm", id: "ou_ops1" }
      }
    },
    // 其他 → main Agent
    {
      agentId: "main",
      match: {
        channel: "feishu"
      }
    }
  ]
}

方案三：话题分流（飞书话题） ​

原理：利用飞书群组的话题功能，不同话题路由到不同 Agent

┌─────────────────────────────────────────────────────────────────────────────────┐
│                         飞书话题分流方案                                        │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                 │
│  飞书群组                                                                      │
│  ├── 主话题 → Agent: main    → 日常对话                                       │
│  ├── #代码审查 → Agent: coder → 技术讨论                                       │
│  ├── #文档协作 → Agent: docs → 文档审核                                       │
│  └── #客服支持 → Agent: support → 客户问题                                     │
│                                                                                 │
│  配置：                                                                       │
│  使用 replyInThread 保持话题内对话                                            │
│                                                                                 │
└─────────────────────────────────────────────────────────────────────────────────┘

配置示例：

json5
{
  channels: {
    feishu: {
      enabled: true,
      replyInThread: "group",  // 群组中回复到话题
      groups: {
        "oc_group_main": {
          allow: true,
          requireMention: false
        }
      }
    }
  },
  bindings: [
    // 默认路由到 main
    {
      agentId: "main",
      match: {
        channel: "feishu",
        peer: { kind: "group", id: "oc_group_main" }
      }
    }
  ]
}

在 Agent 内部根据话题名称分流：

在 main Agent 的 AGENTS.md 中配置：

markdown
# 话题识别分流规则

当收到消息时，识别消息所在话题：

1. 消息包含 "#代码审查" 或 "#技术"
   → 使用 sessions_spawn 调用 coder Agent

2. 消息包含 "#文档" 或 "#写作"
   → 使用 sessions_spawn 调用 docs Agent

3. 消息包含 "#客服" 或 "#支持"
   → 使用 sessions_spawn 调用 support Agent

4. 其他 → 自行处理

四、完整配置示例：企业级多 Agent 客服系统 ​

场景描述 ​

企业飞书群组配置：
- 客服群 (oc_support) → 客服 Agent 处理日常咨询
- 技术群 (oc_tech) → 技术 Agent 处理开发问题
- 运营群 (oc_ops) → 运营 Agent 处理数据分析
- 管理层私聊 → 管理员 Agent 处理特殊请求

配置示例 ​

json5
{
  agents: {
    list: [
      {
        id: "main",
        name: "AI 助手",
        workspace: "~/.openclaw/workspace-main",
        default: true
      },
      {
        id: "support",
        name: "客服专员",
        workspace: "~/.openclaw/workspace-support",
        model: "claude-sonnet-4-5"
      },
      {
        id: "coder",
        name: "技术专家",
        workspace: "~/.openclaw/workspace-coder",
        model: "claude-opus-4-5"
      },
      {
        id: "ops",
        name: "运营助手",
        workspace: "~/.openclaw/workspace-ops",
        model: "claude-sonnet-4-5"
      },
      {
        id: "admin",
        name: "管理员助手",
        workspace: "~/.openclaw/workspace-admin",
        model: "claude-opus-4-5"
      }
    ]
  },
  // 启用跨 Agent 协作
  tools: {
    agentToAgent: {
      enabled: true,
      allow: ["main", "support", "coder", "ops", "admin"]
    }
  },
  channels: {
    feishu: {
      enabled: true,
      dmPolicy: "allowlist",
      allowFrom: [
        "ou_admin1",    // 管理员
        "ou_dev1",      // 开发
        "ou_ops1"       // 运营
      ],
      groupPolicy: "allowlist",
      groupAllowFrom: [
        "oc_support",   // 客服群
        "oc_tech",      // 技术群
        "oc_ops"        // 运营群
      ],
      groups: {
        "oc_support": { allow: true, requireMention: false },
        "oc_tech": { allow: true, requireMention: false },
        "oc_ops": { allow: true, requireMention: false }
      },
      replyInThread: "group"  // 话题内回复
    }
  },
  bindings: [
    // 客服群 → support Agent
    {
      agentId: "support",
      match: {
        channel: "feishu",
        peer: { kind: "group", id: "oc_support" }
      }
    },
    // 技术群 → coder Agent
    {
      agentId: "coder",
      match: {
        channel: "feishu",
        peer: { kind: "group", id: "oc_tech" }
      }
    },
    // 运营群 → ops Agent
    {
      agentId: "ops",
      match: {
        channel: "feishu",
        peer: { kind: "group", id: "oc_ops" }
      }
    },
    // 管理员私聊 → admin Agent
    {
      agentId: "admin",
      match: {
        channel: "feishu",
        peer: { kind: "dm", id: "ou_admin1" }
      }
    },
    // 其他默认 → main Agent
    {
      agentId: "main",
      match: {
        channel: "feishu"
      }
    }

选择建议 ​

需求场景推荐方案
不同群组 → 不同 AgentBinding 路由
同一群组不同话题 → 不同 Agent话题隔离
一个问题需要多 Agent 协作处理主 Agent + sessions_spawn
需要群里多个 Agent 同时回复WhatsApp 广播组（目前唯一）

六、最佳实践 ​

安全配置 ​

json5
{
  // 1. 限制群组访问
  channels: {
    feishu: {
      groupPolicy: "allowlist",
      groupAllowFrom: ["oc_trusted_group1", "oc_trusted_group2"]
    }
  },

  // 2. 限制工具权限
  agents: {
    list: [
      {
        id: "readonly",
        tools: {
          allow: ["read", "sessions_list", "sessions_history"],
          deny: ["exec", "write", "browser"]
        }
      }
    ]
  },

  // 3. 启用审计日志
  session: {
    auditLog: true
  }
}

性能优化 ​

• 使用 parallel 策略处理并发请求
• 为简单任务使用轻量模型
• 限制广播组 Agent 数量（建议 ≤10）
• 合理设置 timeoutSeconds 避免长时间等待

本节小结 ​

  1. Telegram 方案：

    • Binding 路由：基础方案，一个群组一个主 Agent
    • 论坛话题：进阶方案，一个话题一个 Agent

  2. 飞书方案：

    • Binding 路由：按群组/用户分流
    • 话题分流：利用飞书话题功能

  3. 核心差异：

    • WhatsApp：唯一支持真正广播组（多 Agent 同时回复）
    • Telegram/飞书：使用 Binding 或话题隔离实现多 Agent
