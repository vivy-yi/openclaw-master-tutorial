# 元团队架构设计指南

> 本文档介绍如何将元团队定位为"平台层"，为所有业务Agent提供通用能力。
> 日期：2026-03-27
> 来源：DM主控对话

---

## 一、核心概念：一个"群"本质是什么？

```
┌─────────────────────────────────────────────────────────┐
│                        群                                │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐    │
│  │  Agent1 │  │  Agent2 │  │  Agent3 │  │  Agent4 │    │ ← 业务执行层
│  └────┬────┘  └────┬────┘  └────┬────┘  └────┬────┘    │
│       └─────────────┴─────────────┴─────────────┘        │
│                          │                              │
│                    ┌─────┴─────┐                        │
│                    │  工作空间  │                        │ ← 数据隔离层
│                    └───────────┘                        │
└─────────────────────────────────────────────────────────┘
```

**一个群 = 独立的工作空间 + 专属业务Agent矩阵**

---

## 二、各群的共性需求

无论什么群，都需要以下通用能力：

| 需求层次 | 说明 | 例子 |
|---------|------|------|
| **执行** | 业务Agent | 股票分析、内容创作 |
| **调度** | 定时任务 | 早间简报、晚间总结 |
| **监控** | 健康检查 | Agent是否正常 |
| **日志** | 问题排查 | 错误原因 |
| **通知** | 消息触达 | 任务完成通知 |
| **存储** | 数据持久化 | 用户偏好、缓存 |
| **搜索** | 信息获取 | 全网搜索 |

## 三、元团队的定位

### 核心问题：元团队应该服务"群"还是服务"Agent"？

| 视角 | 答案 |
|------|------|
| **服务群** | 每个群都要配监控、日志 → 重复建设 |
| **服务Agent** | Agent是元团队的"客户" → 更合理 |

### 关键洞察

> 元团队不是"第9个业务团队"，而是**平台层**

```
┌─────────────────────────────────────────────────────────┐
│                    用户/群入口层                        │
└───────────────────────┬─────────────────────────────────┘
                        │
    ┌───────────────────┼───────────────────┐
    │                   │                   │
    ▼                   ▼                   ▼
┌─────────┐        ┌─────────┐        ┌─────────┐
│ 生活群  │        │ 金融群  │        │内容运营群│
│(25个A)  │        │(26个A)  │        │(41个A)  │
└────┬────┘        └────┬────┘        └────┬────┘
     │                 │                 │
     └─────────────────┼─────────────────┘
                       │
           ┌───────────┴───────────┐
           │    元团队(平台层)     │
           │  ┌─────┐ ┌─────┐    │
           │  │墨监 │ │墨历 │    │  ← 通用服务
           │  └─────┘ └─────┘    │
           └───────────────────┘
```

## 四、元团队的正确抽象能力

### 核心原则

> **元团队服务Agent，不服务群；元团队提供能力，不提供业务**

| 能力 | 正确理解 | 错误理解 |
|------|---------|---------|
| 监控 | "Agent你还好吗" | "群运营健康吗" |
| 日志 | "出了什么问题" | "群数据报表" |
| 调度 | "触发定时任务" | "群定时推送" |
| 画像 | "用户有什么偏好" | "群用户画像" |

---

### 元团队的6大抽象能力

| 能力 | 名称 | 核心职责 |
|------|------|---------|
| **能力1** | 墨监 | Agent状态守护者：存活检测、任务失败告警、性能指标、Cron状态监控 |
| **能力2** | 墨历 | 问题追溯中心：操作日志、错误追踪、审计合规、日志检索 |
| **能力3** | 墨联 | 信息获取中枢：全网搜索、社媒采集、网页抓取、GitHub |
| **能力4** | 墨邮 | 通知分发中枢：邮件处理、消息推送、定时报告 |
| **能力5** | 墨析 | 个性化服务基座：偏好学习、跨平台画像、历史记录 |
| **能力6** | 墨创 | 快速创建能力：模板生成、自动注册、批量创建 |

## 六、如何支持未来更多群？

### 扩展模型

```
新增一个群 = 定义业务Agent + 接入通用能力

例子：新增"健康群"
  1. 业务层：创建 健康Agent组（墨医、墨康、墨食）
  2. 平台层：复用 元团队能力
     • 墨监 → 监控健康Agent
     • 墨历 → 日志审计
     • 墨析 → 用户健康偏好
     • 墨邮 → 定时健康提醒
```

### 元团队的正确接入方式

| 方式 | 说明 |
|------|------|
| **被调用** | 各群Agent需要时调用元团队能力 |
| **不绑定群** | 元团队不绑定具体群（无binding） |
| **独立工作空间** | 元团队有独立workspace |
| **能力暴露** | 通过skill或子agent调用 |

## 七、最终结论

**元团队 = 平台能力，不是一个"业务群"**

| 属性 | 业务团队 | 元团队 |
|------|---------|--------|
| 绑定群 | ✅ 绑定 | ❌ 不绑定 |
| 子Agent | 业务Agent | 平台能力Agent |
| 服务对象 | 群用户 | 其他Agent |
| 数据 | 群隔离 | 共享 |
| 扩展方式 | 新增业务 | 新增能力 |

## 九、配置管理教训

**⚠️ 重要：配置更新时必须使用 `openclaw config` CLI，禁止直接用 `config.apply` 传入 raw JSON**

正确做法：
1. 优先用 `openclaw config set <path> <value>`
2. 交互式编辑：`openclaw config edit`
3. 必须用 apply 时：先写文件 → 用 `jq` 校验格式 → 确认无误后再 apply
