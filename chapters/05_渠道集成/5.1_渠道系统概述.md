## 8.1.1 什么是通道

### 通道概念

```
┌─────────────────────────────────────────────┐
│           OpenClaw 通道系统                   │
├─────────────────────────────────────────────┤
│                                             │
│  通道 = 消息入口 + 协议适配 + 事件处理        │
│                                             │
│  支持的平台：                                 │
│  ├── 飞书 (IM)                             │
│  ├── Telegram (IM)                         │
│  ├── Discord (社区)                        │
│  ├── Slack (IM)                           │
│  ├── 钉钉 (IM)                             │
│  ├── Web (WebUI)                          │
│  ├── CLI (命令行)                          │
│  └── 自定义 Webhook                        │
│                                             │
└─────────────────────────────────────────────┘
```

![信道架构](./.assets/diagrams/10_channel_architecture.png)



### 通道的作用

```
通道的价值：

1. 多端接入
   ├── 同一 Agent 服务多个平台
   └── 用户选择熟悉的入口

2. 协议适配
   ├── 统一消息格式
   └── 平台特性处理

3. 事件处理
   ├── 消息事件
   ├── 回调事件
   └── 状态事件
```

![信道架构](./.assets/diagrams/10_channel_architecture.png)

## 8.1.4 统一消息格式

### 消息结构

```json
{
  "msg_id": "msg_123456",
  "channel": "feishu",
  "user": {
    "id": "user_123",
    "name": "张三",
    "avatar": "https://..."
  },
  "message": {
    "type": "text",
    "content": "你好"
  },
  "context": {
    "group_id": "group_123",
    "reply_to": "msg_111"
  },
  "timestamp": 1705123456789
}
```

### 响应格式

```json
{
  "msg_id": "reply_123456",
  "channel": "feishu",
  "message": {
    "type": "text",
    "content": "你好，我是 OpenClaw"
  },
  "attachments": []
}
```

## 8.1.5 通道配置

### 全局配置

```json
{
  "gateway": {
    "host": "0.0.0.0",
    "port": 18789,
    "channels": {
      "enabled": true,
      "default": "cli"
    }
  }
}
```

### 通道启用

```json
{
  "channels": {
    "feishu": {
      "enabled": true,
      "config": "./config/feishu.json"
    },
    "telegram": {
      "enabled": true,
      "config": "./config/telegram.json"
    },
    "cli": {
      "enabled": true
    }
  }
}
```

## 8.1.6 常见问题

### Q1: 消息接收不到

```
问题：平台消息无法收到

解决：
1. 检查 Webhook URL 配置
2. 验证回调地址可访问
3. 检查防火墙/白名单
4. 查看通道日志
```

### Q2: 消息发送失败

```
问题：回复消息失败

解决：
1. 检查 API 权限
2. 验证 token 有效性
3. 检查消息格式
4. 查看错误码
```

### Q3: 多通道消息混乱

```
问题：不同通道消息串扰

解决：
1. 隔离会话
2. 区分 sessionKey
3. 配置通道前缀
```
