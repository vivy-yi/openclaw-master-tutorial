9.2.1 定时任务概述 ​

什么是定时任务 ​

定时任务 = 定时自动执行的任务

场景示例：
- 每日数据报告
- 定时健康检查
- 周期性数据同步
- 定时提醒

9.2.2 Cron 表达式 ​

基本格式 ​

Cron 格式：
┌───────────── 分钟 (0 - 59)
│ ┌─────────── 小时 (0 - 23)
│ │ ┌───────── 日 (1 - 31)
│ │ │ ┌─────── 月 (1 - 12)
│ │ │ │ ┌───── 星期 (0 - 6, 0 = 周日)
│ │ │ │ │
* * * * *

特殊字符：
*     任意值
,     值列表  1,3,5
-     范围    1-5
/     步长    */5

常用示例 ​

常用 Cron 表达式：

| 表达式 | 含义 |
|--------|------|
| * * * * * | 每分钟 |
| 0 * * * * | 每小时 |
| 0 0 * * * | 每天午夜 |
| 0 0 * * 0 | 每周日 |
| 0 0 1 * * | 每月1日 |
| 0 9 * * 1-5 | 工作日9点 |
| */15 * * * * | 每15分钟 |
| 0 0 * * * | 每天0点 |
| 0 */2 * * * | 每2小时 |

9.2.3 配置定时任务 (CLI 方式) ​

CLI 基本命令 ​

bash
# 添加一次性提醒
openclaw cron add \
  --name "Reminder" \
  --at "2026-02-01T16:00:00Z" \
  --session main \
  --system-event "Reminder: check the cron docs draft" \
  --wake now \
  --delete-after-run

# 添加循环任务
openclaw cron add \
  --name "Morning brief" \
  --cron "0 7 * * *" \
  --tz "America/Los_Angeles" \
  --session isolated \
  --message "Summarize overnight updates." \
  --announce \
  --channel slack \
  --to "channel:C1234567890"

基础配置 (JSON 格式) ​

Job 存储在 ~/.openclaw/cron/jobs.json，由 Gateway 管理。

json5
{
  "jobs": [
    {
      "jobId": "daily_report",
      "name": "每日报告",
      "schedule": {
        "kind": "cron",
        "expr": "0 8 * * *",
        "tz": "Asia/Shanghai"
      },
      "sessionTarget": "isolated",
      "payload": {
        "kind": "agentTurn",
        "message": "生成昨日数据报告"
      },
      "delivery": {
        "mode": "announce",
        "channel": "slack",
        "to": "channel:C1234567890"
      },
      "enabled": true
    }
  ]
}

完整配置 ​

json5
{
  "cron": {
    "enabled": true,
    "store": "~/.openclaw/cron/jobs.json",
    "maxConcurrentRuns": 1,
    "retry": {
      "maxAttempts": 3,
      "backoffMs": [60000, 120000, 300000],
      "retryOn": ["rate_limit", "overloaded", "network", "server_error"]
    },
    "webhook": "https://example.invalid/legacy",
    "webhookToken": "replace-with-dedicated-webhook-token",
    "sessionRetention": "24h",
    "runLog": {
      "maxBytes": "2mb",
      "keepLines": 2000
    }
  }
}

9.2.4 Payload 类型 ​

System Event (主会话) ​

json5
{
  "sessionTarget": "main",
  "payload": {
    "kind": "systemEvent",
    "text": "Reminder: check the calendar"
  },
  "wakeMode": "now"
}

Agent Turn (独立会话) ​

json5
{
  "sessionTarget": "isolated",
  "payload": {
    "kind": "agentTurn",
    "message": "分析昨日用户数据",
    "model": "opus",
    "thinking": "high",
    "lightContext": true
  },
  "delivery": {
    "mode": "announce",
    "channel": "slack",
    "to": "channel:C1234567890",
    "bestEffort": true
  }
}

模型与思考覆盖 ​

isolated 任务可以覆盖模型和思考级别：

bash
# 使用 Opus 模型，高思考级别
openclaw cron add \
  --name "Deep analysis" \
  --cron "0 6 * * 1" \
  --session isolated \
  --message "Weekly deep analysis" \
  --model "opus" \
  --thinking high \
  --announce

轻量级上下文 ​

对于不需要工作区 bootstrap 的任务，使用 --light-context：

bash
openclaw cron add \
  --name "Lightweight check" \
  --cron "0 * * * *" \
  --session isolated \
  --message "Quick system check" \
  --light-context \
  --no-deliver

9.2.6 Cron Job 核心概念 ​

Job 结构 ​

Cron Job 由以下部分组成：

┌─────────────────────────────────────────────────────────────┐
│                     Cron Job 结构                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. Schedule (何时执行)                                      │
│     ├── at: 一次性时间戳                                     │
│     ├── every: 固定间隔                                      │
│     └── cron: Cron 表达式                                    │
│                                                             │
│  2. Payload (执行什么)                                       │
│     ├── systemEvent: 主会话系统事件                           │
│     └── agentTurn: 独立 Agent 回合                            │
│                                                             │
│  3. Session Target                                          │
│     ├── main: 下次 Heartbeat 时执行                           │
│     └── isolated: 独立会话执行                                │
│                                                             │
│  4. Delivery (输出方式)                                      │
│     ├── announce: 发送到渠道                                 │
│     ├── webhook: HTTP POST 到 URL                            │
│     └── none: 仅内部执行                                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘

Schedule 类型 ​

at：一次性任务 ​

bash
# 一次性提醒 (UTC 时间)
openclaw cron add \
  --name "Reminder" \
  --at "2026-02-01T16:00:00Z" \
  --session main \
  --system-event "Reminder: check the cron docs" \
  --wake now \
  --delete-after-run
bash
# 相对时间 (20分钟后)
openclaw cron add \
  --name "Meeting reminder" \
  --at "20m" \
  --session main \
  --system-event "Meeting starts in 10 minutes" \
  --wake now

every：固定间隔 ​

bash
# 每小时执行
openclaw cron add \
  --name "Hourly check" \
  --every "1h" \
  --session isolated \
  --message "Run hourly system check" \
  --announce

cron：Cron 表达式 ​

bash
# 每天早上 7 点 (美国西部时间)
openclaw cron add \
  --name "Morning brief" \
  --cron "0 7 * * *" \
  --tz "America/Los_Angeles" \
  --session isolated \
  --message "Summarize overnight updates." \
  --announce \
  --channel slack \
  --to "channel:C1234567890"

Session Target ​

Main Session (主会话) ​

  • 在下次 Heartbeat 时执行
  • 共享主会话上下文
  • 适合需要上下文的提醒
bash
openclaw cron add \
  --name "Calendar check" \
  --cron "0 9 * * *" \
  --session main \
  --system-event "Check today's calendar"

Isolated Session (独立会话) ​

  • 独立 Agent 回合，会话 ID 为 cron:<jobId>
  • 每次运行全新会话，无历史
  • 适合频繁、不污染主会话的后台任务
bash
openclaw cron add \
  --name "Background sync" \
  --cron "*/15 * * * *" \
  --session isolated \
  --message "Sync data from API" \
  --announce \
  --channel whatsapp \
  --to "+15551234567"

9.2.7 skipWhenIdle 选项 ​

什么是 skipWhenIdle ​

skipWhenIdle 是 Cron Job 的一个可选配置项，当 Cron 触发时，如果检测到系统处于空闲状态，则跳过本次执行，避免在系统空闲时浪费资源。

配置示例 ​

json5
{
  "jobs": [
    {
      "jobId": "light_task",
      "name": "轻量健康检查",
      "schedule": { "kind": "cron", "expr": "*/15 * * * *" },
      "payload": {
        "kind": "agentTurn",
        "message": "Run health check"
      },
      "skipWhenIdle": true   // ✅ 空闲时跳过
    }
  ]
}

使用场景 ​

适合开启 skipWhenIdle：
- 周期性健康检查（空闲时不需重复确认）
- 日志统计任务（用户不在时无需统计）
- 非紧急数据同步

不建议开启：
- 必须执行的关键提醒
- 定时打卡等有时限任务

9.2.8 ⚠️ Cron 删除后持续投递 Bug（v2026.3.24 #56045） ​

问题现象：使用 cron remove <jobId> 删除一个定时任务后，该 job 的输出仍然持续投递到 session。

根因：删除操作未完全清理 job 的运行时状态。

正确删除流程：

bash
# 1. 先禁用 job
openclaw cron update <jobId> --enabled false

# 2. 确认投递停止（观察几分钟）

# 3. 再执行删除
openclaw cron remove <jobId>

# 4. 手动重启 Gateway（必须！）
openclaw gateway restart

临时应对：如果删除后仍有投递，重复第 4 步重启 Gateway 即可停止。

官方修复跟踪：Issue #56045

9.2.9 重试策略 ​

错误分类 ​

┌─────────────────────────────────────────────────────────────┐
│                      错误分类                                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  瞬态错误 (可重试)                                           │
│  ├── 429 Rate Limit                                        │
│  ├── 提供商过载 (529 overloaded)                            │
│  ├── 网络错误 (超时、ECONNRESET)                            │
│  └── 5xx 服务器错误                                         │
│                                                             │
│  永久错误 (立即禁用)                                         │
│  ├── 认证失败 (无效 API Key)                                │
│  └── 配置错误                                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘

重试行为 ​

一次性任务 (at)：

  • 瞬态错误：指数退避重试 3 次 (30s → 1m → 5m)
  • 永久错误：立即禁用

循环任务 (cron/every)：

  • 任何错误：指数退避 (30s → 1m → 5m → 15m → 60m)
  • 下次成功运行后重置

问题：定时任务没有触发

排查步骤：
1. 检查 cron 是否启用: cron.enabled
2. 确认 Gateway 持续运行
3. 检查时区配置 (--tz vs 主机时区)
4. 查看日志

Q2: 执行时间不准 ​

问题：执行时间有偏差

原因：
- 整点负载分散 (自动 5 分钟窗口)
- 主机时区差异

解决：
1. 使用 --exact 强制精确时间
2. 明确指定时区 --tz
3. 使用 --stagger 设置分散窗口

Q3: Telegram 投递错误 ​

问题：投递到错误位置

解决：
- 使用明确格式: -1001234567890:topic:123
- 避免歧义

9.2.7 多群场景的 sessionKey 配置 ​

核心问题 ​

当一个 OpenClaw Agent 服务多个群时，每个群的 Cron job 需要正确绑定到对应的群：

问题说明
isolated sessionCron 默认使用独立 session（每次运行是新 session）
sessionKey 缺失没有配置 sessionKey 时，无法知道目标群
Agent 漂移子 agents（墨搜/墨版/墨编）属于不同 workspace

sessionKey 配置公式 ​

bash
openclaw cron add \
  --name "墨搜-OpenClaw群-每日采集" \
  --cron "0 23 * * *" \
  --session isolated \
  --sessionKey "agent:openclaw-assistant:telegram:group:-5015409771" \
  --agent "墨搜" \
  --message "执行搜索任务" \
  --announce \
  --channel telegram \
  --to "-5015409771"

配置要素 ​

参数说明示例
--session会话类型isolated
--sessionKey目标群标识agent:openclaw-assistant:telegram:group:-5015409771
--agent子 agent墨搜
--announce投递模式announce
--channel目标渠道telegram
--to目标群 ID-5015409771

典型错误 vs 正确 ​

错误类型问题正确配置
只有 agentId不知道目标群必须配置 sessionKey
sessionKey 错误投递到错误群sessionKey 必须对应目标群
agent 不匹配子 agent 不在对应 workspaceagent 必须在 sessionKey 对应的 workspace

多群命名规范 ​

Cron job 名称应包含群标识，便于识别：

# 好命名
墨搜-OpenClaw群-每日采集
墨搜-金融群-早间报告
墨搜-法律群-时效检查

# 坏命名
每日采集  # 不知道属于哪个群
搜索任务  # 缺少群标识

# 查看详细配置
