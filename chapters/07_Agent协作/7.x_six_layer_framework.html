2. 各层详解 ​

第1层：Agent能力层 (Core) ​

核心问题：Agent本身的能力边界在哪里？

能力维度内容现状
推理能力逻辑推导、多步规划✅ LLM提供
记忆能力短期/长期/向量✅ 六层记忆
工具能力调用外部系统✅ MCP/Tool
角色扮演人格、语气、边界⚠️ 需强化
自主性主动行动 vs 被动响应⚠️ 需设计

关键思考：

  • Agent能力的"天花板"由LLM决定，但"地板"由配置决定
  • 角色扮演能力决定了用户体验的上限
  • 自主性的度：太懒=没用，太自主=危险

第2层：外部知识层 (Knowledge) ​

核心问题：如何让Agent"知道"它应该知道的一切？

知识获取优先级（重要！） ​

┌─────────────────────────────────────────────────────────┐
│                   搜索工具优先级（正确顺序）                 │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  1. 平台原生API  ✅ 优先                               │
│     - Twitter/GitHub/知乎/微博官方API                    │
│     - 精确、无配额、官方支持                              │
│                                                          │
│  2. browser.open()  ✅ 其次                            │
│     - 页面抓取、登录态、交互                             │
│     - 灵活、不消耗API配额                                │
│                                                          │
│  3. web_search (Tavily)  ⚠️ 最后兜底                │
│     - 无特定数据源时使用                                 │
│     - 有配额限制，不滥用                                 │
└─────────────────────────────────────────────────────────┘

知识分类 ​

类型来源示例
内部知识shared/协议、工作流、配置
外部知识平台APITwitter/GitHub
实时信息browser网页抓取
结构化知识RAG/LanceDB古籍、案例库
用户上下文Session历史对话、偏好

关键问题：

  • 知识的一致性：多个Agent如何共享知识而不冲突？
  • 知识的时效性：如何处理"今天的新闻"vs"去年的政策"？
  • 知识的权威性：谁的知识库是"正确答案"？

第3层：Agent团队协作层 (Multi-Agent) ​

核心问题：如何让多个Agent协同工作而不是各自为战？

当前问题 vs 理想协作 ​

当前问题：
┌────────────────────────────────────────┐
│  mo-finance  ←→  mo-law              │
│  金融助手      法律助手                 │
│       ↓              ↓                │
│    各自回答      各自回答              │
│       ↓              ↓                │
│    用户在两个群之间跳跃                 │
│       ↓              ↓                │
│    体验碎片化，信息不连贯              │
└────────────────────────────────────────┘

理想协作：
┌────────────────────────────────────────┐
│  用户 → mo-finance (金融问题)           │
│         ↓                              │
│    需要法律意见 → 触发 mo-law 协作       │
│         ↓                              │
│    金融助手整合法律建议 → 回复用户       │
│         ↓                              │
│    用户获得连贯的跨域服务               │
└────────────────────────────────────────┘

协作模式 ​

模式描述适用场景
串行A完成 → B基于A结果 → 回复用户B依赖A输出
并行同时调用A，B，C → 汇总独立任务
混合A1,A2并行 → 汇总 → B执行先收集后综合

触发机制 ​

@目标Agent 任务描述 → 触发跨群协作
message.send(target: "群ID") → 直接通信

第4层：领域专业能力层 (Domain) ​

核心问题：每个领域助手如何构建"护城河"？

各助手现状与差距 ​

┌──────────────────────────────────────────────────────────┐
│  mo-finance (金融助手)                                  │
│  ✅ 已有：市场数据API、预测追踪、学习机制                 │
│  ❌ 缺少：实时行情、个性化推荐、组合分析                 │
│  🎯 深度：不是"告诉你买什么"，而是"帮你建立投资框架"      │
├──────────────────────────────────────────────────────────┤
│  mo-law (法律助手)                                      │
│  ✅ 已有：法规检索、AI法律新闻                           │
│  ❌ 缺少：案例分析、合同审查、诉讼策略                   │
│  🎯 深度：不是"查法条"，而是"解决法律问题"              │
├──────────────────────────────────────────────────────────┤
│  mo-yunying (内容运营)                                  │
│  ✅ 已有：平台API、竞品监控                             │
│  ❌ 缺少：爆款规律、内容策略、品牌联动                   │
│  🎯 深度：不是"生成内容"，而是"建立内容体系"            │
├──────────────────────────────────────────────────────────┤
│  mo-fortune (算命助手)                                  │
│  ✅ 已有：八字、紫微、塔罗、风水、心理学融合             │
│  ❌ 缺少：实战咨询、案例积累、人设深化                   │
│  🎯 深度：不是"算命"，而是"人生规划参考"              │
└──────────────────────────────────────────────────────────┘

领域深度的三个层次 ​

Level 1: 信息聚合
  - "今天的财经新闻有哪些"
  
Level 2: 分析解读
  - "这些新闻对市场有什么影响"
  
Level 3: 决策支持
  - "基于这些信息，我应该如何调整投资组合"

第5层：合规和风控层 (Compliance) ​

核心问题：如何让Agent"知道什么不该做"？

合规边界设计 ​

┌────────────────────────────────────────┐
│  硬性边界 (绝对不能碰)                  │
│  ├─ 不提供投资建议 (≠分析)              │
│  ├─ 不提供医疗建议 (≠健康)              │
│  ├─ 不提供法律诊断 (≠法律信息)          │
│  └─ 不承诺结果 (≠合理预期)              │
├────────────────────────────────────────┤
│  软性边界 (不建议但可提示)              │
│  ├─ 不过度乐观/悲观                     │
│  ├─ 不代替用户做决定                     │
│  └─ 不隐瞒风险                          │
└────────────────────────────────────────┘

风控机制 ​

风险类型触发条件处理方式
投资建议提到具体股票代码立即打断+合规提示
法律诊断用户问"我能不能赢"提供信息而非判断
隐私泄露涉及个人敏感信息脱敏处理

第6层：用户体验层 (UX) ​

核心问题：如何让用户感觉"这个助手真的懂我"？

用户体验四维度 ​

┌────────────────────────────────────────┐
│  1. 一致性                              │
│     - 回复风格统一                       │
│     - 记忆跨会话延续                     │
│     - "记得上次我们讨论过..."             │
├────────────────────────────────────────┤
│  2. 主动性                              │
│     - 主动预警 (如行情突变)               │
│     - 主动建议 (如季节性机会)             │
│     - 不打扰但有价值                      │
├────────────────────────────────────────┤
│  3. 透明度                              │
│     - 知道Agent能做什么/不能做什么        │
│     - 了解信息来源和置信度                │
│     - "这个结论的依据是..."               │
├────────────────────────────────────────┤
│  4. 情感连接                            │
│     - 不是冷冰冰的问答机                  │
│     - 有温度的专业建议                    │
│     - "理解你的处境"                     │
└────────────────────────────────────────┘

3. 六层协同关系 ​

用户输入
    ↓
第6层 UX解析 → 理解用户真正想问什么
    ↓
第5层 合规检查 → 这个请求是否合规？
    ↓ [通过]
第4层 领域推理 → 用专业知识回答
    ↓ [需要跨域]
第3层 Agent协作 → 调用其他Agent
    ↓ [需要信息]
第2层 知识检索 → 获取相关知识
    ↓
第1层 Agent能力 → 整合所有信息，生成回复
    ↓
回复用户 + 记忆存储

5. 各助手优化建议 ​

助手第1层短板第4层深度第6层体验
mo-finance投资建议边界模糊需要"框架"而非"推荐"主动推送有价值的分析
mo-law法条检索不够精准需要案例+策略"类案推送"
mo-yunying内容策略浮于表面需要"爆款规律"追踪内容效果
mo-fortune心理学融合浅需要"实战咨询"建立信任感

6. 关键原则 ​

  1. 知识优先API：平台原生API > browser > web_search
  2. 协作通过@触发：@目标Agent 任务描述
  3. 合规是底线：先判断能不能答，再想怎么答
  4. 专业有层次：信息聚合 → 分析解读 → 决策支持
  5. 体验要一致：记忆跨会话延续，风格统一
