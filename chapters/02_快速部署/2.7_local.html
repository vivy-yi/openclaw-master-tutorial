2.7 本地部署详解 ​

本节目标 ​

  • 掌握多种本地部署方式
  • 理解 Podman 无根容器部署
  • 掌握从源码编译安装
  • 了解各方案的适用场景

2.7.1 部署方式对比 ​

本地部署方案 ​

方式优点缺点推荐场景
官方安装脚本最简单需要 Node.js快速体验
Docker隔离性好需要 Docker生产环境
Podman无根容器生态较小安全敏感
源码编译可定制门槛高开发者

环境要求 ​

最低配置：
├── Node.js 22 LTS+
├── 2GB 内存
└── 5GB 磁盘空间

推荐配置：
├── Node.js 24 (推荐)
├── 4GB+ 内存
└── 20GB+ SSD 磁盘

2.7.2 官方安装脚本（推荐） ​

macOS / Linux ​

bash
# 一键安装（推荐）
curl -fsSL https://openclaw.ai/install.sh | bash

# 跳过 onboarding
curl -fsSL https://openclaw.ai/install.sh | bash -s -- --no-onboard

Windows (PowerShell) ​

powershell
# PowerShell 安装
iwr -useb https://openclaw.ai/install.ps1 | iex

# 跳过 onboarding
& ([scriptblock]::Create((iwr -useb https://openclaw.ai/install.ps1))) -NoOnboard

验证安装 ​

bash
# 检查版本
openclaw --version

# 查看状态
openclaw status

# 启动 Gateway
openclaw gateway start

# 打开 Dashboard
openclaw dashboard

常见问题 ​

sharp 构建错误 ​

bash
# 如果 libvips 已全局安装导致 sharp 失败
SHARP_IGNORE_GLOBAL_LIBVIPS=1 npm install -g openclaw@latest

# 如果提示需要 node-gyp
npm install -g node-gyp

2.7.3 npm/pnpm 安装 ​

Node.js 版本要求 ​

  • 推荐: Node.js 24
  • 支持: Node.js 22 LTS (22.16+)

使用 npm ​

bash
# 全局安装
npm install -g openclaw@latest

# 启动 onboarding
openclaw onboard --install-daemon

使用 pnpm ​

bash
# 全局安装
pnpm add -g openclaw@latest

# 批准构建脚本
pnpm approve-builds -g

# 启动 onboarding
pnpm approve-builds -g
pnpm onboard --install-daemon

安装最新开发版 ​

bash
# 从 GitHub main 分支安装
npm install -g github:openclaw/openclaw#main

# 或使用 pnpm
pnpm add -g github:openclaw/openclaw#main

2.7.4 Docker 部署（无持久化） ​

快速启动 ​

bash
# 方式一：使用官方镜像
docker run -d \
  --name openclaw \
  -p 18789:18789 \
  openclaw/openclaw:latest

# 方式二：使用 docker-setup.sh（推荐）
git clone https://github.com/openclaw/openclaw.git
cd openclaw
./docker-setup.sh

docker-setup.sh 详解 ​

该脚本会自动完成：

  1. 构建本地镜像（或使用远程镜像）
  2. 运行 onboarding 向导
  3. 启动 Gateway
  4. 生成 token 并写入 .env

可选环境变量 ​

bash
# 使用远程镜像
export OPENCLAW_IMAGE="ghcr.io/openclaw/openclaw:latest"
./docker-setup.sh

# 启用沙箱模式
export OPENCLAW_SANDBOX=1
./docker-setup.sh

# 持久化 /home/node
export OPENCLAW_HOME_VOLUME="openclaw_home"
./docker-setup.sh

# 额外挂载
export OPENCLAW_EXTRA_MOUNTS="$HOME/.codex:/home/node/.codex:ro"
./docker-setup.sh

常用命令 ​

bash
# 启动
docker compose up -d

# 查看状态
docker compose ps

# 查看日志
docker compose logs -f

# 停止
docker compose down

# 重启
docker compose restart

# 获取 Dashboard URL
docker compose run --rm openclaw-cli dashboard --no-open

访问控制 UI ​

bash
# 打开浏览器
open http://127.0.0.1:18789/

# 获取 token
docker compose run --rm openclaw-cli dashboard --no-open

2.7.5 Podman 部署（无根容器） ​

Podman 是 Docker 的无根替代品，无需守护进程。

![代理设置]/assets/diagrams/169_proxy_setup.png)

安装 Podman ​

bash
# macOS
brew install podman

# Linux
sudo apt-get install podman

# Windows (WSL2)
sudo apt-get install podman

初始化 Podman ​

bash
# 初始化
podman machine init
podman machine start

# 检查状态
podman machine list

使用 setup-podman.sh ​

bash
# 克隆仓库
git clone https://github.com/openclaw/openclaw.git
cd openclaw

# 运行 Podman 设置脚本
./setup-podman.sh

手动 Podman 运行 ​

bash
# 创建配置目录
mkdir -p ~/.openclaw

# 运行容器
podman run -d \
  --name openclaw \
  -p 18789:18789 \
  -v ~/.openclaw:/root/.openclaw \
  -e DEEPSEEK_API_KEY=$DEEPSEEK_API_KEY \
  docker.io/openclaw/openclaw:latest

# 开机自启
podman generate systemd --name openclaw > ~/.config/systemd/user/openclaw.service
systemctl --user enable openclaw

Podman vs Docker 命令对照 ​

DockerPodman
docker runpodman run
docker pspodman ps
docker logspodman logs
docker-composepodman-compose

2.7.7 Docker Compose 完整配置 ​

基础配置 ​

yaml
version: '3.8'

services:
  openclaw:
    image: openclaw/openclaw:latest
    container_name: openclaw
    ports:
      - "18789:18789"
    volumes:
      - ./openclaw.json:/root/.openclaw/openclaw.json:ro
      - openclaw-data:/root/.openclaw
    environment:
      - TZ=Asia/Shanghai
      - DEEPSEEK_API_KEY=${DEEPSEEK_API_KEY}
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:18789/healthz"]
      interval: 30s
      timeout: 10s
      retries: 3

volumes:
  openclaw-data:

启用沙箱模式 ​

bash
export OPENCLAW_SANDBOX=1
./docker-setup.sh

配置说明 ​

配置项说明
--bind lan绑定到 0.0.0.0，允许局域网访问
--bind loopback仅本地访问
--allow-unconfigured无配置启动，后续配置

健康检查端点 ​

bash
# 基础健康检查
curl -fsS http://127.0.0.1:18789/healthz

# 就绪检查
curl -fsS http://127.0.0.1:18789/readyz

# 深度健康检查
docker compose exec openclaw-gateway node dist/index.js health --token "$OPENCLAW_GATEWAY_TOKEN"

2.7.8 开机自启配置 ​

systemd (Linux) ​

bash
# 创建服务文件
sudo vim /etc/systemd/system/openclaw.service

# 内容
[Unit]
Description=OpenClaw Gateway
After=network.target

[Service]
Type=simple
User=ubuntu
WorkingDirectory=/home/ubuntu
ExecStart=/usr/local/bin/openclaw gateway start
Restart=always
RestartSec=10

[Install]
WantedBy=multi-user.target

# 启用服务
sudo systemctl daemon-reload
sudo systemctl enable openclaw
sudo systemctl start openclaw

Launchd (macOS) ​

bash
# 创建 plist 文件
vim ~/Library/LaunchAgents/com.openclaw.gateway.plist

# 内容
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>Label</key>
    <string>com.openclaw.gateway</string>
    <key>ProgramArguments</key>
    <array>
        <string>/usr/local/bin/openclaw</string>
        <string>gateway</string>
        <string>start</string>
    </array>
    <key>RunAtLoad</key>
    <true/>
    <key>KeepAlive</key>
    <true/>
</dict>
</plist>

# 加载服务
launchctl load ~/Library/LaunchAgents/com.openclaw.gateway.plist

Docker 开机自启 ​

bash
# 容器自启
docker run -d --name openclaw --restart unless-stopped ...

# docker-compose
# 已在配置中指定 restart: unless-stopped

2.7.9 常见问题 ​

问题一：端口被占用 ​

bash
# 查找占用进程
lsof -i :18789

# 更改端口
openclaw gateway start --port 18889

# Docker 更改端口映射
# docker-compose.yml
ports:
  - "18889:18789"

问题二：Node 版本不匹配 ​

bash
# 检查版本
node -v

# 使用 nvm 管理版本
nvm install 24
nvm use 24

# 或使用 n
n install 24
n use 24

问题三：权限问题 ​

bash
# Docker 权限问题
sudo chown -R 1000:1000 /path/to/openclaw-config

# 检查用户组
groups $USER
sudo usermod -aG docker $USER

问题四：网络连接问题 ​

bash
# 检查防火墙
sudo ufw status

# 开放端口
sudo ufw allow 18789/tcp

# 检查端口监听
netstat -tuln | grep 18789

2.7.7 v2026.3.24 新增：Linux 原生 Gateway Client ​

功能说明 ​

v2026.3.24 新增了 独立 Linux Companion App（Gateway Client），可在 Linux 桌面环境中直接运行 OpenClaw，无需通过终端命令。

与传统部署的区别 ​

┌─────────────────────────────────────────────────────────────┐
│           Linux Gateway Client vs 传统部署                      │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  传统部署（服务端）：                                          │
│  └── 部署在服务器，通过 WebSocket/Telegram 访问               │
│                                                              │
│  Linux Gateway Client（v2026.3.24+）：                        │
│  └── 独立 Linux 桌面应用，本地运行，可脱离浏览器               │
│      支持 Tailscale / SSH 隧道远程访问                        │
│                                                              │
└─────────────────────────────────────────────────────────────┘

系统要求 ​

环境要求：
├── Linux 发行版（Ubuntu 20.04+ / Debian 11+ / Fedora 36+）
├── x86_64 或 ARM64 架构
├── 至少 2GB 内存
├── 至少 500MB 可用磁盘空间
└── 网络连接（用于连接 LLM API）

安装步骤 ​

bash
# 1. 下载最新版本的 AppImage 或 deb 包
# 见 GitHub Releases: https://github.com/openclaw/openclaw/releases

# 2. AppImage 方式（推荐，无需安装）
chmod +x openclaw-linux-client-*.AppImage
./openclaw-linux-client-*.AppImage

# 3. deb 包方式
sudo dpkg -i openclaw-client_*.deb
sudo apt-get -f install  # 安装依赖
openclaw-client           # 启动应用

远程访问配置 ​

Linux Gateway Client 支持通过 Tailscale 或 SSH 隧道远程访问：

方式一：Tailscale（推荐）

bash
# 在 Linux 客户端上安装 Tailscale
curl -fsSL https://tailscale.com/install.sh | sh

# 登录并启动
tailscale up

# Gateway Client 会自动使用 Tailscale IP
# 在手机 App 或其他设备上通过 Tailscale IP:18789 访问

方式二：SSH 隧道

bash
# 从远程设备建立 SSH 隧道
ssh -L 18789:localhost:18789 user@linux-client-host

# 然后本地访问 http://localhost:18789

与 OpenClaw 手机节点的关系 ​

维度Linux Gateway ClientOpenClaw 手机节点
运行环境Linux 桌面iOS/Android
配对方式独立 AppCompanion App
功能Gateway Client全功能节点
远程访问Tailscale/SSHTailscale/SSH
适用场景Linux 用户作为 Gateway移动端随身访问

已知问题 ​

⚠️ Linux Gateway Client 处于早期阶段，部分发行版可能存在兼容性问题。建议通过 GitHub Issues 反馈。

本节小结 ​

  1. 官方脚本：最简部署方式
  2. npm/pnpm：适合已有 Node.js 环境
  3. Docker：隔离性好，生产推荐
  4. Podman：无根容器，更安全
  5. 源码编译：可定制，适合开发者
