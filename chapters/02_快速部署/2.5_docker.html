2.5.3 Docker 基础部署 ​

步骤一：创建配置目录 ​

bash
# 创建配置目录
mkdir -p ~/.openclaw

# 创建配置文件
touch ~/.openclaw/openclaw.json

步骤二：编辑配置文件 ​

json
{
  "model": {
    "provider": "deepseek",
    "model": "deepseek-chat",
    "apiKey": "${DEEPSEEK_API_KEY}"
  },
  "gateway": {
    "port": 18789,
    "channels": {
      "cli": { "enabled": true }
    }
  }
}

步骤三：运行容器 ​

bash
# 基本运行
docker run -d \
  --name openclaw \
  -v ~/.openclaw:/root/.openclaw \
  -p 18789:18789 \
  openclaw/openclaw:latest

# 完整配置（推荐）
docker run -d \
  --name openclaw \
  --restart unless-stopped \
  -v ~/.openclaw:/root/.openclaw \
  -p 18789:18789 \
  -e TZ=Asia/Shanghai \
  -e DEEPSEEK_API_KEY=$DEEPSEEK_API_KEY \
  openclaw/openclaw:latest

参数说明
-d后台运行
--name容器名称

-p端口映射

2.5.4 Docker Compose 部署（推荐） ​

创建 docker-compose.yml ​

yaml
version: '3.8'

services:
  openclaw:
    image: openclaw/openclaw:latest
    container_name: openclaw
    ports:
      - "18789:18789"
    volumes:
      - ./openclaw.json:/root/.openclaw/openclaw.json:ro
      - openclaw-data:/root/.openclaw
    environment:
      - TZ=Asia/Shanghai
      - DEEPSEEK_API_KEY=${DEEPSEEK_API_KEY}
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:18789/health"]
      interval: 30s
      timeout: 10s
      retries: 3

volumes:
  openclaw-data:

环境变量文件 ​

创建 .env 文件：

bash
# .env
DEEPSEEK_API_KEY=sk-你的API密钥

常用命令 ​

bash

# 查看日志
docker-compose logs -f

2.5.5 数据持久化 ​

目录映射 ​

yaml
volumes:
  # 配置文件（只读）
  - ./openclaw.json:/root/.openclaw/openclaw.json:ro

  # 工作目录
  - ./workspace:/root/.openclaw/workspace

  # 记忆数据
  - ./memory:/root/.openclaw/memory

  # 日志
  - ./logs:/root/.openclaw/logs

  # Skills
  - ./skills:/root/.openclaw/skills

完整配置示例 ​

yaml
version: '3.8'

services:
  openclaw:
    image: openclaw/openclaw:latest
    container_name: openclaw
    ports:
      - "18789:18789"
    volumes:
      # 配置（只读，保护配置）
      - ./config/openclaw.json:/root/.openclaw/openclaw.json:ro

      # 数据卷
      - openclaw-workspace:/root/.openclaw/workspace
      - openclaw-memory:/root/.openclaw/memory
      - openclaw-skills:/root/.openclaw/skills
      - openclaw-logs:/root/.openclaw/logs
    environment:
      - TZ=Asia/Shanghai
      # 可以添加多个 API Key
      - DEEPSEEK_API_KEY=${DEEPSEEK_API_KEY}
      - OPENAI_API_KEY=${OPENAI_API_KEY}
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:18789/health"]
      interval: 30s
      timeout: 10s
      retries: 3
    networks:
      - openclaw-network

networks:
  openclaw-network:
    driver: bridge

volumes:
  openclaw-workspace:
  openclaw-memory:
  openclaw-skills:
  openclaw-logs:

2.5.9 备份与恢复 ​

备份脚本 ​

bash
#!/bin/bash
# backup-openclaw.sh

BACKUP_DIR="./backups"
DATE=$(date +%Y%m%d_%H%M%S)

# 创建备份目录
mkdir -p $BACKUP_DIR

# 备份配置文件
cp ~/.openclaw/openclaw.json $BACKUP_DIR/config_$DATE.json

# 备份数据卷
docker run --rm \
  -v openclaw-memory:/data \
  -v $(pwd)/$BACKUP_DIR:/backup \
  alpine tar czf /backup/memory_$DATE.tar.gz -C /data .

echo "备份完成: $DATE"

恢复脚本 ​

bash
#!/bin/bash
# restore-openclaw.sh

BACKUP_FILE=$1

if [ -z "$BACKUP_FILE" ]; then
  echo "用法: ./restore-openclaw.sh <备份文件>"
  exit 1
fi

# 恢复数据
docker run --rm \
  -v openclaw-memory:/data \
  -v $(pwd)/backups:/backup \
  alpine tar xzf /backup/$BACKUP_FILE -C /data

echo "恢复完成"

问题六：CPU 100% 热循环（#55360 setTimeout 溢出） ​

严重程度：🔴 P0 — CPU 满载，Gateway 无响应

问题描述：在 Docker 容器中长时间运行 OpenClaw Gateway（>20天）时，若 Copilot token 的 expires_at 字段值为秒级时间戳（> 100亿，即 > 2021 年的秒级 epoch），OpenClaw 的 Copilot 刷新调度器会将此值误认为毫秒级，导致计算出的 setTimeout 延迟超出 JavaScript 的安全上限（2^31-1 ms，约 24.8 天）。超出上限的延迟被静默截断为 0，触发高频重复调度，形成 CPU 100% 热循环，同时伴随大量 TimeoutOverflowWarning 日志（约 500MB/天）。

根因：

mermaid
graph LR
    A[Copilot token expires_at<br/>秒级 epoch > 10B] --> B[parseCopilotTokenResponse<br/>误判为毫秒]
    B --> C[计算出 setTimeout 延迟<br/>> 2^31-1 ms]
    C --> D[延迟被截断为 0]
    D --> E[高频重复调度]
    E --> F[100% CPU 热循环]
    F --> G[500MB/天日志]
  • 触发条件：Copilot channel 启用 + Gateway 运行时间 > 20 天 + token expires_at 为秒级 epoch
  • 影响范围：使用 Copilot channel 的所有 Docker 部署用户

诊断：

bash
# 1. 检查容器 CPU 使用率
docker stats openclaw --no-stream

# 2. 确认 CPU 是否被 openclaw 进程占满
docker exec openclaw ps aux | grep -E "cpu|openclaw" | head -10

# 3. 查看是否有大量 TimeoutOverflowWarning
docker exec openclaw sh -c 'find /root/.openclaw/logs -name "*.log" -exec grep -l TimeoutOverflow {} \;' 2>/dev/null

# 4. 检查 Gateway 运行时长
docker exec openclaw ps -eo pid,etime,cmd | grep openclaw

修复状态：PR 已开，尚未合并（#55360，fix/settimeout-overflow-clamp 分支），修复方案：

typescript
// 在 scheduleCopilotRefresh() 中增加 clamp
const MAX_SAFE_TIMEOUT = 2 ** 31 - 1; // ~24.8 天
const delay = Math.min(calculatedDelay, MAX_SAFE_TIMEOUT);

恢复方法（无需等 PR）：

bash
# 最快恢复：重启容器（清除状态，CPU 立即恢复正常）
docker-compose restart
# 或
docker restart openclaw

预防措施：

  1. 定期重启容器（推荐每周或每 2 周）

    bash
    # crontab 或 docker-compose.yml 中配置
docker-compose restart  # 每周日凌晨 3 点

  2. 监控 CPU 使用率，设置告警

    bash
    # docker-compose.yml 中添加
deploy:
  resources:
    limits:
      cpus: '2'  # 即使发生热循环，也限制最多占用 2 核

  3. 日志轮转：防止超时热循环期间日志撑爆磁盘

    yaml
    logging:
  driver: "json-file"
  options:
    max-size: "50m"
    max-file: "3"

  4. Copilot channel 敏感用户：关注 #55360 PR 合并进度，合并后升级镜像

2.5.6 v2026.3.24 新功能：容器内运行 OpenClaw（--container） ​

背景 ​

v2026.3.24 新增 --container 和 OPENCLAW_CONTAINER 环境变量支持，允许 OpenClaw 在 Docker 或 Podman 容器内部运行。

典型场景 ​

┌─────────────────────────────────────────────────────────────┐
│                 容器内运行 vs 容器外运行                       │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  容器外运行（直接安装）：                                      │
│  └── 宿主机直接运行 openclaw，二进制在主机命名空间              │
│                                                              │
│  容器内运行（v2026.3.24+）：                                  │
│  └── openclaw 二进制打包进容器，容器内运行 openclaw           │
│      适用于：沙箱隔离、便携部署、统一环境                       │
│                                                              │
└─────────────────────────────────────────────────────────────┘

使用方法 ​

方式一：CLI 参数

bash
openclaw --container

方式二：环境变量

yaml
# docker-compose.yml
environment:
  OPENCLAW_CONTAINER: "true"

方式三：配置文件

json
{
  "container": {
    "enabled": true
  }
}

Podman 支持（v2026.3.24 PR #55388） ​

v2026.3.24 同时更新了 Podman 的安装文档。以下 Podman 配置与 Docker 几乎完全兼容：

yaml
# podman-compose.yml
version: "3.8"

services:
  openclaw:
    image: docker.io/openclaw/openclaw:latest
    container_name: openclaw
    environment:
      - OPENCLAW_CONTAINER=true
      - OPENCLAW_GATEWAY_URL=http://localhost:18789
    volumes:
      - openclaw-data:/root/.openclaw
      - /var/run/podman/podman.sock:/var/run/docker.sock
    ports:
      - "18789:18789"
    restart: unless-stopped

volumes:
  openclaw-data:

与 Docker 直接部署的区别 ​

维度直接部署（docker run）容器内运行（--container）
openclaw 进程位置宿主机容器内
适用场景容器作为宿主机 wrapper沙箱隔离、便携打包
资源控制由 docker 限制由容器 runtime 限制
调试需要进入容器同直接部署

已知问题 ​

⚠️ 在某些 Linux 发行版上，使用 --container 配合 Podman 时，systemd 服务注册可能存在兼容性问题。建议优先使用 Docker 或等待官方修复（跟踪 PR #55388）。
