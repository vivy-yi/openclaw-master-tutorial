# ResearchClawBench 分析报告

## 一、Benchmark 概述

ResearchClawBench 是一个评估 **AI 科学研究 Agent** 能力的基准测试，专门衡量 Agent 完成端到端科研任务的能力。

### 核心定位

| 维度 | 内容 |
|------|------|
| **评估对象** | 自主科研 Agent |
| **任务类型** | 端到端科研任务（从数据到论文） |
| **任务数量** | 40 个任务（10 个科学领域 × 4 个任务） |
| **评估 Agent** | Claude Code, Codex CLI, Nanobot, OpenClaw |

### 科学领域分布

```
┌─────────────────────────────────────────────────────────────────────┐
│                    10 个科学领域                                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  天文学 (Astronomy)      化学 (Chemistry)        地球科学 (Earth)   │
│  能源 (Energy)           信息科学 (Information)    生命科学 (Life)    │
│  材料科学 (Material)     数学 (Math)              神经科学 (Neuroscience) │
│  物理学 (Physics)                                             │
│                                                                     │
│  每个领域 4 个任务，共 40 个任务                                   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 二、任务结构分析

### 每个任务的组成

```
┌─────────────────────────────────────────────────────────────────────┐
│                    任务文件结构                                      │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  task_name/                                                          │
│  ├── INSTRUCTIONS.md        ← 任务指令（给 Agent 的提示词）         │
│  ├── info.json              ← 任务元信息                            │
│  ├── checklist.json         ← 评分检查清单                          │
│  ├── files.json             ← 文件清单                             │
│  ├── paper.pdf              ← 参考论文（可选）                      │
│  ├── data/                 ← 输入数据                              │
│  │   └── *.dat                                                      │
│  ├── code/                  ← Agent 写代码的地方                    │
│  ├── outputs/               ← Agent 输出中间结果                    │
│  ├── report/                ← Agent 写报告的地方                   │
│  │   ├── report.md                                                │
│  │   └── images/                                                 │
│  └── related_work/          ← 参考资料                              │
│      └── *.pdf                                                     │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 任务示例：天文任务 (Astronomy_000)

**任务目标**：利用黑洞超辐射的物理学开发贝叶斯统计框架，约束超轻玻色子的性质。

**输入数据**：
- `IRAS_09149-6206_samples.dat` - 超大质量黑洞的后验分布样本
- `M33_X-7_samples.dat` - 恒星量级黑洞的后验分布样本

**交付物**：
1. 分析代码（保存在 `code/`）
2. 中间结果（保存在 `outputs/`）
3. 研究报告 `report/report.md`
4. 图表（保存在 `report/images/`）

## 三、评分系统分析

### Checklist 结构

每个任务有一个 `checklist.json`，包含多个评分检查点：

```json
[
  {
    "type": "text",
    "content": "检查点描述...",
    "path": null,
    "keywords": ["关键词1", "关键词2", ...],
    "weight": 0.2
  },
  {
    "type": "image",
    "content": "图像检查点描述...",
    "path": "images/figure_name.png",
    "keywords": ["关键词1", ...],
    "weight": 0.3
  }
]
```

### 评分维度

| 类型 | 描述 | 权重示例 |
|------|------|---------|
| **text** | 检查报告文本内容 | 0.2 - 0.5 |
| **image** | 检查生成的图表 | 0.3 |
| **code** | 检查代码质量（如果有） | - |

### 评分机制分析

```
┌─────────────────────────────────────────────────────────────────────┐
│                       评分流程                                       │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  1. Agent 完成任务                                                  │
│     ├── 读取数据                                                    │
│     ├── 分析代码                                                    │
│     ├── 生成图表                                                    │
│     └── 撰写报告                                                    │
│                              │                                      │
│                              ▼                                      │
│  2. 自动评分                                                        │
│     ├── 检查报告文本是否包含关键概念                                │
│     ├── 检查图表是否存在                                           │
│     └── 计算加权得分                                                │
│                              │                                      │
│                              ▼                                      │
│  3. 结果记录                                                        │
│     ├── 保存到 runs/                                               │
│     └── 更新 leaderboard.json                                      │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**注意**：从代码来看，这是一个基于关键词匹配的评分系统，可能不是人工评审。

## 四、Benchmark 优缺点分析

### 优点

| 优点 | 说明 |
|------|------|
| **✅ 端到端评估** | 真正评估 Agent 从数据到论文的全流程能力 |
| **✅ 多领域覆盖** | 10 个科学领域，40 个任务，覆盖广泛 |
| **✅ 真实数据** | 使用真实的科学数据集，非合成数据 |
| **✅ 参考论文** | 提供相关工作论文，模拟真实研究场景 |
| **✅ 结构化评分** | 使用 checklist 进行多维度评分 |
| **✅ 开放架构** | 网页应用，易于查看和扩展 |
| **✅ 多 Agent 对比** | 支持对比多个 Agent 的表现 |

### 缺点

| 缺点 | 说明 |
|------|------|
| **❌ 评分不透明** | 基于关键词匹配，可能不够准确 |
| **❌ 缺乏人工评审** | 没有专家评审论文质量 |
| **❌ 任务数量有限** | 每领域仅 4 个任务，统计意义有限 |
| **❌ 无代码执行** | 只检查代码存在，不运行验证 |
| **❌ 无可复现性验证** | 不验证其他研究者能否复现结果 |
| **❌ 领域有限** | 主要是自然科学，缺少 CS、工程等领域 |
| **❌ 无实验执行** | 任务要求运行代码，但评分不检查执行结果 |

### 关键技术决策

1. **任务来源**：从真实论文中提取可执行的研究任务
2. **数据来源**：使用公开的科学数据集（MCMC 后验样本）
3. **评分方式**：基于关键词匹配 + 图像存在性检查
4. **Agent 接口**：通过标准化文件夹结构定义输入输出

如果让我来制作一个类似的 AI 科学研究 Agent Benchmark，我会按照以下步骤：

```python
    "domains": [

        "Chemistry",
        "Biology",
        "Physics",
        "Mathematics",
    ],
    "tasks_per_domain": 10,   # 增加到 10 个
    "total_tasks": 80,

```

### 第二阶段：任务设计

```
任务选择标准：
├── 1. 有公开数据集
├── 2. 需要代码分析
├── 3. 需要生成可视化
├── 4. 有明确的研究问题
└── 5. 结果可验证
```

**任务类型分类**：
1. **数据分析型** - 统计分析给定数据
2. **模型构建型** - 实现并训练模型
3. **文献综合型** - 综合多篇论文
4. **模拟仿真型** - 运行模拟代码

### 第三阶段：评分系统设计

```python
# 改进的评分系统
SCORING_SYSTEM = {
    "automatic": {
        "keyword_matching": 0.3,    # 关键词匹配

    },
    "semi_automatic": {
        "code_execution": 0.2,      # 运行代码验证
        "result_reproduction": 0.2, # 复现结果
    },
    "human": {
        "report_quality": 0.3,      # 人工评审报告质量
    }
}
```

## 七、针对我们目标的改进建议

如果我们要制作一个评估 **AI 训练 Agent** 的 Benchmark，可以参考以下设计：

### 目标定位

```
┌─────────────────────────────────────────────────────────────────────┐
│               AI 训练 Agent Benchmark vs 科研 Agent                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  科研 Agent Benchmark (现有)     vs    训练 Agent Benchmark (目标)   │
│  ───────────────────────                ─────────────────────       │
│  输入: 研究课题                        输入: 训练任务描述            │
│  输出: 论文                            输出: 训练好的模型            │
│  任务: 数据分析 + 论文写作             任务: 训练 + 评测            │
│                                                                     │
│  评分: 论文质量                        评分: 模型性能指标            │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 训练 Agent 任务设计

```
任务类型：
├── 1. 预训练任务 - 从零训练小模型
├── 2. SFT 任务 - 微调已有模型
├── 3. RL 任务 - 训练奖励模型
├── 4. 评测任务 - 评估模型质量
└── 5. 优化任务 - 优化训练效率
```

### 评分指标

| 指标 | 说明 |
|------|------|
| **Loss 收敛** | 训练 loss 是否下降 |
| **评测分数** | 在验证集上的指标 |
| **训练时间** | 完成任务耗时 |
| **资源效率** | GPU 利用率 |
| **代码质量** | 可读性、可复现性 |
| **错误恢复** | 能否从错误中恢复 |

## 八、结论

ResearchClawBench 是一个有价值的 **AI 科研 Agent 评估基准**，但仍有改进空间：

| 改进方向 | 建议 |
|---------|------|
| **评分质量** | 引入 LLM 辅助评审或人工评审 |
| **任务数量** | 扩大任务库，提高统计显著性 |
| **代码验证** | 实际运行代码，验证结果 |
| **可复现性** | 验证其他研究者能复现结果 |
| **多模态** | 加入实验模拟、仪器控制等 |

**如果我们要制作训练 Agent 的 Benchmark**，核心思路是：
1. 定义标准化的训练任务格式
2. 准备多样化的训练数据集
3. 设计多维度评分（自动化 + 人工）
4. 构建安全的执行环境
