### 2.2 Bug #84327 — Telegram 群聊消息静默丢弃

**来源**: [Issue #84327](https://github.com/openclaw/openclaw/issues/84327)

**标签**: `P2`, `impact:message-loss`

**问题描述**:
在 Telegram 群聊中，当模型生成文本响应但未调用 `message` tool 时，响应被静默丢弃，用户看不到任何内容。这是因为 `sourceReplyDeliveryMode: "message_tool_only"`（群聊/频道聊天的默认值）导致 `dispatch-from-config.ts` 抑制所有缺少 `deliverDespiteSourceReplySuppression` 元数据的 payload。

**根因分析**（两段式问题）:

1. **模型行为**: `claude-opus-4-7` 间歇性地直接生成文本而不调用 `message` tool，特别是在长空闲间隔后（Anthropic 提示缓存变冷时）。

2. **运行时抑制**: 在 `dispatch-from-config.ts`（约第 1608 行）：
   ```typescript
   if (suppressDelivery && !shouldDeliverDespiteSourceReplySuppression(reply)) {
       continue; // payload silently skipped
   }
   ```
   当 `sourceReplyDeliveryMode` 为 `"message_tool_only"` 时，`suppressDelivery` 为 `true`。只有标有 `deliverDespiteSourceReplySuppression` 的 payload（由 `markReplyPayloadForSourceSuppressionDelivery` 设置）才能绕过。`buildEmbeddedRunPayloads` 创建的普通 assistant 文本 payload 从未获得此标志，因此被静默丢弃。

**修复状态**: 🟠 OPEN | PR 即将提交

### 2.5 Bug #84013 — Telegram pollingStallThresholdMs 未生效

**来源**: [PR #84013](https://github.com/openclaw/openclaw/pull/84013)

**问题描述**:
`pollingStallThresholdMs` 在非隔离 Telegram polling 周期中被尊重，但在启用隔离 ingress 时被静默忽略（默认值）。如果隔离 polling worker 卡住，watchdog 永远不会触发，频道会保持死机直到手动重启。

**根因**:
当前主程序有两个 polling 周期：
- **非隔离** (`#runPollingCycle`): 有 `detectStall()` watchdog ✅
- **隔离** (`#runIsolatedIngressCycle`): 无 `detectStall()` watchdog ❌

隔离 ingress 是**默认启用**（`opts.isolatedIngress?.enabled ?? true`），所以大多数 Telegram 用户在没有 stall 检测的情况下运行。

**修复状态**: 🟢 已修复于 PR #84013
